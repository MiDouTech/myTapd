# 手机端响应式优化说明（工单系统）

## 1. 这次做了什么（给产品和开发都能看懂的版本）

你可以把这次优化理解为：  
原来页面像“只给大屏电脑设计的表格”，在手机上会挤在一起；  
现在改成了“手机也能完整点单的外卖页面”——菜单能打开、筛选能填、表格能滑、按钮能点。

本次主要优化两块：

1. `miduo-frontend/src/layouts/MainLayout.vue`  
   - 手机端改成抽屉菜单（从左侧滑出），不再长期占屏遮挡内容。
   - 顶部区域做了手机压缩：隐藏搜索框、隐藏用户名文本、保留关键操作按钮。
   - 通知抽屉在手机端自动变窄（`92vw`），避免超出屏幕。

2. `miduo-frontend/src/views/ticket/TicketListView.vue`  
   - 手机端把“多标签页切换”改成下拉选择，避免文字挤爆。
   - 筛选表单改成单列布局，输入框 100% 宽度。
   - 查询/重置按钮在手机端做并排等宽，手指更容易点中。
   - 表格保留横向滑动能力，确保“操作列（详情）”和所有字段都可访问。

## 2. 功能用途

### 2.1 手机端菜单抽屉
- **用途**：在窄屏中也能打开完整导航并进入任意模块。
- **用户价值**：不需要缩放页面，也不会被左侧菜单遮住内容。

### 2.2 手机端工单筛选区
- **用途**：在手机上完整输入工单编号、标题、分类、状态、优先级、时间范围并查询。
- **用户价值**：筛选字段不丢失、不重叠、可直接操作。

### 2.3 手机端工单表格可操作
- **用途**：通过左右滑动查看完整列，并点击“详情”进入工单详情。
- **用户价值**：保证“能看到 + 能点到 + 能进入下一步”。

### Q2：TypeScript 报 `Parameter 'value' implicitly has an 'any' type`
- **检测**：`TicketListView.vue` 中 `@change="(value) => ..."` 触发类型检查错误。
- **记录（错误类型）**：TS 隐式 any 类型错误（TS7006）。
- **恢复建议**：改为 `@change="handleTabChange"`，复用已声明的函数签名。

## 9. 工单受理自动认领修复说明（后端）

### 9.1 功能用途
- **用途**：当处理人点击“受理”时，系统自动把“处理人”更新为当前点击人。
- **类比理解**：像“接单按钮”，谁点了“接单”，配送单就挂到谁名下；不会出现“单子开始配送了，但负责人还显示别人”。

### 9.2 使用方法（联调/验收）
1. 打开任意处于待受理节点的工单（如：`pending_accept` / `pending_test_accept` / `pending_dev_accept`）。
2. 用用户A点击“受理”。
3. 刷新工单详情页，检查右侧“基础信息 -> 处理人”。
4. 预期结果：处理人应更新为用户A，且状态进入下一节点（如处理中/测试中/开发中）。

### 9.3 参数说明（本次修复相关）
| 参数 | 类型 | 说明 |
|---|---|---|
| `TransitInput.newAssigneeIds` | `List<Long>` | 流转时手动指定处理人列表（优先级最高） |
| `TransitInput.newAssigneeId` | `Long` | 流转时手动指定单个处理人 |
| `operatorId` | `Long` | 当前操作用户ID；本次修复在“受理”且未显式指定处理人时，自动写入该用户 |

### 9.4 返回值说明（接口行为）
| 接口 | 返回值 | 说明 |
|---|---|---|
| `PUT /api/ticket/transit/{id}` | `ApiResult<Void>` | 成功时返回空数据；本次修复后会同步更新工单主处理人 |

### 9.5 常见问题（新增）
#### Q3：为什么工单状态变成“处理中”了，但处理人还是旧人？
- **检测**：查看流转记录中动作为“受理”，但详情页处理人未变化。
- **记录（错误类型）**：状态流转与处理人认领未联动（业务逻辑缺失）。
- **恢复建议**：
  1. 升级到版本 `v1.1.0-ticket-accept-auto-claim` 及以上；
  2. 重新执行“受理”动作；
  3. 若仍异常，检查该工作流动作名/状态映射是否被自定义改动（受理动作需能被识别）。

## 10. Bug简报“确认通过”校验报错修复（后端）

### 10.1 功能用途
- **用途**：修复 Bug 简报在点击“确认通过”时因参数校验过严导致的 9999 报错。
- **类比理解**：以前像“通过申请也必须写长篇理由”，现在改成“通过可以一句话或不写，驳回仍然必须写清楚原因”。

### 10.2 使用方法（联调/验收）
1. 打开一条“待审核”的 Bug 简报详情。
2. 在“审核通过”弹窗中输入短意见（例如“通过”）或留空。
3. 点击“确认通过”。
4. 预期结果：接口成功返回，状态进入“已归档”，不再出现 `MethodArgumentNotValidException`。

### 10.3 参数说明（本次修复相关）
| 接口 | 参数 | 类型 | 说明 |
|---|---|---|---|
| `PUT /api/bug-report/approve/{id}` | `reviewComment` | `String` | **可选**，审核通过意见，不再要求最少 10 个字符 |
| `PUT /api/bug-report/reject/{id}` | `reviewComment` | `String` | **必填且最少 10 个字符**，用于保证驳回理由完整 |

### 10.4 返回值说明（接口行为）
| 接口 | 返回值 | 说明 |
|---|---|---|
| `PUT /api/bug-report/approve/{id}` | `ApiResult<Void>` | 成功时返回空数据，简报状态流转为“已归档” |
| `PUT /api/bug-report/reject/{id}` | `ApiResult<Void>` | 成功时返回空数据，简报状态流转为“已驳回” |

### 10.5 常见问题（新增）
#### Q4：为什么“确认通过”还会提示参数校验失败？
- **检测**：查看返回是否仍包含 `Validation failed for argument`。
- **记录（错误类型）**：后端版本未更新到本次修复代码，或请求仍走旧接口定义。
- **恢复建议**：
  1. 升级到版本 `v1.1.1-bugreport-approve-validation-fix` 及以上；
  2. 重新部署后端并清理前端缓存再试；
  3. 如果是“驳回”操作，请确认审核意见至少 10 个字符。

## 13. 工单详情页信息布局 + 附件查看优化（前端）

### 13.1 功能用途
- **用途1：优化“详细信息”页签下的四块布局（客服信息/测试信息/开发信息/时间追踪）**

- **用途2：优化附件图片信息排版**  
  把“文件名、来源、大小、上传人、时间、操作”分层展示，避免文字竖着挤在一起。
- **用途3：点击‘查看’直接预览图片**  
  不再依赖新窗口打开，改成页内弹窗预览，操作更顺手。
- **用途4：角色英文改中文显示**  
  把 `CUSTOMER_SERVICE / TESTER / DEVELOPER` 等编码统一转成中文，阅读更直观。

### 13.2 使用方法（验收步骤）
1. 打开工单详情页，进入“详细信息”页签。
2. 依次点击“客服信息 / 测试信息 / 开发信息 / 时间追踪”。
3. 在手机宽度（<=768）下检查：
   - 表单标签应显示在输入框上方；
   - 内层页签可横向滑动，不会挤压成难读状态。
4. 滚动到“附件”区域：
   - 文件名应可换行显示；
   - 上传人和上传时间应水平展示，不再竖排拥挤。
5. 点击某张图片的“查看”：

## 15. 企业微信工单详情链接 404 修复（后端）

### 15.1 功能用途
- **用途**：修复企业微信消息里的“查看详情”链接点击后出现 `Whitelabel Error Page 404` 的问题。
- **类比理解**：原来快递单上的地址写成了“省市区+机房门牌号”，导航会找不到；现在系统会自动把地址“纠偏”成可达的标准地址。

### 15.2 根因（用抓包结果说明）
- 抓包请求是：`GET /api/wecom/callback/open/ticket/{ticketNo}`，服务里没有这个路由，所以 404。
- 同时，`wecom.trusted-domain` 在测试/生产默认值被配置成了 `http://xxx/api/wecom/callback`，代码又拼了 `/open/ticket/{ticketNo}`，导致发出去的链接天然容易错。

### 15.3 本次修复内容
1. **新增统一链接构建器**：`WecomPublicLinkBuilder`
   - 自动去掉错误后缀 `/api/wecom/callback`
   - 再拼接正确路径 `/open/ticket/{ticketNo}`
2. **替换 3 处企微回包逻辑**（创建成功、重复工单、交互确认）统一走构建器
3. **新增历史链接兼容接口**：`GET /api/wecom/callback/open/ticket/{ticketNo}`
   - 接口编号：`API000439`
   - 行为：302 跳转到 `/open/ticket/{ticketNo}`
4. **修正配置默认值**
   - `application-test.yml`：`wecom.trusted-domain` 改为 `http://ticket.t.miduonet.com`
   - `application-prod.yml`：`wecom.trusted-domain` 改为 `http://ticket.ebcone.cn`

### 15.4 使用方法（验收步骤）
1. 在企业微信群里重新创建一条工单，让机器人返回新“查看详情”链接。
2. 点击链接，预期可正常打开工单公开页，不再 404。
3. 用历史消息里的旧链接（包含 `/api/wecom/callback/open/ticket/`）再点一次。
4. 预期会自动跳转到 `/open/ticket/{ticketNo}` 并正常展示。

### 15.5 参数说明（本次修复相关）
| 参数/路径 | 类型 | 说明 |
|---|---|---|
| `wecom.trusted-domain` | `String` | 企微可信域名，应该是域名根地址，不应包含 `/api/wecom/callback` |
| `/open/ticket/{ticketNo}` | `GET` | 工单公开详情真实访问路径 |
| `/api/wecom/callback/open/ticket/{ticketNo}` | `GET` | 历史错误链接兼容入口，302 跳转 |

### 15.6 返回值说明（接口行为）
| 接口 | 返回值 | 说明 |
|---|---|---|
| `GET /api/open/ticket/{ticketNo}` | `ApiResult<TicketPublicDetailOutput>` | 返回公开工单详情数据 |
| `GET /api/wecom/callback/open/ticket/{ticketNo}` | `302 Found` | `Location: /open/ticket/{ticketNo}`，用于老链接兼容 |

### 15.7 常见问题（新增）
#### Q11：修复后为什么旧链接仍然 404？
- **检测**：确认后端是否部署了包含兼容接口的版本。
- **记录（错误类型）**：服务版本未更新或流量仍在旧实例。
- **恢复建议**：
  1. 升级到 `v1.1.6-wecom-open-ticket-link-fix` 及以上；
  2. 重启后端服务并确认路由生效；
  3. 再次使用旧链接测试，观察是否返回 302 跳转。

### 15.8 公开详情页跳转系统内处理（前端）
- **用途**：在企业微信里打开的「无需登录工单详情」页（`/open/ticket/{工单号}`）顶部增加「去系统处理」按钮，新标签页打开与当前站点同域的工单处理页（`/ticket/detail/{数据库id}`）。
- **类比**：公开页像贴在门口的「告示」，按钮是「进办公室办手续」——同一栋楼（域名），进去要登录。
- **说明**：链接里的数字 ID 来自公开详情接口已返回的 `id` 字段；若前后端域名不一致，需自行调整部署或反向代理使两处同源。

#### Q12：为什么 `/open/ticket/{工单号}` 打开是一片白？
- **检测**：按 F12 打开开发者工具 →「网络」里看 `GET /api/open/ticket/{工单号}` 是否 401；看「控制台」是否有 JS 报错或静态资源 404。
- **记录（常见原因）**：
  1. **401 被全局拦截**：公开页请求若返回 401，旧逻辑会强制跳转登录；若 SSO 外链异常，可能表现为白屏。前端已改为在 `/open/` 路径下不因 401 跳转登录，页面会显示接口错误提示。
  2. **网关/安全未放行**：`/api/open/ticket/**` 必须在网关与安全配置中允许匿名访问，否则接口 401/403 工单内容加载失败。
  3. **未返回 SPA**：若 Nginx 未对前端路由做 `try_files ... /index.html`，直接访问 `/open/ticket/...` 可能拿到空响应或非 HTML，页面无法挂载。
- **恢复建议**：确认后端 `SecurityConfig` 已放行 `/api/open/ticket/**`；确认前端静态资源与 `index.html` 可加载；接口仍失败时根据页面上的错误文案或网络面板状态码排查。

## 21. 日报推送“凌晨触发 + 重复两条”修复（后端）

### 21.1 功能用途
- **用途**：修复“配置是每天 09:00 + 工作日 18:00，但凌晨 02:00 仍推送，且一次发两条”的问题。
- **类比理解**：以前像“闹钟按了两个规则、还放在错误时区，两个手机一起响”；现在改成“同一个时区校准 + 同一分钟只允许响一次”。

### 21.2 使用方法（验收步骤）
1. 在“日报管理”里配置推送时间为：
   - `每天 09:00`（`0 0 9 * * ?`）
   - `工作日 18:00`（`0 0 18 * * MON-FRI`）
2. 确认系统基础设置中的时区为业务时区（日本可设为 `Asia/Tokyo`）。
3. 到触发时刻观察企微群消息：
   - 预期每个触发时刻仅收到 1 条；
   - 不会在凌晨 02:00 出现“错时推送”。
4. 若有多实例部署（多台后端），同一时刻也应只收到 1 条。

### 21.3 参数说明（本次改动相关）
| 参数/常量 | 类型 | 说明 |
|---|---|---|
| `DailyReportApplicationService.getScheduleTimezone()` | `String` | 读取 `system_config(BASIC/timezone)` 作为日报 cron 计算时区，非法值回退 `Asia/Shanghai` |
| `daily_report_cron` | `String` | 多 cron 配置（`;` 分隔），解析时自动去重，避免同一 cron 重复保存后重复触发 |
| `RedisKeyConstants.DAILY_REPORT_PUSH_DEDUP_PREFIX` | `String` | 分钟级去重 key 前缀，确保同一分钟只推一次（跨实例生效） |

### 21.4 返回值说明（接口行为）
| 接口 | 返回值 | 说明 |
|---|---|---|
| `PUT /api/daily-report/config` | `ApiResult<Void>` | 保存配置成功后，定时任务按新时区和新 cron 重新装载 |
| `POST /api/daily-report/push` | `ApiResult<Void>` | 手动推送仍可正常发送，不受自动任务分钟去重影响 |

### 21.5 常见问题（新增）
#### Q17：为什么会在凌晨 02:00 推送？
- **检测**：检查基础配置里的 `timezone` 是否与业务期望时区一致（如日本应为 `Asia/Tokyo`）。
- **记录（错误类型）**：定时任务按服务器默认时区/错误时区计算 cron，触发时间发生偏移。
- **恢复建议**：
  1. 在系统基础设置中把时区改为正确值；
  2. 保存日报配置后观察日志中的 `timezone` 与 `nextExecuteTime`；
  3. 确认触发时刻与页面设置一致。

#### Q18：为什么同一时刻会收到两条日报？
- **检测**：检查是否配置了重叠 cron（如同一个时刻两条规则）或是否是多实例部署。
- **记录（错误类型）**：同轮多 cron 命中重复触发，或多实例并发触发导致重复推送。
- **恢复建议**：
  1. 升级到版本 `v1.2.2-daily-report-schedule-dedup-timezone-fix` 及以上；
  2. 保留所需 cron，避免逻辑重复配置；
  3. 验证日志中“同一分钟已推送，跳过本轮”去重提示是否生效。

- 这类消息走的是 `WebhookDispatchService` 的企业微信机器人 `text` 消息。
- 旧逻辑只在少数场景拼 `mentioned_list`（企微userid），如果用户资料里 `wecom_userid` 缺失或未同步，就会出现“消息显示了指派给XXX，但没有真正@到人”。

### 25.3 本次修复内容
1. 把@对象收集逻辑升级为统一的 `collectMentionTargets(...)`：
   - 收集创建人、当前处理人、原处理人、操作人等候选人；
   - 一次性批量查用户，避免重复查库。
2. 发送企微机器人 `text` 消息时：
   - **优先**写入 `mentioned_list`（wecom_userid）；
   - **兜底**写入 `mentioned_mobile_list`（手机号，自动清洗 `+86/空格/横杠`）。
3. 处理人没有 `wecom_userid` 时，不再直接漏@，会自动回退到手机号@。

### 25.4 使用方法（验收步骤）
1. 在“新建工单”页面选择一个处理人后提交工单。
2. 到对应企微群查看“工单事件通知”消息。
3. 预期结果：
   - “变更：指派给：XXX”正常显示；
   - 群里会出现真正的@提醒（处理人应收到被@通知）。
4. 建议额外验证两个用户：
   - A用户：有 `wecom_userid`（应走 `mentioned_list`）；
   - B用户：无 `wecom_userid` 但有手机号（应走 `mentioned_mobile_list` 兜底）。

## 31. 企业微信工单事件通知时间少 8 小时修复（后端）

### 31.3 本次修复内容
1. `WebhookDispatchService` 注入 `SystemConfigMapper`，读取基础配置 `BASIC/timezone`。
2. `formatNow()` 改为按业务时区格式化：
   - 优先使用系统配置时区；
   - 配置缺失或非法时，回退 `Asia/Shanghai`。
3. 这样 `[时间]` 与 `变更时间` 都走同一时区口径。

### 31.4 使用方法（验收步骤）
1. 在系统基础设置确认时区（例如 `Asia/Shanghai`）。
2. 执行一次工单状态变更（例如“开发解决中 -> 已完成”）。
3. 查看企业微信群消息中的：
   - `【时间】`
   - `变更时间`
4. 预期结果：显示与你实际操作一致（例如 `2026-04-10 09:29:44`）。

### 31.5 参数说明（本次修复相关）
| 字段/配置 | 类型 | 说明 |
|---|---|---|
| `system_config.config_group` | `String` | 固定 `BASIC` |
| `system_config.config_key` | `String` | 固定 `timezone` |
| `WebhookDispatchService.DEFAULT_TIMEZONE` | `String` | 时区兜底值，`Asia/Shanghai` |
| `WebhookDispatchService.formatNow()` | `String` | 现在按业务时区格式化当前时间 |

### 31.6 返回值说明（接口行为）
| 场景 | 返回值 | 说明 |
|---|---|---|
| Webhook 事件推送 | `msgtype=text` JSON | 文本中的 `【时间】/变更时间` 按业务时区输出 |
| 时区配置非法 | 日志告警 + 正常推送 | 回退 `Asia/Shanghai`，不阻断通知 |

- **用途**：修复 Bug 简报在“待审核通知、审核驳回通知”场景下，站内信能收到但企微应用消息偶发收不到的问题。  
- **类比理解**：以前像“通知员只会走有门牌号的楼栋（textcard 需要详情链接）”，没门牌号就送件失败；现在改成“找不到门牌时也会改打电话（text 消息兜底）”，至少消息一定能到人。

### 42.2 根因说明（对应你反馈的现象）
- 这两类通知确实进入了统一通知分发器（站内 + 企微应用）。
- 但企微应用此前只走 `textcard`，而 `textcard` 对“详情链接”和“文本长度”更敏感：
  - 无可用详情链接时，企微可能拒收；
  - 内容过长时，也可能被拒收。
- 结果就会出现：**站内信成功，企微应用失败**。

### 42.3 本次修复内容
1. 在 `WecomAppMessageSender` 中增加“发送策略分流”：
   - 满足 textcard 条件（链接可用 + 标题/正文长度安全）→ 继续走 `sendTextCardMessage`；
   - 不满足 → 自动降级走 `sendTextMessage`。
2. 增加 UTF-8 字节级长度保护：
   - textcard 标题和描述做安全阈值判断；
   - text 消息超长时自动截断，并追加“内容过长，请到系统查看详情”提示。
3. 保留原有用户判空和 `wecom_userid` 校验，不改变既有权限/偏好逻辑。

### 42.4 使用方法（验收步骤）
1. 提交一条 P0/P1/P2 Bug 简报，触发“待审核通知”。
2. 用审核人账号确认：
   - 站内通知中心可见该通知；
   - 企微应用也能收到通知。
3. 再对同一简报执行“审核驳回”。
4. 用简报责任人账号确认：
   - 站内通知可见“审核驳回”；
   - 企微应用同步收到“审核驳回”。
5. 建议补测一条“超长驳回原因”，确认企微仍能收到（可能走 text 兜底）。

### 42.5 参数说明（本次改动相关）
| 参数/常量 | 类型 | 说明 |
|---|---|---|
| `TEXT_CARD_TITLE_MAX_BYTES` | `int` | textcard 标题安全字节阈值 |
| `TEXT_CARD_DESCRIPTION_MAX_BYTES` | `int` | textcard 描述安全字节阈值 |
| `TEXT_MESSAGE_MAX_BYTES` | `int` | text 消息字节阈值，超长会截断 |
| `isTextCardPayloadSafe(title, description)` | `method` | 判断是否满足 textcard 安全发送条件 |
| `buildTextMessageContent(title, content)` | `method` | 组装 text 兜底消息，并做长度保护 |

### 42.6 返回值说明（接口行为）
| 场景 | 返回值 | 说明 |
|---|---|---|
| textcard 条件满足 | 企微应用 `textcard` 成功 | 保持原卡片样式，可点击详情 |
| textcard 条件不满足 | 自动回退企微应用 `text` 成功 | 不依赖详情链接，保证消息可送达 |
| 用户无 `wecom_userid` | 跳过企微发送并记录日志 | 与原逻辑一致，避免错误投递 |

### 42.7 常见问题（新增）
#### Q43：为什么我还是只看到站内信，没有企微消息？
- **检测**：先检查该用户是否已绑定 `wecom_userid`，且通知偏好里该事件 `wecomEnabled=1`。  
- **记录（错误类型）**：用户账号未绑定企微身份，或个人偏好关闭了企微通道。  
- **恢复建议**：
  1. 让该用户先完成一次企微登录，确保 `wecom_userid` 已写入；
  2. 在“通知偏好”里打开对应事件的“企业微信”开关；
  3. 再触发一次待审核/驳回通知复测。

## 43. 工单终态误操作支持“回退上一步”（前后端）

### 43.1 功能用途
- **用途**：给工单详情页新增“回退上一步”按钮，避免误点“已完成/已关闭”后无法快速恢复。  
- **类比理解**：像文档里的“撤销 Ctrl+Z”，不仅把步骤退回去，还把当时负责人一起恢复，避免“状态回去了、人没回去”。

### 43.2 生效规则（你最关心）
按钮显示条件：
1. 工单状态为 **已关闭**；或  
2. 工单状态为 **已完成** 且存在 **未归档** Bug简报。

点击后效果：
- 状态回退到“进入当前终态前的上一步状态”；
- 处理人恢复为“上一步处理人”。

### 43.3 使用方法（验收步骤）
1. 打开工单详情页，准备两类工单：
   - A类：已关闭工单；
   - B类：已完成且关联简报里至少有一条非 ARCHIVED。
2. 确认顶部操作栏出现“回退上一步”按钮。
3. 点击按钮并确认弹窗。
4. 预期结果：
   - 工单状态回到上一步；
   - 右侧处理人回到上一步处理人；
   - 流转历史新增一条“回退”记录。
5. 反向校验：
   - 已完成但所有简报都已归档时，不显示按钮；
   - 即使绕过前端直接调接口，也会被后端拒绝。

| `ticket_flow_record.from_status` | `string` | 用于定位上一步状态 |

  1. 对历史异常工单补充一次正常流转记录后再回退；

## 44. 企微工单助手应用消息：只发卡片 + 超长自动截断（后端）

### 44.1 功能用途
- **用途**：把工单助手的企微应用通知统一成“应用卡片消息（textcard）”，不再发送普通文本消息（text）。  
- **类比理解**：以前像“同一类通知有时是海报，有时是短信”，看起来很乱；现在统一成“海报样式”，并且海报字太多时会自动缩短标题，保证还能发出去。

### 44.2 本次改动了什么
1. `WecomAppMessageSender` 发送策略从“textcard + text兜底”改为“**仅 textcard**”。  
2. 新增标题安全收敛逻辑：  
   - 标题超长时，优先截断“标题里的描述部分”（例如 `工单状态更新 - xxxxx` 的后半段）。  
   - 仍超长时按 UTF-8 字节截断并追加 `...`。  
3. 新增正文安全收敛逻辑：  
   - textcard 描述超长时，自动压缩并追加“内容过长，已截断”提示。  
4. 详情链接兜底：  
   - 优先使用业务传入 `detailLink`；  
   - 为空时使用可信域名；  
   - 再兜底到 `https://work.weixin.qq.com`，避免 URL 为空导致卡片发送失败。

### 44.3 使用方法（验收步骤）
1. 触发任意会发企微应用通知的场景（如：工单状态变更、分派、评论@、Bug简报提交/驳回）。  
2. 在企微中查看收到的通知样式。  
3. 预期结果：
   - 始终为应用卡片消息；
   - 不会出现纯文本通知。  
4. 构造超长标题场景（例如标题里拼接很长描述）再次触发。  
5. 预期结果：
   - 通知仍发送成功；
   - 标题/正文有截断但不乱码。

## 46. 告警接入新增“仅命中映射规则才创建工单”开关（前后端）

## 49. 后端启动报错修复：JwtAuthenticationFilter 未注册顺序（后端）

### 49.1 功能用途
- **用途**：修复后端启动时报错 `The Filter class ... JwtAuthenticationFilter does not have a registered order`，让 Spring Security 过滤器链可以正常构建。  
- **类比理解**：以前像“先让保安B站到保安A前面，但A还没排进队伍”；现在改成“先把A排进队，再把B插到A前面”。

### 49.2 根因说明（对应本次日志）
- `SecurityConfig` 里原顺序是：
  1. `openApiAppAuthFilter` 插到 `JwtAuthenticationFilter` 前面  
  2. 再注册 `JwtAuthenticationFilter`
- 在 Spring Security 6 中，`addFilterBefore(..., JwtAuthenticationFilter.class)` 这一步要求 `JwtAuthenticationFilter` 已经有已知顺序；否则会直接抛 `IllegalArgumentException`。

### 49.3 本次修复内容
1. 先注册 `jwtAuthenticationFilter` 到 `UsernamePasswordAuthenticationFilter` 前。  
2. 再把 `openApiAppAuthFilter` 插到 `JwtAuthenticationFilter` 前。  
3. `agentApiKeyAuthenticationFilter` 继续保持在 `JwtAuthenticationFilter` 后。  

> 修复文件：`ticket-platform/ticket-bootstrap/src/main/java/com/miduo/cloud/ticket/bootstrap/config/SecurityConfig.java`

### 49.4 使用方法（验收步骤）
1. 重新构建并启动后端容器。  
2. 查看启动日志，确认不再出现：
   - `JwtAuthenticationFilter does not have a registered order`
3. 观察服务是否正常完成启动并监听端口。

### 49.5 参数说明（本次改动相关）
| 参数/对象 | 类型 | 说明 |
|---|---|---|
| `jwtAuthenticationFilter` | `Filter` | JWT 认证过滤器，作为中间锚点先注册 |
| `openApiAppAuthFilter` | `Filter` | OpenAPI 应用鉴权过滤器，放在 JWT 之前 |
| `agentApiKeyAuthenticationFilter` | `Filter` | Agent API Key 鉴权过滤器，放在 JWT 之后 |

### 49.6 返回值说明（接口行为）
| 场景 | 返回值 | 说明 |
|---|---|---|
| Spring Security 构建过滤器链 | `SecurityFilterChain` | 现在可正常创建，不再因过滤器顺序异常中断启动 |

## 51. 节点耗时统计修复：START_PROCESS 正确关前节点并补齐开始/离开时间（后端）

### 51.1 功能用途
- **用途**：修复“待测试受理、待开发受理等节点离开时间一直为空”的问题，保证节点耗时链完整。  
- **类比理解**：以前像“只打了上车时间，没打下车时间”；现在每一站都能打上“开始处理 + 离开时间”，路线就完整了。

### 51.2 根因说明（对应你反馈）
- `START_PROCESS` 在旧逻辑里只做了“开始处理打点”，没有执行“关闭前节点 + 打开下一节点”。  
- 所以像 `pending_test_accept -> testing`、`pending_dev_accept -> developing` 这种流转，会导致前一个节点 `leave_at` 长期为空。

### 51.3 本次修复内容
1. `TicketNodeDurationApplicationService` 新增 `handleStartProcess(...)`：  
   - 当 `fromStatus != toStatus` 时，按“开始前节点 -> 关闭前节点 -> 打开并开始新节点”执行。  
2. 关闭节点时增加兜底：  
   - 如果按 `fromStatus` 找不到开启节点，会降级关闭当前开启节点（避免旧链路遗留的开口节点继续漂移）。  
3. 节点闭环补齐：  
   - 关闭节点时若缺少 `start_process_at`，会按 `first_read_at` 或离开时刻补齐，确保已关闭节点都有开始处理时间和离开时间。

### 51.4 使用方法（验收步骤）
1. 选择一条缺陷工单，执行以下流转：
   - `待测试受理 -> 测试复现中`
   - `待开发受理 -> 开发解决中`
2. 打开工单详情的“节点耗时统计”。
3. 预期结果：
   - `待测试受理`、`待开发受理` 节点的“离开时间”不再为空；
   - 对应节点有“开始处理时间”；
   - 新进入节点也会被正确打开并参与后续耗时统计。

### 51.5 参数说明（本次改动相关）
| 参数/方法 | 类型 | 说明 |
|---|---|---|
| `handleStartProcess(ticketId, fromStatus, toStatus, toUserId, eventTime)` | `method` | 统一处理 START_PROCESS 的跨节点流转 |
| `closeNode(ticketId, status, nextStatus, leaveAt)` | `method` | 关闭节点并计算总耗时/处理耗时，含未命中兜底关闭 |
| `start_process_at` | `datetime` | 节点开始处理时间（本次确保闭环补齐） |
| `leave_at` | `datetime` | 节点离开时间（本次修复中间节点遗漏） |

### 51.6 返回值说明（接口行为）
| 接口 | 返回值 | 说明 |
|---|---|---|
| `GET /api/ticket/node-duration/{id}` | `ApiResult<TicketNodeDurationOutput>` | 修复后中间节点 `leaveAt` 可正确返回，不再长期为 `null` |

## 56. 日报“手动推送”假成功修复（后端）

### 56.1 功能用途
- **用途**：修复“页面提示推送成功，但企业微信没收到”的问题。  
- **类比理解**：以前像“前台说快递已发出”，其实仓库因为开关/地址问题根本没寄；现在改成“要么真寄出，要么当场告诉你哪里寄不出去”。

### 56.2 根因说明（对应你反馈的现象）
- 旧逻辑里，`POST /api/daily-report/push` 复用了自动任务逻辑：
  1. 自动开关关闭时直接跳过；
  2. webhook 发送失败时只记日志，不抛错；
  3. 接口仍返回成功，前端就提示“日报已推送到企微群”。

### 56.3 本次修复内容
1. 新增手动推送专用入口：`pushDailyReportManually()`。  
2. 手动推送**不再受自动开关阻断**（开关关闭也允许手动发）。  
3. 手动推送时，如果以下情况发生会直接返回失败给前端：
   - 未配置 webhook；
   - webhook 全部无效；
   - markdown 为空；
   - 所有 webhook 发送都失败。  
4. 若是“部分成功、部分失败”，接口返回成功，但后端会记录失败明细日志，便于排查。

### 56.4 使用方法（验收步骤）
1. 进入「日报管理 -> 日报预览」。  
2. 点击「手动推送到群」。  
3. 预期结果：
   - 至少一个 webhook 发送成功：页面显示成功，企微群能收到；
   - 全部失败：页面显示失败（不再假成功），并带出失败原因。

### 56.5 参数说明（本次改动相关）
| 参数/方法 | 类型 | 说明 |
|---|---|---|
| `DailyReportApplicationService.pushDailyReport()` | `method` | 自动任务入口（保留原行为，受自动开关控制） |
| `DailyReportApplicationService.pushDailyReportManually()` | `method` | 手动入口（忽略自动开关，失败会抛业务异常） |
| `POST /api/daily-report/push` | API | 现在调用手动入口，失败会返回明确错误 |

## 60. 工作流支持自定义配置 + 安全删除（前后端）

### 60.1 功能用途
- **用途1：支持创建自定义工作流**  
  管理员可以在管理端新增一个工作流，并自行配置“状态”和“流转规则”。
- **用途2：支持安全删除**  
  只有“还没被工单调用过”的自定义工作流才允许删除；一旦被调用，就禁止删除，避免影响历史数据链路。
- **类比理解**：像“菜谱模板管理”：菜谱没被做过可以删；做过一次后就留档，不能随便删。

### 60.2 使用方法（验收步骤）
1. 进入「管理 -> 工作流管理」。
2. 点击「新建工作流」，配置名称、状态、流转规则并保存。
3. 在列表里查看新工作流，确认“调用次数”为 `0`。
4. 点击“删除”：
   - 未调用：可删除成功；
   - 已调用：删除按钮置灰或提示“工作流已被调用，不可删除”。
5. 将某分类绑定该工作流并创建工单后，再尝试删除，预期后端拒绝。

### 60.3 参数说明（本次改动相关）
| 参数/字段 | 类型 | 说明 |
|---|---|---|
| `POST /api/workflow/create` | API | 创建自定义工作流（API000515） |
| `DELETE /api/workflow/delete/{id}` | API | 删除未调用工作流（API000516） |
| `workflow.invocation_count` | `bigint` | 工作流被工单调用次数 |
| `workflow.first_invoked_time` | `datetime` | 首次被调用时间 |
| `workflow.last_invoked_time` | `datetime` | 最近一次被调用时间 |
| `invocationCount` | `number` | 前端列表/详情显示的调用次数 |
| `canDelete` | `boolean` | 前端是否允许删除按钮可用 |
| `deleteBlockedReason` | `string` | 删除受阻原因提示 |

### 60.4 返回值说明（接口行为）
| 接口 | 返回值 | 说明 |
|---|---|---|
| `POST /api/workflow/create` | `ApiResult<Long>` | 返回新建工作流 ID |

### 60.5 常见问题（新增）
#### Q69：为什么删除按钮是灰的？
- **检测**：看该工作流“调用次数”是否大于 0，或是否显示阻塞原因。

  2. 若已被调用：保留该流程，避免破坏历史关联；
  3. 若提示已关联分类：先在分类管理里解绑再重试。

## 71. 工作流详情流程图分支修复（前端）

### 71.1 功能用途
- **用途**：修复「工作流详情 -> 流程图视图」里多条流转线挤在一起、文字重叠的问题，并让一对多流转看起来像分叉而不是一条直线。  
- **类比理解**：以前像“所有车都走一条车道，路牌叠在一起”；现在改成“同一路口按车道分流，每条路有自己的指示牌”。

### 71.2 使用方法（验收步骤）
1. 进入「管理 -> 工作流管理」。
2. 打开「缺陷工单工作流」详情，切到「流程图视图」。
3. 重点观察同一节点发出的多条流转（如“开发解决中”到“临时解决/待验收/已挂起/已关闭”）：
   - 连线应从节点右侧不同位置分叉；
   - 文本标签应分开显示，不再堆叠；
   - 退回线应走上方弧线，不和主线打架。

### 71.3 参数说明（本次改动相关）
| 参数/常量 | 类型 | 说明 |
|---|---|---|
| `EDGE_PORT_PADDING` | `number` | 连线锚点在节点上下预留间距，避免连线都从同一点出发 |
| `EDGE_LABEL_GAP` | `number` | 连线文字与曲线的垂直间隔，减少压线 |
| `outgoingIndex/outgoingCount` | `number` | 同一来源节点的第几条流转和总数，用于分叉排布 |
| `incomingIndex/incomingCount` | `number` | 同一目标节点的第几条入线和总数，用于汇合排布 |
| `pairIndex/pairCount` | `number` | 同一 from→to 多条边时的排序信息，用于标签避让 |

### 71.4 返回值说明（界面行为）
| 场景 | 返回值 | 说明 |
|---|---|---|
| 打开流程图视图 | `void` | 展示分叉后的曲线连线 |
| 一个节点有多个下一节点 | `void` | 连线在起点做“扇形分流”，不再重叠为一条线 |
| 同一路径有多条规则 | `void` | 标签按索引上下错位，避免文字重叠 |
| 退回/回边流转 | `void` | 连线走上方弧线，和正常前进流转视觉区分 |

### 71.5 常见问题（新增）
#### Q87：还是看到标签挤在一起怎么办？
- **检测**：先确认是否为同一节点同一目标存在大量同名流转。  
- **记录（错误类型）**：图形密度过高/命名重复导致视觉拥挤。  
- **恢复建议**：
  1. 优先把重复规则名改成更具体动作名；
  2. 尝试放大浏览器缩放或横向滚动查看完整分叉；
  3. 若仍拥挤，可进一步按业务拆分工作流。

## 74. 处理完成后跳转 Bug 简报时报“仅管理员可查看全部工单”修复（前端）

### 74.1 功能用途
- **用途**：修复「工单详情点处理完成/临时解决确认后跳到 Bug 简报编辑页」时，普通用户看到 `403: 仅管理员可查看全部工单`，导致“关联工单”下拉不可用的问题。

### 74.2 原因说明（给验收同学）
- Bug 简报编辑页搜索工单时，错误传了 `view=all`。
- 后端对 `view=all` 有管理员权限校验，所以普通用户被拦截。
- 本次改为 `view=defect`，只查缺陷分类工单，和简报业务口径一致，也不会触发管理员限制。

### 74.3 使用方法（验收步骤）
1. 打开任意缺陷工单详情页。
2. 执行「处理完成」或「临时解决确认」，跳转到 Bug 简报编辑页。
3. 点击「关联工单」输入框并搜索。
4. **预期**：不再出现“仅管理员可查看全部工单”报错；下拉能正常返回可关联工单。

### 74.4 参数说明
| 参数 | 类型 | 说明 |
|---|---|---|
| `GET /api/ticket/page` 查询参数 `view` | `string` | 从 `all` 调整为 `defect`，仅查询缺陷视图 |
| `GET /api/ticket/page` 查询参数 `linkableForBugReport` | `boolean` | 继续为 `true`，仅保留可关联简报状态工单 |

### 74.5 常见问题（新增）

| 缺陷等级缺失 | `ApiResult` 业务错误 | 返回“确认缺陷转开发前必须填写缺陷等级，请在流转时选择或在测试信息中维护” |
| 缺陷等级非法 | `ApiResult` 业务错误 | 返回“缺陷等级仅支持P0-P4” |
| 缺陷等级合法 | `ApiResult<Void>` | 工单正常转入待开发受理，并同步保存测试信息 |
| 后续创建/编辑 Bug 简报 | `BugReportDetailOutput/BugReportCreateInput` | `severityLevel` 自动取工单测试信息中的等级 |

### 75.5 常见问题（新增）
#### Q91：为什么转开发时多了一个必填项？
- **检测**：确认当前操作是否为缺陷工单的「测试中 -> 待开发受理」。
- **记录（错误类型）**：缺陷闭环信息不完整。
- **恢复建议**：
  1. 由测试根据影响程度选择 P0-P4；
  2. 如已在「测试信息」页维护，弹窗会自动带出；
  3. 保存后再确认转开发。

#### Q92：Bug 简报里等级没有自动带过去怎么办？
- **检测**：回到工单详情「测试信息」页，确认 `缺陷等级` 是否已保存。
- **记录（错误类型）**：测试信息缺等级或使用旧流程历史数据。
- **恢复建议**：
  1. 补填测试信息中的缺陷等级；
  2. 重新进入 Bug 简报编辑页或重新关联该工单；
  3. 若是历史工单，允许人工选择一次等级。

## 76. 仪表盘卡片点击后普通用户误跳“全部工单”403修复（前端）

### 76.1 功能用途
- **用途**：修复普通用户在仪表盘点击统计卡片后，被错误跳到 `/ticket/all` 并触发“仅管理员可查看全部工单”的问题。
- **类比理解**：以前像导航把普通员工直接带到“管理员机房”，门禁当然拦住；现在会先看你的门禁卡，普通员工带去“我的工单工作台”，管理员才去“总控室”。

### 76.2 修改说明
- 仪表盘卡片点击逻辑新增角色判断：
  - **ADMIN/TICKET_ADMIN**：继续跳转 `/ticket/all`；
  - **其他角色**：改为跳转 `/ticket/mine?view=my_todo`。
- 状态筛选联动保持不变：
  - 点击“待受理/处理中/已挂起/已完成”会追加 `status`；
  - 点击“SLA超时”会追加 `slaStatus=BREACHED`；
  - 点击“工单总量”不追加筛选。

### 76.3 使用方法（验收步骤）
1. 用普通账号登录，进入仪表盘。
2. 依次点击“待受理”“处理中”“SLA超时”“工单总量”卡片。
3. **预期**：跳转到 `/ticket/mine`（带 `view=my_todo`），不再出现“仅管理员可查看全部工单”。
4. 用管理员账号重复点击同样卡片。
5. **预期**：管理员仍跳转 `/ticket/all`，功能不受影响。

### 76.4 参数说明
| 参数 | 类型 | 说明 |
|---|---|---|
| `roleCodes` | `string[]` | 前端根据当前用户角色判断跳转路径是否允许进入全部工单 |
| `view` | `string` | 普通用户卡片跳转固定补 `my_todo`，保证进入“我的工单”视图 |
| `status` | `string` | 非“工单总量/SLA超时”卡片时透传对应状态筛选 |
| `slaStatus` | `string` | 点击“SLA超时”卡片时传 `BREACHED` |

### 76.5 常见问题（新增）
#### Q93：修复后还是跳到了“全部工单”怎么办？
- **检测**：打开浏览器 Network 或地址栏，看点击卡片后的目标 URL 是不是仍是 `/ticket/all`。  
- **记录（错误类型）**：前端旧缓存/旧静态资源未更新。  
- **恢复建议**：
  1. 强制刷新（`Ctrl+F5`）；
  2. 清浏览器缓存后重新登录；
  3. 确认当前前端部署版本已包含本次修复分支代码。

## 77. 缺陷等级动态调整解决 SLA（后端）

### 77.1 功能用途
- **用途**：缺陷工单由客成新建时通常不知道 P0-P4，所以响应 SLA 继续按工单分类策略计时；等测试填写缺陷等级后，系统自动按等级调整“解决时限”。
- **类比理解**：像快递先按普通路线揽收，等仓库验货后贴上“特急/普通”标签，后面的送达倒计时就按这个标签重新计算。

### 77.2 规则说明

| 缺陷等级 | 解决时限 | 说明 |
|---|---:|---|
| P0 | 60 分钟 | 1 小时内解决或实施有效临时方案 |
| P1 | 360 分钟 | 6 小时内解决或实施有效临时方案 |
| P2 | 480 分钟 | 1 个工作日内解决或转为临时处理状态 |
| P3 | 1440 分钟 | 3 个工作日内解决或转为临时处理状态 |
| P4 | 1440 分钟 | 3 个工作日内解决或转为临时处理状态 |

> 注意：响应时限不按缺陷等级调整，因为客成新建工单时还没有测试等级结论。

### 77.3 使用方法（验收步骤）
1. 创建一个缺陷工单，确认分类已绑定 SLA 策略。
2. 查询数据库 `sla_timer`，记录该工单 `RESPONSE` 和 `RESOLVE` 的 `threshold_minutes`。
3. 在工单详情的「测试信息」中填写 `缺陷等级=P0` 并保存，或在「确认缺陷转开发」弹窗中填写 P0。
4. 再次查询 `sla_timer`：
   - `RESPONSE` 的 `threshold_minutes` 应保持不变；
   - `RESOLVE` 的 `threshold_minutes` 应变为 `60`。
5. 将等级改为 `P3` 后再次保存：
   - `RESPONSE` 仍保持不变；
   - `RESOLVE` 应变为 `1440`。

### 77.4 参数说明

| 参数/字段 | 类型 | 说明 |
|---|---|---|
| `severityLevel` | `String` | 缺陷等级，支持 `P0/P1/P2/P3/P4` |
| `timer_type=RESPONSE` | `String` | 首次响应计时器，本次不随缺陷等级调整 |
| `timer_type=RESOLVE` | `String` | 解决计时器，本次随缺陷等级调整 |
| `threshold_minutes` | `Integer` | SLA 时限分钟数，会按 P0-P4 映射重算 |
| `deadline` | `Date` | 截止时间，会按剩余工作时间重算 |

### 77.5 返回值说明

| 场景 | 返回值 | 说明 |
|---|---|---|
| 保存测试信息成功 | `ApiResult<Void>` | 等级保存后后台同步重算解决 SLA |
| 确认缺陷转开发成功 | `ApiResult<Void>` | 流转成功，测试等级保存，并同步重算解决 SLA |
| 等级非法 | `ApiResult` 业务错误 | 返回“缺陷等级仅支持P0-P4” |

### 77.6 常见问题（新增）
#### Q94：为什么新建工单时没有立刻按 P0/P1 调整响应时限？
- **检测**：看新建工单页面是否由客成填写，且是否还没有测试等级结论。
- **记录（错误类型）**：业务阶段信息不足，不是系统异常。
- **恢复建议**：
  1. 客成先正常建单；
  2. 测试复现后填写缺陷等级；
  3. 系统会从这一刻开始按等级调整解决 SLA。

#### Q95：改了等级后为什么工单马上超时？
- **检测**：查询 `sla_timer.elapsed_minutes` 是否已经超过新等级的 `threshold_minutes`。
- **记录（错误类型）**：等级收紧导致已消耗时间超过新时限。
- **恢复建议**：
  1. 确认等级是否选错；
  2. 如果等级正确，应尽快解决或转为有效临时方案；
  3. 如需放宽，测试/管理员可修正缺陷等级后重新保存。

## 78. 公开工单详情页 SLA 倒计时（前后端）

### 78.1 功能用途
- **用途**：在无需登录的公开工单详情页 `/open/ticket/{ticketNo}` 展示 SLA 倒计时，让外部链接打开后也能看到响应/解决还剩多久。
- **类比理解**：像外卖订单上的“预计还剩 12 分钟送达”，处理人不用进后台也能快速判断是否快超时。

### 78.2 使用方法（验收步骤）
1. 打开任意有 SLA 的公开工单链接，例如 `/open/ticket/WO-xxxx`。
2. 查看标题卡片下方是否出现“SLA倒计时”卡片。
3. 如果 SLA 正在运行，应看到“剩余 HH:mm:ss”每秒变化。
4. 如果 SLA 已暂停，应看到“暂停中”。
5. 如果 SLA 已完成或已超时，应分别看到“已完成”或“已超时”。

### 78.3 参数说明

| 字段 | 类型 | 说明 |
|---|---|---|
| `slaTimers` | `Array` | 公开详情接口返回的 SLA 计时器列表 |
| `timerType` | `String` | `RESPONSE` 表示首次响应，`RESOLVE` 表示解决 |
| `status` | `String` | `RUNNING/PAUSED/COMPLETED/BREACHED` |
| `thresholdMinutes` | `number` | SLA 总时限分钟数 |
| `elapsedMinutes` | `number` | 已消耗分钟数 |
| `remainingSeconds` | `number` | 剩余秒数 |
| `deadline` | `string` | 截止时间，前端用它计算运行中倒计时 |

### 78.4 返回值说明

| 场景 | 页面展示 |
|---|---|
| 有运行中 SLA | 显示倒计时，例如 `剩余 01:25:36` |
| SLA 暂停 | 显示 `暂停中`，并展示剩余时长 |
| SLA 完成 | 显示 `已完成` |
| SLA 超时 | 显示 `已超时` |
| 无 SLA | 不展示 SLA 空卡片 |

### 78.5 常见问题（新增）
#### Q96：为什么公开页没有 SLA 倒计时？
- **检测**：查询该工单是否有 `sla_timer` 记录。
- **记录（错误类型）**：工单分类未绑定 SLA 策略，或历史工单未生成 SLA 计时器。
- **恢复建议**：
  1. 确认工单分类已绑定 SLA 策略；
  2. 新建一张同分类工单验证；
  3. 历史工单如需展示，需要先补齐 SLA 计时器数据。

#### Q97：为什么暂停状态不跳秒？
- **检测**：查看 `sla_timer.status` 是否为 `PAUSED`。
- **记录（错误类型）**：不是页面卡住，而是 SLA 业务处于暂停计时。
- **恢复建议**：恢复工单处理后，SLA 会重新进入运行状态并继续倒计时。
