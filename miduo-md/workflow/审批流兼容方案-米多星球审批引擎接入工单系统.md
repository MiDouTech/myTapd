# 米多星球审批流引擎 × 工单系统兼容方案

> **文档版本**：v1.0  
> **日期**：2026-06-06  
> **作者**：产品/架构团队  
> **背景**：米多星球（miduo-backend）已落地成熟的自研审批引擎（T3.9.0），工单系统（myTapd）现有审批工单仅靠简单状态机驱动，缺乏多人审批、条件分支、审批人分配等能力。本文分析两系统差异，给出三个渐进式兼容方案，并给出选型建议。

---

## 一、现状对比：两个系统有什么不同？

用一个"快递"类比来理解两个系统：

- **工单系统的工作流** = 快递单的运单状态（揽件 → 转运中 → 派送中 → 已签收），状态由"事件"驱动推进，关注的是**这票货走到哪了**。
- **米多星球的审批引擎** = 银行贷款的审批流（提交申请 → 初审 → 复审 → 总经理签字 → 放款），关注的是**这件事需要谁点头、怎么点头**。

两者不是替代关系，而是**不同粒度的流程管控工具**。

### 1.1 核心模型对比

| 维度 | 工单系统（myTapd） | 米多星球审批引擎 |
|------|-------------------|-----------------|
| **核心驱动** | 状态机（FSM）—— 状态 + 流转规则 | 任务驱动 —— 节点 + 待办任务 |
| **存储结构** | `workflow.states` + `workflow.transitions`（JSON 列） | `wf_form` + `wf_flow_template` + `wf_flow_version.node_config` |
| **审批人** | 处理组成员（轮询/负载均衡/矩阵分派） | 指定成员 / 部门负责人 / 多级领导（逐级向上找） |
| **多人审批** | 无（只有一个 `assignee_id`） | 会签（全部通过）/ 或签（一人通过）/ 依次审批 |
| **条件分支** | 无（流转规则是静态列表） | 基于表单字段值的条件判断（`eq/gte/contains` 等） |
| **操作丰富度** | 流转 / 转派 / 退回 | 同意 / 驳回 / 转交 / 前加签 / 后加签 / 撤回 / 取回重审 / 终止 |
| **版本管理** | 无版本概念（直接更新） | 多版本，历史版本保留，运行中实例绑定发起时的版本 |
| **SLA** | 有，与状态 `slaAction` 绑定 | 有 `due_time` + 超时扫描，但与 SLA 策略独立 |
| **抄送** | 无 | 有专门抄送节点（不阻塞流程） |
| **表单** | 模板字段（`ticket_template.fields_config`） | 自定义表单设计器（`wf_form` + `wf_form_field`） |
| **Spring Events** | `TicketStatusChangedEvent` 等 | `WorkflowInstancePassedEvent` 等 |

### 1.2 工单系统当前审批工单的痛点

工单系统里现有"审批工单"（`workflow_id=2`），状态路径为：

```
submitted → dept_approval → executing → completed
                         ↘ rejected
```

**存在的问题：**

1. **"审批中"是黑箱** — `dept_approval` 状态里谁在审批？多人审批时谁先谁后？系统完全不知道，全靠线下沟通
2. **无审批人队列** — 没有"待我审批"的任务列表，审批人只能靠推送消息记住
3. **审批操作单调** — 只有一次性的"同意/拒绝"，不支持转交、加签
4. **不可扩展** — 如果要支持"金额>10万需要总监再审"，当前状态机完全无法表达
5. **无审批历史轨迹** — 不知道哪个步骤谁审批了多久

## 二、目标：引入什么能力？

| 目标能力 | 现状 | 期望 |
|---------|------|------|
| 审批人明确分配 | 不支持 | 按节点指定审批人（成员/部门负责人/多级领导） |
| 多人审批模式 | 不支持 | 会签/或签/依次审批 |
| 待审批任务中心 | 不支持 | 有"待我审批"列表，角标提醒 |
| 审批时间轴 | 不支持 | 可视化展示每步谁审批了、花了多长时间 |
| 条件分支（可选） | 不支持 | 根据工单字段值走不同审批路径 |
| 转交/加签（可选） | 不支持 | 审批中可转交给他人或追加审批人 |
| 历史版本绑定（可选） | 不支持 | 流程变更不影响进行中的实例 |

## 三、三种兼容方案

### 方案一（推荐）：嵌入式审批任务层

**比喻**：就像快递单的"转运中"阶段内部加了一套分拣流水线，快递的大状态还是"转运中"，但里面有具体的分拣步骤。

#### 核心思路

- 工单的**宏观状态机（FSM）不变**，继续管理工单的生命周期（`pending_assign` → `dept_approval` → `executing`）
- 在某些"审批型"状态（如 `dept_approval`）内，**嵌入一套轻量审批任务层**
- 审批任务全部通过后，**自动触发 FSM 流转**到下一个状态（如 `executing`）
- 任意一个审批任务被驳回，**自动触发 FSM 流转**到 `rejected`

#### 数据模型扩展（新增 3 张表）

```sql
-- 1. 审批流配置（绑定到工作流的某个 transition 上）
CREATE TABLE `ticket_approval_config` (
  `id`             bigint NOT NULL AUTO_INCREMENT,
  `workflow_id`    bigint NOT NULL COMMENT '绑定的工作流 ID',
  `transition_id`  varchar(50) NOT NULL COMMENT '绑定的 transition.id（触发审批的流转）',
  `node_config`    longtext COMMENT '审批节点配置 JSON（结构同米多星球 node_config）',
  `create_time`    datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `update_time`    datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  `deleted`        tinyint NOT NULL DEFAULT 0,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_wf_trans` (`workflow_id`, `transition_id`, `deleted`)
) COMMENT='工单审批流配置';

-- 2. 审批任务（每个待审批步骤一条记录）
CREATE TABLE `ticket_approval_task` (
  `id`            bigint NOT NULL AUTO_INCREMENT,
  `ticket_id`     bigint NOT NULL COMMENT '关联工单 ID',
  `config_id`     bigint NOT NULL COMMENT '关联审批配置 ID',
  `node_key`      varchar(100) NOT NULL COMMENT '节点标识',
  `node_name`     varchar(100) COMMENT '节点名称',
  `approve_mode`  varchar(20) NOT NULL COMMENT 'single/countersign/orsign/sequential',
  `assignee_id`   varchar(64) NOT NULL COMMENT '审批人（企微 userid）',
  `task_status`   varchar(20) NOT NULL DEFAULT 'pending' COMMENT 'pending/approved/rejected/transferred/skipped',
  `sort_order`    int NOT NULL DEFAULT 0 COMMENT '顺序审批时的排序',
  `due_time`      datetime COMMENT '审批截止时间',
  `remark`        varchar(500) COMMENT '审批意见',
  `operate_time`  datetime COMMENT '审批操作时间',
  `create_time`   datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `update_time`   datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  `deleted`       tinyint NOT NULL DEFAULT 0,
  PRIMARY KEY (`id`),
  KEY `idx_ticket_id` (`ticket_id`),
  KEY `idx_assignee` (`assignee_id`, `task_status`)
) COMMENT='工单审批任务';

-- 3. 审批操作记录
CREATE TABLE `ticket_approval_record` (
  `id`           bigint NOT NULL AUTO_INCREMENT,
  `ticket_id`    bigint NOT NULL,
  `task_id`      bigint NOT NULL,
  `action_type`  varchar(20) NOT NULL COMMENT 'approve/reject/transfer/add_sign',
  `operator_id`  varchar(64) NOT NULL COMMENT '操作人企微 userid',
  `operator_name` varchar(100),
  `remark`       varchar(500),
  `target_assignee_id` varchar(64) COMMENT '转交/加签目标人',
  `create_time`  datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  KEY `idx_ticket_id` (`ticket_id`)
) COMMENT='工单审批操作记录';
```

#### 工单系统工作流扩展

```java
// WorkflowTransition 扩展（新增字段，向下兼容）
@Data
public class WorkflowTransition implements Serializable {
    // ... 已有字段 ...
    
    /**
     * 是否需要走审批任务层（true 时本次 transition 提交后不立即生效，
     * 而是创建审批任务，等任务全部通过后才真正流转状态）
     */
    private Boolean requireApproval;
    
    /**
     * 审批通过后自动流转到的目标状态（覆盖 transition.to）
     * 当 requireApproval=true 时有意义
     */
    private String approvalPassedTo;
    
    /**
     * 审批驳回后流转到的目标状态
     */
    private String approvalRejectedTo;
}
```

#### 流转流程

```
用户发起"提交审批"操作
    ↓
TicketWorkflowAppService.transit(ticketId, "submit")
    ↓
检测 transition.requireApproval = true
    ↓
不立即改变 ticket.status（或先改为"审批中"占位状态）
    ↓
TicketApprovalService.startApproval(ticketId, configId)
  → 解析 node_config
  → 创建 ticket_approval_task（每个节点每个审批人一条）
  → 推送企微消息通知第一个审批人
    ↓
审批人操作（同意/驳回/转交）
    ↓
TicketApprovalService.performApprove(taskId, action, remark)
  → 更新 task_status
  → 写 ticket_approval_record
  → 判断当前节点是否完成（会签需全部通过/或签需一人通过）
  → 若所有节点完成 → 触发 FSM 流转到 approvalPassedTo 状态
  → 若有节点驳回 → 触发 FSM 流转到 approvalRejectedTo 状态
```

#### 前端新增模块

| 页面/组件 | 说明 |
|---------|------|
| 工单详情 → 审批流程 Tab | 展示审批任务列表和时间轴（复用米多星球 `WorkflowProgressTimeline.vue` 思路） |
| 待我审批 | 汇总所有需要当前用户操作的 `ticket_approval_task`（角标计数） |
| 审批操作弹窗 | 同意/驳回/转交操作 |
| 工作流配置 → 审批节点配置 | 管理员配置某个 transition 的审批流 |

#### 方案一优缺点

| | |
|-|-|
| **优点** | • 完全内聚在工单系统内，不依赖外部服务<br>• FSM 生命周期管控不变，现有逻辑零破坏<br>• 可按需启用（只有配置了 `requireApproval=true` 的 transition 才走审批层）<br>• 数据模型借鉴米多星球设计，概念对齐 |
| **缺点** | • 需新写一套轻量审批引擎（约 1000-1500 行服务层代码）<br>• 不支持复杂条件分支（如需要可后续扩展）<br>• 与米多星球的数据不互通（各自独立） |
| **适用场景** | 工单系统需要**独立运行**，不依赖米多星球，但希望有类似审批体验 |

### 方案二：审批流代理模式（调用米多星球 API）

**比喻**：就像顺丰快递在某些地区用"加盟商"来完成末端配送，快递单状态仍由顺丰总系统维护，但具体配送动作由加盟商执行。

#### 核心思路

- 工单系统**不实现审批逻辑**，仅维护工单状态机
- 当需要审批时，工单系统通过**内部 HTTP 调用**将审批任务**委托给米多星球审批引擎**
- 米多星球审批引擎处理完后，通过**回调接口**通知工单系统更新状态

#### 集成架构

```
工单系统                              米多星球审批引擎
    │                                      │
    │  1. 提交工单 → 需要审批               │
    │  ─────────────────────────────────>  │
    │     POST /api/workflow/form/submission/save
    │     (ticket_id 作为 businessRelatedId)│
    │                                      │
    │  ← ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─  │
    │  2. 返回 submissionId                 │
    │                                      │
    │  3. 审批人在米多星球操作审批           │
    │                                      │
    │  4. 审批通过 → 触发 Spring Event      │
    │      ↓                               │
    │  WorkflowInstancePassedEvent         │
    │  ─────────────────────────────────>  │
    │  TicketApprovalCallbackController    │
    │  5. 工单系统收到回调 → 执行 FSM 流转  │
```

#### 工单系统新增内容

```java
// 新增回调 Controller
@RestController
@RequestMapping("/api/internal/ticket/approval-callback")
public class TicketApprovalCallbackController {
    
    @PostMapping("/passed")
    public void onApprovalPassed(@RequestBody ApprovalCallbackRequest request) {
        // 根据 businessRelatedId 找到工单
        // 执行 FSM 流转（dept_approval → executing）
    }
    
    @PostMapping("/rejected")
    public void onApprovalRejected(@RequestBody ApprovalCallbackRequest request) {
        // 执行 FSM 流转（dept_approval → rejected）
    }
}

// 工单系统新增字段（ticket 表）
// approval_submission_id bigint  -- 关联米多星球的表单提交 ID
```

#### 米多星球需要的适配

1. `WorkflowInstancePassedEvent` 中增加 `businessSystemType` 标识（区分来自工单系统的提交）
2. `XxxAfterWorkflowPassedListener` 中新增 `TicketApprovalAfterWorkflowPassedListener`，负责回调工单系统
3. 或者：米多星球通过 Webhook/HTTP 回调工单系统（更解耦）

#### 方案二优缺点

| | |
|-|-|
| **优点** | • 完全复用米多星球成熟引擎（无需重写审批逻辑）<br>• 审批体验全局统一（同一个"待办"入口）<br>• 复杂功能（条件分支/加签/撤回）立即可用 |
| **缺点** | • **强依赖米多星球服务**（一旦米多星球不可用，工单审批卡死）<br>• 数据割裂（审批记录在米多星球，工单数据在工单系统）<br>• 两个系统的用户体系需统一（企微 userid 对齐）<br>• 网络调用带来延迟和故障点<br>• 联调复杂度较高 |
| **适用场景** | 两个系统**同一公司同一用户体系**，且米多星球是"公司内部标准审批平台" |

### 方案三：增强型 Transition（最轻量）

**比喻**：快递在"等待签收"状态里，增加了一个"必须本人或家属亲签，不能代签"的规则，但整体状态机不变。

#### 核心思路

仅在现有 `WorkflowTransition` 中扩展 `approvalConfig` 字段，支持**简单的多人审批配置**，不引入完整的节点概念，以最小代价支持最常见的"N 人审批"场景。

#### WorkflowTransition 扩展

```json
{
  "id": "t03",
  "from": "pending_approval",
  "to": "executing",
  "name": "审批通过",
  "allowedRoles": ["ADMIN"],
  "requireApproval": true,
  "approvalConfig": {
    "mode": "countersign",
    "approvers": [
      { "type": "member", "userId": "zhangsan_wework" },
      { "type": "deptLeader" }
    ],
    "dueHours": 48,
    "rejectTo": "rejected"
  }
}
```

新增 `ticket_approval_task` 表（简化版，无 node_config）：

```sql
CREATE TABLE `ticket_approval_task` (
  `id`           bigint AUTO_INCREMENT PRIMARY KEY,
  `ticket_id`    bigint NOT NULL,
  `transition_id` varchar(50) NOT NULL,
  `assignee_id`  varchar(64) NOT NULL,
  `status`       varchar(20) DEFAULT 'pending',
  `remark`       varchar(500),
  `operate_time` datetime,
  `create_time`  datetime DEFAULT CURRENT_TIMESTAMP
);
```

#### 方案三优缺点

| | |
|-|-|
| **优点** | • 改动极小（扩展一个 JSON 字段 + 加一张表）<br>• 不破坏现有工作流定义结构<br>• 适合快速验证需求 |
| **缺点** | • 功能较弱（无条件分支/无版本/无加签）<br>• 后期如果需求扩展，很可能推倒重来<br>• 仅支持单节点审批，无法表达多步串行审批 |
| **适用场景** | 当前审批需求确实简单（就是"需要某人点头"），短期不会有复杂场景 |

## 四、方案对比与选型建议

| 维度 | 方案一（嵌入式） | 方案二（代理模式） | 方案三（增强 Transition） |
|------|---------------|----------------|------------------------|
| **开发量** | 中（约 1200 行后端 + 前端 3 个组件） | 中（两端各约 300 行 + 联调） | 小（约 300 行） |
| **功能完整度** | ★★★★☆ | ★★★★★ | ★★☆☆☆ |
| **运行独立性** | 完全独立 | 依赖米多星球 | 完全独立 |
| **数据统一性** | 各自独立 | 审批数据在米多星球 | 各自独立 |
| **可扩展性** | 高（可逐步补齐条件分支等） | 受米多星球限制 | 低 |
| **用户体验统一性** | 独立待办入口 | 共用米多星球待办 | 独立待办入口 |
| **风险** | 低 | 高（服务依赖） | 低 |

### 推荐选型：**方案一（嵌入式审批任务层）**

**理由：**

1. **工单系统是独立产品**，不应该强依赖米多星球的可用性。方案二的服务依赖会形成"单点故障链"。
2. **方案一借鉴了米多星球的成熟设计**（`node_config` JSON结构、审批模式枚举、`countersign/orsign/sequential` 概念），不是重新发明，是有据可查的设计。
3. **渐进落地**：Phase 1 先实现最核心的"多人审批 + 待审批列表"；Phase 2 再补"条件分支"；Phase 3 视需要再对接米多星球统一待办入口。
4. **方案三的问题**：结构太轻，一旦出现"A 审批通过后再转 B 审批"的串行需求，整个结构就撑不住，不如一次做对。

## 五、推荐方案的分阶段实施计划

### Phase 1：核心审批能力（优先级：高）

**目标**：支持多人审批的"审批工单"场景，用户能看到"待我审批"列表

| 任务 | 说明 |
|------|------|
| 新增 3 张数据库表 | `ticket_approval_config` / `ticket_approval_task` / `ticket_approval_record` |
| `WorkflowTransition` 扩展 `requireApproval` | 布尔字段，向下兼容 |
| `TicketApprovalService` | 核心服务：启动审批/执行审批/节点完成判断/自动流转触发 |
| 审批人解析器 | 支持 `member`（指定成员）/ `deptLeader`（部门负责人）两种类型 |
| 企微消息推送 | 创建任务时通知审批人 |
| 前端：工单详情审批面板 | 展示当前审批任务列表 + 操作按钮（同意/驳回） |
| 前端：待我审批列表 | 聚合入口，支持快速操作 |

**涉及改动范围：**

- `ticket-domain`：`WorkflowTransition` 模型扩展
