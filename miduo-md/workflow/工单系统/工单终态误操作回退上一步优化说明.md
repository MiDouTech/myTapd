# 工单终态误操作回退上一步优化说明

## 1. 背景与问题

在工单详情页中，用户可能误点将工单推进到终态（如“已完成”“已关闭”）。  
一旦误点，当前流程里没有直观的“一键撤回”入口，容易造成处理卡死、责任人错位。

典型现象：
- 工单状态已进入“已完成”或“已关闭”；
- 处理人仍停留在终态操作后的值，无法快速恢复到上一步责任人；
- 用户需要人工介入数据库或走额外操作链，成本高、风险大。

## 2. 目标与非目标

### 2.1 目标
- 在工单详情页提供“回退上一步”按钮，减少误操作恢复成本。
- 回退时同时恢复：
  - 状态：恢复到进入终态前的上一步状态；
  - 处理人：恢复到上一步处理人。
- 对“已完成”增加业务保护：仅当关联 Bug 简报存在“未归档”时允许回退。

### 2.2 非目标
- 不新增独立 API，复用 `API000017`（`PUT /api/ticket/return/{id}`）。
- 不改变既有工作流配置结构（`isReturn=true` 的标准退回规则保持不变）。
- 不引入数据库结构变更。

## 3. 方案概要

## 3.1 前端展示规则（工单详情）
- 显示“回退上一步”按钮的条件：
  1. 工单状态为“已关闭”；或
  2. 工单状态为“已完成”，且关联 Bug 简报存在“未归档”记录。

点击后调用：
- `PUT /api/ticket/return/{id}`（API000017）
- 请求体：`{ reason: "误操作回退上一步" }`

## 3.2 后端执行规则（TicketWorkflowAppService.returnTicket）
优先走原有工作流退回逻辑：
- 若当前节点存在 `isReturn=true` 的合法流转，保持原行为。

终态兜底回退逻辑（本次新增）：
- 当当前状态无 `isReturn=true` 路径时，若满足以下条件则允许“回退上一步”：
  - 当前状态 = `closed`；或
  - 当前状态 = `completed` 且存在未归档 Bug 简报。
- 从 `ticket_flow_record` 中查找“最近一次进入当前状态的状态变更记录”（`from_status != to_status`）。
- 回退目标：
  - `targetStatus = 上述记录.from_status`
  - `targetAssignee = 上述记录.from_assignee_id`
- 落库与审计：
  - 更新工单状态；
  - 同步恢复处理人；
  - 写 `ticket_log`、`ticket_flow_record`（`flow_type=RETURN`）；
  - 记录时间追踪与状态变更事件。

## 4. 影响范围

### 4.1 前端
- `miduo-frontend/src/views/ticket/TicketDetailView.vue`
  - 新增“回退上一步”按钮显示条件与点击逻辑。

### 4.2 后端
- `ticket-platform/ticket-application/.../TicketWorkflowAppService.java`
  - `returnTicket` 增加终态兜底回退能力。
- `ticket-platform/ticket-infrastructure/.../TicketFlowRecordMapper.java`
- `ticket-platform/ticket-infrastructure/.../TicketFlowRecordMapper.xml`
  - 新增查询“最近一次进入当前状态的流转记录”。

## 5. 验收标准

- 场景A：状态=已完成，且关联简报未归档  
  - 详情页显示“回退上一步”按钮；
  - 点击后状态回到上一步，处理人恢复为上一步处理人。

- 场景B：状态=已关闭  
  - 详情页显示“回退上一步”按钮；
  - 点击后状态回到上一步，处理人恢复为上一步处理人。

- 场景C：状态=已完成，但关联简报全部已归档  
  - 不显示“回退上一步”按钮；
  - 后端接口也拒绝回退（防绕过）。

## 6. 回滚与兼容策略

- 兼容性：
  - 标准工作流退回（`isReturn=true`）不受影响；
  - 仅在“无标准退回路径”时触发终态兜底逻辑。
- 回滚策略：
  - 可回滚到本改动前版本，恢复为“仅支持工作流显式退回”模式。
