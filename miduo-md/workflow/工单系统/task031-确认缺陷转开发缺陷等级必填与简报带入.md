# Task031：确认缺陷转开发缺陷等级必填与简报带入

> **业务模块**：工单系统（缺陷流转 + Bug简报）  
> **对应产品文档**：4.4.5 缺陷工单工作流、4.12 Bug简报管理  
> **关联接口**：API000015 状态流转、API000022 更新缺陷工单测试信息、API000022/API000023 Bug简报创建与更新（历史编号按既有文档保留）

---

## 一、背景

测试在「确认缺陷转开发」时已经完成缺陷判断，此时缺陷等级是后续开发排期、审核流和 Bug 简报归因的重要标签。  
如果这个标签没有在转开发时固定下来，后面填写 Bug 简报时就容易像“快递单缺少易碎标签”一样，需要人工再补，甚至补错。

## 二、现状问题

1. 「确认缺陷转开发」弹窗只强制填写复现环境，没有强制填写缺陷等级。
2. 后端状态流转接口只接收并保存 `reproduceEnv`，没有接收并保存 `severityLevel`。
3. Bug 简报后续虽然会从缺陷测试信息读取 `severityLevel`，但前置链路可能没有写入该字段。
4. 前端 Bug 简报多工单预填时缺陷等级排序未覆盖 P4，只有 P4 时可能被默认 P2 覆盖。

## 三、目标

1. 测试从「测试中」执行「确认缺陷转开发」时，必须填写缺陷等级。
2. 缺陷等级支持 `P0/P1/P2/P3/P4`，保存到缺陷工单测试信息。
3. 后续工单完成后生成/填写 Bug 简报时，自动带入该缺陷等级。
4. 前端手动关联工单填写 Bug 简报时，P4 也能被正确带入，不被默认等级覆盖。

## 四、非目标

1. 不新增数据库字段。
2. 不新增 API 编号，只扩展既有 API000015 的请求体字段。
3. 不改变 Bug 简报审核规则：P0/P1/P2 仍走审核，P3/P4 仍可直接归档。

## 五、设计方案

### 5.1 前端流转弹窗

- 在「确认缺陷转开发」弹窗中新增必填项：`缺陷等级`。
- 默认读取「测试信息」页已保存的 `severityLevel`。
- 未填写时阻止提交，并提示“确认缺陷转开发前请选择缺陷等级”。

### 5.2 后端流转兜底

- `TransitInput` 新增 `severityLevel` 字段。
- `TicketWorkflowAppService` 在 `testing -> pending_dev_accept` 流转时，同时校验：
  - 复现环境必须存在且有效；
  - 缺陷等级必须存在且有效。
- `TicketBugApplicationService` 将流转入参中的缺陷等级规范化为 `P0/P1/P2/P3/P4` 并保存到 `ticket_bug_test_info.severity_level`。

### 5.3 Bug 简报自动带入

- 自动草稿：继续通过 `BugReportApplicationService.applyAutoPrefill(...)` 从测试信息批量读取 `severityLevel`。
- 手动编辑：前端 `BugReportEditView` 的等级优先级补齐 P4，保证 P4 不被默认 P2 覆盖。

## 六、验收标准

- [ ] 缺陷工单在「测试中」点击「确认缺陷转开发」时，弹窗显示必填的「缺陷等级」。
- [ ] 不填写缺陷等级无法提交，页面给出明确提示。
- [ ] 填写 P0/P1/P2/P3/P4 任一等级均可正常转开发，并在「测试信息」中保存。
- [ ] 工单后续完成后自动创建 Bug 简报草稿时，简报 `severityLevel` 与转开发时填写的等级一致。
- [ ] 手动创建/编辑 Bug 简报并关联该工单时，等级自动预填；仅 P4 的场景也保持 P4。

## 七、回滚与兼容策略

- 已保存的历史测试信息不做迁移。
- 若历史工单未填写缺陷等级，后续 Bug 简报仍保持既有默认/手填逻辑。
- 如需回滚，只需移除 API000015 的 `severityLevel` 入参校验与前端必填项，不影响数据库结构。
