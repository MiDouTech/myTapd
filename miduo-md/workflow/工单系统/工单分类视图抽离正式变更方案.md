# 工单分类视图抽离正式变更方案

## 1. 方案概要

本方案用于将当前「所有工单」页面中的「功能缺陷/技术缺陷」和「监控告警」抽离为独立工作入口。

用一句话说明：  
原来所有工单像一个大筐，普通问题、缺陷、告警都扔在里面；本次要把它改成“分拣货架”，普通工单、缺陷工单、告警工单分别进入对应货架，但底层仍使用同一套工单详情和流转能力。

推荐目标导航：

```text
工单中心
├─ 我的工单
├─ 分类工单
│  ├─ 一级分类A
│  ├─ 一级分类B
│  └─ ...（分类管理有多少一级分类，就显示多少入口）
├─ 全部工单（管理员可见）
└─ 工单看板
```

核心原则：

1. **列表分开，详情复用**：只拆列表入口，不拆工单主表和详情页。
2. **按分类管理配置识别，不按名称写死**：兼容测试环境「功能缺陷」和生产环境「技术缺陷」命名差异。
3. **告警优先使用来源识别**：告警列表以 `source=alert` 为主，分类 ID 为辅助。
4. **分类入口跟随分类管理**：分类管理有多少一级分类，前端就显示多少个分类工单入口。
5. **先做最小闭环，再做字段定制**：第一阶段复用现有列表组件，第二阶段再按缺陷/告警优化列字段。

## 4. 目标方案

### 4.1 新增视图编码

建议在 `TicketView` 中新增以下视图：

| 视图编码 | 中文名称 | 说明 |
|---|---|---|
| `general` | 通用工单 | 排除缺陷和告警 |
| `defect` | 缺陷工单 | 只看缺陷分类 |
| `alert` | 告警工单 | 只看告警来源/告警分类 |
| `all` | 全部工单 | 管理员全量查询 |

说明：  
当前已有 `all`，建议把菜单文案从「所有工单」收敛为「全部工单」或仅管理员可见；普通用户主入口改为「通用工单」。

### 4.2 新增路由

| 路由 | 页面 | view |
|---|---|---|
| `/ticket/category/{categoryId}` | 动态分类工单 | `category` |
| `/ticket/general` | 通用工单 | `general` |
| `/ticket/defect` | 缺陷工单 | `defect` |
| `/ticket/alert` | 告警工单 | `alert` |
| `/ticket/all` | 全部工单 | `all` |

第一阶段继续复用 `TicketListView.vue`。动态分类入口使用分类管理中的一级分类 ID 作为 `categoryGroupId`，后端查询该分类及所有子分类下的工单。

### 4.3 分类管理口径设计

不新增“缺陷分类 ID/告警分类 ID”系统配置。分类归属直接来源于「管理 -> 分类管理」中的工作流绑定。

| 分类管理配置 | 归属视图 | 说明 |
|---|---|---|
| 分类绑定「缺陷工单工作流」（内置工作流 id=3） | 缺陷工单 | 分类名称可为功能缺陷、技术缺陷或其它业务命名 |
| 分类绑定「告警工单流转」（内置工作流 id=4） | 告警工单 | 告警视图同时按 `source=alert` 兜底 |
| 分类绑定其它工作流 | 通用工单 | 普通协作类工单 |

兼容策略：

1. 分类自身未绑定工作流时，使用最近父级分类绑定的工作流作为有效工作流。
2. 没有任何分类绑定缺陷工作流时，缺陷工单列表返回空，避免错误展示全量。
3. 没有任何分类绑定告警工作流时，告警工单仍可按 `source=alert` 查询。
4. 通用工单自动排除有效工作流为缺陷/告警的分类，并默认排除 `source=alert`。

### 4.4 后端查询规则

#### 通用工单 `view=general`

SQL 条件建议：

```text
category_id NOT IN (defectCategoryIds + alertCategoryIds)
AND source != 'alert'
```

注意：

1. `defectCategoryIds` / `alertCategoryIds` 来自分类管理中的有效工作流绑定。
2. `source` 为空的历史数据应保留在通用工单中。

#### 缺陷工单 `view=defect`

SQL 条件建议：

```text
category_id IN (defectCategoryIds)
```

`defectCategoryIds` 由分类管理中绑定「缺陷工单工作流」的分类自动推导。后续增强可增加缺陷扩展表存在性判断，但第一阶段不建议强依赖侧表，避免历史缺陷数据漏查。

#### 告警工单 `view=alert`

SQL 条件建议：

```text
source = 'alert'
OR category_id IN (alertCategoryIds)
```

原因：  
告警映射可能把告警分到不同分类；只按分类容易漏，只按来源可能漏掉历史手工补录告警。分类部分从分类管理绑定「告警工单流转」的分类自动推导，两者组合更稳。

#### 全部工单 `view=all`

保持当前全量口径，但需要增加权限保护。

### 4.5 前端页面规则

#### 通用工单

1. 页面标题：`通用工单`
2. 分类下拉：默认隐藏缺陷/告警分类，或显示但标注为其他入口处理。
3. 筛选项保留：工单编号、标题、公司名称、分类、状态、优先级、时间范围。
4. 列字段第一阶段复用现有工单列表。

#### 缺陷工单

1. 页面标题：`缺陷工单`
2. 分类下拉只展示缺陷分类树。
3. 状态下拉优先展示缺陷流程状态。
4. 第二阶段建议列字段调整为：
   - 工单编号
   - 缺陷标题
   - 公司名称
   - 缺陷分类
   - 优先级/严重程度
   - 状态
   - 测试受理人
   - 开发负责人
   - 有效反馈

#### 告警工单

1. 页面标题：`告警工单`
2. 分类下拉只展示告警分类树。
3. 状态下拉优先展示告警状态。
4. 第二阶段建议列字段调整为：
   - 工单编号
   - 告警标题
   - 告警等级
   - 来源系统
   - 最近触发时间
   - 状态
   - SLA
   - 处理人

#### 全部工单

1. 页面标题：`全部工单`
2. 仅管理员或有授权用户可见。
3. 可增加筛选开关：
   - 包含缺陷工单
   - 包含告警工单

### 4.6 顶部搜索规则

当前顶部搜索默认跳转 `/ticket/all?q=...`。本次建议：

| 场景 | 推荐规则 |
|---|---|
| 普通用户搜索 | 跳转 `/ticket/general?q=...` |
| 管理员搜索 | 跳转 `/ticket/all?q=...` |
| 搜索结果为空 | 提示可切换到缺陷/告警/全部工单继续查 |

如果第一阶段不做角色分流，建议顶部搜索保留跳转 `/ticket/all`，但将菜单中的「全部工单」标为管理员视角。

### 4.7 权限规则

| 页面 | 权限建议 |
|---|---|
| 通用工单 | 原「所有工单」普通权限 |
| 缺陷工单 | 测试、开发、工单管理员、系统管理员 |
| 告警工单 | 运维/值班、工单管理员、系统管理员 |
| 全部工单 | 工单管理员、系统管理员 |

第一阶段若权限体系暂不细分，可先保持菜单可见，但全部工单至少应限制管理员。

## 5. 接口设计

### 5.1 复用现有分页接口

优先复用：

```text
GET /api/ticket/page
```

新增或扩展请求参数：

| 参数 | 类型 | 说明 |
|---|---|---|
| `view` | `String` | 新增 `general`、`defect`、`alert` |
| `categoryGroupId` | `Long` | 分类分组ID：查询该分类及所有子分类下的工单 |
| `categoryIds` | `List<Long>` | 多分类包含过滤，后端内部可使用 |
| `excludeCategoryIds` | `List<Long>` | 多分类排除过滤，后端内部可使用 |
| `source` | `String` | 可选来源过滤，如 `alert` |
| `excludeSource` | `String` | 可选来源排除，如排除 `alert` |

说明：  
前端不直接传 `categoryIds/excludeCategoryIds`，后端根据 `view` 和分类管理中的工作流绑定自动计算，减少前端知道过多业务口径。

### 5.2 分类管理配置方式

无需新增配置接口。管理员在现有「分类管理」里维护分类绑定的工作流即可：

| 操作 | 结果 |
|---|---|
| 分类绑定「缺陷工单工作流」 | 该分类及继承该工作流的子分类进入缺陷工单 |
| 分类绑定「告警工单流转」 | 该分类及继承该工作流的子分类进入告警工单 |
| 分类绑定其它工作流 | 默认进入通用工单 |

## 7. 实施步骤

### 阶段一：最小闭环

1. 后端扩展 `TicketView`：
   - `category`
   - `general`
   - `defect`
   - `alert`
2. `TicketApplicationService` 支持 `categoryGroupId`，自动查询该分类及所有子分类。
3. `TicketApplicationService` 根据 view 从分类管理的工作流绑定推导缺陷/告警分类。
4. `TicketMapper.xml` 增加分类包含/排除与来源过滤。
5. 前端从分类树接口动态生成分类分组菜单。
6. `TicketListView.vue` 根据 view 展示不同标题并调用分页接口。
7. 完成动态分类工单和全部工单入口的基本验证。

### 7.1 第一阶段代码落地记录（2026-06-06）

本次已完成第一阶段最小闭环：

1. 后端 `TicketView` 新增：
   - `category`：分类工单
   - `general`：通用工单
   - `defect`：缺陷工单
   - `alert`：告警工单
   - `all`：全部工单
2. 后端 `GET /api/ticket/page` 复用 API000007，按 `view` 自动读取分类管理中的工作流绑定并过滤：
   - `view=general`：排除缺陷/告警分类，并默认排除 `source=alert`
   - `view=defect`：只包含分类管理中绑定「缺陷工单工作流」的分类
   - `view=alert`：包含 `source=alert` 或分类管理中绑定「告警工单流转」的分类
3. 后端分页新增 `categoryGroupId`：查询该分类及所有子分类下的工单。
4. 前端新增动态分类路由和左侧菜单：
   - `/ticket/category/{categoryId}`：分类工单
   - 分类管理有多少一级分类，左侧「分类工单」下就显示多少个入口
   - `/ticket/all`：全部工单
5. 顶部搜索默认在当前分类分组内搜索；不在分类页时默认进入第一个分类分组。

### 阶段二：体验增强

1. 缺陷工单列表定制缺陷字段。
2. 告警工单列表定制告警字段。
3. 分类下拉按视图过滤。
4. 状态下拉按视图过滤。
5. 顶部搜索支持选择搜索范围。

### 阶段三：分类管理体验增强

1. 在分类管理中对工作流选择增加解释说明，例如“绑定缺陷工作流后会进入缺陷工单入口”。
2. 分类树可增加视图归属标签，帮助管理员快速识别该分类属于通用/缺陷/告警。
3. 分类下拉按视图过滤时，直接复用分类管理中的工作流绑定结果。

## 8. 验证计划

### 8.1 数据准备

至少准备以下真实数据：

| 数据类型 | 要求 |
|---|---|
| 分类分组工单 | 分类管理中的任意一级分类，含其所有子分类 |
| 普通工单 | 分类有效工作流不是缺陷/告警，且 `source` 不是 `alert` |
| 缺陷工单 | 分类管理中绑定「缺陷工单工作流」 |
| 告警工单A | `source=alert` |
| 告警工单B | 分类管理中绑定「告警工单流转」 |
| 管理员账号 | 可查看全部工单 |
| 普通账号 | 不应看到全部工单入口 |

### 8.2 页面验收

| 编号 | 操作 | 预期 |
|---|---|---|
| FE01 | 打开任意分类工单入口 | 只展示该分类分组及子分类下的工单 |
| FE02 | 分类管理新增一级分类后刷新页面 | 左侧「分类工单」下出现对应入口 |
| FE03 | 打开告警工单 | 展示告警来源或告警分类工单 |
| FE04 | 打开全部工单 | 管理员可查看所有类型 |
| FE05 | 点击任意工单编号 | 正常进入原详情页 |
| FE06 | 使用分类筛选 | 分类选项符合当前视图范围 |
| FE07 | 使用状态筛选 | 状态筛选不影响视图基础口径 |

### 8.3 接口验收

| 编号 | 接口 | 参数 | 预期 |
|---|---|---|---|
| API01 | `/api/ticket/page` | `view=general` | 排除缺陷和告警 |
| API02 | `/api/ticket/page` | `view=defect` | 只返回缺陷分类 |
| API03 | `/api/ticket/page` | `view=alert` | 返回 `source=alert` 或告警分类 |
| API04 | `/api/ticket/page` | `view=all` | 管理员返回全量 |

### 8.4 回归验收

1. 我的工单各页签不受影响。
2. 待出简报工单不受影响。
3. 工单详情不受影响。
4. 告警 Webhook 建单不受影响。
5. 日报统计按分类 ID 过滤不受影响。
6. 工单看板如仍使用全量口径，需要明确是否同步拆分。

## 9. 发布准备

发布前必须确认：

1. 测试/生产环境中，缺陷类分类是否已绑定「缺陷工单工作流」。
2. 告警类分类是否已绑定「告警工单流转」。
3. 分类子级是否需要继承父级工作流，或已单独绑定工作流。
4. 是否存在告警映射到非告警分类的情况。
5. 哪些角色可以查看「全部工单」。
6. 是否需要在页面增加“为什么少了部分工单”的说明提示。

## 10. 回滚方案

### 10.1 快速回滚

1. 将菜单入口恢复为原「所有工单」。
2. 将默认入口重新指向 `/ticket/all`。

### 10.2 代码回滚

如果新视图逻辑导致查询异常：

1. 回退前端路由和菜单变更。
2. 回退后端 `TicketView` 和分页查询条件变更。

## 11. 风险清单

| 风险 | 等级 | 说明 | 应对 |
|---|---|---|---|
| 分类工作流绑定错误 | 高 | 会导致列表漏单或错分 | 发布前在分类管理核对缺陷/告警分类绑定的工作流 |
| 告警漏查 | 高 | 只按分类不够 | 告警视图同时按 `source=alert` 查询 |
| 普通用户找不到旧工单 | 中 | 原所有工单口径改变 | 提供「全部工单」权限入口和说明 |
| 顶部搜索口径不一致 | 中 | 搜索结果与当前页面不一致 | 搜索目标按角色或范围选择 |
| 看板统计与列表不一致 | 中 | 看板仍可能全量 | 明确看板是否同步拆分 |
| SQL 条件空列表异常 | 中 | `IN ()` 或 `NOT IN ()` 报错 | 后端拼接前判空 |

| 告警是否只按 `source=alert`？ | 建议 `source=alert OR 分类绑定告警工作流` |
| 通用工单是否排除手工创建的告警分类？ | 建议排除 |
| 顶部搜索默认搜哪里？ | 普通用户搜通用，管理员搜全部 |

## 13. 结论

推荐采用「独立菜单入口 + 后端按分类管理工作流绑定推导 view 口径」方案。

这是当前成本和收益最均衡的做法：

1. 用户理解成本低：像医院分科室，一眼知道去哪处理。
2. 技术改动可控：复用现有分页接口和详情页。
3. 跨环境稳定：按分类管理里的工作流绑定识别，不怕分类名称不同。
4. 可分阶段上线：先拆入口，再定制字段和配置页面。
