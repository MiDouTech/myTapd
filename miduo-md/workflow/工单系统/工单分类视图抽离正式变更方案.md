# 工单分类视图抽离正式变更方案

## 1. 方案概要

本方案用于将当前「所有工单」页面中的「功能缺陷/技术缺陷」和「监控告警」抽离为独立工作入口。

用一句话说明：  
原来所有工单像一个大筐，普通问题、缺陷、告警都扔在里面；本次要把它改成“分拣货架”，普通工单、缺陷工单、告警工单分别进入对应货架，但底层仍使用同一套工单详情和流转能力。

推荐目标导航：

```text
工单中心
├─ 我的工单
├─ 通用工单
├─ 缺陷工单
├─ 告警工单
├─ 全部工单（管理员可见）
└─ 工单看板
```

核心原则：

1. **列表分开，详情复用**：只拆列表入口，不拆工单主表和详情页。
2. **按分类 ID 配置，不按名称写死**：兼容测试环境「功能缺陷」和生产环境「技术缺陷」命名差异。
3. **告警优先使用来源识别**：告警列表以 `source=alert` 为主，分类 ID 为辅助。
4. **普通用户少看噪音，管理员保留全量**：通用工单默认排除缺陷/告警，全部工单仅授权角色可见。
5. **先做最小闭环，再做字段定制**：第一阶段复用现有列表组件，第二阶段再按缺陷/告警优化列字段。

## 4. 目标方案

### 4.1 新增视图编码

建议在 `TicketView` 中新增以下视图：

| 视图编码 | 中文名称 | 说明 |
|---|---|---|
| `general` | 通用工单 | 排除缺陷和告警 |
| `defect` | 缺陷工单 | 只看缺陷分类 |
| `alert` | 告警工单 | 只看告警来源/告警分类 |
| `all` | 全部工单 | 管理员全量查询 |

说明：  
当前已有 `all`，建议把菜单文案从「所有工单」收敛为「全部工单」或仅管理员可见；普通用户主入口改为「通用工单」。

### 4.2 新增路由

| 路由 | 页面 | view |
|---|---|---|
| `/ticket/general` | 通用工单 | `general` |
| `/ticket/defect` | 缺陷工单 | `defect` |
| `/ticket/alert` | 告警工单 | `alert` |
| `/ticket/all` | 全部工单 | `all` |

第一阶段可以继续复用 `TicketListView.vue`，通过路由 meta 或 query 自动设置 `view`。

### 4.3 配置项设计

建议新增系统配置，统一放入 `system_config`。

| 配置分组 | 配置键 | 示例 | 说明 |
|---|---|---|---|
| `TICKET_VIEW` | `ticket_defect_category_ids` | `12,13,14` | 缺陷分类 ID 集合 |
| `TICKET_VIEW` | `ticket_alert_category_ids` | `21,22` | 告警分类 ID 集合 |
| `TICKET_VIEW` | `ticket_general_exclude_alert_source` | `true` | 通用工单是否排除告警来源 |

兼容策略：

1. `ticket_defect_category_ids` 为空时，缺陷工单列表返回空并提示管理员配置，避免错误展示全量。
2. `ticket_alert_category_ids` 为空时，告警工单仍可按 `source=alert` 查询。
3. `ticket_general_exclude_alert_source` 缺失时默认 `true`。
4. 通用工单排除分类时，将缺陷分类和告警分类合并为 `excludeCategoryIds`。

### 4.4 后端查询规则

#### 通用工单 `view=general`

SQL 条件建议：

```text
category_id NOT IN (defectCategoryIds + alertCategoryIds)
AND source != 'alert'（当配置开启时）
```

注意：

1. 分类 ID 列表为空时，不拼接空 `NOT IN`。
2. `source` 为空的历史数据应保留在通用工单中。

#### 缺陷工单 `view=defect`

SQL 条件建议：

```text
category_id IN (defectCategoryIds)
```

后续增强可增加缺陷扩展表存在性判断，但第一阶段不建议强依赖侧表，避免历史缺陷数据漏查。

#### 告警工单 `view=alert`

SQL 条件建议：

```text
source = 'alert'
OR category_id IN (alertCategoryIds)
```

原因：  
告警映射可能把告警分到不同分类；只按分类容易漏，只按来源可能漏掉历史手工补录告警。两者组合更稳。

#### 全部工单 `view=all`

保持当前全量口径，但需要增加权限保护。

### 4.5 前端页面规则

#### 通用工单

1. 页面标题：`通用工单`
2. 分类下拉：默认隐藏缺陷/告警分类，或显示但标注为其他入口处理。
3. 筛选项保留：工单编号、标题、公司名称、分类、状态、优先级、时间范围。
4. 列字段第一阶段复用现有工单列表。

#### 缺陷工单

1. 页面标题：`缺陷工单`
2. 分类下拉只展示缺陷分类树。
3. 状态下拉优先展示缺陷流程状态。
4. 第二阶段建议列字段调整为：
   - 工单编号
   - 缺陷标题
   - 公司名称
   - 缺陷分类
   - 优先级/严重程度
   - 状态
   - 测试受理人
   - 开发负责人
   - 有效反馈

#### 告警工单

1. 页面标题：`告警工单`
2. 分类下拉只展示告警分类树。
3. 状态下拉优先展示告警状态。
4. 第二阶段建议列字段调整为：
   - 工单编号
   - 告警标题
   - 告警等级
   - 来源系统
   - 最近触发时间
   - 状态
   - SLA
   - 处理人

#### 全部工单

1. 页面标题：`全部工单`
2. 仅管理员或有授权用户可见。
3. 可增加筛选开关：
   - 包含缺陷工单
   - 包含告警工单

### 4.6 顶部搜索规则

当前顶部搜索默认跳转 `/ticket/all?q=...`。本次建议：

| 场景 | 推荐规则 |
|---|---|
| 普通用户搜索 | 跳转 `/ticket/general?q=...` |
| 管理员搜索 | 跳转 `/ticket/all?q=...` |
| 搜索结果为空 | 提示可切换到缺陷/告警/全部工单继续查 |

如果第一阶段不做角色分流，建议顶部搜索保留跳转 `/ticket/all`，但将菜单中的「全部工单」标为管理员视角。

### 4.7 权限规则

| 页面 | 权限建议 |
|---|---|
| 通用工单 | 原「所有工单」普通权限 |
| 缺陷工单 | 测试、开发、工单管理员、系统管理员 |
| 告警工单 | 运维/值班、工单管理员、系统管理员 |
| 全部工单 | 工单管理员、系统管理员 |

第一阶段若权限体系暂不细分，可先保持菜单可见，但全部工单至少应限制管理员。

## 5. 接口设计

### 5.1 复用现有分页接口

优先复用：

```text
GET /api/ticket/page
```

新增或扩展请求参数：

| 参数 | 类型 | 说明 |
|---|---|---|
| `view` | `String` | 新增 `general`、`defect`、`alert` |
| `categoryIds` | `List<Long>` | 多分类包含过滤，后端内部可使用 |
| `excludeCategoryIds` | `List<Long>` | 多分类排除过滤，后端内部可使用 |
| `source` | `String` | 可选来源过滤，如 `alert` |
| `excludeSource` | `String` | 可选来源排除，如排除 `alert` |

说明：  
前端不一定直接传 `categoryIds/excludeCategoryIds`，更推荐后端根据 `view` 和配置计算，减少前端知道过多业务口径。

### 5.2 配置接口

可复用现有系统配置能力；如果管理端需要可视化配置，建议新增“工单视图配置”页面或并入分类管理。

建议接口：

| 接口名称 | 方法 | 路径 | 说明 |
|---|---|---|---|
| 查询工单视图配置 | GET | `/api/ticket-view/config` | 返回缺陷/告警分类 ID 配置 |
| 保存工单视图配置 | PUT | `/api/ticket-view/config` | 保存缺陷/告警分类 ID 配置 |

> 若后续实际新增接口，必须按 API 编号规范分配接口编号，并更新 `功能接口对应关系.md`。本方案阶段不预占编号，避免和并行开发冲突。

1. 新增系统配置键：
   - `ticket_defect_category_ids`
   - `ticket_alert_category_ids`
   - `ticket_general_exclude_alert_source`

   - `general`
   - `defect`

7. 完成通用/缺陷/告警/全部四个入口的基本验证。

## 9. 发布准备

发布前必须确认：

1. 测试环境缺陷分类 ID。
2. 生产环境缺陷分类 ID。
3. 告警分类 ID。
4. 是否存在告警映射到非告警分类的情况。
5. 哪些角色可以查看「全部工单」。
6. 是否需要在页面增加“为什么少了部分工单”的说明提示。

## 10. 回滚方案

### 10.1 快速回滚

1. 将菜单入口恢复为原「所有工单」。
2. 清空：
   - `ticket_defect_category_ids`
   - `ticket_alert_category_ids`
3. 将默认入口重新指向 `/ticket/all`。

### 10.2 代码回滚

如果新视图逻辑导致查询异常：

1. 回退前端路由和菜单变更。
2. 回退后端 `TicketView` 和分页查询条件变更。
3. 保留配置项不影响旧逻辑，后续可复用。

## 11. 风险清单

| 风险 | 等级 | 说明 | 应对 |
|---|---|---|---|
| 分类 ID 配错 | 高 | 会导致列表漏单 | 发布前用分类树接口核对 |
| 告警漏查 | 高 | 只按分类不够 | 告警视图同时按 `source=alert` 查询 |
| 普通用户找不到旧工单 | 中 | 原所有工单口径改变 | 提供「全部工单」权限入口和说明 |
| 顶部搜索口径不一致 | 中 | 搜索结果与当前页面不一致 | 搜索目标按角色或范围选择 |
| 看板统计与列表不一致 | 中 | 看板仍可能全量 | 明确看板是否同步拆分 |
| SQL 条件空列表异常 | 中 | `IN ()` 或 `NOT IN ()` 报错 | 后端拼接前判空 |

## 12. 待确认问题

| 问题 | 推荐答案 |
|---|---|
| 「全部工单」是否普通用户可见？ | 建议仅管理员/工单管理员可见 |
| 缺陷工单是否包含所有子分类？ | 建议配置父分类时自动包含子分类 |
| 告警是否只按 `source=alert`？ | 建议 `source=alert OR 告警分类ID` |
| 通用工单是否排除手工创建的告警分类？ | 建议排除 |
| 顶部搜索默认搜哪里？ | 普通用户搜通用，管理员搜全部 |

## 13. 结论

推荐采用「独立菜单入口 + 后端 view 口径配置化」方案。

这是当前成本和收益最均衡的做法：

1. 用户理解成本低：像医院分科室，一眼知道去哪处理。
2. 技术改动可控：复用现有分页接口和详情页。
3. 跨环境稳定：按分类 ID 配置，不怕分类名称不同。
4. 可分阶段上线：先拆入口，再定制字段和配置页面。
