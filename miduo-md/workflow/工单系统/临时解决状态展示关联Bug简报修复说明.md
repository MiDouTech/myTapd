# 临时解决状态展示关联 Bug 简报修复说明

## 一、背景与现状问题

当前工单详情页「关联 Bug 简报」区块本身没有按工单状态做前端隐藏，只要详情接口返回 `bugReports` 数据就会展示。

但系统自动创建简报草稿的触发点仅覆盖工单进入终态（主要是 `completed`）。  
当缺陷工单进入 **临时解决**（`temp_resolved`）时，若此前没有手动关联简报，常出现「工单详情页看不到关联 Bug 简报」的问题。

这会导致用户在“临时解决”节点无法直接进入简报补充，链路体验割裂。

## 二、目标

- 缺陷工单流转到 **临时解决**（`temp_resolved`）后，也能自动建立（或复用已有）Bug 简报关联；
- 工单详情页在“临时解决”状态下可正常展示「关联 Bug 简报」；
- 保持已有行为不变：`completed` 仍可自动创建/关联；`closed` 仍不自动创建。

## 三、非目标

- 不调整前端「关联 Bug 简报」区块样式与交互；
- 不修改手动创建/手动关联简报流程；
- 不清理历史数据。

## 四、设计方案（单一最小实现）

### 4.1 关键决策

在 `BugReportEventListener` 新增对 `TicketStatusChangedEvent` 的监听：

- 当 `newStatus == temp_resolved` 时，调用既有方法 `createDraftFromClosedTicket(ticketId, operatorId)`；
- 该方法内部已有幂等保护（已有关联则直接返回），可避免重复创建；
- 该方法内部已有缺陷工单判断（非缺陷工单直接返回）。

### 4.2 影响范围

- 后端事件监听层：`ticket-application` 模块；
- 详情接口无结构变更；
- 前端无需改接口契约。

### 4.3 兼容与回滚策略

- 兼容：不改数据库结构，不改 API 入参与出参；
- 回滚：若需回退，仅移除新增 `temp_resolved` 监听方法，保留原有终态监听逻辑。

## 五、验收标准

1. 缺陷工单从处理中/开发中流转到 `temp_resolved` 后，工单详情页可看到「关联 Bug 简报」区块；
2. 同一工单重复进出 `temp_resolved` 时，不会重复生成多条自动关联；
3. 缺陷工单进入 `completed` 行为保持原样；
4. 缺陷工单进入 `closed` 行为保持原样（不自动创建简报）。
