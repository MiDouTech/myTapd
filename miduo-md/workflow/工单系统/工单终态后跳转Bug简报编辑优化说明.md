# 工单终态后跳转 Bug 简报编辑优化说明

## 1. 背景与现状问题

在工单详情页中，用户点击「处理完成」或「临时解决确认」后，系统原先只做“滚动到关联简报区块并高亮入口”的引导。  
这会带来两个问题：

1. 用户还要再点一次「查看简报」，操作链路长；
2. 若没有现成草稿，用户需要手动去新建简报并重新选择当前工单，容易漏关联。

类比：像你下单后系统只把你“滚动到购物车位置”，但不帮你“直接打开结算页并带好商品”。

## 2. 目标与非目标

### 2.1 目标
- 在工单详情页点击「处理完成」或「临时解决确认」成功后，自动跳转到 Bug 简报编辑页；
- 自动关联当前工单，减少重复选择；
- 兼容“已有待填写草稿”和“暂无草稿”两种情况。

### 2.2 非目标
- 不改后端流转接口；
- 不改 Bug 简报字段校验规则；
- 不改现有简报详情与审核流程。

## 3. 方案概要

### 3.1 工单详情页跳转策略
位置：`miduo-frontend/src/views/ticket/TicketDetailView.vue`

- 在流转成功并刷新详情后，识别本次动作：
  - 动作名包含「处理完成」或「临时解决确认」（兼容历史「验证完成」）；
  - 若动作名缺失，降级按目标状态 `temp_resolved/completed` 判断。
- 命中后执行跳转：
  1. 若当前工单已有“待填写”简报草稿，直接跳 `/bug-report/edit/{reportId}`；
  2. 否则跳 `/bug-report/edit?ticketId={当前工单ID}`。

### 3.2 Bug 简报编辑页自动关联策略
位置：`miduo-frontend/src/views/bugreport/BugReportEditView.vue`

- 仅在新建模式（无 `:id`）下处理路由参数；
- 支持三种入参形式：
  - `ticketId=123`
  - `ticketIds=123,456`
  - `ticketIds=123&ticketIds=456`
- 页面初始化时自动把参数里的工单 ID 写入 `form.ticketIds`，并补查工单详情用于：
  - 显示下拉文案（工单编号+标题）；
  - 写入工单状态映射；
  - 触发已有自动预填逻辑（问题描述、影响范围等）。

## 4. 影响范围

- 前端页面：
  - `TicketDetailView.vue`
  - `BugReportEditView.vue`
- 文档：
  - 本说明文档
  - 根目录 `readme.md`

## 5. 兼容与回滚策略

### 5.1 兼容策略
- 若后端未返回动作名，仍可按状态码兜底判断是否跳转；
- 若路由参数中的某个工单查询失败，不阻断页面，用户仍可手动搜索补选。

### 5.2 回滚策略
- 若需要回滚，恢复到“流转后仅滚动引导”的旧前端逻辑；
- 不涉及数据库结构和后端接口变更，可独立前端回退。

## 6. 验收标准

1. 在工单详情点击「处理完成」后，成功流转即跳转到 Bug 简报编辑页；
2. 在工单详情点击「临时解决确认」后，成功流转即跳转到 Bug 简报编辑页；
3. 跳转后的编辑页中，“关联工单”默认已包含当前工单；
4. 若已有待填写草稿，直接进入该草稿编辑页，不再停留工单详情页。
