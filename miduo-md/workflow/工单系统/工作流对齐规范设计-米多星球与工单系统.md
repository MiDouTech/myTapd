> **版本**：v1.0  
> **日期**：2026-06-07  
> **范围**：工单系统（myTapd）工作流与米多星球（miduo-backend）审批引擎的规范对齐、差距分析、改造路径

## 一、两套系统的根本差异（先想清楚）

```
米多星球                          工单系统
────────────────────────────────────────────────
表单提交驱动的审批流               状态机（FSM）驱动的工单流转
↓                                 ↓
wf_form → wf_flow_template        workflow.states + transitions
→ node_config.flowSteps           → TicketWorkflowAppService.transit()
(start→审批→抄送→条件→end)        (pending_assign→processing→completed)
↓                                 ↓
wf_flow_task（每人一任务）         ticket + assignee（一个处理人）
↓                                 ↓
WorkflowInstancePassedEvent        TicketStatusChangedEvent
→ 业务 Listener 回写              → SLA/通知/自动分派
```

**结论**：两者**不是同一种东西**，不能直接替换。  
正确的姿势是：**工单系统的 FSM 管大流程，米多星球的审批引擎作为某个 transition 的「内嵌审批层」**。  
这正是 PR #176 已经实现的架构。

## 三、通用节点/状态/字段标准定义

### 3.1 审批节点（ApprovalNode）标准字段

参考米多星球 `approval` 节点，工单系统 `ApprovalNode` 的标准字段定义：

| 字段 | 类型 | 必填 | 说明 | 米多星球对应 |
|------|------|------|------|-------------|
| `nodeKey` | String | 是 | transition 内唯一标识，如 `node_01` | `id` |
| `nodeName` | String | 是 | 展示名，如「部门负责人审批」 | `name` |
| `approveMode` | enum | 是 | `single`/`countersign`/`orsign`/`sequential` | `multiAssigneeMode` |
| `assigneeType` | enum | 是 | `member`（指定人）/ `groupLeader`（处理组长） | `assigneeType`（含更多类型） |
| `assigneeIds` | List\<Long\> | 条件 | `assigneeType=member` 时必填 | `assigneeNames[].id` |
| `dueHours` | Integer | 否 | 节点超时时间（小时），可选 | `nodeDeadline`+`nodeDeadlineUnit` |

**差距（工单系统 vs 米多星球）：**

| 能力 | 工单系统 | 米多星球 | 是否需要补充 |
|------|---------|---------|------------|
| assigneeType | member / groupLeader | member / deptLeader / multiLevelLeader | 建议 Phase 2 补 deptLeader |
| 按钮自定义 | 无 | buttonSettings（同意/驳回自定义名） | 可选，低优先级 |
| 审批意见必填 | 无全局配置 | `agree_comment_required` 模板级开关 | 可选 |
| 自动审批（发起人=审批人） | 无 | `auto_approve_when_initiator` | 可选，防止卡死问题 |

### 3.2 审批模式（approveMode）标准行为

| 模式 | 完成条件 | 米多星球对应 |
|------|----------|------------|
| `single` | 唯一审批人完成 | `FlowApprovalTypeEnum.SINGLE` |
| `countersign` | 全部 pending 人 agree | `FlowApprovalTypeEnum.COUNTERSIGN` |
| `orsign` | 任一 pending 人 agree → 其余 skipped | `FlowApprovalTypeEnum.ORSIGN` |
| `sequential` | 按 sortOrder 依次激活，一人完成后下一人 pending | `FlowApprovalTypeEnum.SEQUENTIAL` |

### 3.3 审批任务状态（ApprovalTaskStatus）标准

| 状态 | 含义 | 米多星球对应 |
|------|------|------------|
| `pending` | 待处理（当前轮到） | `FlowTaskStatusEnum.PENDING` |
| `waiting` | 等待中（sequential 未轮到） | `FlowTaskStatusEnum.WAITING` |
| `approved` | 已通过 | `FlowTaskStatusEnum.DONE` |
| `rejected` | 已驳回 | `FlowTaskStatusEnum.DONE`（type=reject） |
| `transferred` | 已转交 | `FlowTaskStatusEnum.DONE`（transfer） |
| `skipped` | 已跳过（orsign 他人通过） | 无对应（`DONE`+关闭逻辑） |

### 3.4 ApprovalConfig 完整 JSON 规范

```json
{
  "passedStatus": "executing",
  "rejectedStatus": "rejected",
  "nodes": [
    {
      "nodeKey": "node_01",
      "nodeName": "部门负责人审批",
      "approveMode": "single",
      "assigneeType": "groupLeader",
      "assigneeIds": [],
      "dueHours": 48
    },
    {
      "nodeKey": "node_02",
      "nodeName": "总经理审批",
      "approveMode": "countersign",
      "assigneeType": "member",
      "assigneeIds": [101, 102],
      "dueHours": 72
    }
  ]
}
```

**约束规则：**
- `passedStatus` 和 `rejectedStatus` 必须是当前工作流 `states` 中存在的状态码
- `nodes` 至少 1 个
- `assigneeType=member` 时 `assigneeIds` 至少 1 个
- `nodeKey` 在同一 `transition` 内唯一

## 六、通用部分 vs 工单专属部分

### 6.1 通用部分（可抽象为公共能力）

```
审批任务引擎：
├─ 任务创建（单人/会签/或签/依次）
├─ 执行操作（agree/reject/transfer）
├─ 完成判断（会签全过/或签一过）
└─ 结果回调（passedStatus/rejectedStatus 触发 FSM）

可抽象为 ApprovalEngine 接口，供：
- 工单系统（ticket_approval_task 表驱动）
- 米多星球 OA 场景（wf_flow_task 表驱动）
复用同一套逻辑
```

### 6.2 工单专属部分（不通用）

```
工单专属：
├─ FSM 状态机（workflow.states + transitions JSON）
├─ 分派策略（ROUND_ROBIN/LOAD_BALANCE/MATRIX）
├─ SLA 计时器（按状态 slaAction 驱动）
├─ 缺陷/告警专用状态（investigating/alert_triggered 等）
└─ 企微建单/群机器人/自动分派
```
