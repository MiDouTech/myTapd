# 企微智能机器人视频解密内存优化

## 背景与问题

智能机器人 `video.url` 与 `image.url` 相同，为 AES-256-CBC 加密体。此前视频处理使用 `HttpUtil.downloadBytes` + `Cipher.doFinal` 全量进堆，大视频（官方上限约 100MB）时密文与明文并存，易引发 OOM 或 GC 压力。

## 目标

- 大视频场景将堆占用控制在缓冲区级别（常量级），避免密文+明文双份大数组常驻堆。
- 小视频保持低延迟（可选内存路径）。
- 上传完成后删除临时文件，不长期占用磁盘。

## 非目标

- 不改变七牛存储路径与对外 URL 形态。
- 不改变图片消息处理路径（仍以 byte[] 为主，体量通常较小）。

## 方案概要

1. **下载**：HTTP 流写入临时文件（`wecom-vid-enc-*.bin`），避免整段密文进堆。
2. **解密**：
   - 若加密文件大小 ≤ `wecom.image.video-decrypt-in-memory-max-mb`：读入内存后 `doFinal` + PKCS#7(块32) 去填充。
   - 若超过阈值：`Cipher.update` 按 **16 字节 AES 块**对齐分块解密写入临时明文文件（`wecom-vid-dec-*.mp4`），再对明文文件尾部做 PKCS#7 校验并 `RandomAccessFile#setLength` 截断。
3. **上传**：`FileInputStream` 交给现有 `QiniuUploadService.uploadStream`，由 SDK 读取上传。
4. **清理**：`finally` 中 `Files.deleteIfExists`；可选配置 `wecom.image.video-temp-directory` 指定临时目录（自动 `createDirectories`）。

## 配置项

| 配置键 | 说明 | 默认 |
|--------|------|------|
| `wecom.image.video-decrypt-in-memory-max-mb` | 加密临时文件小于等于该值（MB）时走内存解密 | 8 |
| `wecom.image.video-temp-directory` | 临时文件目录，空则 `java.io.tmpdir` | 空 |

## 验收要点

- 大于阈值视频处理完成后，临时目录中无残留 `wecom-vid-enc-*` / `wecom-vid-dec-*`（正常结束路径）。
- 并发处理视频时堆增长不明显随单视频体积线性翻倍（可通过 JVM 堆 dump 或监控观察）。

## 回滚

恢复仅内存解密实现即可；配置项可保留兼容。
