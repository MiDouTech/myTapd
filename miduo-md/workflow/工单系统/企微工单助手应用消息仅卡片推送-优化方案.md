# 企微工单助手应用消息仅卡片推送-优化方案

## 方案概要

将 `WecomAppMessageSender` 的发送策略从“卡片优先 + 文本兜底”调整为“仅卡片发送”，并在发送前对标题与描述进行 UTF-8 字节级收敛，确保满足企微 textcard 限制后再发送。

## 影响范围

| 模块 | 文件 | 影响说明 |
|---|---|---|
| 应用层通知 | `ticket-application/.../WecomAppMessageSender.java` | 发送策略调整为仅 textcard，新增标题与描述自动截断 |
| 基础设施 | `ticket-infrastructure/.../WecomClient.java` | 无接口变更，仅复用已有 `sendTextCardMessage` |
| 其他通知渠道 | 站内信/邮件/群机器人 | 无行为变更 |

## 关键决策

1. **只保留 textcard 发送**
   - 删除 text 类型应用消息兜底分支，避免同类通知样式漂移。
2. **按“先标题后正文”策略收敛长度**
   - 先处理标题：优先截断“标题中的描述部分”（如 `工单状态更新 - 标题...` 中 ` - ` 后部分）；
   - 再处理正文：若描述超长，逐步压缩正文，必要时替换为精简提示。
3. **字节级安全截断**
   - 以 UTF-8 字节长度为准截断，避免中文字符被半截裁断导致乱码或发送失败。

## 详细设计

### 1）标题收敛策略

1. 标题为空时回退为“工单通知”；
2. 超过 textcard 标题上限时，优先保留前缀（如事件名、工单号），截断后半段描述；
3. 若无结构化分隔符可用，则直接按字节截断整段标题，并追加截断标记。

### 2）正文收敛策略

1. 保留原有 HTML 转义与分行规则；
2. 超过 textcard 描述上限时，进行字节截断并追加“内容过长，已截断”提示；
3. 若提示后仍超限，进一步压缩为最短可发送提示文本。

### 3）链接策略

1. 优先使用入参 `detailLink`；
2. 为空时回退 `trustedDomain` 兜底；
3. 保证传给 textcard 的 URL 不为空字符串。

## 风险与兼容

### 风险

1. 过长标题会被截断，部分细节不在标题中展示；
2. 超长正文会被压缩，完整信息需进入系统详情页查看。

### 兼容策略

1. 保留原有通知触发链路，不影响消息发送时机；
2. 截断提示明确告知“内容已截断”，降低误解风险。

## 回滚方案

如出现线上兼容问题，可回滚 `WecomAppMessageSender` 本次提交，恢复“textcard + text兜底”旧逻辑；不涉及数据库变更，无需执行数据回滚。

## 验收清单

- [ ] 企微应用消息不再出现 text 类型消息；
- [ ] 超长标题场景下依然发送成功；
- [ ] 超长正文场景下依然发送成功，且可见截断提示；
- [ ] 后端编译通过（`mvn clean install -DskipTests` 或等效模块构建）。
