# 工单详情-已关闭显示有效反馈优化说明

## 1. 变更需求收敛

### 1.1 背景
- 当前工单详情右侧信息区中，“有效报告”字段依赖 `bugSummaryInfo` 才会渲染。
- 当工单状态已是“已关闭”，但该工单未关联 Bug 简报且尚未设置手工值时，页面不会显示该字段，用户无法直接补录。

### 1.2 现状问题
- 用户在“已关闭”状态下看不到“有效反馈”入口，信息采集不完整。
- 需要通过其他绕行路径（如先关联简报）才能出现该字段，操作成本高。

### 1.3 优化目标
1. 当工单状态为“已关闭”时，固定显示“有效反馈”字段。
2. 字段支持下拉选择，仅提供“是/否”两个选项。
3. 选择后可保存，刷新后保持显示。

### 1.4 非目标
- 不改动后端接口协议。
- 不改动“非已关闭状态”下的展示规则。
- 不新增新的状态流转规则。

### 1.5 验收标准
1. 打开任意“已关闭”工单，右侧信息区可看到“有效反馈”字段。
2. 点击字段进入编辑后，下拉项仅包含“是”“否”。
3. 保存后刷新页面，值与保存结果一致。
4. 非“已关闭”状态，不出现该字段的可编辑入口。

## 2. 设计方案更新

### 2.1 方案概要
- 前端工单详情右侧面板改造字段显示条件：由“依赖 `bugSummaryInfo`”调整为“状态为 `closed` 即显示”。
- 展示值优先级：
  1. `bugSummaryInfo.isValidReport`（后端摘要值）
  2. `bugCustomerInfo.manualValidReport`（手工值）
  3. 空值展示 `-`
- 关闭态识别补强：
  1. 状态码先做 `trim + lowerCase + normalize`
  2. 若状态码缺失或历史值异常，回退使用 `statusLabel` 中文判定（包含“关闭”）
  3. 若状态与标签都缺失，使用 `closedAt` 作为兜底信号

### 2.2 影响范围
- `miduo-frontend/src/views/ticket/components/bug/BugDetailInfoPanel.vue`

### 2.3 关键决策
1. 文案统一为“有效反馈”，与业务沟通口径一致。
2. 编辑控件继续使用 `el-select`，避免引入新组件学习成本。
3. 保存沿用现有 `updateBugCustomerInfo` 接口，减少联调范围和风险。

### 2.4 兼容与回滚策略
- 兼容：未设置过值时显示 `-`，不影响历史数据读取。
- 回滚：如出现异常，可回退 `BugDetailInfoPanel.vue` 到变更前版本，恢复原显示条件。

## 3. 问题复盘（刷新后无变化）

### 3.1 现象
- 页面顶部状态标签显示“已关闭”，但右侧“有效反馈”字段未出现。

### 3.2 原因
- 前端显示条件仅使用状态码精确匹配，历史/异常数据场景下可能不命中。

### 3.3 修复策略
1. 关闭态判定改为“状态码标准化 + 中文状态（status/statusLabel）+ closedAt 兜底”三重保险。
2. 增加验收检查项：同时验证 `status`、`statusLabel`、`closedAt` 任一命中关闭信号即可显示字段。

## 4. 服务端兜底补充（前后端版本错位兼容）

### 4.1 补充背景
- 线上可能存在前端静态资源缓存或灰度版本错位，导致页面仍按旧逻辑（依赖 `bugSummaryInfo`）决定是否显示“有效反馈”。

### 4.2 兜底策略
1. 工单为“已关闭”且未关联 Bug 简报时，后端仍返回非空 `bugSummaryInfo` 对象。
2. 若有手工值（`manualValidReport`）则写入摘要；无手工值也返回空摘要对象（`isOverdue=false`）。

### 4.3 预期收益
- 即使前端命中旧渲染逻辑，也能稳定显示“有效反馈”入口，避免用户看不到字段。
