# 工单评论 @ 提及与企微推送（优化）

## 一、变更需求

### 1.1 背景

工单详情「评论与处理记录」需要支持协作场景：评论中 @ 指定同事，对方通过 **站内通知 + 企业微信应用消息** 及时知晓，体验参考语雀评论区 @ 人。

### 1.2 现状问题

- 评论接口仅接收正文，无法稳定识别被通知人。
- 发表评论后未对被 @ 用户走企微 `message/send`。

### 1.3 目标

- 支持随评论提交被 @ 用户 ID 列表；服务端校验并去重后发送站内通知与企微文本消息（在偏好开启且已绑定企微账号时）。
- 前端评论框支持 @ 选人，与正文联动。

### 1.4 非目标

- 企微群内 @、机器人解析评论。
- 独立「@ 我的」页面（依赖通知中心即可）。

### 1.5 验收标准

1. 评论可 @ 同事并成功提交。
2. 被 @ 用户收到站内通知；已绑定企微且开启推送时收到企微消息。
3. 未绑定企微或关闭推送时不报错，评论仍成功。
4. 无 @ 时与现网行为一致。

## 二、设计方案概要

### 2.1 接口

- 扩展 `POST /api/ticket/{id}/comment`（接口编号 **API000508**）请求体：增加可选字段 `mentionedUserIds: number[]`。
- 若现有页面无合适用户搜索接口，可新增轻量用户搜索 API（按 `api-interface-numbering.mdc` 分配新编号并更新功能接口对应关系表）。

### 2.2 后端

- `addComment` 持久化评论后：批量查询用户 → 写站内通知 → 异步发企微文本（失败仅日志）。
- 用户查询、通知写入均避免循环内单条 SQL。

### 2.3 前端

- 富文本编辑器扩展 @ 下拉；提交时携带 `mentionedUserIds`。
- 组件结构满足前端嵌套与 props 透传规范。

### 2.4 回滚与兼容

- 旧客户端不传 `mentionedUserIds` 时行为不变。
- 可关闭异步企微发送或配置开关（若后续增加）。

### 2.5 实施记录（2026-04-07）

- 后端：`TicketCommentInput.mentionedUserIds`；`addComment` 写入评论后发布 `TicketCommentMentionEvent`，`TicketEventNotificationListener` 在事务提交后 `@Async` 调用 `NotificationOrchestrator.dispatch(..., COMMENT_MENTION, ...)`（站内 + 企微文本卡片，遵循用户偏好）。
- 通知类型新增 `COMMENT_MENTION`，且该类型**不参与** Redis 通知合并，避免连续 @ 被吞。
- 前端：工单详情评论 `@同事` 弹层 + `getUserList`（API000403）；`RichTextEditor` 暴露 `insertHtml` 插入带 `data-user-id` 的 @ 片段。
