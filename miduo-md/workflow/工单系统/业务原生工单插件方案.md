## 一、方案概述

### 1.1 一句话定义

**业务原生工单插件** = 嵌入业务系统前端的轻量 JS SDK + 工单平台开放网关，让用户在星球/BDE 等业务系统内 **3 步提工单、全程不跳页**，处理人员在工单系统内 **上下文全带、直接定位**。

### 1.2 设计哲学

| 视角 | 道德经隐喻 | 设计原则 |
|------|-----------|---------|
| 终端用户 | 为道日损 | 用户只写问题描述，其余信息自动携带 |
| 业务接入方 | 我无为而民自化 | 引入 + 初始化 + 按需 setContext，半天接入 |
| 工单处理人 | 不出户，知天下 | 打开工单即见用户/系统/业务/环境全量上下文 |

### 1.3 与现有系统的关系

本方案 **不推翻** 现有工单核心（分类→模板→工作流→SLA→通知），而是在其之上新增：

```
┌─────────────────────────────────────────────────────────────┐
│  业务系统（星球 / BDE / …）                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  ticket-sdk.min.js（弹窗/侧边栏/悬浮球）              │   │
│  └──────────────────────────┬──────────────────────────┘   │
└─────────────────────────────┼───────────────────────────────┘
                              │ HTTPS
┌─────────────────────────────▼───────────────────────────────┐
│  工单平台 — 插件开放网关（/api/open/v1/plugin/**）          │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────┐  │
│  │ 接入应用管理  │  │ 插件建单 API  │  │ 状态 Webhook 出站 │  │
│  │ integration  │  │ 复用核心服务  │  │ 复用现有派发器    │  │
│  └──────────────┘  └──────────────┘  └──────────────────┘  │
│  ┌──────────────────────────────────────────────────────┐  │
│  │ TicketApplicationService.createTicket()（已有）       │  │
│  │ WebhookDispatchService（已有）                        │  │
│  │ OpenApiAppAuthFilter（已有，扩展写权限）               │  │
│  └──────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

**复用清单**（已实现，直接继承）：

| 能力 | 现有实现 | 插件复用方式 |
|------|---------|-------------|
| 工单创建核心逻辑 | `TicketApplicationService.createTicket()` | 插件 API 透传调用 |
| 分类/模板/动态字段 | `category` + `template.fields_config` + `customFields` | SDK 按接入配置预填分类，上下文写入 `customFields` |
| AppKey + HMAC 签名 | `OpenApiAppAuthFilter` + `task029` 签名规范 | 插件写接口走同一路径 `/api/open/v1/**` |
| 出站 Webhook | `WebhookDispatchService` + `WebhookEventType` | 按应用维度订阅，payload 增加 `pluginContext` |
| 公开工单链接 | `GET /api/open/ticket/{ticketNo}` | SDK「我的工单」详情跳转 |
| 用户身份 | 米多 SSO / 企微 OAuth JWT | SDK 用业务系统传入的 user + 平台用户映射 |

**需新建**（本方案核心增量）：

| 能力 | 说明 |
|------|------|
| 接入应用注册（DB） | 替代 YAML 单 AppKey，支持多业务系统、每应用独立配置 |
| 插件写接口 | AppKey 鉴权下的建单、查我的工单、上传附件 |
| JS SDK 包 | `ticket-sdk.min.js`，modal/sidebar/float 三种模式 |
| 插件上下文字段 | `plugin_context` JSON 列 + 标准字段映射 |
| 按应用 Webhook | `integration_app.callback_url` 绑定出站地址 |

## 二、现状差距分析

### 2.1 三视角体验差距

| 环节 | 目标体验 | 现状 | 差距 |
|------|---------|------|------|
| 用户找入口 | 当前页 1 秒点开 | 需跳转工单系统独立域名 | 无 embed SDK |
| 用户填表 | 只写描述，其余自动带 | Web 端 3 步选手动填分类/模板/字段 | 无上下文自动采集 |
| 接入方集成 | 十几行代码半天搞定 | 需 JWT/API Key 代表用户调 `/api/v1/tickets` | AppKey 仅只读 |
| 状态回同步 | 业务系统内看进度 | 全局 Webhook，无 per-app 绑定 | 缺应用级回调 |
| 处理人看上下文 | 用户/系统/页面/单据号齐全 | 仅有 `customFields` 文本，无结构化上下文 | 缺 `plugin_context` |

### 2.2 鉴权模型差距（关键）

当前三种鉴权各自服务不同场景：

| 鉴权 | 路径 | 能力 | 插件是否适用 |
|------|------|------|-------------|
| JWT | `/api/**` | 完整用户态 | SDK 需用户登录工单系统，**违背不跳页** |
| 个人 API Key | `/api/**`、`/api/v1/**` | 代理某用户 | 适合 IDE Agent，**不适合业务系统插件** |
| AppKey + 签名 | `/api/open/v1/**` | 系统级只读 | 缺写接口 |

**结论**：插件建单必须走 **AppKey 写权限 + 用户身份透传** 的新模式，不能复用个人 API Key。

## 三、总体架构

### 3.1 端到端闭环

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  业务系统内   │────▶│  ticket-sdk  │────▶│  用户描述问题 │
│  遇到问题     │     │  自动填信息   │     │  一键提交     │
└──────────────┘     └──────────────┘     └──────┬───────┘
                                                 │
                                                 ▼
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  状态同步回去  │◀────│  工单系统处理  │◀────│  插件开放网关  │
│  用户看进度   │     │  上下文全带   │     │  分类路由     │
└──────────────┘     └──────────────┘     └──────────────┘
```

### 3.2 模块划分

| 模块 | 职责 | 交付物 |
|------|------|--------|
| **ticket-sdk**（前端 npm/CDN） | UI 壳、上下文采集、API 调用 | `ticket-sdk.min.js` |
| **plugin-gateway**（后端） | 接入应用管理、插件专用 API | Controller + Service |
| **plugin-context**（数据） | 结构化上下文存储与展示 | DB 字段 + 详情 DTO |
| **plugin-webhook**（出站） | 按应用推送状态变更 | 扩展 `WebhookDispatchService` |

### 3.3 SDK 部署形态

| 形态 | 适用场景 | 说明 |
|------|---------|------|
| **CDN `<script>`** | 星球、BDE 等老系统 | **默认主推**，零构建，一行引入 |
| **npm `@miduo/ticket-sdk`** | Vue/React 新业务 | Tree-shaking、TypeScript 类型 |
| **iframe 降级** | CSP 严格环境 | 平台托管页面，postMessage 通信（P3 可选） |

> **分发决策（2026-06-24）**：暂不做管理后台离线包下载。SDK 统一通过 CDN / npm 发布，接入方在管理后台仅申请 **AppKey 凭证与业务配置**，不下载 SDK 文件。

## 四、终端用户视角：3 步提工单

### 4.1 交互流程

```
第1步：点「提交工单」/ 悬浮球
         ↓
第2步：弹窗打开 — 信息已预填，用户只写描述（+ 可选截图/附件）
         ↓
第3步：提交 → 成功提示 → 弹窗关闭（≤30 秒）
```

### 4.2 SDK 展示模式

| mode | 行为 | 默认 |
|------|------|------|
| `modal` | 居中弹窗 | ✅ 默认 |
| `sidebar` | 右侧滑出面板 | 宽表单场景 |
| `float` | 右下角悬浮球 | 无 entry 绑定时启用 |

### 4.3 自动携带信息（SDK 采集）

SDK 在 `init()` 和 `setContext()` 时组装 `pluginContext`，提交时一并发送：

| 分类 | 字段 | 采集方式 |
|------|------|---------|
| **用户信息** | `userId`, `userName`, `dept`, `role`, `mobile` | `init({ user })` 由业务系统传入 |
| **系统信息** | `system`, `module`, `page`, `pageUrl`, `pageTitle` | `init({ system })` + `setContext({ module, page })` + `location` 自动读 |
| **业务信息** | `bizId`, `bizType`, `bizNo`, `extra` | `setContext()` 由业务页面传入 |
| **环境信息** | `browser`, `os`, `screen`, `userAgent`, `network` | SDK 自动采集（`navigator` / `screen`） |
| **辅助信息** | `screenshot`, `attachments`, `clientTime` | 用户一键截图 / 粘贴上传 |

### 4.4 表单简化策略

插件模式 **跳过分类选择**（对用户不可见）：

1. 接入应用在管理端绑定 **默认分类 ID**（或按 `bizType` 映射多分类）
2. 标题自动生成：`[{system}/{module}] {用户首句描述前 50 字}`
3. 优先级默认 `medium`，用户可在弹窗中切换（可选展示）
4. 模板动态字段：仅展示接入方配置的「用户可见字段」，其余由 `pluginContext` 静默写入 `customFields`

### 4.5 「我的工单」入口

SDK 提供 `TicketSDK.openMyTickets()`：

- 在业务系统内打开侧边栏列表（调用 `GET /api/open/v1/plugin/tickets/mine`）
- 点击工单项 → 弹窗展示摘要 + 外链到公开详情页 `/open/ticket/{ticketNo}`
- 状态角标通过 Webhook 推送到业务系统后，由 SDK 事件 `onTicketUpdated` 刷新

## 五、接入方视角：4 步接入

### 5.1 第 1 步：申请接入

管理端「集成配置 → 接入应用」新建应用：

| 配置项 | 说明 |
|--------|------|
| 应用名称 | 如「米多星球」 |
| 系统标识 `system` | 全局唯一，如 `xingqiu` |
| AppKey / AppSecret | 自动生成，可轮换 |
| 默认分类 | 绑定工单分类（决定模板与工作流） |
| 分类映射（可选） | `bizType → categoryId` JSON |
| 回调地址 | 状态变更 Webhook URL |
| 允许域名 | SDK 来源白名单（CORS） |
| 用户映射策略 | `externalUserId → sys_user` 自动创建/关联 |

### 5.2 第 2 步：引入 SDK

```html
<script src="https://ticket.miduo.com/sdk/v1/ticket-sdk.min.js"></script>
<script>
  TicketSDK.init({
    appKey: 'ak_xxxxxxxx',
    system: 'xingqiu',
    user: {
      id: '10086',
      name: '张三',
      dept: '渠道部',
      role: '运营'
    },
    mode: 'modal',
    entry: '#submit-ticket-btn'   // 可选；不传则显示悬浮球
  });
</script>
```

> **安全说明**：`appSecret` **不得**写在前端。SDK 只持 `appKey`；建单请求由 SDK 向插件网关发起，网关侧用 **短期 launchToken**（见 §7.2）完成用户身份绑定，避免 Secret 泄露。

### 5.3 第 3 步：配置业务上下文

```javascript
// 渠道管理详情页
TicketSDK.setContext({
  module: '渠道管理',
  page: '渠道详情',
  bizId: 'Q20260624001',
  bizType: 'channel',
  extra: { channelName: '华东一区' }
});
```

### 5.4 第 4 步：联调上线

| 检查项 | 验收标准 |
|--------|---------|
| 建单成功 | 返回 `ticketNo`，工单 `source=plugin` |
| 上下文完整 | 详情页「插件上下文」面板字段齐全 |
| 分类/工作流 | 自动进入正确分类的处理流程 |
| Webhook | 状态变更后业务系统收到回调 |
| 我的工单 | 用户只看到自己在该应用下提交的工单 |

## 七、安全与鉴权设计

### 7.1 双层鉴权模型

```
业务前端 SDK                插件网关                    工单核心
─────────────              ──────────                  ─────────
appKey（公开）    ──▶   校验 AppKey + 域名白名单
launchToken（短期） ──▶   校验 Token + 用户映射  ──▶  createTicket(creatorId)
```

### 7.2 LaunchToken 机制

因 `appSecret` 不能放前端，采用 **服务端签发短期令牌**：

1. 业务系统后端调用 `POST /api/open/v1/plugin/launch-token`（AppKey + 签名，**服务端调用**）
2. 请求体带 `externalUserId`, `userName`, `dept` 等
3. 返回 `launchToken`（JWT，有效期 5 分钟，单次使用）
4. 前端 SDK `init({ launchToken })` 或 `TicketSDK.refreshToken()` 由业务后端注入
5. SDK 建单时带 `Authorization: Bearer {launchToken}`

**类比**：像游乐园手腕票——门口（业务后端）用员工卡（AppSecret）换票，游客（前端 SDK）凭手腕票进场，不用揣员工卡。

### 7.3 写接口 AppKey 权限扩展

在 `integration_app` 表增加 `permissions` 字段：

| 权限码 | 说明 |
|--------|------|
| `plugin:ticket:create` | 插件建单 |
| `plugin:ticket:read-mine` | 查我的工单 |
| `plugin:attachment:upload` | 上传附件 |
| `open:ticket:export` | 已有只读导出 |

`OpenApiAppAuthFilter` 增加路径级权限校验。

### 7.4 签名规范

**完全复用** [task029](./task029-外部系统工单开放拉取接口.md) 的 HMAC-SHA256 签名规则，不新增协议。

## 八、数据模型设计

### 8.1 新增表：`integration_app`（接入应用）

```sql
CREATE TABLE `integration_app` (
  `id` bigint NOT NULL AUTO_INCREMENT COMMENT '主键',
  `app_name` varchar(100) NOT NULL COMMENT '应用名称',
  `app_key` varchar(64) NOT NULL COMMENT 'AppKey',

  `default_category_id` bigint NOT NULL COMMENT '默认工单分类',

  `callback_secret` varchar(128) DEFAULT NULL COMMENT '回调签名密钥',
  `allowed_origins` varchar(1000) DEFAULT NULL COMMENT 'CORS 白名单，逗号分隔',
  `permissions` varchar(500) NOT NULL DEFAULT 'plugin:ticket:create' COMMENT '权限列表',
  `status` tinyint NOT NULL DEFAULT 1 COMMENT '1启用 0禁用',
  `create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `update_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  `create_by` varchar(50) NOT NULL DEFAULT '',
  `update_by` varchar(50) NOT NULL DEFAULT '',
  `deleted` tinyint NOT NULL DEFAULT 0,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_app_key` (`app_key`),
  UNIQUE KEY `uk_system_code` (`system_code`),
  KEY `idx_deleted` (`deleted`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='插件接入应用';
```

- 防止用户连点、网络重试导致重复工单

## 十、Webhook 状态同步

### 10.1 按应用回调

扩展 `WebhookDispatchService`：

1. 工单 `integration_app_id` 非空时，除全局 `webhook_config` 外，额外向 `integration_app.callback_url` 推送
2. 请求头增加 `X-App-Key`、`X-Webhook-Signature`（HMAC-SHA256 of body + `callback_secret`）
3. payload 增加 `pluginContext` 摘要 + `externalTicketRef`

## 十一、ticket-sdk 技术设计

### 11.1 包结构

```
ticket-sdk/
├── src/
│   ├── index.ts          # 入口：init / setContext / open / destroy
│   ├── api.ts            # HTTP 客户端
│   ├── context.ts        # 上下文采集（env、page）
│   ├── ui/
│   │   ├── Modal.ts
│   │   ├── Sidebar.ts
│   │   └── FloatButton.ts
│   ├── form.ts           # 表单渲染（描述、优先级、附件）
│   └── events.ts         # EventEmitter
├── dist/
│   └── ticket-sdk.min.js # UMD，目标 < 30KB gzip
└── package.json          # @miduo/ticket-sdk
```

### 11.2 核心 API

| 方法 | 说明 |
|------|------|
| `TicketSDK.init(options)` | 初始化，挂载 UI；`options.autoReport` 可启用 HTTP 错误自动捕获（v1.1.0+） |
| `TicketSDK.setContext(ctx)` | 设置/合并业务上下文 |
| `TicketSDK.open(prefillDescription?)` | 手动打开提交面板；可传入预填描述（HTTP 自动捕获场景） |
| `TicketSDK.openMyTickets()` | 打开我的工单 |
| `TicketSDK.reportHttpError(error)` | 手动上报 HTTP 错误并打开弹窗（axios 或业务代码调用） |
| `TicketSDK.createAxiosInterceptor()` | 返回 axios 响应错误拦截器 |
| `TicketSDK.on(event, handler)` | 订阅事件 |
| `TicketSDK.destroy()` | 卸载 SDK |
| `TicketSDK.setUser(user)` | 用户切换时更新（SPA 登出登入） |

**init.autoReport 配置**（默认关闭，向后兼容）：

| 字段 | 类型 | 默认 | 说明 |
|------|------|------|------|
| `enabled` | `boolean` | `false` | 是否启用 fetch/XHR 全局监听 |
| `httpStatus` | `number[]` | `[403,500,502,503]` | 触发捕获的状态码 |
| `confirmBeforeSubmit` | `boolean` | `true` | 仅预填描述并打开弹窗，不静默建单 |
| `debounceMs` | `number` | `300000` | 同 URL+状态码防抖间隔（毫秒） |
| `apiPatterns` | `string[]` | — | 仅匹配 URL 含这些片段的请求 |

```javascript
// 方式一：init 时开启（推荐）
TicketSDK.init({
  // ...appKey、system、user 等
  autoReport: {
    enabled: true,
    httpStatus: [403, 500, 502, 503],
    confirmBeforeSubmit: true,
    debounceMs: 300000,
    apiPatterns: ['/api/'],
  },
})

// 方式二：axios 项目手动挂载
axios.interceptors.response.use(null, TicketSDK.createAxiosInterceptor())
```

### 13.1 分期建议

| 阶段 | 范围 | 交付 |
|------|------|------|
| **P0** | 后端接入应用 + 插件建单 API + LaunchToken | 可用 curl 建单，上下文入库 |

| **P2** | 我的工单 + 按应用 Webhook + 详情上下文面板 | 完整闭环 |

### 13.2 验收标准（整体）

- [ ] 业务系统引入 SDK 后，用户 30 秒内可完成提单，全程不跳页
- [ ] 工单详情可查看完整插件上下文，处理人无需追问用户信息/页面/单据号
- [ ] 状态变更 30 秒内推送到业务系统回调地址
- [ ] 同一 `externalTicketRef` 幂等，不重复建单
- [ ] AppSecret 不出现在前端代码/网络请求中

## 十四、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 外部用户未在工单系统开户 | 无法关联 `creatorId` | 按 `externalUserId` 自动创建影子用户 |
| CSP 阻止内联脚本 | SDK 无法加载 | 提供 iframe 降级方案 |
| 业务系统不传 launchToken | 前端无法建单 | 文档强调必须后端换票；提供联调工具 |
| 上下文字段膨胀 | 存储/展示混乱 | JSON Schema 校验 + 标准字段 + extra 扩展 |
| 多应用 AppKey 管理复杂 | 运维成本 | 管理端 CRUD + 密钥轮换 + 审计日志 |

## 十五、与传统方式对比总结

| 环节 | 传统方式 | 插件方式 |
|------|---------|---------|
| 找入口 | 跳系统 → 登录 → 找入口（2–3 分钟） | 当前页 1 秒 |
| 填信息 | 手动填 10+ 字段（3–5 分钟） | 只写描述（30 秒） |
| 上下文 | 处理人反复追问 | 自动携带，打开即见 |
| 看进度 | 跳工单系统查 | 业务系统内嵌「我的工单」 |
| 接入成本 | 对接多个 API + 自建表单 | SDK 十几行代码 |
