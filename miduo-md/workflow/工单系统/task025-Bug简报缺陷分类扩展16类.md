# Task025：Bug简报缺陷分类扩展为16类

> **业务模块**：工单系统（Bug简报）  
> **变更类型**：优化（不新增接口，调整字典口径）  
> **关联接口**：API000030 `GET /api/dict/defect-category`  
> **关联页面**：`/bug-report/edit/:id?`（编辑页缺陷分类下拉 + 右侧说明）

---

## 一、需求收敛（背景 / 目标 / 非目标）

### 1.1 背景

当前 Bug 简报“缺陷分类”仅有 7 个分类，无法覆盖测试与复盘中常见的问题类型，导致多人填写时口径不统一，统计分析粒度不足。

### 1.2 现状问题

1. 现有分类过粗，多个不同问题被迫归为同一类。  
2. 右侧分类说明与团队实际分类标准存在偏差。  
3. 统计看板“缺陷分类分布”无法准确反映问题结构。

### 1.3 目标

将缺陷分类统一升级为以下 16 类（按展示顺序）：

1. 页面异常  
2. 交互异常  
3. 设计问题  
4. 错误逻辑  
5. 错误代码  
6. 错误数据  
7. 错误配置  
8. 错误发布  
9. 基础服务  
10. 基础设施  
11. 兼容问题  
12. 架构问题  
13. 性能问题  
14. 安全问题  
15. 安装部署  
16. 第三方原因

### 1.4 非目标

1. 不修改 API 路径、请求参数、响应结构。  
2. 不修改 Bug 简报主流程状态机（创建/提交/审核/归档）。  
3. 不对历史简报数据做自动改写映射（保持原值可追溯）。

---

## 二、方案设计（影响范围 / 决策 / 兼容与回滚）

### 2.1 方案概要

通过数据库迁移脚本更新 `dict_defect_category` 字典表数据，前端继续复用同一接口 API000030 动态渲染下拉与说明，无需新增前端接口或改动页面交互。

### 2.2 影响范围

- 后端：Flyway 迁移脚本（字典数据从 7 类升级为 16 类）。  
- 前端：无需改业务逻辑，页面将自动显示新的 16 类。  
- 文档：产品 PRD 与任务文档同步更新分类口径。

### 2.3 关键决策

1. **保留 API000030 编号不变**：接口契约未变，仅字典内容扩展。  
2. **字典全量重建排序**：统一使用 `sort_order=1..16`，确保前端展示顺序可控。  
3. **软停用旧分类**：将旧字典项置为 `is_active=0`，避免脏数据干扰下拉。

### 2.4 兼容策略

`bug_report.defect_category` 为文本字段，历史值不依赖外键约束：  
- 历史简报仍可正常展示原分类文本；  
- 新建/编辑简报只能选择新 16 类，逐步完成口径切换。

### 2.5 回滚策略

若出现业务争议，可通过补充迁移脚本将 `dict_defect_category` 恢复为上一版本分类集合（7 类），不影响已有简报记录。

---

## 三、验收标准

- [ ] 编辑页“缺陷分类”下拉展示 16 项，且顺序与目标一致。  
- [ ] 编辑页右侧“缺陷分类说明”与下拉同源一致。  
- [ ] `GET /api/dict/defect-category` 返回 16 项有效分类（`is_active=1`）。  
- [ ] 历史简报详情页可正常显示原有分类文本，不报错。

---

## 四、测试要点（最小闭环）

1. 打开 Bug 简报编辑页，检查下拉分类数量与顺序。  
2. 对照右侧说明，确认名称与描述一致。  
3. 调用 API000030，确认返回项数量为 16。  
4. 随机抽查历史简报详情，确认旧分类可读。
