# 企微工单事件通知：待受理处理人与变更时间

## 背景与目标

企业微信群机器人「工单事件通知」在**工单状态变更**场景下，需更易读、可行动：
- 进入**待受理类**状态（待分派、待受理、待测试受理、待开发受理）时，应直接看到**当前处理人**。
- **变更内容**区块末尾补充**变更时间**，与事件时间对齐，避免仅顶部一行时间时扫读遗漏。

## 方案概要

| 项 | 说明 |
|----|------|
| 实现位置 | `ticket-platform` → `WebhookDispatchService#buildWecomRobotTextPayload` |
| 待受理判定 | `TicketStatus#isPendingAcceptanceLike()`（枚举层统一维护） |
| 当前处理人 | 工单快照 `assigneeId` → `SysUserMapper` 解析姓名；无则 `-` |
| 变更时间 | 存在「变更内容」时，在编号行之后追加 `变更时间：{eventTime}`，缺省用组装时刻 |

## 非目标

- 不改变非企微机器人 Webhook 的 JSON 负载结构。
- 不在无「变更内容」的事件中强行插入变更时间块。

## 验收

1. 将工单流转到「待开发受理」等待受理类状态，群消息「工单信息」含 **5) 当前处理人：xxx**。
2. 同一条消息「变更内容」最后有一行 **变更时间**，与顶部「【时间】」一致。

## 分派类型中文展示（assignType）

**问题**：工单分派 Webhook 中「分派类型」曾直接输出英文码（如 `ACCEPT_CLAIM`），可读性差。

**方案**：在 `ticket-common` 中维护 `TicketAssignType` 枚举（码与中文含义）；`WebhookDispatchService#resolveAssignTypeLabel` 优先按该枚举解析；自动分派策略码仍走 `DispatchStrategy`。

| 码 | 展示文案 |
|----|----------|
| CREATE_ASSIGN | 创建时指派 |
| MANUAL_ASSIGN | 手动指派 |
| TRANSFER_ON_TRANSIT | 流转时转派 |
| TRANSFER | 转派 |
| ACCEPT_CLAIM | 流转认领 |

**验收**：群消息「变更内容」中「分派类型」为上述中文，不再出现裸英文码（未知码仍可原样回退便于排查）。

## 追加处理人通知修复（指派给 + @追加人）

### 背景与现状问题

在“工单详情 → 追加处理人”场景，企业微信群机器人通知出现两个体验问题：

1. 「变更内容」里的 **指派给** 仍显示旧主处理人，无法直观看到本次追加的人；
2. 消息没有真正 @ 到新增处理人，导致新增协同人收不到及时提醒。

### 目标

- **目标1**：分派类通知中的「指派给」展示工单当前处理人全集（主处理人 + 协同处理人）。
- **目标2**：群消息 @ 名单包含本次追加后的全部当前处理人，确保新增处理人被提醒到。
- **目标3**：不改动外部接口契约，不影响非企业微信机器人地址的普通 Webhook 负载。

### 非目标

- 不调整“谁有权限追加处理人”的业务规则；
- 不改变工单分派事件类型与事件名；
- 不新增前端页面交互，仅修复通知内容与@命中。

### 方案概要

| 项 | 说明 |
|---|---|
| 实现位置 | `ticket-platform/ticket-application/.../WebhookDispatchService` |
| 关键数据源 | `ticket_assignee` 当前有效处理人列表（按 `sort_order`） |
| 指派给展示 | 分派通知优先展示“当前处理人列表姓名”，无列表时回退单人 `assigneeId` |
| @名单收集 | 在原“创建人/主处理人/操作人/原处理人”基础上，补充“当前处理人列表” |
| 兼容策略 | 保留原有 `assigneeId` 解析链路作为兜底，避免历史数据缺字段导致空展示 |

### 影响范围

- 企业微信机器人文本消息排版与@名单组装（`msgtype=text`）；
- 工单分派事件（`TICKET_ASSIGNED`）为主要收益场景，其他事件仅增强“当前处理人解析能力”。

### 风险与回滚

- 风险：若 `ticket_assignee` 数据异常（空或脏数据），可能导致“指派给”回退到单人显示；
- 回滚：可回退 `WebhookDispatchService` 本次改动，恢复只按 `assigneeId` 显示与@。

### 验收标准

1. 先有处理人A，再追加处理人B后触发工单分派通知：
   - 「变更内容」中的 **指派给** 应包含 B（建议显示 `A、B`）；
   - 通知应真正 @ 到 B。
2. 仅单处理人场景通知行为与历史一致（无回归）。
3. 非企微机器人 Webhook 配置保持原始 JSON 推送，不受影响。
