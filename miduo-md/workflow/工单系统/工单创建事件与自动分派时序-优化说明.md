# 工单创建事件与自动分派时序 — 优化说明

**日期**：2026-04-01  
**类型**：体验改进 / 集成一致性（不改变 Webhook 事件类型与路径，仅调整推送时机与负载语义）

## 背景与现状

- 创建工单时发布 `TicketCreatedEvent`；未指定处理人时另发布 `TicketAutoDispatchEvent`，在事务提交后由 `TicketAutoDispatchListener` 调用 `DispatchAppService.autoDispatch`（独立事务 `REQUIRES_NEW`）完成自动分派。
- `TicketWebhookEventListener`、`TicketEventNotificationListener` 对 `TicketCreatedEvent` 使用 `@Async` + `AFTER_COMMIT`，与同步的自动分派监听**无先后顺序保证**。
- 结果：Webhook / 站内通知在构建负载时读取 `ticket` 快照，可能仍看到**待分派**，而自动分派已随后把状态与处理人更新，造成集成方与群内消息**与真实状态不一致**。

## 目标

- 对「创建后还会触发自动分派」的工单：**创建类对外推送**（Webhook `TICKET_CREATED`、站内「工单创建」通知、企微群推送）在自动分派尝试**结束之后**再发送，负载中的状态与处理人与库表一致。
- 创建时已指定处理人、或不触发自动分派的场景：行为与现网一致，仍为提交后立即推送。

## 非目标

- 不新增/变更 HTTP 接口与 Webhook 事件枚举值。
- 不改变 `TICKET_STATUS_CHANGED` / `TICKET_ASSIGNED` 等后续事件的语义（自动分派仍会发布既有领域事件）。

## 设计方案

1. **`TicketCreatedEvent` 增加 `pendingAutoDispatch`**：为 `true` 时表示将由自动分派链路负责「延后」的创建类推送；Webhook 与通知监听器在该标志为 `true` 时**跳过**。
2. **新增 `TicketCreatedAfterAutoDispatchEvent`**：在 `TicketAutoDispatchListener` 中，`autoDispatch` 执行完毕（成功、跳过或异常捕获后）于 `finally` 中发布，携带 `ticketId`、`categoryId`、`priority`（提交后从库读取），供原创建推送逻辑使用。
3. **监听器**：对 `TicketCreatedAfterAutoDispatchEvent` 复用与 `TicketCreatedEvent` 相同的 Webhook/通知构建路径（事件类型仍为 `TICKET_CREATED` / `NotificationType.TICKET_CREATED`）。

## 验收标准

- 无自动分派（已指定处理人）：创建后仍立即收到创建类 Webhook/通知，内容与库一致。
- 有自动分派且分派成功：创建类 Webhook/通知中工单状态与处理人**不为**仅「待分派且无处理人」的过时快照（与自动分派提交后一致）。
- 有自动分派但规则未命中或失败：仍收到创建类推送，内容与当前库一致（可为待分派）。
