# Task027：Bug简报缺陷分类16类口径精确对齐

> **业务模块**：工单系统（Bug简报）  
> **变更类型**：优化（不新增接口，不改变接口结构）  
> **关联接口**：API000030 `GET /api/dict/defect-category`  
> **关联页面**：`/bug-report/edit/:id?`（缺陷分类下拉 + 右侧缺陷分类说明）

---

## 一、需求收敛（背景 / 目标 / 非目标）

### 1.1 背景

当前“缺陷分类”虽然已扩展到 16 类，但部分分类描述文案与业务口径存在细微偏差，导致填写人员理解不一致，分类结果稳定性不足。

### 1.2 现状问题

1. 同一分类在不同文档/SQL脚本中的描述文本不完全一致。  
2. 页面下拉与右侧说明虽然同源，但“源数据文案”存在历史版本差异。  
3. 复盘统计时，容易出现“同名分类，不同理解”的偏差。

### 1.3 目标

将 Bug 简报缺陷分类统一为以下 16 类，并使用同一版标准说明（顺序固定）：

1. 页面异常：由于页面排版不合理、样式、文案错误，页面显示重叠等导致的页面性错误。  
2. 交互异常：由于前端用户交互体验、接口调用导致的错误。  
3. 设计问题：逻辑删除无“确认窗”、用户体验感低、注释描述有歧义或存在错别字、相关提示语缺失、光标位置不正确、文字排列不整齐，相关功能场景未考虑等。  
4. 错误逻辑：由于业务逻辑、代码逻辑异常导致的错误。  
5. 错误代码：逻辑正确的情况下由于编码问题导致的错误。  
6. 错误数据：由于数据与业务不兼容导致的错误。  
7. 错误配置：由于配置文件异常导致（参数配置错误等）的错误。  
8. 错误发布：由于灰度或线上发布流程异常导致的错误。  
9. 基础服务：由于基础组件、服务网关、任务调度、ES 搜索引擎等服务故障导致的错误。  
10. 基础设施：数据库相关（如数据库配置、数据库定时作业运行故障、数据库脚本操作、数据库高可用性架构等）、缓存 Redis 相关、Nginx 反向代理、web 负载均衡等。  
11. 兼容问题：由于浏览器（不同的浏览器、或者浏览器版本太低）、移动设备或数据库版本不兼容导致的错误。  
12. 架构问题：由于架构不适用导致的错误，如无法支持大数据量存储、系统扩展性差等。  
13. 性能问题：并发产生的问题、页面响应时间长、服务器资源耗用过高、死锁、内存泄露、难以满足系统性能要求等。  
14. 安全问题：账号越权、sql 注入漏洞、频率未做限制（比如短信发送次数，登录密码次数未做限制）等。  
15. 安装部署：服务器初始化（操作系统安装，基础环境支持）、软件安装（比如数据库管理系统，ES 程序等）、编译后代码部署等产生的问题。  
16. 第三方原因：由于第三方服务异常导致的错误。如：腾讯云服务器网络、七牛云存储、大禹高防及 SSL 证书等。

### 1.4 非目标

1. 不新增接口，不修改 API000030 的路径、方法、返回结构。  
2. 不改 Bug 简报状态流转和提交流程。  
3. 不自动改写历史 Bug 简报中已保存的旧分类文本。

---

## 二、方案设计（影响范围 / 决策 / 回滚）

### 2.1 方案概要

通过数据库迁移脚本统一 `dict_defect_category` 的 16 类分类名称与描述文案；前端继续使用 API000030 动态加载，自动同步下拉与右侧说明。

### 2.2 影响范围

- 后端：Flyway 新增迁移脚本，统一字典数据文案。  
- 前端：无需改接口调用逻辑，页面展示自动生效。  
- 文档：流程文档与 readme 同步更新口径，避免后续漂移。

### 2.3 关键决策

1. 使用“停用旧字典 + 清理同名 + 重建16类”策略，确保数据最终一致。  
2. 保持 `sort_order = 1..16`，保证页面显示顺序稳定。  
3. 接口编号 API000030 保持不变，保证前后端契约兼容。

### 2.4 回滚策略

如需回滚，可新增补丁迁移脚本恢复上一版描述文本，不影响历史简报记录读取。

---

## 三、验收标准

- [ ] 缺陷分类下拉框可见且仅可选 16 项，顺序与目标一致。  
- [ ] 右侧“缺陷分类说明”与下拉名称、描述一一对应。  
- [ ] `GET /api/dict/defect-category` 返回 16 条启用数据（`is_active=1`）。  
- [ ] 历史简报详情可正常显示历史分类文本。

---

## 四、测试要点

1. 打开 Bug 简报编辑页，检查“缺陷分类”下拉项数量和顺序。  
2. 对照右侧说明，核对 16 项描述文本是否与本任务文档一致。  
3. 调用 API000030，验证返回条数与排序。  
4. 抽查历史简报详情，确认可正常展示。
