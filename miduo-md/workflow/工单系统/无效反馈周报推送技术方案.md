# 无效反馈周报推送技术方案

## 1. 方案概要

在现有“日报管理”能力基础上，新增一条“无效反馈周报”链路，复用：

1. system_config 配置模型；
2. 企微 webhook 发送器；
3. 定时任务轮询 + Redis 分钟级去重机制。

---

## 2. 影响范围

## 2.1 后端

- `ticket-application`：新增周报应用服务；

- 新增周报 API 封装和类型定义；

## 2.3 文档

- 新增需求文档与技术方案文档；

## 3. 关键设计决策

1. **口径复用现有字段**：无效反馈直接使用 `manual_valid_report = NO`，不新增判定字段；
2. **统计按周窗口**：本周一 00:00:00 到当前推送时间；
3. **反馈人口径固定**：使用 `ticket.creator_id` 映射用户姓名；
4. **推送链路复用**：沿用日报的 webhook 与去重机制，降低新功能风险；
5. **接口独立命名空间**：新建 `/api/weekly-invalid-report/*`，避免和日报接口耦合。

---

## 4. 数据与SQL设计

新增三类查询：

1. `selectWeeklyInvalidFeedbackTotal`：统计总数；
2. `selectWeeklyInvalidFeedbackByReporter`：按反馈人统计数量；
3. `selectWeeklyInvalidFeedbackDetail`：查询明细列表（按关闭时间倒序）。

查询条件统一包含：

- `ticket.deleted = 0`；
- `ticket_bug_info.deleted = 0`；
- `ticket.status = 'closed'`；
- `ticket_bug_info.manual_valid_report = 'NO'`；
- 关闭时间在周范围内；
- 分类 ID 过滤（可选）。

---

## 5. 配置与调度

新增配置键（`WEEKLY_INVALID_REPORT` 分组）：

- `weekly_invalid_report_enabled`
- `weekly_invalid_report_cron`
- `weekly_invalid_report_webhook_urls`
- `weekly_invalid_report_stat_category_ids`
- `weekly_invalid_report_max_detail_count`
- `weekly_invalid_report_timezone`

调度策略：

- 每 30 秒轮询配置 cron；
- 命中后抢 Redis 分钟桶去重锁；
- 抢锁成功才执行推送。

## 7. 风险与回滚

## 7.1 风险

1. 明细过长触发企微单条消息长度限制；
2. 分类 ID 配置错误导致“统计为 0”；
3. 多实例部署导致重复推送风险。

## 7.2 缓解措施

1. 增加 `max_detail_count` 与消息长度保护；
2. 配置页明确提示“按分类 ID 生效”；
3. 使用 Redis 分钟去重锁防重。

## 7.3 回滚策略

1. 配置层回滚：关闭 `weekly_invalid_report_enabled`；
2. 代码层回滚：下线周报 job 与 controller，保留配置和 migration 不影响存量功能。
