# Task028：CD部署触发与Flyway版本冲突修复

> **业务模块**：工单系统（发布链路 / 数据库迁移）  
> **变更类型**：优化（不新增业务接口，不改变对外API契约）  
> **关联现象**：代码已合并、CD显示成功，但线上页面仍是旧内容

---

## 一、需求收敛（背景 / 目标 / 非目标）

### 1.1 背景

近期出现“PR合并后 CD 成功，但线上修改不生效”的问题。经排查，发布链路存在“构建完成但部署触发不稳定”的风险点，同时后端迁移目录存在 Flyway 版本号重复，可能导致新镜像启动迁移失败，最终线上仍跑旧实例。

### 1.2 现状问题

1. `cd.yml` 在更新版本仓库时，`testVersion` 写入逻辑被注释，部署触发链路存在断点。  
2. Flyway 迁移脚本出现重复版本号（`V44`、`V46` 各有两份不同脚本），新环境或新镜像启动时可能报冲突。  
3. 运维侧看到“构建成功”，但业务侧看到“页面不生效”，形成认知差。

### 1.3 目标

1. 恢复并加固 CD 的部署触发版本写入，确保“构建成功→可触发部署更新”。  
2. 消除 Flyway 重复版本号冲突，保证后端新版本可稳定启动并执行迁移。  
3. 保持业务接口、页面交互、数据模型对外契约不变。

### 1.4 非目标

1. 不新增或变更业务 API。  
2. 不调整工单/Bug简报业务流程。  
3. 不改动线上运维平台的非本仓库配置。

---

## 二、方案设计（影响范围 / 决策 / 回滚）

### 2.1 方案概要

1. 在 `.github/workflows/cd.yml` 中恢复 `testVersion` 更新逻辑，并做“文件存在+键存在/不存在”兜底处理。  
2. 将冲突的迁移脚本改为唯一递增版本：  
   - `V46__expand_bugreport_defect_category_to_16.sql` → `V48__expand_bugreport_defect_category_to_16.sql`  
   - `V47__align_bugreport_defect_category_wording.sql` → `V49__align_bugreport_defect_category_wording.sql`  
   - `V44__init_sso_session.sql` → `V50__init_sso_session.sql`

### 2.2 影响范围

- CI/CD：版本仓库写入行为（部署触发链路）  
- 后端：Flyway 迁移装载顺序  
- 文档：readme 与流程任务文档中的迁移编号说明

### 2.3 关键决策

1. 保留历史已发布迁移的编号（如用户API密钥统计的 `V46`），避免已上线环境重复执行同一DDL。  
2. 将 Bug 简报相关迁移后移到 `V48/V49`，并将 SSO 会话迁移后移到 `V50`，确保版本全局唯一且顺序正确。  
3. `testVersion` 写入采用“存在则替换、不存在则追加”，降低环境差异导致的失败概率。

### 2.4 回滚策略

1. CI/CD 回滚：恢复 `cd.yml` 到上一个提交。  
2. Flyway 回滚：保持数据库已执行迁移不回退，仅通过新增补丁迁移修正。  
3. 若部署触发异常，可手动执行部署脚本并锁定镜像 tag。

---

## 三、验收标准

- [ ] `CD – Build & Push Docker Images` 完成后，版本仓库中的 `testVersion` 与本次 `sha-xxxxxxx` 一致。  
- [ ] 后端启动不再出现 Flyway duplicate version 报错。  
- [ ] Bug简报缺陷分类字典迁移在新环境可按 `V48/V49` 正常执行。  
- [ ] 线上环境切换后，页面可见本次改动内容。

---

## 四、测试要点

1. 检查 CD 日志中 `web_trigger.yml` 与 `.cnb_version.yml` 是否写入并提交。  
2. 检查后端启动日志 Flyway 执行顺序，确认无重复版本报错。  
3. 校验 `dict_defect_category` 16类数据是否正确。  
4. 前端强刷后验证页面显示口径是否生效。
