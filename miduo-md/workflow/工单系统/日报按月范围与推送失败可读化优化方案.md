# 日报按月范围与推送失败可读化优化方案

## 1. 方案概要

本次优化聚焦“**看准范围**”和“**看懂失败**”两件事：

1. 将日报统计口径从历史全量收敛为“当前月份”；
2. 将手动推送失败原因做可读化映射，避免把技术错误原文直接抛给用户。

## 2. 影响范围

### 2.1 后端

- `ticket-platform/ticket-application/.../DailyReportApplicationService.java`
  - 新增“当月起止时间”计算；
  - 日报汇总与明细查询改为按当月时间范围统计；
  - 推送失败原因增加语义化转换（重点处理企微 4096 长度超限）。
- `ticket-platform/ticket-infrastructure/.../DailyReportMapper.java`
  - 相关明细/汇总查询方法补充日期范围参数。
- `ticket-platform/ticket-infrastructure/.../DailyReportMapper.xml`
  - 对待解决、临时解决、挂起、总反馈等查询增加当月 `create_time` 区间过滤。

### 2.2 前端

- `miduo-frontend/src/views/manage/DailyReportView.vue`
  - 手动推送失败时展示后端返回的可读原因，避免固定“推送失败”笼统提示。

### 2.3 文档

- `readme.md`
  - 追加本次优化说明、验收方式、常见问题与版本记录。

## 3. 关键决策

1. **口径优先统一**：日报标题和内容使用同一“当月”窗口，避免跨月数据误读。
2. **失败信息分层**：日志保留技术细节，接口返回面向业务语言，兼顾排障与可用性。
3. **保持接口兼容**：不改 API 出参结构，减少前后端联调风险。

## 4. 风险与回滚

### 4.1 风险

1. 业务方可能看到“待解决/挂起”数量下降（因为跨月历史单被排除）。
2. 个别异常信息被语义化后，技术细节在前端不再完整展示（需从日志追踪）。

### 4.2 回滚策略

1. 若业务确认需恢复历史全量口径，可回滚到改造前查询条件；
2. 推送失败可读化如出现误判，可回滚到原始错误透出策略。

## 5. 验收步骤

1. 进入“日报管理 -> 日报预览”，确认仅出现当前月工单。
2. 构造超长内容触发手动推送失败，确认提示文案可读、可执行。
3. 配置错误 Webhook 再次推送，确认失败原因简洁清晰。
