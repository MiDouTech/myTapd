## 一、执行摘要

我们正在为工单系统**企业微信集成**模块构建**图片消息自动关联工单**能力。当前企微回调仅处理 `text` 类型消息，大量用户随消息附上的截图被直接丢弃，导致工单信息不完整，研发复现问题困难。

**目标**：支持企微群和私聊场景下，用户发送的图片消息（截图）自动持久化并关联到对应工单，实现「文字 + 截图」一体化建单体验。

**核心方案**：新增 `wecom_pending_image` 暂存表，通过「5 分钟时间窗口 + 同用户 + 同群」的关联策略，兼容「先文后图」和「先图后文」两种常见习惯；图片经企微 API 下载后上传到七牛云持久化存储；超时未关联图片按可配置策略处理。

**预期影响**：
- 缺陷工单信息完整率提升（附图比例从 0 → 目标 60%+）
- 用户「二次补传截图」操作减少 80%
- 工单系统回调接入图片消息能力，为后续 OCR 辅助建单打下基础

## 二、问题陈述

### 谁有这个问题？

**客服/产品经理、研发/测试人员**，在通过企业微信群或私聊向工单助手提交问题时，习惯同时发送截图辅助描述。

| 角色 | 使用入口 | 图片诉求 |
|------|----------|----------|
| 客服/产品 | 企微群 @工单助手 | 发送客户问题截图，辅助建单 |
| 研发/测试 | 企微群 @工单助手 | 发送 Bug 复现步骤截图、报错界面 |
| 任意员工 | 企微私聊工单助手 | 发送申请/问题相关图片 |

### 问题是什么？

当前系统在 `WecomCallbackApplicationService` 中对消息类型做了硬性过滤：只处理 `msgType == "text"` 的消息，`image` 类型消息被直接丢弃。用户发送的截图无法被采集、存储或关联到工单。

此外，企微消息是**逐条独立回调**，同一用户「发文字建单」和「发截图」是两次独立的 Webhook 事件，系统没有任何机制将它们关联在一起。

### 为什么痛？

- **用户体验割裂**：用户发了截图，但工单里没有图片；只能到 WEB 端再次上传，操作繁琐
- **缺陷复现困难**：研发看不到错误截图，沟通成本高，修复周期延长
- **信息遗漏不可感知**：用户不知道截图被丢弃，误以为工单附带了图片，造成信息误解

### 证据

- 产品方案文档中明确指出：`msgType != "text"` 即丢弃（第一章第 1.1 节）
- 业务场景分析表明，缺陷工单中「截图」是高频刚需，用户发送「文字 + 截图」是常见习惯
- 现有代码中 `WecomCallbackApplicationService` 存在硬性过滤逻辑，不处理非文本消息

## 四、战略背景

### 4.1 业务目标对齐

本功能对应以下核心业务目标：

1. **提升工单信息完整率**：工单附图功能完善，降低研发与客服之间「截图找回」的沟通成本
2. **扩展企微集成能力**：在 Task023（自然语言建单）基础上，进一步提升企微作为建单渠道的能力
3. **为未来 OCR 建单铺垫**：持久化图片 URL 是后续「截图识别文字辅助建单」的前提基础

### 4.2 竞争背景

业界主流工单系统（Jira、飞书项目）均已支持 IM 侧截图直接进入工单。本项目通过时间窗口关联策略，可在不改变企微消息逐条回调特性的前提下，实现近似的截图自动关联体验。

### 4.3 为什么是现在？

- Task023（企微文本消息自动创建工单）完成后，企微建单能力大幅提升，用户在企微侧的建单频次上升，截图关联需求更加迫切
- 七牛云存储能力已具备，无需引入新的存储依赖
- 企微图片 MediaId API 已在 Task006 中有调用基础，改造成本低

## 五、解决方案概述

### 5.1 核心设计

系统新增**图片消息处理分支**，在企微回调入口处对 `MsgType=image` 的消息进行拦截处理，而非丢弃：

```
企微回调 (POST /api/wecom/callback)
         │
         ├── MsgType = text
         │     └── [现有流程] 解析 → 创建工单 → 触发图片关联检查
         │
         └── MsgType = image  ← 新增处理分支
               │
               ├── 异步下载图片（MediaId API，PicUrl 降级）
               ├── 上传到七牛云持久化
               ├── 暂存到 wecom_pending_image（status=PENDING）
               └── 尝试关联到近期工单（时间窗口）
                     ├── 找到 → 直接关联，status=LINKED
                     └── 未找到 → 等待后续文字消息触发关联
```

### 5.2 三大核心场景

**场景一：先描述后发图（推荐流程）**

```
T+0s  用户：@工单助手 #缺陷 登录页面崩溃
T+3s  用户：[发送截图1]
T+5s  用户：[发送截图2]

工单助手回复：
  ✅ 工单已创建：WO-20260313-001
  📋 标题：登录页面崩溃
  🖼️ 图片附件：2张（已上传）
  👉 处理人：张三 | 优先级：高
```

**场景二：先发图后描述（常见习惯）**

```
T+0s  用户：[发送截图]
T+10s 用户：@工单助手 #缺陷 上面是错误弹窗，商户10004506反馈的

工单助手回复：
  ✅ 工单已创建：WO-20260313-002
  📋 标题：错误弹窗（商户10004506反馈）
  🖼️ 图片附件：1张（已上传）
```

**场景三：仅发图，超时未关联**

```
T+0s  用户：[仅发送截图，未发文字]
（5分钟内无文字建单）

默认策略（CREATE_TICKET）：
  静默创建「仅有图片」的工单，标题待补充
```

### 5.3 时间窗口关联规则

| 规则 | 触发条件 | 处理逻辑 |
|------|----------|----------|
| **规则A：先文后图** | 图片发送时间在工单创建时间 [0, +5分钟] 内，且同一 chatId + 同一 fromUserId | 自动关联到该工单 |
| **规则B：先图后文** | 文字消息触发工单创建时，该用户 5 分钟内有 PENDING 图片 | 批量关联所有 PENDING 图片 |
| **规则C：超时未关联** | PENDING 状态超过 5 分钟 | 定时任务按策略处理 |

### 5.4 关键技术决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| 图片下载方式 | 优先 MediaId API，降级 PicUrl HTTP | MediaId 更稳定，PicUrl 是临时外链 |
| 图片存储 | 七牛云（复用现有能力） | 无需引入新依赖 |
| 下载方式 | `@Async` 异步 | 不阻塞主回调流程，避免 5 秒响应超时 |
| 关联策略 | 时间窗口 + 同用户 + 同群 | 兼容先文后图 / 先图后文两种习惯 |
| 超时策略 | 默认静默创建工单 | 比丢弃图片用户体验更好 |

## 七、用户故事与需求

### 7.1 Epic 假设

我们相信，通过新增企微图片消息处理与时间窗口自动关联机制，将使缺陷工单信息完整率从 0% 提升至 60%+，因为当前用户发送截图的习惯已经存在，只是系统没有接收。我们将在上线后 30 天通过「图片关联成功率」和「工单附图比例」两项指标来验证。

### 7.2 用户故事

---

**Story 1：处理企微图片消息，异步下载并持久化**

作为系统，当收到企微回调的 `MsgType=image` 消息时，我需要异步下载图片并存储到七牛云，以确保图片不因企微临时 URL 过期而丢失。

**验收标准**：
- [ ] 收到 `MsgType=image` 回调时，系统不丢弃消息，触发异步下载
- [ ] 优先通过企微 MediaId API 下载图片，失败时降级 PicUrl HTTP 下载
- [ ] 下载成功后上传到七牛云，获得持久化 `qiniu_url`
- [ ] 写入 `wecom_pending_image` 表，状态为 `PENDING`，包含 `chat_id`、`from_user_id`、`msg_id`、`qiniu_url`、`expire_time`
- [ ] 下载全部失败时标记为 `FAILED`，记录日志，主回调流程正常返回
- [ ] 主回调响应时间不因图片下载而延迟（异步执行）

---

**Story 2：先文后图 — 图片自动关联到已创建工单**

作为用户，当我先发文字建单、再发截图时，截图可自动关联到刚创建的工单。

**验收标准**：
- [ ] 图片消息收到时，检查同一 `chatId` + 同一 `fromUserId` 下，5 分钟内是否有已创建的工单
- [ ] 找到匹配工单时，自动创建 `ticket_attachment` 记录，图片 `source=WECOM_BOT`，工单 `pending_image` 状态更新为 `LINKED`
- [ ] 已关联的图片数量超过 `max-images-per-ticket` 时，超出部分不关联，记录日志
- [ ] 成功关联后，企微回调无须单独回复用户（静默处理）

---

**Story 3：先图后文 — 文字建单时自动拉取暂存图片**

作为用户，当我先发截图、再发文字建单时，工单创建后可自动关联之前的截图。

**验收标准**：
- [ ] 文字消息触发工单创建成功后，立即查询同一 `chatId` + 同一 `fromUserId` 下，5 分钟内状态为 `PENDING` 的所有暂存图片
- [ ] 找到暂存图片时，批量创建 `ticket_attachment` 记录，批量将图片状态更新为 `LINKED`
- [ ] 在工单创建成功回复中，追加图片数量信息（`🖼️ 图片附件：N张（已上传）`）
- [ ] 无暂存图片时，回复内容不变，不展示图片信息

---

**Story 4：超时未关联图片的策略处理**

作为系统管理员，我希望图片在超过时间窗口后有明确的处理策略，避免 `PENDING` 状态永久积压。

**验收标准**：
- [ ] 定时任务每分钟扫描 `wecom_pending_image` 中超过 `expire_time` 且状态为 `PENDING` 的记录
- [ ] 按 `timeout-strategy` 配置执行对应策略：
  - `CREATE_TICKET`（默认）：静默创建「仅有图片」的工单，标题为「图片工单（待补充描述）」，状态更新为 `LINKED`
  - `EXPIRE`：状态更新为 `EXPIRED`，不创建工单
  - `NOTIFY_USER`：回复用户「图片已接收，请补充文字描述」，状态保持 `PENDING`（重置 expire_time 等待一次）
- [ ] 定时任务幂等，同一图片不重复处理

---

**Story 5：仅发图片时的即时反馈（可选）**

作为用户，当我仅发了截图但未附文字时，希望收到提示，告知截图已被接收并可关联。

**验收标准**：
- [ ] 配置 `notify-on-pending=true` 时，图片收到后立即回复：「📷 图片已接收，请在5分钟内补充文字描述，图片将自动关联到新工单。」
- [ ] 配置 `notify-on-pending=false`（默认）时，静默处理，不回复任何消息
- [ ] 同一用户连续发多张图时，仅发一次提示（基于 `chat_id + from_user_id` 去重，5分钟内不重复提示）

---

**Story 6：工单详情展示企微来源图片附件**

作为工单处理人，在查看工单详情时，我可以看到来自企微的图片附件，区分于 Web 端上传的附件。

**验收标准**：
- [ ] 工单详情页 `ticket_attachment` 列表中展示所有附件，包括企微来源图片
- [ ] 企微来源图片显示来源标识「企微」（`source=WECOM_BOT`）
- [ ] WEB 端上传图片显示来源标识「Web」（`source=WEB`）
- [ ] 图片支持预览（点击放大）和下载
- [ ] 附件按上传时间升序排列

---

**Story 7：Nacos 配置化支持（系统管理员）**

作为系统管理员，我可以通过 Nacos 配置图片消息处理的关键参数，无需重新部署。

**验收标准**：
- [ ] `wecom.image.enabled`（默认 `true`）：关闭后不处理任何 image 类型消息，直接丢弃
- [ ] `wecom.image.association-window-minutes`（默认 `5`）：时间窗口可配置，范围 1-60 分钟
- [ ] `wecom.image.timeout-strategy`（默认 `CREATE_TICKET`）：超时策略三选一
- [ ] `wecom.image.notify-on-pending`（默认 `false`）：收到图片时是否立即回复
- [ ] `wecom.image.max-images-per-ticket`（默认 `10`）：单工单最大图片数
- [ ] 配置热更新后立即生效，无需重启服务

### 7.3 边界条件与约束

| 场景 | 处理方式 |
|------|----------|
| MediaId 下载失败 | 降级 PicUrl 下载；均失败标记 `FAILED`，工单正常创建但无图片 |
| 七牛云上传失败 | 标记 `FAILED`，记录日志；不影响工单创建流程 |
| 单工单图片数超上限 | 超出部分不关联，记录日志，已关联的正常显示 |
| `msg_id` 重复回调 | 基于 `msg_id` 唯一索引去重，幂等处理 |
| 同一用户同一群发多张图 | 按时间顺序分别暂存，关联时批量关联 |
| 图片文件过大 | 遵循七牛云当前限制，超大文件不上传，标记 `FAILED` |
| 企微回调超时（5秒） | 图片下载通过 `@Async` 异步执行，主流程快速返回 200 |

## 九、数据设计

### 9.1 新增表：wecom_pending_image

```sql
CREATE TABLE `wecom_pending_image` (
  `id`           BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `chat_id`      VARCHAR(100) NOT NULL COMMENT '企微群ID（私聊时为用户ID）',
  `from_user_id` VARCHAR(100) NOT NULL COMMENT '发送人企微UserID',
  `msg_id`       VARCHAR(100) NOT NULL COMMENT '企微消息ID（去重）',
  `media_id`     VARCHAR(200) DEFAULT NULL COMMENT '企微MediaId（用于API下载）',
  `pic_url`      VARCHAR(500) DEFAULT NULL COMMENT '企微图片临时预览URL',
  `qiniu_url`    VARCHAR(500) DEFAULT NULL COMMENT '七牛云持久化URL',
  `ticket_id`    BIGINT DEFAULT NULL COMMENT '关联工单ID（关联后填入）',
  `status`       VARCHAR(20) NOT NULL DEFAULT 'PENDING' COMMENT '状态：PENDING/LINKED/EXPIRED/FAILED',
  `expire_time`  DATETIME NOT NULL COMMENT '过期时间（create_time + window_minutes）',
  `create_time`  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `update_time`  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `create_by`    VARCHAR(50) NOT NULL DEFAULT '' COMMENT '创建人',
  `update_by`    VARCHAR(50) NOT NULL DEFAULT '' COMMENT '更新人',
  `deleted`      TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除标志（0:未删除 1:已删除）',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_msg_id` (`msg_id`),
  KEY `idx_chat_user_status` (`chat_id`, `from_user_id`, `status`),
  KEY `idx_expire_time` (`expire_time`),
  KEY `idx_ticket_id` (`ticket_id`),
  KEY `idx_create_time` (`create_time`),
  KEY `idx_deleted` (`deleted`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='企微图片消息暂存表';
```

## 十三、异常处理

| 异常场景 | 处理方式 | 用户感知 |
|----------|----------|----------|
| 企微 MediaId 下载失败 | 降级 PicUrl，若均失败则标记 FAILED | 工单正常创建，该图片未关联 |
| 七牛云上传失败 | 标记 FAILED，记录日志，不阻塞主流程 | 同上 |
| 时间窗口内无工单可关联 | 保持 PENDING，等待后续文字建单 | 可选：收到「图片已接收」提示 |
| 单工单图片数超限 | 超出部分不关联，已接收的正常显示 | 仅前 N 张关联 |
| 企微回调响应超时 | 图片下载 @Async 异步，主流程快速返回 | 用户无感知，图片稍晚关联 |
| msg_id 重复（重试回调） | 唯一索引保护，幂等处理，不创建重复记录 | 无感知 |
