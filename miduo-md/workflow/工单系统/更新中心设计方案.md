# 动态设计方案（原更新中心）

> 业务模块：工单系统 / 动态
> 版本：v1.0  
> 日期：2026-06-14  
> 关联任务：Task061

---

## 1. 背景

`prd_agent` 已有“更新中心”，可以把仓库里的更新碎片、正式版本日志和 Git 提交记录集中展示。工单系统对用户展示为“动态”。
工单系统当前缺少同类入口，管理员想看“最近改了什么”时，只能去翻提交记录或问开发同学。

可以把本功能理解为“工单系统的更新公告栏”：  
每次代码变更像一张小便签，更新中心负责把便签按日期、版本和提交记录整理好。

---

## 2. 现状问题

1. 管理菜单下没有“更新中心”入口。
2. 仓库没有统一的 `CHANGELOG.md` 和 `changelogs/` 更新碎片规范。
3. 前端无法直接查看待发布变更、已发布版本和 GitHub 提交记录。
4. `prd_agent` 的周报页依赖 DocumentStore；工单系统当前使用仓库 `doc/report.*.md` 作为周报来源。
5. `prd_agent` 的 AI 大事依赖 AI 网关，工单系统暂未具备同等底座。

---

## 3. 目标与非目标

### 3.1 目标

1. 在工单系统左侧一级菜单新增“动态”，位置位于“管理”下方。
2. 提供“已发布、待发布、周报、GitHub提交”四个核心视图。
3. 后端读取仓库根目录 `CHANGELOG.md`、`changelogs/*.md` 和本地 Git 日志。
4. 前端按 `prd_agent` 更新中心的核心逻辑实现筛选、刷新、统计、周报列表和 Markdown 正文展示。
5. 新增接口编号并保持前后端注释与文档一致。

### 3.2 非目标

1. 本次不接入 AI 总结；原因是工单系统没有 `prd_agent` 同款 AI 网关。
2. 本次不接入周报来源配置；原因是工单系统没有 DocumentStore 文档库，先固定读取仓库 `doc/report.*.md`。
3. 本次不做 SSE 实时推送；先通过页面刷新和 GitHub 提交轮询满足核心查看需求。
4. 本次不新增数据库表；更新数据来自仓库文件与 Git 历史。

---

## 4. 方案概述

### 4.1 数据来源

| 数据 | 来源 | 说明 |
|---|---|---|
| 待发布更新 | `changelogs/*.md` | 按文件名日期倒序展示，每个文件是一组待发布更新 |
| 已发布更新 | `CHANGELOG.md` | 按 `## [版本] - 日期` 解析正式版本 |
| 周报 | `doc/report.*.md` | 按文件名倒序展示周报列表，选中后展示 Markdown 正文 |
| GitHub提交 | 本地 `git log` | 展示最近提交，支持 `before` 游标继续加载 |

### 4.2 文件格式

`changelogs/*.md` 和 `CHANGELOG.md` 中的更新表格统一使用：

```markdown
| 类型 | 模块 | 描述 |
|---|---|---|
| feat | 更新中心 | 新增管理端更新中心页面 |
```

### 4.3 后端分层

1. Controller：只接收请求，调用应用服务，返回 `ApiResult`。
2. ApplicationService：负责读取文件、解析 Markdown、读取 Git 日志。
3. DTO：统一放在 `ticket-entity` 的 `updatecenter` 包中。

### 4.4 前端交互

1. 在左侧一级菜单新增“动态”，位置位于“管理”下方；旧路径 `/manage/update-center` 保留重定向兼容。
2. 页面顶部显示统计卡片：已发布版本数、待发布条目数、Git 提交数。
3. 四个 Tab：
   - 已发布：展示版本、日期、亮点和更新条目。
   - 待发布：展示 `changelogs/` 中尚未合并发布的碎片。
   - 周报：左侧展示 `doc/report.*.md` 列表，右侧渲染 Markdown 正文。
   - GitHub提交：展示提交 SHA、作者、时间和提交说明。
4. 支持按更新类型筛选、手动刷新、加载更多。

## 5. 接口设计

| 接口编号 | 接口 | 方法 | 路径 | 说明 |
|---|---|---|---|---|
| API000521 | 查询待发布更新 | GET | /api/update-center/current-week | 读取 `changelogs/*.md` |
| API000522 | 查询已发布更新 | GET | /api/update-center/releases | 读取 `CHANGELOG.md` |
| API000523 | 查询指定版本更新详情 | GET | /api/update-center/releases/detail/{version} | 按版本号读取完整版本条目 |
| API000524 | 查询Git提交日志 | GET | /api/update-center/github-logs | 读取本地 Git 提交记录 |
| API000525 | 查询周报列表 | GET | /api/update-center/weekly-reports | 读取 `doc/report.*.md` |
| API000526 | 查询周报详情 | GET | /api/update-center/weekly-reports/detail/{fileName} | 读取指定周报 Markdown 正文 |

## 7. 风险与回滚

### 7.1 风险

1. 如果仓库没有 `CHANGELOG.md` 或 `changelogs/`，页面会显示空状态。
2. 如果运行环境没有 `.git` 目录，GitHub 提交 Tab 会显示数据源不可用。
3. Markdown 表格格式不规范时，该行会被跳过。

### 7.2 回滚策略

1. 前端可移除菜单和路由，接口保留不影响现有功能。
2. 后端接口都是只读接口，不涉及数据写入；回滚风险低。
3. 若页面异常，可先隐藏“管理 / 更新中心”菜单入口。
