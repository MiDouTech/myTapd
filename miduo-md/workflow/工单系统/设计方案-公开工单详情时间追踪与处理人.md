# 设计方案：公开工单详情时间追踪与当前处理人

## 方案概要

1. **当前处理人**：公开详情构建逻辑与内网 `buildDetailOutput` 对齐，通过 `TicketAssigneeSyncService.listActiveUserIds` 获取活跃处理人 ID 列表，批量查用户表后将 `assigneeName` 设为 `String.join("、", names)`；无处理人时仍为「待分配」语义（空则前端展示待分配，与现网一致）。
2. **时间追踪摘要**：在 `TicketTimeTrackApplicationService` 中新增 `listPublicTimeTrackItems(Long ticketId)`，仅查询 `ticket_time_track` 表并映射为公开 DTO 列表，**不**调用变更历史关联逻辑（避免 `BugChangeHistory`、字段级明细外泄与性能开销）。
3. **过滤规则**：排除动作码 `READ`、`FOLLOW`、`UNFOLLOW`、`COMMENT`；其余轨迹点映射为含 `actionLabel`、状态中文标签、操作人/转派双方姓名、时间、备注的条目。

## 影响范围

| 层级 | 文件/模块 |
|------|-----------|
| 契约 | `TicketPublicDetailOutput` 增加 `timeTrackItems`（`List<TicketPublicTimeTrackItemOutput>`） |
| 应用服务 | `TicketTimeTrackApplicationService`、`TicketApplicationService#buildPublicDetailOutput` |
| 前端类型 | `miduo-frontend/src/types/ticket.ts` 中 `TicketPublicDetailOutput` |
| 前端页面 | `TicketPublicView.vue` 增加时间线展示区块 |

## 关键决策

- **为何不复用 `getTimeTrack`**：内网接口会拉取并关联 `ticket_log` 缺陷变更历史，对外仅需轨迹摘要且需避免字段级明细与额外查询。
- **为何过滤 READ/FOLLOW/COMMENT**：对外用户不关心「谁点开过工单」；评论已在「处理记录」中展示，避免重复与噪音。

## 兼容与回滚

- 响应仅**新增**字段 `timeTrackItems`，旧客户端忽略即可。
- 回滚：移除 DTO 字段与前端区块即可，数据库无变更。
