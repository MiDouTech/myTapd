# Task026：Bug简报逻辑归因与右侧说明12类对齐

> **业务模块**：工单系统（Bug简报）  
> **变更类型**：优化（不新增接口，前端同源渲染）  
> **关联页面**：`/bug-report/edit/:id?`（逻辑归因下拉 + 右侧说明）

---

## 一、需求收敛（背景 / 目标 / 非目标）

### 1.1 背景

当前“逻辑归因下拉”使用后端字典树（7个一级+二级），而右侧说明使用前端静态 12 类说明，导致用户看到的“可选项”和“说明项”不一致。

### 1.2 目标

以右侧说明为准，将“逻辑归因下拉”统一为同一套 12 类，保证页面展示口径一致：

1. 配置错误  
2. 编码缺陷  
3. 设计缺陷  
4. 第三方问题  
5. 数据问题  
6. 运营问题  
7. 注释与代码不一致  
8. 兼容性问题  
9. 安全漏洞  
10. 性能问题  
11. 重构风险  
12. 接口问题

### 1.3 非目标

1. 不改 API 编号与接口路径。  
2. 不改 Bug 简报字段结构（仍使用 `logicCauseLevel1/logicCauseLevel2`）。  
3. 不修改历史简报文本值。

---

## 二、方案设计（影响范围 / 决策 / 兼容策略）

### 2.1 方案概要

在前端抽取“逻辑归因12类共享配置”作为单一数据源：

- 右侧说明直接渲染该配置；
- 下拉选项也直接使用该配置（单选一级，写入 `logicCauseLevel1`）；
- `logicCauseLevel2` 不再由下拉选择，保持空值或历史值兼容展示。

### 2.2 影响范围

- 前端：
  - `BugReportInstructionContent.vue`
  - `BugReportEditView.vue`
  - 新增共享配置文件（逻辑归因12类）
- 文档：
  - 本任务文档
  - `readme.md` 变更记录

### 2.3 关键决策

1. **页面口径优先一致**：以用户可见说明为准，避免“选项/说明”分裂。  
2. **字段兼容不破坏**：提交时继续写 `logicCauseLevel1`，`logicCauseLevel2` 保持兼容，不影响后端接口契约。  
3. **单一事实源**：逻辑归因说明和下拉都从同一配置读取，避免后续再次漂移。

### 2.4 回滚策略

若业务决定恢复后端字典树方案，可将下拉数据源切回 API000029，同时保留本次共享配置作为右侧展示基础。

---

## 三、验收标准

- [ ] 编辑页“逻辑归因”下拉项与右侧说明完全一致（12类、顺序一致）。  
- [ ] 选择任一逻辑归因后可正常保存/提交简报。  
- [ ] 历史简报详情可正常展示，不影响已存数据读取。  
- [ ] 页面无报错，交互可用。
