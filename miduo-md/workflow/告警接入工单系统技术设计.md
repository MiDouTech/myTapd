## 一、方案概要

基于夜莺（Nightingale）v7 的 Webhook 回调能力，在工单系统中新增告警接收端点。夜莺在告警触发时通过 HTTP POST 将 `AlertCurEvent` JSON 推送到工单系统，工单系统解析后根据映射规则自动创建工单并指派处理人。

### 核心数据流

```
夜莺告警引擎 --HTTP POST--> 工单系统 Webhook 端点
    --> Token 鉴权
    --> 解析 AlertCurEvent JSON
    --> 判断 is_recovered（恢复事件走恢复流程）
    --> 查询告警规则映射配置
    --> 去重检查（hash + 时间窗口）
    --> 创建工单（复用 TicketApplicationService.createTicket）
    --> 记录告警事件日志
```

## 四、关键决策

1. **鉴权方式**：采用 URL Token 参数鉴权（`?token=xxx`），Token 存储在系统配置中。夜莺 Webhook 不支持 JWT/OAuth。
2. **去重策略**：基于 `hash` + 时间窗口，在 `alert_event_log` 中查询最近创建过工单的记录。
3. **默认映射**：未找到映射配置时，使用系统默认分类创建工单，优先级按 P1→urgent, P2→high, P3→medium 映射。
4. **恢复事件**：不自动关闭工单，仅添加工单评论记录恢复信息。
5. **兼容策略**：告警工单复用现有 `createTicket` 流程，与现有工单体系完全兼容。
6. **处理人指派策略**：优先从夜莺推送的 `notify_users_obj` 中解析通知用户，按 phone → email → nickname/name 优先级与 `sys_user` 匹配，支持多人指派（通过 `TicketAssigneeSyncService.replaceAndPersist`）。匹配不到时回退到映射配置的 `assigneeId`，再无则走自动分派。批量加载活跃用户到内存 Map 匹配，避免循环查库。
7. **告警专用工作流**：`V42__alert_ticket_workflow.sql` 写入 **`workflow.id=4` 告警工单流转**；`V43__alert_workflow_oncall_states.sql` 将其更新为 **OnCall 专用状态机**（`alert_triggered` 等），**不再**与通用工单 id=1 对齐。告警映射绑定的分类应将 `workflow_id=4`。详见 [告警工单流转-工作流设计.md](./告警工单流转-工作流设计.md)。
8. **恢复事件终态判定**：关联工单的「是否可写恢复评论」使用 `WorkflowApplicationService.isTerminalStatus(ticket.workflowId, ticket.status)`，以工作流 JSON 为准（含 `alert_resolved` / `alert_suppressed` / `closed` 等终态）。

## 六、回滚/兼容策略

- 所有变更为新增，不修改任何已有表结构
- `TicketSource` 枚举新增值向后兼容
- Webhook 端点默认关闭（需配置 Token 并添加映射规则后才生效）
- Flyway 迁移脚本为增量脚本，不影响现有数据
