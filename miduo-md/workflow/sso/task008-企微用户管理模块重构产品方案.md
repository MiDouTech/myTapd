# Task008-企微用户管理模块重构产品方案

## 1. 背景与问题

当前“组织账号管理”已具备部门树、员工列表、手动同步能力，但在企业微信高频变更场景下仍存在以下问题：

1. 同步过程透明度不足：只能看到总成功/失败，不知道部门和用户分别发生了什么变更；
2. 同步健康度不可直观筛选：无法快速识别“已离职但仍在本地”“同步失败待处理”等异常账号；
3. 同步链路效率不佳：历史实现按部门逐个拉成员，组织规模变大后同步耗时明显上升；
4. 组织负责人信息利用不足：部门负责人未回写，影响后续派单与组织治理能力。

## 2. 设计目标

### 2.1 业务目标

1. 建立“可观测、可追溯、可治理”的企微用户管理模块；
2. 让管理员可以在 1 个页面完成：同步触发、同步排障、账号筛选、成员查看；
3. 将企微通讯录变更稳定映射到本地账号体系（新增、更新、失活）。

### 2.2 体验目标

1. 手动同步后立即看到“部门变更 + 用户变更”细分结果；
2. 员工列表支持按“同步状态”筛选，快速定位异常；
3. 员工列表可直观看到“同步状态 + 最近同步时间”。

## 3. 用户角色与核心诉求

### 3.1 系统管理员

- 关注点：同步是否成功、失败在哪里、哪些对象发生变化；
- 关键动作：手动同步、筛选同步失败用户、查看同步日志。

### 3.2 业务管理员（工单管理员/部门负责人）

- 关注点：成员是否在职、是否已同步、部门结构是否最新；
- 关键动作：按部门筛选成员、查看成员详情、核对同步时间。

## 4. 功能设计

## 4.1 同步运维能力升级

### 4.1.1 手动同步反馈升级

在保留“总数/成功/失败”的基础上，新增分项统计：

- 部门：新增、更新、停用；
- 用户：新增、更新、离职失活。

### 4.1.2 同步健康度治理

新增“同步状态”筛选维度：

- `0=未同步`
- `1=同步成功`
- `2=同步失败/失效`

用于快速筛出异常账号进行治理。

## 4.2 员工管理视图升级

员工列表新增展示字段：

1. 同步状态（标签化展示）；
2. 最近同步时间（便于判断数据新鲜度）。

员工详情新增展示：

1. 同步状态；
2. 最近同步时间。

## 4.3 组织治理能力升级

同步部门时回写部门负责人（企微 `department_leader`），为后续“按部门负责人派单/通知”预留数据基础。

## 5. 核心业务规则

1. **同步来源优先级**：企微通讯录为主，本地为镜像；
2. **用户状态映射**：
   - 企微 `status=1` → 本地在职；
   - 企微 `status=2` → 本地停用；
   - 企微 `status=4/5` → 本地离职；
3. **差异收敛规则**：
   - 企微存在、本地不存在 → 新增；
   - 企微存在、本地存在且字段变化 → 更新；
   - 本地存在、企微不存在 → 标记离职失活（逻辑保留，不物理删除）；
4. **部门差异规则**：
   - 企微缺失的本地部门设为停用，保留历史关联；
5. **敏感信息展示规则**：手机号/邮箱/企微userid仍保持脱敏展示。

## 6. 交互流程

## 6.1 手动同步流程

1. 管理员点击“同步企业微信”；
2. 系统执行全量同步并返回总览结果；
3. 页面提示：
   - 总处理数、成功数、失败数；
   - 部门新增/更新/停用；
   - 用户新增/更新/离职失活；
4. 页面自动刷新：组织树、成员列表、最近同步状态。

## 6.2 异常治理流程

1. 管理员在员工列表将“同步状态”筛选为“失败/失效”；
2. 查看异常人员及最近同步时间；
3. 结合同步日志排查（配置错误、权限错误、数据问题）；
4. 修复后再次触发手动同步闭环。

## 7. 验收标准

1. 手动同步提示包含分项统计且数据正确；
2. 员工分页支持“同步状态”筛选；
3. 员工列表与详情可展示同步状态、同步时间；
4. 同步后能正确处理“企微已移除用户/部门”的失活收敛；
5. 同步链路在中大型组织下耗时较旧方案明显下降（接口调用次数显著减少）。

## 8. 本期不做

1. 企微通讯录变更回调（增量事件驱动）；
2. 成员多部门关系的完整建模与展示；
3. 自动冲突工单（同步失败自动派发运维工单）。
