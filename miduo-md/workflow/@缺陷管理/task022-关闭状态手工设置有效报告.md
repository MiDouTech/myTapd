# task022-关闭状态手工设置有效报告

> 日期：2026-05-24  
> 状态：🚧进行中  
> 类型：优化（不新增独立接口，扩展存量接口能力）

---

## 1. 变更背景（需求收敛）

当前“有效报告”字段由系统自动计算：  
是否为“是”取决于关联 Bug 简报是否已归档（`ARCHIVED`）。

该规则在“工单已关闭”场景下存在业务痛点：

1. 关闭后需要按人工复盘口径修正“有效/无效”，但页面无法编辑；
2. 简报状态只能反映流程状态，不能完全表达最终复盘结论；
3. 运营和管理侧希望在关闭阶段对“有效报告”做最终确认。

---

## 2. 目标与非目标

### 2.1 目标

1. 工单状态为 `closed` 时，支持人工选择“有效报告=是/否”；
2. 人工设置后，工单详情右侧“缺陷信息”按人工值优先展示；
3. 变更写入变更历史，便于审计；
4. 不新增 Controller 路径，复用现有 `API000021`（更新客服信息）。

### 2.2 非目标

1. 不改 Bug 简报状态机（DRAFT/PENDING_REVIEW/ARCHIVED 等）；
2. 不改简报详情页“有效报告”口径；
3. 不做批量回填历史工单数据。

---

## 3. 方案设计

## 3.1 数据模型

在 `ticket_bug_info` 新增字段：

- `manual_valid_report`：手工有效报告，取值 `YES/NO`，可空。

可空含义：
- `NULL`：未手工指定，沿用系统自动计算口径；
- `YES/NO`：人工已确认，关闭状态下优先展示该值。

## 3.2 口径优先级

工单详情“有效报告”展示规则：

1. 若工单状态为 `closed` 且 `manual_valid_report` 有值（YES/NO）  
   → 展示人工值（优先级最高）；
2. 否则沿用原规则：关联简报状态 `ARCHIVED` => 是，其他 => 否；
3. 若无关联简报且无人工值，则显示 `-`。

## 3.3 权限与状态约束

手工更新仅允许在 `closed` 状态执行：

- 非 `closed` 状态提交 `manualValidReport` 变更，后端返回参数错误；
- 其他客服信息字段维持原有更新逻辑。

## 3.4 接口影响

本次不新增 API 编号，扩展存量接口：

1. `API000021 /api/ticket/bug/customer-info/{id}`  
   - 入参新增：`manualValidReport`（YES/NO，可空）。
2. `API000008 /api/ticket/detail/{id}`  
   - `bugSummaryInfo.isValidReport / isValidReportLabel` 在关闭状态下可返回人工值。

---

## 4. 兼容与回滚

## 4.1 向后兼容

1. 新字段默认空，不影响历史工单；
2. 未设置人工值时，行为与当前线上一致；
3. 前端旧版本不传 `manualValidReport` 也可正常使用。

## 4.2 回滚策略

若需回滚展示逻辑：
- 可保留字段不删，仅移除“人工值优先”逻辑，自动回到旧口径。

---

## 5. 验收标准

1. 工单非 `closed`：页面不可保存人工“有效报告”变更；
2. 工单 `closed`：可在详情页选择“是/否”并保存成功；
3. 保存后刷新详情，字段展示与选择一致；
4. 变更历史可看到“有效报告”字段从旧值到新值的变化；
5. 不影响原有简报状态流转与详情接口稳定性。
