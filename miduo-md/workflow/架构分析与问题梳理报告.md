# 米多工单系统 — 架构分析与问题梳理报告

> 生成时间：2026-03-12  
> 范围：ticket-platform（后端）+ miduo-frontend（前端）  
> 目的：梳理当前系统架构、识别存在的设计问题和 Bug，为优化迭代提供输入

## 一、项目整体架构概览

### 1.1 技术栈

| 层次 | 技术选型 |
|------|---------|
| 前端 | Vue 3.5 + TypeScript 5.9 + Vite 7 + Element Plus + Pinia + Axios |
| 后端 | Spring Boot 2.7.18 + MyBatis-Plus 3.5.3 + JDK 8 |
| 数据库 | MySQL 8.0 + Redis 7 |
| 文件存储 | MinIO |
| 构建/部署 | Docker Compose + GitHub Actions + Watchtower 自动更新 |
| 认证 | JWT（无状态）+ 企业微信 OAuth |

### 1.2 后端模块分层

```
ticket-platform（父 POM）
│
├── ticket-common         ← 公共基础：枚举、常量、工具类、通用 DTO
├── ticket-entity         ← 接口 DTO（Input/Output）
├── ticket-domain         ← 领域模型、领域事件、Repository 接口、工作流引擎
├── ticket-infrastructure ← MyBatis-Plus PO/Mapper、外部集成（企微 API / Redis / WebSocket）
├── ticket-application    ← 业务编排 ApplicationService、事件监听器、定时任务
├── ticket-controller     ← REST Controller
├── ticket-job            ← 定时任务（SLA 检查、Bug 简报提醒、企微同步）
└── ticket-bootstrap      ← 启动类、Security 配置、JWT 过滤器、全局异常处理、Flyway
```

**设计意图**：依赖方向单向向上（Common → Entity/Domain → Infrastructure → Application → Controller），`domain` 层不允许依赖 `infrastructure` 和任何 DB 框架。

### 1.3 核心功能模块

## 二、架构设计问题（需优化）

### 2.1 【高优先级】Application 层直接操作大量 Mapper，绕过 Domain Repository

**位置**：
- `ticket-application/...ticket/TicketApplicationService.java`（注入 13 个 Mapper）
- `ticket-application/...bugreport/BugReportApplicationService.java`（注入 12 个 Mapper）
- `CategoryApplicationService.java`、`DispatchAppService.java` 等

**问题描述**：项目已在 `ticket-domain` 层定义了 `UserRepository`、`DepartmentRepository` 接口（正确的 DDD 做法），但绝大多数 Application Service 绕过这一机制，直接依赖 `ticket-infrastructure` 的 Mapper，破坏了 Domain 层的隔离性。

**影响**：领域逻辑与数据访问细节强耦合，后续难以替换底层存储实现或进行单元测试。

**建议**：扩展 `ticket-domain` 中的 Repository 接口，Application Service 通过 Repository 访问数据，而非直接操作 Mapper。

---

### 2.2 【高优先级】`TicketApplicationService` 职责过度集中

**位置**：`ticket-application/...ticket/TicketApplicationService.java`（848 行）

**问题描述**：单个服务同时处理：工单 CRUD、Bug 扩展信息组装、时间追踪、Bug 简报关联等多个领域职责，违反单一职责原则。已有 `TicketBugApplicationService`、`TicketTimeTrackApplicationService` 分担部分逻辑，但拆分不彻底。

**建议**：
- 将 Bug 扩展信息组装逻辑全部迁至 `TicketBugApplicationService`
- 将 `bugSummaryInfo` 组装独立为 `TicketBugSummaryService`
- `TicketApplicationService` 专注工单核心状态流转

---

### 2.3 【中优先级】Controller 层包含业务逻辑私有方法（违反规范 25.1）

**位置**：`ticket-controller/...openapi/v1/OpenTicketV1Controller.java` 第 84-96 行

**问题描述**：`toOpenApiCreateInput()` 是私有方法，内含字段映射和 `TicketSource.API` 枚举赋值等业务逻辑，违反 Controller 层规范。

**建议**：将该方法迁移至 `ticket-application` 层或 `ticket-common` 的 util 包。

---

### 2.4 【中优先级】接口路径设计不符合 RESTful 规范

| 接口 | 当前路径 | 问题 | 建议路径 |
|------|---------|------|---------|
| SLA 更新 | `PUT /api/sla/policy/update` | 路径含动词，缺少 `/{id}` | `PUT /api/sla/policy/update/{id}` |
| 企微配置保存 | `POST /api/wecom/config/save` | 路径含动词 `save`，语义为更新 | `PUT /api/wecom/config` |

---

### 2.5 【低优先级】RabbitMQ 两套实现并存，代码维护风险

**位置**：`ticket-infrastructure/...external/wework/`

**问题描述**：`AsyncWecomMessagePublisher` 和 `RabbitWecomMessagePublisher` 两套 MQ 实现并存，当前通过 `wecom.mq.enabled=false` 默认禁用。若后续启用 MQ，双实现并存可能引发混乱。

**建议**：明确一套主实现，将另一套标记为废弃或删除。

## 三、代码规范问题（违反项目红线）

### 3.1 【高优先级·红线】循环内单次数据库查询

#### 问题 1：`DispatchAppService` 负载均衡分派

**位置**：`ticket-application/...workflow/DispatchAppService.java` 第 163-167 行

```java

    long count = ticketMapper.selectCount(

            .eq(TicketPO::getAssigneeId, memberId)
            .notIn(TicketPO::getStatus, "COMPLETED", "CLOSED")

}
```

**建议**：一次性用 `GROUP BY assignee_id WHERE assignee_id IN (...)` 查询所有成员未结工单数，内存组装为 Map 后处理。

#### 问题 2：`CategoryApplicationService.buildFullPathName`

**位置**：`ticket-application/...category/CategoryApplicationService.java` 第 283-293 行

```java
// ❌ 违规：while 循环内逐层查询父分类

    names.add(0, current.getName());
    if (current.getParentId() != null) {
        current = categoryMapper.selectById(current.getParentId());
    }

```

**建议**：一次性查出全量分类树缓存到内存（或 Redis），路径拼接在内存中完成。

#### 问题 3：`TicketApplicationService.createTicket` 自定义字段写入

**位置**：`ticket-application/...ticket/TicketApplicationService.java` 第 122-133 行

**建议**：改为批量 insert（`saveBatch` 或自定义批量 SQL）。

### 3.3 【中优先级·红线】注解类直接写 SQL（应放 mapper.xml）

**位置**：

```java
// ❌ BugReportTicketMapper.java
@Delete("DELETE FROM bug_report_ticket WHERE report_id = #{reportId}")
int hardDeleteByReportId(@Param("reportId") Long reportId);

// ❌ BugReportResponsibleMapper.java
```

**建议**：迁至 mapper.xml，同时改为逻辑删除（`UPDATE ... SET deleted=1`）。

### 3.4 【中优先级】`@Transactional` 缺少 `rollbackFor`

**位置**：
- `AuthApplicationService.java` 第 53、107 行
- `WecomSyncService.java` 第 52、60 行

**问题描述**：Spring 默认只回滚 `RuntimeException`，对受检异常（`Exception` 子类）不回滚，存在数据不一致风险。

**建议**：统一改为 `@Transactional(rollbackFor = Exception.class)`。

## 四、安全问题（需修复）

### 4.1 【高优先级】所有接口缺乏角色权限控制（RBAC 未实现）

**问题描述**：系统已定义 ADMIN / TICKET_ADMIN / HANDLER / SUBMITTER / OBSERVER 五个角色，`@EnableGlobalMethodSecurity(prePostEnabled = true)` 已配置，但**所有接口均对全部已认证用户开放**，没有任何接口使用 `@PreAuthorize` 注解进行角色控制。

**具体风险**：
- 普通 SUBMITTER 用户可以调用 SLA 策略创建/更新接口
- 普通用户可以修改工作流配置
- 普通用户可以更新 Webhook/企微配置

**建议**：对管理员接口（分类管理/工作流/SLA/Webhook/企微配置/组织管理）增加 `@PreAuthorize("hasRole('ADMIN') or hasRole('TICKET_ADMIN')")`。

---

### 4.2 【中优先级】`/actuator/**` 全部端点对外开放

**位置**：`ticket-bootstrap/.../SecurityConfig.java` 第 42-52 行

**问题描述**：`/actuator/health` 配置了 `show-details: always`，会暴露数据库连接状态、Redis 状态等敏感信息，且全部 actuator 端点均无认证要求。

**建议**：将 actuator 接口限制在内网访问，或要求 ADMIN 角色认证。

---

### 4.3 【中优先级】开发模式登录接口在生产环境的风险

**问题描述**：`application-prod.yml` 中未显式声明 `dev-login.enabled: false`，仅依赖 `@Value("${dev-login.enabled:false}")` 的默认值保护。若生产环境 profile 配置出错，开发登录接口将暴露。同时，测试账号密码 `admin/admin2026` 直接硬编码在代码中。

**建议**：
1. 在 `application-prod.yml` 中显式声明 `dev-login.enabled: false`
2. 将开发账号密码移至环境变量注入，不提交到代码库

---

### 4.4 【中优先级】生产环境 JWT Secret 使用占位符

**位置**：`application-prod.yml`

**问题描述**：JWT secret 使用 `your_jwt_secret_key_must_be_at_least_64_characters_long_replace_this` 占位符，与开发配置通过 `${ENV_VAR:default}` 模式注入不一致，依赖运维手动替换。

**建议**：改为环境变量注入模式：`jwt.secret: ${JWT_SECRET}`，强制要求部署时提供。

---

### 4.5 【中优先级】GET 请求参数校验失败未统一处理

**问题描述**：全局异常处理（`ValidationAdvice`）只处理了 `MethodArgumentNotValidException`（`@RequestBody` + `@Valid`），对 GET 请求 `@RequestParam`/`@PathVariable` 的校验失败会抛出 `ConstraintViolationException`，未被捕获，将返回 500 错误。

**建议**：在全局异常处理中补充对 `ConstraintViolationException` 的处理。

---

### 4.6 【低优先级】`TicketChangeHistoryController` 获取 userId 但未做权限校验

**位置**：`ticket-controller/.../TicketChangeHistoryController.java` 第 41-43 行

**问题描述**：方法内获取了 `currentUserId` 但完全未使用，也没有任何权限验证逻辑，OBSERVER 角色用户可随意查看所有工单的变更历史。

---

### 4.7 【低优先级】`WecomSyncController` userId 为 null 时降级处理不合理

**位置**：`ticket-controller/.../WecomSyncController.java` 第 41-43 行

```java
// ❌ 手动同步是管理员操作，userId 为 null 说明未认证，不应降级为 "system"
Long userId = SecurityUtil.getCurrentUserId();
String triggerBy = userId == null ? "system" : String.valueOf(userId);
```

## 六、数据库设计问题

### 6.1 【中优先级】`TicketMapper.xml` 使用前置模糊查询（违反索引规范）

**位置**：`ticket-infrastructure/src/main/resources/mapper/ticket/TicketMapper.xml` 第 32-33 行

```xml
<!-- ❌ 前置模糊查询，无法使用索引，大数据量时全表扫描 -->
AND t.title LIKE CONCAT('%', #{title}, '%')
```

**建议**：改为后置模糊 `LIKE CONCAT(#{title}, '%')` 或引入全文索引（`FULLTEXT INDEX`）。

---

### 6.2 【低优先级】`ticket_log` 表缺少复合索引

**位置**：`V2__init_ticket_core.sql` 第 143-155 行

**问题描述**：按工单查询日志是最高频操作，当前只有 `idx_ticket_id` 单列索引，没有包含 `create_time` 的复合索引。

**建议**：增加 `KEY idx_ticket_id_create_time (ticket_id, create_time DESC)` 复合索引。

## 七、性能与内存问题

### 7.1 【中优先级】`BugReportApplicationService.statistics` 全量加载到内存

**位置**：`ticket-application/...bugreport/BugReportApplicationService.java` 第 430 行

```java
// ❌ 全量拉取到内存做统计，数据量大时有 OOM 风险
List<BugReportPO> reports = bugReportMapper.selectList(wrapper);
```

**建议**：改为数据库聚合查询（`GROUP BY` + `COUNT` / `SUM`）返回统计结果，避免大量数据加载到内存。

---

### 7.2 【中优先级】`BugReportApplicationService.isBugTicket` 串行 3 次 selectCount

**位置**：`ticket-application/...bugreport/BugReportApplicationService.java` 第 997-1011 行

**问题描述**：对三张缺陷扩展表串行执行 `selectCount` 判断是否为缺陷工单，虽有短路返回，但最坏情况是 3 次独立数据库查询。

**建议**：改为一次 JOIN 查询，或使用 `CompletableFuture` 并行执行三个查询。

---

### 7.3 【低优先级】`CategoryApplicationService.convertToDetailOutput` 5 次串行单查询

**位置**：`ticket-application/...category/CategoryApplicationService.java` 第 247-278 行

**问题描述**：每次分类详情转换，最多发起 5 次串行单查询（父分类、模板、工作流、SLA 策略、处理组）。

**建议**：使用 `CompletableFuture` 并行发起 5 个查询，或缓存常用配置数据到 Redis。

---

### 7.4 【低优先级】MyBatis 全局 SQL 日志在生产配置中有泄露风险

**位置**：`ticket-bootstrap/src/main/resources/application.yml` 第 33-36 行

```yaml
# 全局配置，在 application.yml 中配置 StdOutImpl，会打印所有 SQL 语句
configuration:
  log-impl: org.apache.ibatis.logging.stdout.StdOutImpl
```

虽然 `application-prod.yml` 中有覆盖，但全局配置优先级问题存在风险。建议将 `application.yml` 中该配置移除，在各 profile 中按需配置。

## 十、总体评价

### 优点

1. **模块分层清晰**：严格的多模块 DDD 架构，依赖方向单向，`ticket-domain` 层零基础设施依赖
2. **工作流灵活**：状态机 JSON 化存储在数据库，支持运行时配置无需重部署
3. **事件驱动解耦**：通知、SLA、Webhook、时间追踪均通过 Spring 领域事件解耦，扩展性好
4. **实时推送可靠**：WebSocket 主 + 轮询降级双保险通知机制
5. **数据库规范统一**：所有表包含审计字段（create_by/update_by/create_time/update_time/deleted），逻辑删除统一
6. **CI/CD 完整**：多镜像仓库推送，Watchtower 自动更新
7. **接口文档完备**：全部接口携带编号注释，有完整设计文档

### 主要短板

1. **RBAC 权限控制缺失**：角色已定义但未落地到接口授权，是最大的安全隐患
2. **DB 红线违规**：多处循环内查询、前置模糊查询，数据量增长后将成为性能瓶颈
3. **Application 层职责失控**：`TicketApplicationService` 过大，且大量 ApplicationService 绕过 Domain Repository 直接操作 Mapper
4. **前端功能完整性**：搜索框无功能、缺陷工单字段保存 Bug 等影响用户体验
