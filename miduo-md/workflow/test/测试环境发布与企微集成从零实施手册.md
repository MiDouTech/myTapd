## 1. 先说结论（基于文档 + 代码的现状分析）

本手册是基于以下内容综合整理的：

- 产品方案：`doc/工单系统产品设计方案.md`

- 实际代码：`ticket-platform/`（后端）、`miduo-frontend/`（前端）

### 1.1 当前“已具备”的能力（代码已落地）

1. **企微扫码登录后端接口已具备**  
   - `POST /api/auth/wecom/login`（API000400）  
   - `POST /api/auth/refresh`（API000401）
2. **企微回调与群消息处理已具备**  
   - `GET/POST /api/wecom/callback`（API000020）  
   - 支持签名校验、AES 解密、消息去重、异步处理
3. **企微群绑定配置接口已具备**  
   - `/api/wecom/group-binding/list|create|update`
4. **企微消息推送链路已具备**  
   - 个人应用消息（textcard）  
   - 群 webhook markdown 推送

### 1.2 当前需要你明确的“配置动作”

代码不是“零配置开箱即用”，你必须准备以下参数：

- `wecom.corp-id`
- `wecom.agent-id`
- `wecom.secret`
- `wecom.contact-secret`
- `wecom.callback-token`
- `wecom.callback-aes-key`
- `wecom.trusted-domain`

以及前端参数：

- `VITE_WECOM_OAUTH_URL`
- `VITE_API_BASE_URL`

### 1.3 文档与代码存在的差异（已替你踩坑）

1. 技术分析里写过 Java 17 / Spring Boot 3.x，但**实际代码是 Java 8 + Spring Boot 2.7.18**。  
2. `application-test.yml` 默认数据库是 `ticket_platform_test`，而 docker-compose 默认建的是 `ticket_platform`。  
   - 本手册提供的后端发布脚本已经自动兜底创建 `ticket_platform_test`。  
3. 企微 MQ 默认关闭（`wecom.mq.enabled=false`），默认走本地异步处理，这对测试环境是友好的（不强依赖 RabbitMQ）。

### 1.4 任务执行情况（基于 task 文档 + 接口清单交叉判断）

## 2. 发布前先做环境自检（小白必做）

先在服务器执行：

```bash
command -v java
command -v mvn
command -v node
command -v npm
command -v docker
```

至少要满足：

1. 后端发布：`java + mvn`  
2. 前端发布：`node + npm`  
3. 需要脚本自动拉起 MySQL/Redis/MinIO 时：`docker`

### 2.1 Ubuntu 常用安装命令（示例）

```bash
sudo apt-get update
sudo apt-get install -y maven docker.io docker-compose-plugin
```

Node 建议使用 nvm 安装 LTS 版本（若未安装）。

## 5. 企微扫码登录配置教程（超详细）

## 5.1 在企业微信管理后台创建自建应用

路径（大致）：

1. 企业微信管理后台
2. 应用管理
3. 自建应用
4. 新建应用

拿到 4 个关键参数：

1. CorpID（企业 ID）
2. AgentID（应用 ID）
3. Secret（应用密钥）
4. 通讯录 Secret（用于通讯录同步）

> 注意：通讯录 secret 可能和应用 secret 不同，别填反。

## 5.2 配可信域名

在企微应用里配置可信域名：  
`test-ticket.company.com`（示例）

这个域名必须和你前端访问域名一致。

## 5.3 生成 OAuth 链接（不用手工拼）

执行：

```bash
cd /workspace/ticket-platform
bash scripts/gen-wecom-oauth-url.sh \
  "ww你的CorpID" \
  "1000002" \
  "https://test-ticket.company.com/login" \
  "ticket_test_state"
```

把输出结果填到前端：

```bash
VITE_WECOM_OAUTH_URL=刚才输出的完整URL
```

## 5.4 后端配置企微参数

在 `ticket-platform/scripts/.env.test.local` 填写：

- `WECOM_CORP_ID`
- `WECOM_AGENT_ID`
- `WECOM_SECRET`
- `WECOM_CONTACT_SECRET`
- `WECOM_TRUSTED_DOMAIN`

重启后端脚本生效。

## 5.5 联调验证（最小闭环）

1. 浏览器打开前端登录页 `/login`
2. 点击“前往企业微信授权页”
3. 企业微信扫码确认
4. 回调到 `/login?code=xxx`
5. 前端自动调 `POST /api/auth/wecom/login`
6. 登录成功跳转 `/dashboard`

如果失败，先看：

- 前端浏览器控制台
- 后端日志：`ticket-platform/.run/logs/ticket-bootstrap.log`

## 6. 企微机器人配置教程（群里 @工单助手 建单）

> 你要配置的是“应用回调 + 群绑定 +（可选）群 webhook 推送”三件事。

## 6.1 配置企微回调 URL（应用接收消息）

回调地址固定为：

```text
https://你的后端域名/api/wecom/callback
```

在企微后台“应用接收消息”处填写：

1. URL：上面这个地址
2. Token：与后端 `WECOM_CALLBACK_TOKEN` 完全一致
3. EncodingAESKey：与后端 `WECOM_CALLBACK_AES_KEY` 完全一致

企微后台会做“URL 验证”，通过后才算配置成功。

## 6.2 把应用拉进目标群

1. 进入企业微信群
2. 添加该自建应用（工单助手）
3. 确认群成员可 `@工单助手`

## 6.3 在系统里绑定“群 ChatID 与默认分类”

先拿登录 token（页面登录后从浏览器 localStorage 取 `ticket_access_token`），然后调用：

```bash
curl -X POST "https://你的后端域名/api/wecom/group-binding/create" \
  -H "Authorization: Bearer 你的token" \
  -H "Content-Type: application/json" \
  -d '{
    "chatId": "wrfxxxxxxxx",
    "chatName": "测试客服群",
    "defaultCategoryId": 12,
    "webhookUrl": "https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=xxx",
    "isActive": 1
  }'
```

## 6.4 群内测试命令

按顺序测：

1. `@工单助手 帮助`
2. `@工单助手 分类`
3. `@工单助手 #IT支持/网络问题 VPN无法连接`
4. `@工单助手 查询 WO-xxxx`
5. `@工单助手 我的工单`

## 7. 企微信息推送配置教程（个人消息 + 群消息）

## 7.1 个人企微消息推送前提

系统要知道目标用户的 `wecom_userid`。有两种方式：

1. 用户至少登录过一次（会自动绑定/自动建号）  
2. 等待定时同步（每天凌晨 1 点）或运维手动补齐用户映射

## 7.2 群消息推送前提

1. 群绑定里配置了 `webhookUrl`
2. 工单满足“来源群”或“分类绑定命中”

## 7.3 触发一条推送做验证

最简单：创建一个工单，随后做一次分派或状态变更。  
系统会触发：

- 站内信
- 企微应用消息（给个人）
- 群 webhook 消息（给群）

## 7.4 如果你收不到消息，按这个顺序查

1. 用户是否有 `wecom_userid`
2. 企微参数是否完整（corp/agent/secret）
3. 用户通知偏好里 `wecomEnabled` 是否为 1
4. 群 webhook 地址是否可用
5. 后端日志里是否有 `WECOM_API_ERROR`

## 8. 常见报错与处理

### 8.1 回调验签失败

现象：企微后台提示 URL 验证失败。  
排查：

1. `WECOM_CALLBACK_TOKEN` 是否与后台完全一致
2. `WECOM_CALLBACK_AES_KEY` 是否 43 位并且一致
3. 回调 URL 是否 HTTPS 可公网访问

### 8.2 登录失败：无法获取企微用户身份

排查：

1. `WECOM_CORP_ID / WECOM_SECRET` 是否正确
2. `VITE_WECOM_OAUTH_URL` 的 corpId/agentId/redirect_uri 是否正确
3. code 是否过期（code 只能短时间使用）

### 8.3 机器人有回复但没建单

排查：

1. 分类是否存在（`@工单助手 分类` 先看列表）
2. 模板格式是否满足要求
3. 发消息的人是否已绑定系统账号（先扫码登录一次）

## 10. 最后给你的执行建议（小白版）

严格按这条顺序：

1. 先发后端（能过 health）
2. 再发前端（能打开登录页）
3. 再配企微扫码（先测登录）

5. 最后测消息推送（创建工单触发）

只要你按这个顺序走，出现问题时定位会非常快，不会乱。
