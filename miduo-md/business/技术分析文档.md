## 二、技术架构设计分析

### 2.1 架构选型：单体优先

**决策**：第一期采用**模块化单体架构**（Modular Monolith），不做微服务拆分。

**理由**：
- 并发要求 500+ 人同时在线，单体完全可承载
- 内部系统，用户量有天花板（公司人员规模）
- 团队规模有限，微服务带来的运维复杂度不合理
- DDD 分层 + 模块化设计为未来拆分保留余地

**架构分层**（沿用现有 DDD 六边形架构）：

```
┌─────────────────────────────────────────────────────┐
│                    入站适配器层                        │
│  REST Controller / 企微回调 / WebSocket / 定时任务     │
├─────────────────────────────────────────────────────┤
│                    应用服务层                          │
│  工单服务 / 工作流服务 / SLA 服务 / 通知编排 / 简报服务  │
├─────────────────────────────────────────────────────┤
│                    领域层（核心）                      │
│  工单聚合 / 工作流聚合 / SLA 策略 / 模板聚合 / 简报聚合 │
│  领域事件 / 领域服务 / 仓储接口                        │
├─────────────────────────────────────────────────────┤
│                    出站适配器层                        │
│  MySQL 持久化 / Redis 缓存 / 企微 API / MQ 生产者     │
│  MinIO 文件存储 / 邮件服务                             │
└─────────────────────────────────────────────────────┘
```

### 2.3 新增中间件决策分析

#### 2.3.1 消息队列：RabbitMQ vs Redis Stream

| 对比维度 | RabbitMQ | Redis Stream |
|---------|----------|-------------|
| 消息可靠性 | ACK 机制、持久化、死信队列 | 持久化支持，但故障恢复能力弱于 MQ |
| 消息路由 | Exchange + Routing Key，灵活路由 | 简单的消费者组模型 |
| 延迟消息 | 插件支持（SLA 超时场景需要） | 不原生支持 |
| 运维复杂度 | 中等，需独立部署 | 低，复用现有 Redis |
| 适配场景 | 通知推送、SLA 预警、异步解耦 | 简单的异步队列 |

**决策**：选择 **RabbitMQ**。

理由：SLA 超时预警需要延迟消息能力；通知推送需要灵活的路由分发（不同事件发往不同渠道）；消息可靠性要求高（工单状态流转通知不能丢失）。

**降级方案**：若第一期不引入 RabbitMQ，可先用 **Redis + 定时轮询** 实现 SLA 检查和通知推送，后续再迁移至 MQ。

#### 2.3.2 文件存储：MinIO vs 本地存储

| 对比维度 | MinIO | 本地存储 |
|---------|-------|---------|
| 扩展性 | 分布式对象存储，无容量限制 | 受限于磁盘 |
| 高可用 | 支持纠删码、跨节点复制 | 单点风险 |
| 访问方式 | S3 兼容 API，支持预签名 URL | 需自行实现文件服务 |
| 部署复杂度 | 中等，Docker 一键部署 | 低 |

**决策**：选择 **MinIO**。

理由：工单附件（截图、日志文件）是核心资产，不能有丢失风险；MinIO Docker 部署极简；S3 兼容 API 便于未来迁移至云存储。

### 3.2 核心聚合设计

#### 3.2.1 工单聚合（Ticket Aggregate）— 核心聚合根

```
Ticket（聚合根）
├── TicketId              值对象 - 工单唯一标识
├── TicketNo              值对象 - 工单编号（业务可读）
├── TicketTitle           值对象 - 标题
├── TicketDescription     值对象 - 描述（富文本）
├── Category              实体引用 - 分类（三级）
├── Priority              枚举 - 优先级
├── Status                值对象 - 当前状态（由工作流驱动）
├── Creator               值对象 - 创建人
├── Assignee              值对象 - 当前处理人
├── Source                枚举 - 来源（Web/企微群/API）
├── CustomFieldValues[]   值对象集合 - 自定义字段值
├── Attachments[]         值对象集合 - 附件
├── TimeTrack[]           实体集合 - 时间追踪记录
├── Comments[]            实体集合 - 评论/处理记录
└── SlaSnapshot           值对象 - SLA 快照（响应时限/解决时限/当前状态）
```

**设计要点**：
- `TimeTrack` 作为实体（有独立生命周期）而非值对象，因为追踪记录需要独立查询和统计
- `CustomFieldValues` 是按模板动态渲染的键值对集合，存储为 JSON
- `SlaSnapshot` 嵌入工单聚合而非独立聚合，避免跨聚合事务

#### 3.2.2 缺陷工单扩展（策略模式）

缺陷工单在通用工单基础上扩展了分角色信息区，建议采用**策略模式**而非继承：

```
Ticket（通用工单聚合根）
└── BugTicketExtension（缺陷扩展 - 可选组合）
    ├── BugCustomerInfo    值对象 - 客服信息（商户编号、公司名称、问题描述等）
    ├── BugTestInfo        值对象 - 测试信息（复现步骤、环境、缺陷等级等）
    └── BugDevInfo         值对象 - 开发信息（缺陷原因、修复方案、关联分支等）
```

**存储策略**：缺陷扩展信息拆分为三张独立表（`ticket_bug_info`、`ticket_bug_test_info`、`ticket_bug_dev_info`），与产品方案数据模型一致。通过 `ticket_id` 外键关联，按需加载。

#### 3.2.3 工作流聚合（Workflow Aggregate）

```
WorkflowDefinition（聚合根）
├── WorkflowId            值对象 - 工作流标识
├── Name                  工作流名称
├── Mode                  枚举 - 简单模式/高级模式
├── States[]              实体集合 - 状态定义
│   ├── StateId
│   ├── StateName
│   ├── StateType         枚举 - 初始/中间/终止
│   └── SlaTimerAction    枚举 - 开始计时/暂停/结束
└── Transitions[]         实体集合 - 流转规则
    ├── FromState
    ├── ToState
    ├── AllowedRoles[]    可操作角色
    ├── RequiredFields[]  必填字段校验
    ├── Conditions[]      前置条件（SpEL 表达式）
    └── AutoActions[]     自动化动作
```

**技术决策**：

| 方案 | 说明 | 优劣 | 决策 |
|------|------|------|------|
| 自研状态机 | 基于数据库配置驱动的状态机 | 轻量可控，但高级功能需自行实现 | **第一期采用** |
| Spring Statemachine | Spring 官方状态机框架 | 功能完善但配置复杂，与 Spring 强绑定 | 备选 |
| Flowable / Camunda | 成熟的 BPMN 流程引擎 | 功能极强但太重，学习成本高 | **不推荐** |

**自研状态机设计思路**：

```java
// 核心接口设计
public interface WorkflowEngine {
    
    /**
     * 执行状态流转
     * @return 流转结果（包含新状态、触发的自动化动作）
     */
    TransitionResult transit(Ticket ticket, TransitionCommand command);
    
    /**
     * 获取当前状态可用的操作列表（基于角色和条件过滤）
     */
    List<AvailableAction> getAvailableActions(Ticket ticket, User currentUser);
    
    /**
     * 校验流转是否合法
     */
    ValidationResult validate(Ticket ticket, TransitionCommand command);
}
```

状态流转规则存储在 `workflow` 表的 `states` 和 `transitions` JSON 字段中，引擎在运行时解析 JSON 配置驱动流转。

### 3.3 领域事件设计

系统中大量的跨上下文协作通过**领域事件**实现异步解耦：

| 领域事件 | 发布上下文 | 消费上下文 | 触发场景 |
|---------|-----------|-----------|---------|
| `TicketCreatedEvent` | 工单 | 工作流、SLA、通知 | 工单创建成功 |
| `TicketStatusChangedEvent` | 工单 | SLA、通知、统计 | 工单状态变更 |
| `TicketAssignedEvent` | 工单 | 通知 | 工单分派/转派 |
| `TicketReadEvent` | 工单 | 时间追踪 | 处理人首次打开工单 |
| `SlaWarningEvent` | SLA | 通知 | SLA 即将超时 |
| `SlaBreachedEvent` | SLA | 通知、工单 | SLA 已超时 |
| `TicketCompletedEvent` | 工单 | 简报 | 缺陷工单修复完成 |
| `BugReportSubmittedEvent` | 简报 | 通知 | 简报提交审核 |
| `BugReportReviewedEvent` | 简报 | 通知 | 简报审核完成 |
| `WecomMessageReceivedEvent` | 企微 | 工单 | 群机器人收到消息 |

**事件实现方案**：

- **进程内**：Spring ApplicationEvent（第一期，简单可靠）
- **进程间**：RabbitMQ（第二期引入，支持重试和死信）

```java
// 第一期：Spring 事件发布
@Service
public class TicketApplicationService {
    
    private final ApplicationEventPublisher eventPublisher;
    
    @Transactional
    public TicketId createTicket(CreateTicketCommand command) {
        Ticket ticket = ticketFactory.create(command);
        ticketRepository.save(ticket);
        
        eventPublisher.publishEvent(new TicketCreatedEvent(
            ticket.getId(), ticket.getCategoryId(), ticket.getPriority()
        ));
        
        return ticket.getId();
    }
}
```

### 4.2 数据量评估与容量规划

| 数据维度 | 预估（1年） | 预估（3年） | 增长模式 |
|---------|------------|------------|---------|
| 工单数 | ~5,000~10,000 | ~20,000~30,000 | 线性增长 |
| 时间追踪记录 | ~100,000 | ~300,000~500,000 | 随工单线性增长 |
| 评论/处理记录 | ~50,000 | ~150,000 | 随工单线性增长 |
| 通知记录 | ~500,000 | ~1,500,000 | 高频，考虑归档 |
| 附件存储 | ~50GB | ~150GB | 与工单正相关 |

**结论**：以内部系统的数据量级，MySQL 单库完全可以承载 3~5 年的数据增长。**不需要分库分表**。仅需对 `notification` 表和 `ticket_time_track` 表制定定期归档策略（如 6 个月以上数据迁移至归档表）。

### 4.3 自定义字段存储方案

产品方案要求支持 10 种自定义字段类型，这是系统灵活性的关键。评估两种方案：

| 方案 | 存储方式 | 查询能力 | 灵活性 | 性能 |
|------|---------|---------|--------|------|
| **方案 A：EAV 表** | `ticket_custom_field(ticket_id, field_key, field_value)` | 需拼接查询 | 高 | 中 |
| **方案 B：JSON 列** | `ticket.custom_fields JSON` | MySQL 8.0 JSON 函数查询 | 高 | 中高 |

**决策**：采用 **方案 B（JSON 列）** 作为主存储 + **方案 A（EAV 表）** 作为搜索索引的混合方案。

**理由**：
- 自定义字段主要用于展示和填写，按工单 ID 读取时 JSON 列性能优于 EAV 多行查询
- MySQL 8.0 原生支持 JSON 索引（`JSON_EXTRACT` + 生成列 + 索引）
- 对于需要按自定义字段筛选的场景，将热点字段同步写入 EAV 表作为搜索索引

```sql
-- ticket 表增加 JSON 列
ALTER TABLE ticket ADD COLUMN custom_fields JSON DEFAULT NULL;

-- 需要按自定义字段筛选时，创建虚拟生成列 + 索引
ALTER TABLE ticket 
ADD COLUMN severity_level VARCHAR(20) 
    GENERATED ALWAYS AS (JSON_UNQUOTE(JSON_EXTRACT(custom_fields, '$.severity_level'))) VIRTUAL,
ADD INDEX idx_severity_level (severity_level);
```

```java
// 伪代码 - 状态流转核心
@Service
public class WorkflowEngineImpl implements WorkflowEngine {

    @Transactional
    public TransitionResult transit(Ticket ticket, TransitionCommand cmd) {
        // 1. 加载工作流定义
        WorkflowDefinition workflow = workflowRepository.findById(ticket.getWorkflowId());
        
        // 2. 查找匹配的流转规则
        Transition transition = workflow.findTransition(
            ticket.getStatus(), cmd.getTargetStatus()
        );
        
        // 3. 权限校验：当前用户是否有操作权限
        if (!transition.isAllowedFor(cmd.getOperator().getRole())) {
            throw BusinessException.of(ErrorCode.FORBIDDEN);
        }
        
        // 4. 前置校验：必填字段、业务条件
        ValidationResult validation = transition.validate(ticket, cmd);
        if (!validation.isValid()) {
            throw BusinessException.of(ErrorCode.PARAM_ERROR, validation.getMessage());
        }
        
        // 5. 执行状态变更
        String oldStatus = ticket.getStatus();
        ticket.changeStatus(cmd.getTargetStatus(), cmd.getOperator());
        
        // 6. 执行自动化动作（通知、字段更新等）
        List<AutoAction> autoActions = transition.getAutoActions();
        autoActions.forEach(action -> action.execute(ticket));
        
        // 7. 记录时间追踪
        ticketTimeTracker.record(ticket, cmd);
        
        // 8. 发布领域事件
        eventPublisher.publishEvent(new TicketStatusChangedEvent(
            ticket.getId(), oldStatus, cmd.getTargetStatus(), cmd.getOperator()
        ));
        
        return TransitionResult.success(ticket);
    }
}
```

#### 5.2.3 SLA 检查方案

| 方案 | 实现 | 精度 | 复杂度 |
|------|------|------|--------|
| 定时轮询 | `@Scheduled` 每分钟扫描运行中的 SLA | 1 分钟 | 低 |
| 延迟消息 | 创建工单时向 MQ 发送延迟消息（到期触发） | 精确 | 中 |
| Redis 有序集合 | 将超时时间作为 score 存入 ZSet，定时弹出 | 精确 | 中 |

**决策**：第一期采用**定时轮询**（每分钟执行），第二期引入 RabbitMQ 后升级为**延迟消息**。

```java
// 第一期 SLA 检查任务
@Scheduled(fixedRate = 60_000)
public void checkSlaTimers() {
    List<SlaTimer> runningTimers = slaTimerRepository.findRunning();
    
    for (SlaTimer timer : runningTimers) {
        long elapsed = workingTimeCalculator.calculateWorkingMinutes(
            timer.getStartAt(), LocalDateTime.now()
        );
        
        double percentage = (double) elapsed / timer.getThresholdMinutes() * 100;
        
        if (percentage >= 100 && !timer.isBreached()) {
            timer.markBreached();
            eventPublisher.publishEvent(new SlaBreachedEvent(timer));
        } else if (percentage >= 75 && !timer.isWarned()) {
            timer.markWarned();
            eventPublisher.publishEvent(new SlaWarningEvent(timer));
        }
    }
}

### 5.3 企业微信集成

#### 5.3.1 技术风险分析

| 风险点 | 风险等级 | 说明 | 应对措施 |
|--------|---------|------|---------|
| access_token 过期 | 中 | 有效期 2 小时，频繁请求可能触发限流 | Redis 缓存 + 过期前 5 分钟刷新 + 失败自动重试 |
| 回调 5 秒超时 | **高** | 企微回调要求 5 秒内响应，复杂消息解析可能超时 | 回调仅做签名验证和消息入队，异步处理 |
| 消息加解密 | 中 | AES-256-CBC 加解密错误会导致消息丢失 | 使用企微官方 SDK 封装加解密 |
| 重复回调 | 中 | 企微可能重复推送同一消息 | 基于 MsgId 去重（Redis SET + 过期） |
| 群机器人消息解析 | 中 | 自然语言格式解析可能出错 | 严格模板匹配 + 友好的错误提示 + 帮助指令 |

#### 5.3.2 回调服务架构

```

#### 5.3.3 access_token 管理

```java
// access_token 缓存管理
@Service
public class WecomTokenManager {
    
    private static final String TOKEN_KEY = "wecom:access_token";
    private static final long REFRESH_BEFORE_EXPIRE = 300; // 提前5分钟刷新
    
    private final StringRedisTemplate redis;
    private final WecomApiClient wecomApi;
    
    public String getAccessToken() {
        String token = redis.opsForValue().get(TOKEN_KEY);
        if (token != null) {
            return token;
        }
        return refreshToken();
    }
    
    private synchronized String refreshToken() {
        // 双重检查
        String token = redis.opsForValue().get(TOKEN_KEY);
        if (token != null) return token;
        
        WecomTokenResponse resp = wecomApi.getToken(corpId, secret);
        redis.opsForValue().set(TOKEN_KEY, resp.getAccessToken(),
            resp.getExpiresIn() - REFRESH_BEFORE_EXPIRE, TimeUnit.SECONDS);
        return resp.getAccessToken();
    }
}
```

    private static final Pattern BUG_PATTERN = Pattern.compile(
        "^#缺陷\\s*\\n" +
        "商户编号[：:]\\s*(.+)\\n" +
        "公司名称[：:]\\s*(.+)\\n" +
        "商户账号[：:]\\s*(.+)\\n" +
        "问题描述[：:]\\s*(.+)\\n" +
        "预期结果[：:]\\s*(.+)\\n" +
        "(?:场景码[：:]\\s*(.+))?",
        Pattern.DOTALL
    );
    
    // 通用工单模板解析
    private static final Pattern GENERAL_PATTERN = Pattern.compile(
        "^#(.+?)\\s+(.+?)$\\n?" +
        "(?:优先级[：:]\\s*(.+?)\\n)?" +
        "(?:描述[：:]\\s*(.+))?",
        Pattern.MULTILINE | Pattern.DOTALL
    );
    
    public ParseResult parse(String content) {
        // 1. 优先匹配缺陷模板
        Matcher bugMatcher = BUG_PATTERN.matcher(content);
        if (bugMatcher.find()) {
            return parseBugTicket(bugMatcher);
        }
        
        // 2. 匹配通用工单模板
        Matcher generalMatcher = GENERAL_PATTERN.matcher(content);
        if (generalMatcher.find()) {
            return parseGeneralTicket(generalMatcher);
        }
        
        // 3. 解析失败
        return ParseResult.failure("消息格式无法识别");
    }
}

#### 5.4.1 阅读时间记录方案

产品方案要求记录"首次阅读时间"，实现需前后端配合：

```java
@PostMapping("/api/tickets/{id}/track/read")

    Long currentUserId = SecurityContext.getCurrentUserId();
    ticketTimeTracker.recordRead(id, currentUserId);

}
```

**注意事项**：
- 防刷：同一用户对同一工单的同一节点，仅记录首次打开（`is_first_read=true`）
- 性能：使用 Redis `SETNX` 做去重，避免高频写入 DB
- 异步：阅读事件写入 MQ 异步持久化，不阻塞用户操作

#### 5.4.2 耗时统计方案

`ticket_node_duration` 汇总表的更新策略：

| 方案 | 实现 | 优劣 |
|------|------|------|
| 实时更新 | 每次流转时重新计算并更新汇总表 | 实时性好，但计算逻辑分散 |
| 事件驱动 | 监听 `TicketStatusChangedEvent`，异步更新汇总表 | 解耦，但有短暂延迟 |
| 查询时计算 | 不存汇总表，查询时从 `ticket_time_track` 实时聚合 | 简单但查询慢 |

**决策**：采用**事件驱动**方案。流转事件触发时异步计算并更新 `ticket_node_duration`，兼顾实时性和解耦。

| 性能 | 500 人并发、500ms 响应时间目标可达 |
| 最大挑战 | 工作流引擎自研 + SLA 工作时间计算 + 企微回调可靠性 |

### 11.2 关键架构决策汇总

| # | 决策 | 选择 | 理由 |
|---|------|------|------|
| AD-1 | 架构风格 | 模块化单体 | 用户规模有限，微服务过度设计 |
| AD-2 | 工作流引擎 | 自研状态机 | 轻量可控，Flowable 太重 |
| AD-3 | 消息队列 | RabbitMQ | SLA 延迟消息 + 通知路由 + 消息可靠性 |
| AD-4 | 文件存储 | MinIO | S3 兼容、Docker 部署、未来可迁移云存储 |
| AD-5 | 自定义字段存储 | JSON 列 + EAV 索引混合 | 兼顾灵活性和查询性能 |
| AD-6 | SLA 检查 | 定时轮询（一期）→ 延迟消息（二期） | 渐进式升级 |
| AD-7 | 企微回调 | 异步处理 + MQ 入队 | 5 秒超时硬约束 |
| AD-8 | 领域事件 | Spring Event（一期）→ MQ（二期） | 渐进式解耦 |
| AD-9 | 数据库迁移 | Flyway | DDL 版本化管理 |
| AD-10 | 认证方案 | 企微 OAuth2.0 + 双 Token JWT | 零密码登录 + 安全刷新 |
