## 1. 需求背景与目标

### 1.1 背景

当前仪表盘（`/dashboard`）采用固定布局，展示工单概览、趋势、分类分布、效率指标、SLA 达成率、人员工作量共 **6 大模块**，布局顺序和位置由前端代码硬编码决定，无法自定义。

不同用户的工作职责和关注侧重不同：处理人更关注自己的工作量和待处理工单，管理员更关注 SLA 和整体效率，运营人员更关注趋势和分类分布。**每个用户应能按自己的习惯自主调整仪表盘模块排列顺序**，调整结果只影响自己，不影响他人。

### 1.2 核心目标

| 目标 | 说明 |
|------|------|
| 所有登录用户均可拖拽排序 | 仪表盘页面提供"编辑布局"入口，任意登录用户均可进入编辑模式拖拽调整顺序 |
| 布局个人独立保存 | 每人的布局配置独立存储，互不影响 |
| 布局后端持久化 | 布局存储在后端数据库（关联用户 ID），不依赖浏览器 localStorage |
| 支持恢复系统默认 | 任何用户均可一键恢复为系统预设的默认布局顺序 |
| 首次使用自动初始化 | 用户首次访问仪表盘时，若无个人布局记录，自动按系统默认顺序展示 |

### 1.3 不在本期范围内

- 模块的显示/隐藏控制（本期只做排序，不做显隐）
- 模块尺寸/宽高自定义
- 仪表盘模块内容的自定义（如更换图表类型）
- 管理员强制为他人设置布局
- 布局模板分享功能

## 2. 用户角色与权限说明

本期**所有已登录用户**均拥有相同的布局编辑权限，无角色区分：

| 角色 | 布局编辑权限 | 说明 |
|------|------------|------|
| 系统管理员（ADMIN） | ✅ 可编辑自己的仪表盘布局 | 个人布局，不影响他人 |
| 工单管理员（TICKET_ADMIN） | ✅ 可编辑自己的仪表盘布局 | 同上 |
| 处理人（HANDLER） | ✅ 可编辑自己的仪表盘布局 | 同上 |
| 提交人（SUBMITTER） | ✅ 可编辑自己的仪表盘布局 | 同上 |
| 观察者（OBSERVER） | ✅ 可编辑自己的仪表盘布局 | 同上 |

**后端鉴权说明**：布局相关接口只需校验用户已登录（JWT 有效），通过 JWT 中的 `userId` 隔离各用户数据，无需额外角色校验。

## 3. 功能需求详述

### 3.1 查看仪表盘（所有登录用户）

**描述**：用户进入仪表盘时，按其个人保存的布局配置渲染各模块。

**触发条件**：用户访问 `/dashboard` 路由。

**流程**：
1. 前端调用 `GET /api/dashboard/layout` 获取当前用户的模块排列顺序；
2. 后端从 `dashboard_user_layout` 表查询该用户（`user_id`）的布局记录；
3. 若该用户无个人布局记录，返回系统默认排序（见 §4.2）；
4. 前端按返回的 `sortOrder` 值渲染模块；
5. 若接口异常，使用前端内置的默认顺序降级展示，不影响数据接口调用；
6. 各模块数据接口并行请求（现有 API000405–API000410 不变）。

### 3.2 进入布局编辑模式

**描述**：任意登录用户在仪表盘右上角看到"编辑布局"按钮，点击后进入编辑模式。

**进入编辑模式的视觉变化**：
- 按钮切换为"保存布局"和"取消"两个按钮，同时显示"恢复默认"按钮（文字链样式）；
- 仪表盘切换为编辑态：每个可拖拽行组顶部出现拖拽把手图标（`☰` grip icon）；
- 可拖拽行组出现高亮边框（蓝色虚线 `1px dashed #1675d1`）；
- 页面顶部出现提示条："拖拽模块可调整顺序，仅对您自己生效"；
- 各模块内的数据正常展示（不影响查看）。

### 3.3 拖拽排序（编辑模式）

**描述**：用户在编辑模式下，通过拖拽把手对行组模块进行重新排序。

**交互规则**：
- 拖拽单位：以**行组**为整体拖拽（见 §4.1，一个行组包含 1 或 2 个模块卡片）；
- 统计卡片行（6 个小卡片，`overview`）作为固定第一行，**不参与拖拽排序，始终置顶**；
- 拖拽指示：被拖行组半透明（`opacity: 0.5`），目标位置出现蓝色 2px 横线占位；
- 拖拽范围：只能在可拖拽行组之间互换位置，不能拖到 `overview` 行之上；
- 拖拽结果仅在本地生效，**点击"保存布局"后才持久化**。

### 3.4 保存布局

**描述**：用户点击"保存布局"，将当前拖拽后的排序持久化到后端。

**流程**：
1. 前端收集当前模块排序（`rowGroupKey` + `sortOrder`）；
2. 调用 `PUT /api/dashboard/layout` 提交排序数据（携带用户 JWT）；
3. 后端以 `user_id`（从 JWT 解析）为 Key，覆盖写入该用户的布局配置；
4. 成功后：提示"布局已保存"，退出编辑模式；
5. 失败后：提示错误信息，**不退出编辑模式**，保留当前拖拽状态；
6. 其他用户：不受任何影响，各自展示自己的布局。

### 3.5 取消编辑

**描述**：用户点击"取消"按钮，放弃本次拖拽修改。

**逻辑**：
- 前端恢复为进入编辑模式前的布局顺序（从进入时的快照还原）；
- 不调用任何后端接口；
- 退出编辑模式。

### 3.6 恢复默认布局

**描述**：用户在编辑模式下点击"恢复默认"，清除个人自定义布局。

**流程**：
1. 弹出确认对话框："确认恢复为系统默认布局？您当前的自定义布局将被清除。"；
2. 用户确认后调用 `DELETE /api/dashboard/layout`；
3. 后端删除该用户的个人布局记录（软删除）；
4. 前端重新按系统默认顺序渲染；
5. 退出编辑模式，提示"已恢复默认布局"。

## 4. 模块定义与布局规则

### 4.1 可配置行组清单

仪表盘以**行组**为拖拽排序单位，每个行组内的列布局固定不变：

| 行组 Key | 显示名称 | 包含模块 | 默认排序 | 是否可拖拽 |
|----------|---------|---------|---------|----------|
| `overview` | 工单概览 | 6 张统计卡 | 0（固定置顶） | ❌ 不可拖拽 |
| `trend_category` | 趋势 + 分类 | 工单趋势（左14列）+ 分类分布（右10列） | 1 | ✅ |
| `efficiency_workload` | 效率 + 工作量 | 处理效率与SLA（左10列）+ 人员工作量TOP10（右14列） | 2 | ✅ |

> **扩展说明**：未来新增模块时，新建一个行组 Key，在数据库 `dashboard_default_layout` 表插入默认排序记录，前端新增对应行组渲染组件，无需改动排序逻辑。

### 4.2 系统默认布局顺序

| 行组 Key | sort_order | 说明 |
|----------|-----------|------|
| `overview` | 0 | 固定，不可拖拽 |
| `trend_category` | 1 | 默认第一可拖拽行 |
| `efficiency_workload` | 2 | 默认第二可拖拽行 |

### 4.3 布局配置优先级

```
用户个人布局记录（dashboard_user_layout）
      ↓ 若不存在
系统默认布局顺序（dashboard_default_layout 表或代码内置常量）
      ↓ 若接口异常
前端代码内置 fallback 顺序（与系统默认一致）
```

### 5.3 移动端处理

- 移动端（屏幕宽度 < 768px）**不显示"编辑布局"按钮**，不支持拖拽排序；
- 移动端按该用户已保存的布局顺序展示（只读）；
- 未来可扩展支持移动端长按拖拽，本期不做。

### 5.4 数据隔离说明

- `GET /api/dashboard/layout`：后端从 JWT 解析 `userId`，只返回该用户的布局；
- `PUT /api/dashboard/layout`：只更新当前登录用户的布局；
- `DELETE /api/dashboard/layout`：只删除当前登录用户的布局记录；
- 后端无"读取他人布局"的接口，彻底隔离。

## 6. 数据设计

### 6.1 数据库表设计

#### `dashboard_user_layout`（用户个人仪表盘布局配置表）

```sql
CREATE TABLE `dashboard_user_layout` (
  `id`            bigint(20)   NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `user_id`       bigint(20)   NOT NULL COMMENT '用户ID（关联 sys_user.id）',
  `row_group_key` varchar(64)  NOT NULL COMMENT '行组标识Key（overview / trend_category / efficiency_workload）',
  `sort_order`    int(11)      NOT NULL DEFAULT 0 COMMENT '排列顺序（数值越小越靠前）',
  `is_fixed`      tinyint(1)   NOT NULL DEFAULT 0 COMMENT '是否固定不可拖拽（1:固定 0:可拖拽）',
  `create_time`   datetime     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `update_time`   datetime     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `create_by`     varchar(50)  NOT NULL DEFAULT '' COMMENT '创建人',
  `update_by`     varchar(50)  NOT NULL DEFAULT '' COMMENT '更新人',
  `deleted`       tinyint(4)   NOT NULL DEFAULT 0 COMMENT '逻辑删除（0:未删除 1:已删除）',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_user_row_group` (`user_id`, `row_group_key`),
  KEY `idx_user_id` (`user_id`),
  KEY `idx_sort_order` (`sort_order`),
  KEY `idx_deleted` (`deleted`)
) ENGINE=InnoDB AUTO_INCREMENT=1 DEFAULT CHARSET=utf8mb4 COMMENT='用户个人仪表盘布局配置表';
```

**字段说明**：
- `user_id`：关联 `sys_user.id`，每个用户独立一套布局记录；
- `row_group_key`：行组标识，允许的值：`overview`、`trend_category`、`efficiency_workload`；
- `sort_order`：排列序号，`overview` 固定为 0，其余从 1 开始；
- `is_fixed`：`overview` 为 1（固定），其余为 0；
- `(user_id, row_group_key)` 联合唯一，防止重复记录。

**不预先为每个用户初始化数据**：用户首次访问时，若无记录，后端返回系统默认顺序；仅在用户**首次主动保存**时才写入数据库。

### 6.2 Redis 缓存键设计

```
dashboard:layout:user:{userId}   →  该用户的布局列表（JSON）
TTL: 30 分钟
失效策略: 用户保存/恢复默认时主动 DEL 该 Key
```

> 使用用户维度缓存（而非全局），避免缓存污染。

### 6.3 系统默认布局（代码常量）

后端 `DashboardLayoutConstants` 类维护系统默认顺序，作为用户无记录时的 fallback：

```java
// 系统默认行组排序
public static final List<DashboardLayoutItemOutput> DEFAULT_LAYOUT = List.of(
    new DashboardLayoutItemOutput("overview",              0, true),
    new DashboardLayoutItemOutput("trend_category",        1, false),
    new DashboardLayoutItemOutput("efficiency_workload",   2, false)
);
```

## 7. 接口设计

### 7.1 接口总览

| 接口编号 | HTTP方法 | 接口路径 | 描述 | 权限 |
|---------|---------|---------|------|------|
| API000411 | GET | `/api/dashboard/layout` | 获取当前用户个人布局配置 | 所有登录用户 |

| API000413 | DELETE | `/api/dashboard/layout` | 恢复当前用户布局为系统默认 | 所有登录用户 |

### 7.2 接口详情

---

#### API000411 · 获取用户个人仪表盘布局配置

```
GET /api/dashboard/layout
Authorization: Bearer {token}
```

**请求参数**：无（用户身份从 JWT 获取）

**响应体**：

```json
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "rowGroupKey": "overview",
      "sortOrder": 0,
      "isFixed": true
    },
    {
      "rowGroupKey": "efficiency_workload",
      "sortOrder": 1,
      "isFixed": false
    },
    {
      "rowGroupKey": "trend_category",
      "sortOrder": 2,
      "isFixed": false
    }
  ]
}
```

**业务规则**：
- 从 JWT 解析 `userId`；
- 查询 `dashboard_user_layout` 表中该用户的记录；
- 若无记录，返回系统默认布局（代码常量）；
- 返回结果按 `sortOrder` 升序排列；
- 支持 Redis 缓存（Key：`dashboard:layout:user:{userId}`，TTL 30 分钟）。

---

#### API000412 · 保存用户个人仪表盘布局配置

```
PUT /api/dashboard/layout
Authorization: Bearer {token}
Content-Type: application/json
```

**请求体**：

```json
{
  "layouts": [
    { "rowGroupKey": "overview",              "sortOrder": 0 },
    { "rowGroupKey": "efficiency_workload",   "sortOrder": 1 },
    { "rowGroupKey": "trend_category",        "sortOrder": 2 }
  ]
}
```

**请求体字段说明**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `layouts` | Array | 是 | 布局排序列表，需包含全部行组 Key |
| `layouts[].rowGroupKey` | String | 是 | 行组 Key，须为已知值 |
| `layouts[].sortOrder` | Integer | 是 | 排列顺序，0 起始，同一用户内不可重复 |

**响应体**：

```json
{
  "code": 200,
  "message": "布局保存成功",
  "data": null
}
```

**业务规则**：
- 从 JWT 解析 `userId`；
- `overview` 的 `sortOrder` 强制校正为 0，`is_fixed` 为 1（即使请求中传了其他值也忽略）；
- `rowGroupKey` 不在枚举范围内返回 400；
- 采用**覆盖写入**：先软删除该用户旧记录，再批量插入新记录（事务保障）；
- 成功后主动删除 Redis Key `dashboard:layout:user:{userId}`。

---

#### API000413 · 恢复用户仪表盘为系统默认布局

```
DELETE /api/dashboard/layout
Authorization: Bearer {token}
```

**请求参数**：无

**响应体**：

```json
{
  "code": 200,
  "message": "已恢复默认布局",
  "data": null
}
```

**业务规则**：
- 从 JWT 解析 `userId`；
- 软删除 `dashboard_user_layout` 表中该用户所有记录（`deleted = 1`）；
- 删除 Redis Key `dashboard:layout:user:{userId}`；
- 前端收到成功响应后重新调用 `GET /api/dashboard/layout`（此时将返回系统默认顺序）。

## 8. 非功能性需求

| 项目 | 要求 |
|------|------|
| 性能 | `GET /api/dashboard/layout` 响应 ≤ 100ms（Redis 命中）；≤ 200ms（DB 查询，单用户少量行） |
| 并发 | 多用户同时保存各自布局，数据库以 `user_id` 隔离，无锁竞争 |
| 可用性 | 布局接口失败时前端降级展示默认顺序，仪表盘数据正常加载 |
| 安全 | 所有接口强制 JWT 鉴权；用户只能读写自己的布局，无越权风险 |
| 兼容性 | 桌面端支持 Chrome/Edge 最新 2 个大版本；移动端只读不支持拖拽 |
| 扩展性 | 新增行组只需加枚举值和前端渲染组件，无需改动排序核心逻辑 |
| 存储 | 每用户最多 10 条布局记录（按当前行组数量），存储开销极小 |

## 10. 验收标准

### 10.1 功能验收

| 验收点 | 验收条件 |
|--------|---------|
| 所有用户显示编辑入口 | 任意登录用户均可看到"编辑布局"按钮 |
| 进入编辑模式 | 提示条文字含"仅对您自己生效"；出现拖拽把手和蓝色虚线边框 |
| 拖拽排序 | `overview` 行固定；其余行组可拖拽；有视觉反馈（半透明+占位线） |
| 保存布局 | 后端按 `userId` 写入；刷新后该用户看到新布局；其他用户不受影响 |
| 数据隔离 | 用户A保存布局后，用户B仪表盘不变；用户B独立调整不影响用户A |
| 恢复默认 | 弹出确认框；确认后删除个人记录；刷新后显示系统默认顺序 |
| 首次访问 | 新用户（无布局记录）进入仪表盘，按系统默认顺序展示 |
| 取消编辑 | 不调用接口；恢复进入编辑前的顺序 |
| 布局降级 | 布局接口 500 时，仪表盘按默认顺序展示，无白屏 |
| 移动端 | 不显示"编辑布局"按钮；按已保存布局只读展示 |

### 10.2 性能验收

| 验收点 | 目标值 |
|--------|--------|
| 布局接口响应时间（Redis 命中） | ≤ 100ms |
| 布局接口响应时间（DB 查询） | ≤ 200ms |
| 保存布局操作耗时 | ≤ 500ms |
| 仪表盘首屏数据加载时间 | ≤ 2s（布局+数据并行加载） |
