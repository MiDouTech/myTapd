## 1. 需求背景与目标

### 1.1 背景

当前仪表盘（`/dashboard`）采用固定布局，展示工单概览、趋势、分类分布、效率指标、SLA 达成率、人员工作量共 **6 大模块**，布局顺序和位置由前端代码硬编码决定，无法自定义。

随着系统在不同团队中使用，不同管理员对各模块的关注优先级存在差异（有人首要关注 SLA，有人首要关注趋势），需要支持**仪表盘模块的个性化排布**，同时要求管理员调整后的布局**统一同步至所有账号**，确保全局一致性。

### 1.2 核心目标

| 目标 | 说明 |
|------|------|
| 管理员可拖拽排序 | 仪表盘页面顶部提供"编辑布局"入口，管理员进入编辑模式后可拖拽各模块调整顺序 |
| 布局全局生效 | 管理员保存布局后，所有登录账号下次进入仪表盘时均使用最新布局 |
| 布局实时持久化 | 布局配置存储在后端数据库，不依赖浏览器 localStorage |
| 支持恢复默认 | 提供"恢复默认布局"按钮，可一键还原系统预设布局 |
| 普通用户只读 | 非管理员用户仪表盘按当前全局布局展示，无拖拽入口 |

### 1.3 不在本期范围内

- 模块的显示/隐藏控制（本期只做排序，不做显隐）
- 模块尺寸/宽高自定义
- 个人级别的私有布局（本期只有全局布局）
- 仪表盘模块内容的自定义（如更换图表类型）

## 2. 用户角色与权限说明

| 角色 | 权限 | 说明 |
|------|------|------|
| 系统管理员（admin） | 可进入编辑模式、拖拽排序、保存布局、恢复默认 | 当前系统角色 `ROLE_ADMIN` |
| 普通用户 | 只能查看仪表盘，布局只读 | 按全局布局展示 |

**判断管理员的逻辑**：后端通过 JWT 中的角色信息判断，前端从 `userStore` 中的 `roles` 字段读取，包含 `ROLE_ADMIN` 时显示编辑入口。

## 3. 功能需求详述

### 3.1 查看仪表盘（所有用户）

**描述**：用户进入仪表盘时，按当前全局布局配置渲染各模块，布局配置从后端接口获取。

**触发条件**：用户访问 `/dashboard` 路由。

**流程**：
1. 前端调用 `GET /api/dashboard/layout` 获取当前模块排列顺序；
2. 按返回的 `sortOrder` 值对模块进行排序渲染；
3. 若接口异常或返回为空，使用系统默认排序（见 §4.2）；
4. 各模块数据接口并行请求（现有 API000405–API000410 不变）。

### 3.2 进入布局编辑模式（管理员）

**描述**：管理员在仪表盘右上角看到"编辑布局"按钮，点击后进入编辑模式。

**进入编辑模式的视觉变化**：
- 按钮切换为"保存布局"和"取消"两个按钮；
- 仪表盘切换为编辑态：每个模块卡片顶部出现拖拽把手图标（`☰` grip icon）；
- 模块卡片出现高亮边框（蓝色虚线 `1px dashed #1675d1`）；
- 页面顶部出现提示条："拖拽模块可调整顺序，保存后对所有账号生效"；
- 各模块内的数据正常展示（不影响查看）。

### 3.3 拖拽排序（管理员·编辑模式）

**描述**：管理员在编辑模式下，可通过拖拽把手对模块进行重新排序。

**交互规则**：
- 拖拽目标：每个模块卡片整体可拖拽；
- 拖拽指示：被拖拽的卡片半透明（opacity: 0.6），目标位置出现蓝色占位线；
- 布局结构：仪表盘为**行式布局**（每行可包含 1 或多列），拖拽以整行为排序单位；
- 统计卡片行（6 个小卡片）作为固定第一行，**不参与拖拽排序**，始终置顶；
- 可拖拽的模块为以下 5 个区块（见 §4.1）：
  - `trend`：工单趋势
  - `category`：分类分布
  - `efficiency`：处理效率与 SLA
  - `workload`：人员工作量 TOP10
  - *(后续扩展留位)*

**注意**：工单趋势（`trend`）与分类分布（`category`）在默认布局中同属一行（左右排列），人员工作量（`workload`）与处理效率（`efficiency`）同属一行。拖拽以**行组**为单位：每次拖拽移动的是整个行组。

### 3.4 保存布局（管理员）

**描述**：管理员点击"保存布局"后，当前拖拽后的顺序持久化到后端。

**流程**：
1. 前端收集当前模块排序（`moduleKey` + `sortOrder`）；
2. 调用 `PUT /api/dashboard/layout` 提交排序数据；
3. 后端更新全局布局配置（覆盖写入）；
4. 成功后：提示"布局已保存，对所有账号立即生效"，退出编辑模式；
5. 失败后：提示错误信息，**不退出编辑模式**，保留用户当前拖拽状态；
6. 其他在线用户：下次刷新/访问仪表盘时生效（不强制推送）。

### 3.5 取消编辑（管理员）

**描述**：管理员点击"取消"按钮，放弃本次拖拽修改。

**逻辑**：
- 前端直接恢复为进入编辑模式前的布局顺序；
- 不调用任何后端接口；
- 退出编辑模式。

### 3.6 恢复默认布局（管理员）

**描述**：管理员在编辑模式下可点击"恢复默认"按钮。

**流程**：
1. 弹出确认对话框："确认恢复为系统默认布局？该操作将对所有账号生效。"；
2. 用户确认后调用 `DELETE /api/dashboard/layout`；
3. 后端删除自定义布局记录（或将所有模块 `sortOrder` 重置为默认值）；
4. 前端重新按默认顺序渲染；
5. 退出编辑模式，提示"已恢复默认布局"。

## 4. 模块定义与布局规则

### 4.1 可配置模块清单

| 模块 Key | 显示名称 | 所属行组（默认） | 所占列宽 |
|----------|---------|--------------|---------|
| `overview` | 工单概览（6张统计卡） | 行组 0（固定，不可拖拽） | 全行 24 列 |
| `trend` | 工单趋势（近14天） | 行组 1（左侧） | 14/24 列（lg: 14） |
| `category` | 分类分布 | 行组 1（右侧） | 10/24 列（lg: 10） |
| `efficiency` | 处理效率与SLA | 行组 2（左侧） | 10/24 列（lg: 10） |
| `workload` | 人员工作量 TOP10 | 行组 2（右侧） | 14/24 列（lg: 14） |

> **说明**：行组（rowGroup）为两个模块的组合。每个行组的内部列布局固定，行组之间支持上下拖拽排序。

### 4.2 默认布局顺序

系统默认布局（`sortOrder` 从 1 开始）：

```
sortOrder=0  → overview（固定，不参与排序）
sortOrder=1  → 行组1：trend（左）+ category（右）
sortOrder=2  → 行组2：efficiency（左）+ workload（右）
```

用数据库记录表示：

| module_key | sort_order |
|------------|-----------|
| `overview` | 0 |
| `trend_category` | 1 |
| `efficiency_workload` | 2 |

> **简化设计**：行组以组合 Key 存储（`trend_category`、`efficiency_workload`），这样前端解析时直接按行分组渲染，每个行组内的列布局不变。

### 4.3 扩展说明

未来若新增模块，在数据库插入新的 `module_key` 记录，设置最大 `sort_order`，前端增加对应渲染组件即可，不需要改动排序逻辑。

## 5. 交互设计说明

### 5.1 页面布局结构（编辑态）

```
┌─────────────────────────────────────────────────────────────┐
│ 仪表盘                    [编辑布局] / [保存布局][取消][恢复默认] │
├─────────────────────────────────────────────────────────────┤
│ ⚠ 提示条（编辑模式）：拖拽调整模块顺序，保存后对所有账号生效         │
├─────────────────────────────────────────────────────────────┤
│ ┌──────────────────────────────────────────────────────────┐│
│ │ ☰ 工单概览（固定，不可拖拽）                                ││
│ │ [待受理7] [处理中3] [已挂起0] [已完成2] [SLA超时0] [总量12] ││
│ └──────────────────────────────────────────────────────────┘│
│                                                             │
│ ┌──────────────────────────────────────────────────────────┐│  ← 可拖拽行组（拖拽把手在标题区）
│ │ ☰ 工单趋势（近14天）         │ ☰ 分类分布                  ││
│ │  [趋势进度条...]             │  [分类进度条...]              ││
│ └──────────────────────────────────────────────────────────┘│
│                                                             │
│ ┌──────────────────────────────────────────────────────────┐│  ← 可拖拽行组
│ │ ☰ 处理效率与SLA              │ ☰ 人员工作量 TOP10           ││
│ │  [环形图...]                 │  [表格...]                   ││
│ └──────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────┘
```

### 5.2 拖拽交互细节

| 状态 | 视觉表现 |
|------|---------|
| 默认态 | 无拖拽把手，正常渲染 |
| 编辑模式·静止 | 行组顶部显示 `☰` 把手，蓝色虚线边框，鼠标悬停把手时变为 `grab` |
| 拖拽中 | 被拖行组 `opacity: 0.5`，`box-shadow` 加深，跟随鼠标；目标位置出现蓝色 2px 横线占位 |
| 放置后 | 行组吸附到新位置，动画过渡 200ms |
| 保存中 | 按钮 loading 状态，模块不可操作 |

### 5.3 权限控制（前端）

- 从 `userStore.roles` 判断是否包含 `ROLE_ADMIN`；
- 非管理员：不渲染"编辑布局"按钮，`DashboardView` 组件不挂载拖拽相关逻辑；
- 管理员：仪表盘组件根据 `isEditMode` 响应式变量切换编辑/查看两种渲染模式。

### 5.4 全局生效机制

- 布局配置存储于后端 `dashboard_layout_config` 表；
- 所有用户每次访问仪表盘时通过 `GET /api/dashboard/layout` 实时拉取最新配置；
- 无需 WebSocket 推送，采用**请求时拉取**（pull）方式，确保简单可靠；
- 考虑到仪表盘访问频率高，布局配置接口响应需缓存（Redis，TTL 60 秒，管理员保存时主动失效）。

## 6. 数据设计

### 6.1 数据库表设计

#### `dashboard_layout_config`（仪表盘布局配置表）

```sql
CREATE TABLE `dashboard_layout_config` (
  `id`          bigint(20)    NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `module_key`  varchar(64)   NOT NULL COMMENT '模块标识Key（如 trend_category、efficiency_workload）',
  `sort_order`  int(11)       NOT NULL DEFAULT 0 COMMENT '排列顺序（0最顶部，越小越靠前；0为固定overview）',
  `is_fixed`    tinyint(1)    NOT NULL DEFAULT 0 COMMENT '是否固定不可拖拽（1:固定 0:可拖拽）',
  `create_time` datetime      NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `update_time` datetime      NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `create_by`   varchar(50)   NOT NULL DEFAULT '' COMMENT '创建人',
  `update_by`   varchar(50)   NOT NULL DEFAULT '' COMMENT '更新人',
  `deleted`     tinyint(4)    NOT NULL DEFAULT 0 COMMENT '逻辑删除（0:未删除 1:已删除）',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_module_key` (`module_key`),
  KEY `idx_sort_order` (`sort_order`),
  KEY `idx_deleted` (`deleted`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='仪表盘布局配置表';
```

#### 初始化数据（默认布局）

```sql
INSERT INTO `dashboard_layout_config` (`module_key`, `sort_order`, `is_fixed`, `create_by`, `update_by`) VALUES
('overview',              0, 1, 'system', 'system'),
('trend_category',        1, 0, 'system', 'system'),
('efficiency_workload',   2, 0, 'system', 'system');
```

### 6.2 数据模型说明

**模块 Key 设计选择**：

采用"行组 Key"而非"单模块 Key"存储，原因：
1. 当前布局行内列布局固定（trend:14列 + category:10列），无需动态调整列比例；
2. 简化前端渲染逻辑，一个行组 Key 对应一个固定的行渲染组件；
3. 排序操作的粒度与用户视觉感知一致（以行为单位移动）。

**扩展路径**：若未来需要支持模块显隐或单模块排序，可在 `dashboard_layout_config` 表增加 `is_visible` 字段，并将 Key 改为单模块 Key，同时调整前端分组逻辑。

### 6.3 Redis 缓存键设计

```
dashboard:layout:global   →  存储全局布局列表（JSON）
TTL: 60秒
失效策略: 管理员保存/恢复默认时主动 DELETE
```

## 7. 接口设计

### 7.1 接口总览

| 接口编号 | HTTP方法 | 接口路径 | 描述 | 权限 |
|---------|---------|---------|------|------|
| API000411 | GET | `/api/dashboard/layout` | 获取当前全局布局配置 | 所有登录用户 |
| API000412 | PUT | `/api/dashboard/layout` | 保存全局布局配置 | 管理员 |
| API000413 | DELETE | `/api/dashboard/layout` | 恢复默认布局 | 管理员 |

### 7.2 接口详情

#### API000411 · 获取仪表盘布局配置

```
GET /api/dashboard/layout
```

**请求参数**：无

**响应体**：

```json
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "moduleKey": "overview",
      "sortOrder": 0,
      "isFixed": true
    },
    {
      "moduleKey": "trend_category",
      "sortOrder": 1,
      "isFixed": false
    },
    {
      "moduleKey": "efficiency_workload",
      "sortOrder": 2,
      "isFixed": false
    }
  ]
}
```

**业务规则**：
- 按 `sortOrder` 升序返回；
- 接口支持 Redis 缓存（TTL 60s）；
- 未配置时返回系统默认顺序。

---

#### API000412 · 保存仪表盘布局配置

```
PUT /api/dashboard/layout
```

**请求体**：

```json
{
  "layouts": [
    { "moduleKey": "overview",            "sortOrder": 0 },
    { "moduleKey": "efficiency_workload", "sortOrder": 1 },
    { "moduleKey": "trend_category",      "sortOrder": 2 }
  ]
}
```

**请求体字段说明**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `layouts` | Array | 是 | 布局排序列表，须包含全部非固定模块 |
| `layouts[].moduleKey` | String | 是 | 模块 Key，须为已知值 |
| `layouts[].sortOrder` | Integer | 是 | 排列顺序，0 起始，不重复 |

**响应体**：

```json
{
  "code": 200,
  "message": "布局保存成功",
  "data": null
}
```

**业务规则**：
- 仅管理员可调用，否则返回 403；
- `overview` 的 `sortOrder` 固定为 0，提交中若包含 `overview` 则忽略；
- `moduleKey` 不在已知范围内返回 400；
- 保存成功后主动清除 Redis 缓存 `dashboard:layout:global`；
- 采用**覆盖写入**（全量更新），不做增量 diff。

---

#### API000413 · 恢复默认仪表盘布局

```
DELETE /api/dashboard/layout
```

**请求参数**：无

**响应体**：

```json
{
  "code": 200,
  "message": "已恢复默认布局",
  "data": null
}
```

**业务规则**：
- 仅管理员可调用；
- 后端将所有模块的 `sort_order` 重置为系统默认值；
- 保存后主动清除 Redis 缓存；
- 前端收到成功响应后重新调用 `GET /api/dashboard/layout` 刷新。

## 8. 非功能性需求

| 项目 | 要求 |
|------|------|
| 性能 | `GET /api/dashboard/layout` 响应时间 ≤ 100ms（Redis 命中）；≤ 300ms（DB 查询） |
| 并发 | 多用户同时访问仪表盘不影响布局读取，布局保存为串行操作 |
| 可用性 | 布局接口失败时前端降级展示默认顺序，不影响数据展示 |
| 安全 | 管理员权限通过后端 JWT 鉴权，前端展示控制只做 UX 层保护 |
| 兼容性 | 支持 Chrome/Edge 最新 2 个大版本，移动端仅查看（不支持拖拽） |
| 扩展性 | 新增模块只需插入数据库记录 + 前端增加组件，排序逻辑不变 |

## 10. 验收标准

### 10.1 功能验收

| 验收点 | 验收条件 |
|--------|---------|
| 普通用户查看仪表盘 | 不显示"编辑布局"按钮；按当前全局布局渲染各模块 |
| 管理员查看仪表盘 | 显示"编辑布局"按钮 |
| 进入编辑模式 | 出现提示条、拖拽把手、蓝色虚线边框；按钮变为"保存/取消/恢复默认" |
| 拖拽排序 | overview 行固定，其余行组可拖拽；拖拽有视觉反馈（半透明+占位线） |
| 保存布局 | 后端写入成功；其他账号刷新后看到新布局；提示"保存成功" |
| 取消编辑 | 不调用接口，恢复拖拽前顺序 |
| 恢复默认 | 弹出确认框；确认后调用 DELETE 接口；所有账号恢复默认顺序 |
| 布局降级 | `GET /api/dashboard/layout` 接口 500 时，仪表盘按默认顺序展示，无白屏 |
| 移动端 | 仪表盘正常展示，不显示编辑入口 |

### 10.2 性能验收

| 验收点 | 目标值 |
|--------|--------|
| 布局接口响应时间（Redis 命中） | ≤ 100ms |
| 保存布局操作耗时 | ≤ 500ms |
| 仪表盘首屏数据加载时间 | ≤ 2s（含布局+数据并行请求） |
