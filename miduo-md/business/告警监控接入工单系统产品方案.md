# 告警监控接入工单系统产品方案

## 版本信息

| 版本 | 日期 | 变更内容 | 作者 |
|------|------|---------|------|
| v1.0 | 2026-04-01 | 初始版本 | 系统 |

---

## 一、背景

公司运维监控平台（夜莺 Nightingale v7）负责对服务器、数据库、磁盘等基础设施进行实时监控。当监控指标触发告警规则后，当前流程是推送消息到企业微信群，由运维人员手动跟进处理。

**现状问题**：
1. 告警消息仅推送到企微群，缺乏工单化的跟踪闭环
2. 无法明确告警的责任人指派和处理进度
3. 告警可能被遗漏，缺少SLA保障
4. 缺乏历史告警与处理结果的统计分析

## 二、目标

将运维监控平台的告警通过 Webhook 回调方式接入工单系统，实现：
1. **告警自动创建工单**：监控平台触发告警后，自动在工单系统中创建对应工单
2. **基于夜莺推送规则指派处理人**：从夜莺推送的 `notify_users_obj` 中解析通知用户，自动按手机号/邮箱/姓名匹配工单系统用户，支持多人指派
3. **回退处理人**：当夜莺推送用户匹配不到时，使用映射配置中的回退处理人
4. **告警去重**：同一告警事件在短时间窗口内不重复创建工单
5. **告警恢复关联**：告警恢复时自动在工单中记录恢复信息
6. **可配置映射**：支持在工单系统管理后台配置告警规则与工单分类/优先级的映射关系

## 三、非目标

1. 不改变监控平台本身的告警规则配置（在夜莺平台配置）
2. 不替代企微群告警通知（两者并行）
3. 本期不涉及告警聚合降噪（由监控平台自身能力处理）

## 四、接入方式

### 4.1 夜莺 Webhook 回调

- **协议**：夜莺在告警触发时，通过 HTTP POST 方式将告警事件 JSON 发送到工单系统提供的 Webhook 地址
- **超时**：请求超时5秒，失败后每200ms重试，最多3次
- **参考文档**：https://flashcat.cloud/docs/content/flashcat-monitor/nightingale-v7/monitor/notify-webhook/

### 4.2 告警事件关键字段

| 字段 | 类型 | 说明 |
|------|------|------|
| rule_id | int | 告警规则ID |
| rule_name | string | 规则名称（如"磁盘使用率告警"） |
| rule_note | string | 规则备注 |
| severity | int | 告警级别：1=P1, 2=P2, 3=P3 |
| target_ident | string | 监控对象标识（如"md-sh-win12-1-db-AY001"） |
| trigger_value | string | 触发时值（如"98.06949"） |
| trigger_time | int | 告警触发时间戳 |
| first_trigger_time | int | 首次触发时间戳 |
| tags | string[] | 标签列表 |
| tags_map | object | 标签键值对 |
| hash | string | 事件唯一哈希 |
| is_recovered | boolean | 是否为恢复事件 |

## 五、功能设计

### 5.1 Webhook 接收端点

- **路径**：`POST /api/open/alert/nightingale/webhook`
- **鉴权**：通过 URL 参数携带 token 进行简单鉴权（`?token=xxx`）
- **响应**：返回标准 `ApiResult` JSON

### 5.2 告警规则映射配置（管理后台）

在工单系统「设置」菜单下新增「告警接入」配置页面：

| 配置项 | 说明 |
|--------|------|
| 规则名称匹配 | 夜莺告警规则名称（支持精确匹配或前缀匹配） |
| 工单分类 | 告警创建工单时使用的工单分类 |
| 优先级映射 | severity 1/2/3 分别映射的工单优先级 |
| 回退处理人 | 夜莺推送用户匹配不到时使用的回退指派人 |
| 去重时间窗口 | 同一 hash 值在该时间窗口内不重复创建工单（默认30分钟） |
| 是否启用 | 该映射规则是否启用 |

### 5.2.1 处理人指派策略

处理人优先从夜莺告警推送的通知用户（`notify_users_obj`）中自动匹配，匹配规则：
1. 按手机号匹配 → 2. 按邮箱匹配 → 3. 按姓名/昵称匹配
4. 支持多人指派：夜莺推送的多个通知用户均可匹配为工单处理人
5. 全部匹配不到时，使用映射配置中的回退处理人
6. 若回退处理人也未设置，则走自动分派流程

### 5.3 告警去重策略

- 基于告警事件的 `hash` 字段进行去重
- 在可配置的时间窗口内（默认30分钟），同一 hash 不重复创建工单
- 超过时间窗口后再次收到同一告警，创建新工单

### 5.4 告警恢复处理

- 收到 `is_recovered=true` 的事件时，查找关联的未关闭工单
- 在工单中添加评论记录告警已恢复
- 不自动关闭工单（仍需人工确认）

### 5.5 工单内容映射

- **工单标题**：`[告警] {rule_name} - {target_ident}`
- **工单描述**：包含告警级别、规则备注、监控指标、触发值、触发时间等完整信息
- **工单来源**：`ALERT`（新增工单来源类型）
- **优先级映射**：P1→紧急, P2→高, P3→中
- **处理人**：优先从夜莺 `notify_users_obj` 自动匹配（支持多人），匹配不到时使用回退处理人

## 六、验收标准

1. 夜莺配置 Webhook 回调地址后，告警自动创建工单
2. 工单包含完整的告警信息（规则名称、监控对象、触发值、时间等）
3. 同一告警在时间窗口内不重复创建工单
4. 告警恢复事件自动在工单中记录
5. 管理后台可配置告警规则与工单分类的映射关系
6. 未配置映射的告警规则使用默认配置创建工单
