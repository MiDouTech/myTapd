## 一、项目背景与目标

### 1.1 项目背景

当前公司内部跨部门协作主要依赖即时通讯工具（如微信、企业微信）和邮件进行需求提交与问题跟踪，存在以下痛点：

- **信息分散**：需求、问题散落在聊天记录中，难以追溯和统计
- **流程不透明**：提交人无法实时了解处理进度，频繁催问
- **权责不清**：没有明确的分派机制和时效约束，导致工单积压或遗漏
- **缺乏度量**：无法量化各部门的响应效率和处理质量

### 1.2 项目目标

构建一套内部工单系统，实现：

| 目标 | 描述 |
|------|------|
| 流程标准化 | 统一工单提交、流转、处理和关闭的全生命周期流程 |
| 协作高效化 | 自动分派、状态流转提醒、SLA 预警，减少人工干预 |
| 管理可视化 | 提供多维度数据看板，支撑管理决策 |
| 灵活可扩展 | 支持自定义工单类型、字段、工作流，适配不同业务场景 |

## 二、TAPD 平台调研总结

### 2.1 TAPD 核心能力概览

腾讯 TAPD 是沉淀了腾讯 10+ 年研发实践经验的敏捷协作平台，服务 30,000+ 家企业。其核心能力如下：

| 模块 | 能力 |
|------|------|
| 工作项管理 | 需求、任务、缺陷等多类型工作项，支持自定义字段（8 种字段类型：文本、下拉、复选、单选、多行文本、人名、日期、数值） |
| 双工作流引擎 | 敏捷状态机模式（简单模式，状态间任意切换）+ 多分支流程节点模式（高级模式，灵活配置流转规则） |
| 看板（故事墙） | 按工作流状态分列展示，支持自定义卡片信息、状态聚合、多看板切换 |
| 计划管理 | 发布计划 + 迭代管理 + 甘特图，全链路交付管控 |
| 自动化规则 | "触发条件 + 执行动作"编排：逾期预警、状态联动、字段变更监控、定时通知 |
| 报表与仪表盘 | 个人仪表盘、多维度统计报表、可视化数据看板 |
| 权限管理 | 按角色/用户组/人名字段/邮件组配置流转操作权限 |
| 开放能力 | API 接口、Webhook、SSO、插件，支持集成企业微信、GitLab、Jenkins 等 |
| 通知机制 | 系统内通知、企业微信推送、Webhook 推送第三方应用 |

### 2.2 TAPD 设计理念借鉴

我们从 TAPD 中提炼出以下关键设计理念，应用于工单系统设计：

1. **场景化分类**：拒绝"万能工单"，为不同业务场景提供专属模板和流程
2. **灵活定制**：自定义字段 + 自定义工作流，适配组织的实际管理颗粒度
3. **自动化驱动**：通过规则引擎减少人工操作，提升流转效率
4. **数据驱动决策**：通过度量指标和可视化报表辅助管理层把控全局

### 4.2 工单管理（核心模块）

#### 4.2.1 工单创建

**入口**：顶部导航 "新建工单" 按钮 / 快捷入口 / API 创建 / 企微群 @机器人创建

**创建流程**：

```
选择工单分类 → 加载对应模板表单 → 填写工单信息 → 提交
                                                    ↓
                                              触发工作流
                                              自动分派处理人
                                              发送通知
```

**基础字段**（所有工单通用）：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 工单标题 | 文本 | 是 | 简明概括工单内容 |
| 工单分类 | 级联选择 | 是 | 支持最多 3 级分类 |
| 优先级 | 下拉选择 | 是 | 紧急 / 高 / 中 / 低 |
| 描述 | 富文本 | 是 | 支持图片、附件、@提及 |
| 期望完成时间 | 日期 | 否 | 提交人的期望时间 |
| 附件 | 文件上传 | 否 | 支持多文件，单文件 ≤ 20MB |
| 关联工单 | 工单选择 | 否 | 关联已有工单 |

**扩展字段**：根据工单分类模板动态加载（见 4.3 模板管理）

#### 4.2.2 工单列表与筛选

提供多视图维度：

| 视图 | 说明 |
|------|------|
| 我创建的 | 当前用户提交的所有工单 |
| 我待办的 | 分配给当前用户待处理的工单 |
| 我参与的 | 当前用户曾参与处理的工单 |
| 我关注的 | 当前用户主动关注的工单 |
| 所有工单 | 按权限范围可见的所有工单（管理员可见全部） |

**筛选条件**：工单编号、标题关键词、分类、状态、优先级、创建人、处理人、创建时间范围、SLA 状态

**排序**：创建时间、更新时间、优先级、SLA 剩余时间

**列表展示**：支持自定义展示列，可拖拽调整列宽和顺序

**核心操作**：

| 操作 | 说明 | 权限 |
|------|------|------|
| 处理 | 填写处理意见，流转到下一状态 | 当前处理人 |
| 转派 | 将工单转交给其他人处理 | 当前处理人 / 管理员 |
| 加签 | 邀请他人协助处理（会签） | 当前处理人 |
| 挂起 | 暂时搁置（需填写原因，SLA 暂停计时） | 当前处理人 |
| 退回 | 退回到上一节点（需填写退回原因） | 当前处理人 |
| 催办 | 提交人对超时工单进行催促 | 创建人 |
| 关闭 | 确认完成，关闭工单 | 创建人 / 管理员 |
| 重新打开 | 关闭后问题复现，重新激活 | 创建人 / 管理员 |
| 关注 | 关注工单动态 | 所有人 |

### 4.3 分类与模板管理

#### 4.3.1 工单分类体系

采用**三级分类**结构，示例：

```
一级分类          二级分类          三级分类
──────────       ──────────       ──────────
IT 支持          网络问题          VPN 连接异常
                                  WiFi 无法连接
                                  内网访问故障
                 硬件问题          电脑维修
                                  外设申领
                 软件问题          系统安装
                                  账号权限
──────────       ──────────       ──────────
行政服务          办公环境          空调/照明
                                  座位调整
                 物资申领          办公用品
                                  设备借用
──────────       ──────────       ──────────
人事服务          考勤管理          请假/加班
                                  打卡异常
                 薪资福利          工资疑问
                                  福利申请
──────────       ──────────       ──────────
研发需求          功能需求          新功能开发
                                  功能优化
                 缺陷修复          线上 Bug
                                  测试 Bug
```

每个分类可独立配置：
- 绑定的工单模板
- 绑定的工作流
- 默认处理组/处理人
- SLA 策略

#### 4.3.2 工单模板与自定义字段

**自定义字段类型**（参考 TAPD）：

| 字段类型 | 说明 | 示例 |
|----------|------|------|
| 单行文本 | 短文本输入 | IP 地址、设备编号 |
| 多行文本 | 长文本输入 | 问题详细描述 |
| 下拉选择 | 单选下拉 | 故障等级、所属系统 |
| 多选框 | 复选选择 | 影响范围 |
| 单选框 | 单选按钮组 | 是否影响生产 |
| 日期 | 日期选择器 | 故障发生时间 |
| 数值 | 数字输入 | 影响人数 |
| 人员选择 | 用户选择器 | 审批人、协助人 |
| 级联选择 | 多级联动 | 部门选择 |
| 附件 | 文件上传 | 截图、日志文件 |

### 4.4 工作流引擎

#### 4.4.1 设计理念

借鉴 TAPD 的**双工作流引擎**思路，提供两种模式：

- **简单模式（状态机）**：状态之间可自由切换，适合轻量级场景
- **高级模式（流程引擎）**：严格定义流转路径和条件，适合需要审批和多部门协作的场景

#### 4.4.2 内置工作流 — 通用工单

```
                    ┌───────┐
                    │ 待受理 │ ← 工单创建后默认状态
                    └───┬───┘
                        │ 受理
                    ┌───▼───┐
              ┌─────│ 处理中 │──────┐
              │     └───┬───┘      │
              │ 挂起    │ 处理完成  │ 转派
          ┌───▼───┐ ┌───▼───┐  ┌───▼───┐
          │ 已挂起 │ │待验收  │  │ 待受理 │(新处理人)
          └───┬───┘ └───┬───┘  └───────┘
              │ 恢复    │
              └────►────┤
                        │ 验收通过      验收不通过
                    ┌───▼───┐      ┌───▼───┐
                    │ 已完成 │      │ 处理中 │ (退回)
                    └───┬───┘      └───────┘
                        │ 重新打开
                    ┌───▼───┐
                    │ 待受理 │
                    └───────┘
```

**状态定义**：

| 状态 | 含义 | SLA 计时 |
|------|------|----------|
| 待受理 | 工单已创建，等待处理人接单 | 响应计时中 |
| 处理中 | 处理人已接单，正在处理 | 解决计时中 |
| 已挂起 | 暂时搁置，等待外部条件 | 暂停计时 |
| 待验收 | 处理完成，等待创建人确认 | 暂停计时 |
| 已完成 | 创建人确认解决，工单关闭 | 计时结束 |
| 已关闭 | 无需处理/重复工单，直接关闭 | 计时结束 |

#### 4.4.3 高级流转配置

每条流转规则可配置：

| 配置项 | 说明 |
|--------|------|
| 可见条件 | 哪些角色/人员可以看到该操作按钮 |
| 操作权限 | 哪些角色/人员有权执行该操作 |
| 校验条件 | 执行前的前置校验（如必填字段检查） |
| 必填内容 | 流转时必须填写的字段（如处理意见） |
| 自动化动作 | 流转后自动执行的动作（通知、字段更新等） |

#### 4.4.4 内置工作流 — 审批工单

```
提交 → 部门审批 → 【通过】→ 执行中 → 完成
                  【驳回】→ 已驳回 → (修改后重新提交)
```

适用于需要上级审批的工单类型（如物资申领、权限申请等）。

#### 4.4.5 内置工作流 — 缺陷工单（★ 核心业务流程）

缺陷工单是本系统最高频的使用场景，涉及客户、客服、测试、开发四种角色的协作流转，且支持同角色间的横向转派。

**完整流转路径**：

```
客户反馈问题（群内/口头/邮件等）
    │
    ▼
┌─────────┐
│ ① 客服   │  客服整理为标准缺陷模板，在企微群 @机器人 提交
│  提交    │  → 机器人自动创建工单，返回工单ID和链接到群
└────┬────┘
     │  系统自动分派给测试组
     ▼                              ┌──────────────────┐
┌─────────┐                         │ 测试B             │
│ ② 测试   │  ── 转派给其他测试 ──→  │ (记录转派时间)     │
│  受理    │  ←─────────────────────│ 流程同②           │
│          │                         └──────────────────┘
│  点开工单 → 记录【阅读时间】
│  复现问题 → 补充复现步骤、截图、环境信息
│  确认缺陷 → 流转给开发
│          │  → 记录【测试→开发 流转时间】
└────┬────┘
     │
     ▼                              ┌──────────────────┐
┌─────────┐                         │ 开发B             │
│ ③ 开发   │  ── 转派给其他开发 ──→  │ (记录转派时间)     │
│  处理    │  ←─────────────────────│ 流程同③           │
│          │                         └──────────────────┘
│  点开工单 → 记录【阅读时间】
│  修复缺陷 → 填写修复说明、关联代码/分支
│  修复完成 → 流转给测试验收
│          │  → 记录【开发→测试 流转时间】
└────┬────┘
     │
     ▼
┌─────────┐
│ ④ 测试   │
│  验收    │  点开工单 → 记录【阅读时间】
│          │
│  ├─ 验收通过 → 流转给客服确认
│  │             → 记录【测试→客服 流转时间】
│  │
│  └─ 验收不通过 → 退回开发，附不通过原因
│                  → 记录【退回时间】，回到步骤③
└────┬────┘
     │ 验收通过
     ▼
┌─────────┐
│ ⑤ 客服   │  点开工单 → 记录【阅读时间】
│  确认    │  反馈给客户，确认问题已修复
│          │
│  ├─ 确认关闭 → 工单完成 → 进入步骤⑥
│  └─ 客户仍有问题 → 退回测试，回到步骤②
└────┬────┘
     │ 确认关闭
     ▼
┌─────────┐
│ ⑥ 技术   │  工单完成后，系统自动通知责任人（开发）填写 Bug 简报
│  简报    │  → 企微消息推送 + 站内通知
│          │  → 自动创建简报草稿，预填部分字段（来自工单信息）
│          │
│  责任人填写简报 → 提交审核 → 审核人审核
│  ├─ 审核通过 → 简报归档，与工单关联完成
│  └─ 审核不通过 → 退回修改，重新提交
│          │
│  ⚠ 超过N天未填写 → 系统自动催促（通知责任人+技术主管）
└─────────┘

**缺陷工单状态定义**：

| 状态 | 当前责任角色 | 含义 | SLA 计时 |
|------|-------------|------|----------|
| 待分派 | 系统/管理员 | 工单刚创建，等待分派给测试 | 响应计时 |
| 待测试受理 | 测试 | 已分派给测试，等待测试查看 | 响应计时 |
| 测试中 | 测试 | 测试正在复现和分析问题 | 解决计时 |
| 待开发受理 | 开发 | 测试确认缺陷，已流转给开发 | 解决计时 |
| 开发中 | 开发 | 开发正在修复缺陷 | 解决计时 |
| 待验收 | 测试 | 开发修复完成，等待测试验收 | 解决计时 |
| 待客服确认 | 客服 | 测试验收通过，等待客服反馈客户 | 解决计时 |
| 已完成 | - | 客服确认问题解决，工单关闭 | 计时结束 |
| 已关闭 | - | 非缺陷/重复/无法复现，直接关闭 | 计时结束 |

**同角色转派规则**：

| 转派类型 | 说明 | 约束 |
|----------|------|------|
| 测试 → 测试 | 当前测试可将工单转派给其他测试人员 | 必须填写转派原因；工单状态保持不变；新测试需重新记录阅读时间 |
| 开发 → 开发 | 当前开发可将工单转派给其他开发人员 | 必须填写转派原因；工单状态保持不变；新开发需重新记录阅读时间 |

转派不改变工单状态，但会在时间追踪链中产生新的节点记录（详见 4.4.6）。

**退回机制**：

| 退回路径 | 说明 |
|----------|------|
| 验收不通过 → 开发 | 测试验收失败，退回开发重新修复。必须填写不通过原因和复现步骤 |
| 客服确认不通过 → 测试 | 客户反馈问题仍存在，退回测试重新验证 |

退回后 SLA 继续计时，不重置。退回次数作为质量指标纳入统计。

#### 4.4.6 全链路时间追踪机制

系统在工单流转的**每一个环节**自动记录精确时间戳，形成完整的处理时间链条，用于效率度量和瓶颈分析。

**追踪的时间节点**：

| 时间节点 | 触发时机 | 记录方式 | 说明 |
|----------|----------|----------|------|
| 创建时间 | 工单创建成功 | 自动 | 工单进入系统的时间 |
| 分派时间 | 工单被分派给处理人 | 自动 | 从创建到分派的响应时间 |
| 首次阅读时间 | 处理人首次打开工单详情页 | 自动（页面打开事件） | 从分派到被查看的等待时间 |
| 开始处理时间 | 处理人点击"开始处理"或修改工单内容 | 半自动 | 从阅读到开始实际操作的时间 |
| 流转时间 | 处理人将工单流转到下一角色 | 自动 | 当前角色的处理耗时 |
| 转派时间 | 同角色内转派给其他人 | 自动 | 转派发生的时间点 |
| 退回时间 | 验收不通过退回 | 自动 | 退回发生的时间点 |
| 完成时间 | 工单最终关闭 | 自动 | 从创建到关闭的总耗时 |

**阅读时间记录机制**：

```
处理人点击工单详情
    │
    ├── 前端发送 "打开工单" 事件 → POST /api/tickets/{id}/track/read
    │
    ├── 后端记录：
    │   {
    │     ticket_id: "WO-20260228-003",
    │     user_id: "user_123",
    │     action: "read",
    │     role: "tester",              // 当前角色
    │     workflow_node: "待测试受理",   // 当前工单状态
    │     timestamp: "2026-02-28 14:30:25",
    │     is_first_read: true          // 是否为该节点的首次阅读
    │   }
    │
    └── 如果是该节点的首次阅读 → 自动更新工单状态
        例：待测试受理 → 测试中
```

**时间追踪链示例**（一条缺陷工单的完整生命周期）：

```
时间轴 ──────────────────────────────────────────────────────────────────→

14:00       14:02       14:05       14:08      15:30       15:35
  │           │           │           │          │           │
  ▼           ▼           ▼           ▼          ▼           ▼
客服提交    分派测试A    测试A阅读   测试A转派   测试B阅读   测试B流转
(创建)     (自动分派)   (首次打开)  给测试B    (首次打开)  给开发C
                        ⏱3min                  ⏱5min       ⏱填写复现步骤

15:40       16:20       16:25       16:50       16:55       17:00

  ▼           ▼           ▼           ▼          ▼           ▼

**各环节耗时自动统计**：

| 统计指标 | 计算方式 | 用途 |
|----------|----------|------|
| 响应时间 | 分派时间 - 创建时间 | 衡量工单调度效率 |
| 阅读等待时间 | 首次阅读时间 - 分派/流转到达时间 | 衡量处理人响应速度 |
| 各角色处理耗时 | 流转出去时间 - 首次阅读时间 | 衡量各角色处理效率 |
| 测试耗时 | 测试所有处理时段之和（含复现 + 验收） | 测试环节总耗时 |
| 开发耗时 | 开发所有处理时段之和 | 开发环节总耗时 |
| 退回次数 & 耗时 | 退回次数、每次退回后的重新处理时间 | 衡量修复质量 |
| 端到端总耗时 | 完成时间 - 创建时间 | 衡量整体处理效率 |
| 流转次数 | 状态变更总次数 | 衡量流程复杂度 |

### 4.6 企业微信集成（★ 新增模块）

本系统以企业微信作为统一身份入口和核心消息通道，实现登录零密码、消息实时达、群内快速建单三大能力。

#### 4.6.1 企微扫码登录（OAuth2.0）

**前置配置**：

| 配置项 | 说明 |
|--------|------|
| 企业 ID（CorpID） | 在企业微信管理后台 → 我的企业中获取 |
| 自建应用 AgentID | 创建自建应用后获取 |
| 应用 Secret | 自建应用的密钥 |
| 可信域名 | 工单系统部署域名（如 ticket.company.com） |
| 授权回调地址 | `https://ticket.company.com/auth/wecom/callback` |

**登录流程**：

```
用户                    工单系统前端               工单系统后端              企业微信服务器
 │                        │                        │                        │
 │  1.点击"企微扫码登录"  │                        │                        │
 │───────────────────────>│                        │                        │
 │                        │  2.加载企微登录组件     │                        │
 │                        │  (WWLoginPanel)        │                        │
 │  3.打开企业微信扫码     │                        │                        │
 │<───────────────────────│                        │                        │
 │                        │                        │                        │
 │  4.用户扫码确认授权     │                        │                        │
 │────────────────────────────────────────────────────────────────────────>│
 │                        │                        │                        │
 │                        │  5.回调携带 auth_code   │                        │
 │                        │───────────────────────>│                        │
 │                        │                        │  6.用code换取user_id   │
 │                        │                        │───────────────────────>│
 │                        │                        │  7.返回 UserId         │
 │                        │                        │<───────────────────────│
 │                        │                        │                        │
 │                        │                        │  8.匹配本地用户         │
 │                        │                        │     生成 JWT Token     │
 │                        │  9.返回Token+用户信息   │                        │
 │                        │<───────────────────────│                        │
 │  10.登录成功，进入系统   │                        │                        │
 │<───────────────────────│                        │                        │
```

**前端实现要点**：

```javascript
// 安装企微JSSDK: npm install @wecom/jssdk
import { createWWLoginPanel } from '@wecom/jssdk'

createWWLoginPanel({
  el: '#ww_login_container',
  params: {
    login_type: 'CorpApp',
    appid: 'YOUR_CORP_ID',
    agentid: 'YOUR_AGENT_ID',
    redirect_uri: encodeURIComponent('https://ticket.company.com/auth/wecom/callback'),
    state: 'random_state_string',
  },
  onLoginSuccess({ code }) {
    // 将 code 发送到后端换取 Token
    fetch('/api/auth/wecom/login', {
      method: 'POST',
      body: JSON.stringify({ code })
    })
  },
  onLoginFail(err) {
    console.error('扫码登录失败', err)
  }
})
```

**后端处理逻辑**：

```
接收 auth_code
    │
    ├── 调用企微API获取 access_token
    │   POST https://qyapi.weixin.qq.com/cgi-bin/gettoken
    │        ?corpid=CORP_ID&corpsecret=SECRET
    │
    ├── 用 code 换取用户身份
    │   GET https://qyapi.weixin.qq.com/cgi-bin/auth/getuserinfo
    │       ?access_token=TOKEN&code=AUTH_CODE
    │   → 返回 UserId
    │
    ├── 通过 UserId 查询本地用户表（wecom_userid 字段）
    │   ├── 已绑定 → 直接生成 JWT Token
    │   └── 未绑定 → 调用通讯录API获取用户详情
    │               → 自动创建本地账号并绑定
    │
    └── 返回 JWT Token + 用户信息
```

**用户绑定策略**：

| 场景 | 处理方式 |
|------|----------|
| 首次登录，通讯录已同步 | 通过 wecom_userid 自动匹配，直接登录 |
| 首次登录，通讯录未同步 | 调用通讯录 API 获取姓名/部门/手机号，自动创建账号 |
| 企微用户离职 | 通过通讯录变更事件回调，自动禁用本地账号 |

#### 4.6.2 企微通讯录同步

系统定时或通过事件回调同步企业微信通讯录，作为用户数据的唯一来源。

**同步方式**：

| 方式 | 说明 | 频率 |
|------|------|------|
| 全量同步 | 调用通讯录 API 拉取全部部门和成员 | 每天凌晨 1 次 |
| 增量回调 | 配置通讯录变更事件回调（成员创建/修改/离职、部门变更） | 实时 |

**同步数据映射**：

| 企微字段 | 本地字段 | 说明 |
|----------|----------|------|
| userid | wecom_userid | 企微用户唯一标识，用于登录匹配 |
| name | name | 姓名 |
| department[] | department_id | 部门（取主部门） |
| mobile | phone | 手机号 |
| email | email | 邮箱 |
| position | position | 职位 |
| avatar | avatar_url | 头像 |
| status | account_status | 1=已激活 2=已禁用 4=未激活 |

#### 4.6.3 企微应用消息推送

通过企业微信自建应用的消息推送 API，将工单相关通知实时送达用户的企业微信。

**推送架构**：

```
工单事件触发
    │
    ├── 写入消息队列（异步解耦）
    │
    └── 消息推送消费者
        │
        ├── 获取/刷新 access_token（缓存至 Redis，有效期 2h）

        │
        ├── 组装消息体（按事件类型选择模板）
        │
        └── 调用企微 API 推送
            POST https://qyapi.weixin.qq.com/cgi-bin/message/send
                 ?access_token=TOKEN
```

**消息模板设计**：

以**文本卡片消息**为主要推送格式，兼顾信息展示和交互便捷性：

**各事件推送模板**：

| 事件 | 标题 | 描述内容 | 推送对象 |
|------|------|----------|----------|
| 工单创建 | 新工单待处理 | 编号、标题、优先级、创建人、分类 | 处理人/处理组 |

- 非工作时间的低优先级通知延迟至次日工作时间推送（紧急/高优先级不受限制）

#### 4.6.4 企微群机器人工单（★ 核心创新）

在企业微信群中部署工单机器人，群成员可直接在群内按模板格式 @机器人创建工单，机器人自动解析并回复工单信息和链接。

**实现方案选择**：

| 方案 | 原理 | 优劣 | 推荐 |
|------|------|------|------|
| 方案 A：自建应用 + 群聊回调 | 在企微管理后台创建自建应用，配置接收消息回调 URL，应用可接收用户发来的消息并被动回复 | 功能完整、可双向交互、支持富文本回复 | **推荐** |
| 方案 B：群 Webhook 机器人 | 群内添加 Webhook 机器人，仅支持主动推送消息到群 | 简单易用但只能单向推送，无法接收和解析群消息 | 辅助方案 |

**推荐采用方案 A + B 组合**：方案 A 实现群内@创建工单的双向交互，方案 B 的 Webhook 用于工单状态变更等场景的群内通知推送。

**方案 A 技术架构**：

```
企业微信群
    │
    │  用户发送：@工单助手 #IT支持/网络问题 VPN连接异常
    │           优先级：紧急
    │           描述：全公司VPN今天上午10点开始无法连接
    │
    ▼
企业微信服务器
    │
    │  POST 回调请求（XML 加密消息体）
    │
    ▼
工单系统 - 机器人回调服务
    │
    ├── 1. 签名验证 + 消息解密（AES）
    │
    ├── 2. 解析消息内容
    │   ├── 提取发送人 UserId → 映射为系统用户（工单创建人）
    │   ├── 解析消息文本 → 识别模板字段
    │   │   ├── 分类：#IT支持/网络问题
    │   │   ├── 标题：VPN连接异常
    │   │   ├── 优先级：紧急
    │   │   └── 描述：全公司VPN今天上午10点开始无法连接
    │   └── 字段校验（分类是否合法、必填项是否完整）
    │
    ├── 3. 调用工单创建 API → 生成工单
    │
    ├── 4. 触发工作流（自动分派处理人、SLA 计时开始）
    │
    └── 5. 被动回复消息到群 / 通过 Webhook 推送结果到群
        │
        └── 回复内容：
            ┌─────────────────────────────────────┐
            │  ✅ 工单创建成功                       │
            │                                     │
            │  工单编号：WO-20260228-003            │
            │  标题：VPN连接异常                     │
            │  分类：IT支持 > 网络问题               │
            │  优先级：紧急                          │
            │  处理人：张三（IT运维组）               │
            │  SLA：15分钟内响应，2小时内解决         │
            │                                     │
            │  🔗 查看详情：                        │
            │  https://ticket.company.com/t/WO-003│
            └─────────────────────────────────────┘
```

**消息模板格式定义**：

群成员 @机器人 时，按以下格式发送消息即可创建工单：

```
@工单助手 #分类路径 工单标题
优先级：紧急/高/中/低
描述：问题的详细描述内容
```

**字段解析规则**：

| 字段 | 格式 | 必填 | 解析规则 |
|------|------|------|----------|
| 分类 | `#一级分类/二级分类` 或 `#一级分类/二级分类/三级分类` | 是 | 以 `#` 开头，`/` 分隔层级 |
| 标题 | 分类后同一行的文本 | 是 | `#分类` 之后到行末的文本 |
| 优先级 | `优先级：xxx` | 否 | 匹配关键字，默认"中" |
| 描述 | `描述：xxx`（支持多行） | 否 | `描述：` 之后的所有文本 |

**简化格式**（快速创建）：

```
@工单助手 #IT支持/网络问题 VPN无法连接，全公司受影响
```

仅提供分类和标题时，优先级默认"中"，标题同时作为描述内容。

**异常处理**：

| 异常场景 | 机器人回复 |
|----------|-----------|
| 格式无法解析 | "❌ 消息格式无法识别，请按模板发送：\n@工单助手 #分类 标题\n优先级：紧急/高/中/低\n描述：详细描述\n\n发送"@工单助手 帮助"查看完整指南" |

### 4.7 SLA 管理

#### 4.7.1 SLA 策略配置

| 优先级 | 首次响应时限 | 解决时限 | 适用说明 |
|--------|-------------|---------|----------|
| 紧急 | 15 分钟 | 2 小时 | 生产系统故障、全公司影响 |
| 高 | 30 分钟 | 4 小时 | 部门级影响、核心功能故障 |
| 中 | 2 小时 | 8 小时（1 工作日） | 个人工作受影响 |
| 低 | 4 小时 | 24 小时（3 工作日） | 咨询类、优化建议 |

> SLA 时限按**工作时间**计算，非工作时间不计入。

#### 4.7.2 预警与升级机制

```
SLA 剩余时间
    │
    ├─ > 50% ─── 正常（绿色标识）
    │
    ├─ 25%~50% ── 预警（黄色标识，通知处理人）
    │
    ├─ < 25% ─── 告警（橙色标识，通知处理人 + 组长）
    │
    └─ 超时 ───── 超时（红色标识，通知处理人 + 组长 + 上级主管）
                   自动提升优先级
```

#### 4.7.3 SLA 暂停规则

以下情况 SLA 计时暂停：
- 工单状态为"已挂起"（等待外部条件）
- 工单状态为"待验收"（等待提交人确认）

### 4.12 Bug 简报管理（★ 新增模块）

缺陷修复上线后，技术人员需要出具 **Bug 简报**，对缺陷的根因、分类、解决方案、影响范围等进行结构化总结归档。Bug 简报与缺陷工单一一关联，形成从"发现缺陷 → 修复缺陷 → 归因总结"的完整闭环。

#### 4.12.1 业务流程

```
缺陷工单修复上线（工单状态进入"已完成"）
    │
    ▼
┌──────────────────────┐
│ ① 系统自动提醒       │  工单关闭后，系统自动通知责任人填写 Bug 简报
│    创建简报           │  → 企微应用消息 + 站内通知
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│ ② 责任人填写简报      │  技术人员（开发/测试）登录系统填写简报
│                      │  → 从关联工单自动预填部分字段
│  填写内容：           │  → 问题描述、逻辑归因、缺陷分类、解决方案等
│  关联工单：           │  → 自动关联触发简报的缺陷工单，也可手动关联多个工单
└──────────┬───────────┘
           │  提交审核
           ▼
┌──────────────────────┐
│ ③ 审核人审核          │  指定审核人（通常为技术主管/项目负责人）
│                      │
│  ├─ 审核通过 ─────────── → 简报归档，状态变为"已归档"
│  │                   │
│  └─ 审核不通过 ──────── → 退回责任人修改，附审核意见
└──────────────────────┘
```

#### 4.12.3 Bug 简报字段设计

基于实际 Bug 简报格式设计，所有字段含义和枚举值参考公司现有规范。

**简报字段定义**：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| 简报编号 | 系统生成 | 自动 | 格式：BR-YYYYMMDD-序号，如 BR-20260228-015 |
| 关联工单 | 工单选择（多选） | 是 | 关联一个或多个缺陷工单，支持搜索和级联选择 |
| 问题描述 | 多行文本 | 是 | 缺陷的现象描述，从关联工单自动预填，可修改补充 |
| 逻辑归因 | 级联选择 | 是 | 二级归因分类（一级归因类型 > 二级具体原因），见下方枚举 |
| 缺陷分类 | 下拉选择 | 是 | 缺陷的技术分类，见下方枚举 |
| 引入项目 | 下拉选择 + 手动输入 | 是 | 引入该缺陷的项目/模块名称，支持从项目库选择或手动输入 |
| 开始时间 | 日期 | 是 | 缺陷被发现/报告的时间，自动从关联工单创建时间预填 |
| 解决时间 | 日期 | 是 | 缺陷修复上线的时间，自动从关联工单完成时间预填 |
| 解决方案 | 多行文本 | 是 | 具体的修复措施和技术方案 |
| 影响范围 | 多行文本 | 是 | 该缺陷对用户/业务的影响描述 |
| 缺陷等级 | 下拉选择 | 是 | P0（致命）/ P1（严重）/ P2（一般）/ P3（轻微）/ P4（建议） |
| 反馈人 | 人员选择 | 是 | 最初反馈该问题的人员 |
| 审核人 | 人员选择 | 是 | 负责审核该简报的技术主管 |
| 责任人 | 人员选择（多选） | 是 | 该缺陷的相关责任人，支持选择多人 |
| 附件 | 文件上传 | 否 | 相关截图、日志、分析文档等 |
| 备注 | 多行文本 | 否 | 其他需要补充说明的信息 |

**逻辑归因枚举**（二级级联）：

| 一级归因 | 二级归因 |
|----------|----------|
| 配置错误 | 参数配置错误、环境配置错误、权限配置错误、开关配置遗漏 |
| 编码缺陷 | 逻辑错误、空指针/异常处理、边界条件遗漏、并发问题、内存泄漏 |
| 设计缺陷 | 需求理解偏差、架构设计不合理、接口设计不当、兼容性考虑不足 |
| 第三方问题 | 第三方 SDK 缺陷、第三方接口变更、第三方服务不稳定 |
| 数据问题 | 脏数据、数据迁移错误、数据库设计缺陷、缓存一致性问题 |
| 运维问题 | 部署操作失误、资源不足、网络故障、证书/域名过期 |
| 测试遗漏 | 测试用例覆盖不足、回归测试遗漏、特定环境未测试 |

**缺陷分类枚举**：

| 分类 | 说明 |
|------|------|
| 功能异常 | 功能逻辑不符合预期，操作无响应或结果错误 |
| 交互异常 | 界面展示异常、控件行为不符合预期、页面空白/错位 |
| 性能问题 | 响应缓慢、卡顿、超时、资源占用异常 |
| 数据异常 | 数据丢失、数据错误、数据不一致 |
| 安全漏洞 | 权限绕过、注入攻击、信息泄露 |
| 兼容性问题 | 浏览器/设备/系统版本兼容性问题 |
| 接口异常 | API 返回异常、接口超时、参数校验失败 |

**缺陷等级定义**：

| 等级 | 名称 | 定义 | 示例 |
|------|------|------|------|
| P0 | 致命 | 系统崩溃、核心功能完全不可用、数据丢失 | 支付功能全部失败、数据库宕机 |
| P1 | 严重 | 核心功能部分不可用、严重影响用户使用 | 部分商户无法下单、页面频繁白屏 |
| P2 | 一般 | 非核心功能异常、有临时解决方案 | 报表导出格式错误、筛选条件失效 |
| P3 | 轻微 | 界面细节问题、不影响功能使用 | 文案错误、对齐偏差、图标显示异常 |
| P4 | 建议 | 优化建议、体验改进 | 操作步骤可简化、提示文案可优化 |

#### 4.12.4 Bug 简报审核流程

```
┌───────────┐     ┌───────────┐     ┌───────────┐     ┌───────────┐
│  待填写    │────▶│  待审核    │────▶│  已归档    │     │  已作废    │
│ (Draft)   │     │ (Pending) │     │ (Archived)│     │ (Voided)  │
└───────────┘     └─────┬─────┘     └───────────┘     └───────────┘
                        │                                   ▲
                        │  审核不通过                         │
                        ▼                                   │
                  ┌───────────┐                              │

                  │ (Rejected)│                              │

```

**简报状态定义**：

| 状态 | 含义 | 可执行操作 |
|------|------|-----------|
| 待填写 | 系统自动创建草稿，等待责任人填写 | 编辑、提交审核、删除草稿 |
| 待审核 | 责任人已提交，等待审核人审核 | 审核通过、审核不通过（退回） |
| 已退回 | 审核不通过，退回责任人修改 | 编辑、重新提交审核 |
| 已归档 | 审核通过，正式归档 | 仅查看（管理员可作废） |
| 已作废 | 简报被作废（如重复、误建） | 仅查看 |

**审核操作说明**：

| 操作 | 操作人 | 必填内容 | 后续动作 |
|------|--------|----------|----------|
| 提交审核 | 责任人 | 所有必填字段 | 通知审核人进行审核 |
| 审核通过 | 审核人 | 审核意见（选填） | 简报归档，通知责任人 |
| 审核不通过 | 审核人 | 退回原因（必填） | 退回责任人，通知修改 |
| 作废 | 管理员 | 作废原因（必填） | 简报作废，通知相关人 |

#### 4.12.6 简报自动预填机制

| 简报字段 | 预填来源 | 说明 |

| 关联工单 | 触发工单 | 自动关联当前缺陷工单 |

| 解决时间 | 工单完成时间 | 取工单关闭/完成的日期 |
| 反馈人 | 工单创建人 | 取工单的原始提交人 |
| 缺陷等级 | 测试填写的缺陷等级 | 映射转换：致命→P0、严重→P1、一般→P2、轻微→P3 |
| 解决方案 | 开发填写的修复方案 | 从开发信息区的"修复方案"字段预填 |
| 影响范围 | 测试填写的影响范围 + 客服问题描述 | 综合测试和客服的描述 |
| 责任人 | 工单处理人（开发） | 默认预填开发阶段的处理人 |

#### 4.12.7 简报通知机制

| 事件 | 通知对象 | 站内信 | 企微应用消息 | 说明 |
|------|----------|--------|-------------|------|
| 工单关闭触发简报 | 责任人（开发） | ✅ | ✅ | 提醒填写 Bug 简报 |
| 简报提交审核 | 审核人 | ✅ | ✅ | 通知审核人进行审核 |
| 审核通过 | 责任人、反馈人 | ✅ | ✅ | 通知简报已归档 |
| 审核不通过 | 责任人 | ✅ | ✅ | 通知退回原因，需修改重提 |
| 简报超期未填写 | 责任人 + 技术主管 | ✅ | ✅ | 工单关闭后超过 N 天未填写简报，自动催促 |

| TAPD | 双工作流引擎、自定义字段强大、自动化规则丰富 | 工作流配置、自定义字段体系、自动化规则编排 |
