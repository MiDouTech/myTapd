# 优化需求：公开工单详情补充时间追踪与当前处理人

## 背景

外网通过工单编号访问的公开详情页（`/open/ticket/{ticketNo}`）用于客户或协作方了解工单进展。当前页面虽展示「处理人」字段，但与内网一致的多人协作场景下仅同步了主处理人；同时缺少与内网「时间追踪」对应的处理过程时间线，外部无法直观看到节点流转与处理动作。

## 现状问题

1. **当前处理人**：`assigneeName` 仅来自工单主表 `assignee_id`，多人处理时未体现全部在办处理人。
2. **处理过程**：无基于 `ticket_time_track` 的对外可读摘要，与产品「全链路时间追踪」能力不对齐。

## 目标

1. 公开详情接口与页面展示**完整当前处理人**（与内网详情一致：活跃处理人列表，多名时用「、」拼接展示）。
2. 公开详情接口与页面展示**时间追踪处理摘要**：按时间顺序列出关键处理动作（操作类型、操作人、状态变化、涉及人员、时间、备注），**不包含**缺陷字段级变更明细、独立变更批次等内网专属敏感/冗余项。

## 非目标

- 不新增独立开放 API 路径；在现有 `GET /open/ticket/{ticketNo}` 响应体中扩展字段。
- 不暴露阅读记录（READ）、关注（FOLLOW/UNFOLLOW）、与评论区重复的 COMMENT 轨迹点。
- 不改变登录态工单详情、开放 API V1 等其它接口契约（除非另有需求）。

## 验收标准

1. 多人协作工单在公开页「处理人」与内网详情中活跃处理人列表一致（顺序、人员一致，展示为拼接姓名）。
2. 公开页新增「处理进度」或等价区块，时间线顺序与内网时间链一致（过滤上述排除动作后），且状态展示为中文标签。
3. 无登录接口响应新增字段向后兼容：老前端忽略新字段仍可运行；新前端消费新字段。

## 衡量方式

- 手工：同一工单在内网详情与公开链接对比处理人与时间线条目数量（过滤规则一致后应一致或公开为子集）。
- 回归：`GET /open/ticket/{ticketNo}` 仍返回 200，且不含 `fieldChanges` / 缺陷变更历史结构。
