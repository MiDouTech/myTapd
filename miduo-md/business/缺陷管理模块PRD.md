## 一、背景与目标

### 1.1 背景

米多内部工单系统已具备完整的工单生命周期管理、Bug简报体系和多角色协作能力。现有缺陷工单详情页（`TicketDetailView`）已支持客服/测试/开发三段信息填写，但在用户体验上与 TAPD 等行业标杆工具存在明显差距，核心缺口在于：

- **变更历史不可见**：操作日志（`ticket_log`）仅记录操作类型（action），未记录字段级新旧值，无法还原"谁在什么时间改了哪个字段，从什么值改成了什么值"。
- **详情页信息密度不足**：关键缺陷字段（有效报告、缺陷划分、责任人、预计结束时间、是否逾期）未在详情页基础信息面板集中展示。
- **缺陷状态体系与页面布局不匹配**：页面顶部无清晰的状态徽章和工单编号的突出显示，操作入口分散。

### 1.2 目标

1. **对齐 TAPD 缺陷详情体验**：实现变更历史 Tab、字段新旧值对比展示、状态徽章、顶部操作按钮组。
2. **补全缺陷基础信息字段**：在右侧面板集中展示所有缺陷维度字段。
3. **数据完整性**：所有字段变更均有记录，支持按字段名、变更方式过滤。

- 位置：页面左上角
- 展示：当前工单状态（中文标签）

| 状态 | 中文 | 颜色 |
|------|------|------|

| pending_accept | 待受理 | `#909399`（灰色） |

| pending_test_accept | 待测试受理 | `#E6A23C`（橙色） |
| testing | 测试中 | `#E6A23C`（橙色） |
| pending_dev_accept | 待开发受理 | `#6B7280`（蓝灰） |

### 3.3 右侧基础信息面板字段规格

#### 3.3.1 字段完整清单

| 字段名 | 字段标识 | 数据来源 | 编辑权限 | 说明 |
|--------|---------|---------|---------|------|
| 状态 | status | ticket.status | 通过工作流流转 | 只读展示 |
| 公司名称 | companyName | ticket_bug_info.company_name | 客服/管理员 | 可编辑 |
| 商户编号 | merchantNo | ticket_bug_info.merchant_no | 客服/管理员 | 可编辑 |
| 反馈人 | reporterName | ticket.create_by | 只读 | 创建人姓名 |
| 反馈时间 | reportTime | ticket.create_time | 只读 | 工单创建时间 |
| 创建人 | creatorName | ticket.creator_id → sys_user | 只读 | 系统锁定 |
| 创建时间 | createTime | ticket.create_time | 只读 | 系统锁定 |
| 引入项目 | projectName | bug_report.project → dict_project | 管理员 | 可选，关联项目字典 |
| 影响范围 | impactScope | ticket_bug_test_info.impact_scope | 测试/管理员 | 枚举：单一/部分/全部商户 |
| 缺陷等级 | severityLevel | ticket_bug_test_info.severity_level | 测试/管理员 | P0/P1/P2/P3/P4 |
| 缺陷划分 | defectCategory | bug_report.defect_category | 管理员 | 关联字典 |
| 有效报告 | isValidReport | bug_report（从简报关联取） | 管理员 | 是/否 |
| 处理人 | assigneeName | ticket.assignee_id → sys_user | 管理员/分派 | 可通过分派操作修改 |
| 责任人 | responsibleName | bug_report_responsible | 管理员 | 从简报获取 |
| 预计结束 | expectedTime | ticket.expected_time | 客服/管理员 | 可编辑 |
| 解决时间 | resolvedAt | ticket.resolved_at | 只读 | 系统自动填充 |
| 是否逾期 | isOverdue | 计算字段 | 只读 | expected_time < now() AND status != closed |

**数据来源说明**：
- 「缺陷划分」「有效报告」「责任人」来自 Bug简报关联数据，若工单未关联简报则显示「-」
- 「是否逾期」为前端/后端计算字段，无需存储

#### 3.3.2 字段展示交互规则
- 字段值为空时统一展示「-」
- 可编辑字段：鼠标悬停显示铅笔图标，点击弹出行内编辑；
- 不可编辑字段：无鼠标悬停效果，背景无变化；
- 日期时间字段：格式为 `yyyy-MM-dd HH:mm`；
- 枚举字段：展示中文标签，颜色匹配状态体系。

### 3.4 变更历史 Tab

这是本次最核心的新功能，完全对标 TAPD 变更历史能力。

#### 3.4.1 功能描述

记录缺陷工单从创建到关闭的**所有字段级变更**，包括：
- 工单主表字段变更（标题、状态、优先级、处理人、预计时间等）
- 缺陷扩展字段变更（客服信息、测试信息、开发信息）
- 评论新增
- 附件上传/删除

每一次"保存操作"如果涉及多个字段变更，合并为同一个"操作批次"展示，批次内按字段分行。

#### 3.4.2 列表展示规格

```
筛选栏：[变更内容: 全部 ▼]   [变更方式: 全部 ▼]

序号 │ 变更时间                │ 变更人    │ 变更类型  │ 变更字段（及新旧值）
─────┼────────────────────────┼──────────┼──────────┼──────────────────────
  1  │ 2026-03-06 09:46:53    │ 陈嘉颖    │ 手动变更  │ 有效报告
     │                         │           │           │   - → 否
─────┼────────────────────────┼──────────┼──────────┼──────────────────────
  2  │ 2026-03-06 09:46:49    │ 陈嘉颖    │ 手动变更  │ 缺陷划分
     │                         │           │           │   - → 非缺陷
─────┼────────────────────────┼──────────┼──────────┼──────────────────────
  3  │ 2026-03-06 09:46:44    │ 陈嘉颖    │ 手动变更  │ 状态
     │                         │           │           │   处理中 → 已关闭
     │                         │           │           │ 关闭时间
     │                         │           │           │   - → 2026-03-06 09:46:44
─────┼────────────────────────┼──────────┼──────────┼──────────────────────
  6  │ 2026-03-05 10:06:31    │ 周香      │ 创建缺陷  │ 详细描述 / 公司名称 /
     │                         │           │           │ 商户编号 / 反馈人 ...
```

**序号规则**：按时间**倒序**显示，序号从 1 开始顺序标注（序号1为最新操作）。

#### 3.4.3 筛选功能

| 筛选维度 | 选项 |
|---------|------|
| 变更内容（字段） | 全部 / 状态 / 处理人 / 缺陷等级 / 有效报告 / 缺陷划分 / 评论 / 附件 / 其他字段 |
| 变更方式（类型） | 全部 / 手动变更 / 状态流转 / 系统自动 / 创建缺陷 |

筛选为前端本地过滤（不触发新的接口请求），保证响应速度。

#### 3.4.4 数据加载

- 一次性加载该工单全部变更历史（一般不超过 100 条），不分页；
- 如超过 100 条，分页加载，每页 20 条，分页组件置于列表底部。

### 3.5 详细信息 Tab

保持现有三段式结构（客服信息 / 测试信息 / 开发信息），在现有 `TicketDetailView` 基础上优化以下细节：

1. 每段信息区增加「最后修改人 + 修改时间」的轻量说明；
2. 附件支持预览（图片直接展示，其他文件显示文件类型图标）；
3. 描述/步骤等富文本字段支持折叠/展开（超过 5 行时默认折叠）。

---

### 3.6 评论 Tab

复用现有评论功能，增加以下能力：

1. **@ 提及**：评论框支持 `@用户名` 触发下拉选人，被@用户收到系统通知；
2. **评论数角标**：Tab 标题展示评论总数，如 `评论(3)`；
3. **评论时间轴**：按时间顺序展示，最新评论在底部。

---

### 3.7 更多 Tab（预留）

预留以下功能入口（当前版本不实现，保留空 Tab）：

- 关联工单（同类 Bug 关联）
- 子任务（拆分为多个子 Bug）
- 变更申请（触发 Bug 等级升降）

### 3.9 缺陷导航（上一条/下一条）

- 导航范围：与当前页面进入时的列表查询条件一致（继承筛选参数）；
- 在列表最首条时，「上一条」按钮置灰禁用；
- 在列表最末条时，「下一条」按钮置灰禁用；
- 实现方式：前端从路由 query 参数传递 `prevId` / `nextId`，后端提供 navigate 接口辅助。

## 四、非功能性需求

### 4.1 性能

| 指标 | 目标 |
|------|------|
| 详情页首屏加载 | ≤ 800ms（P99） |
| 变更历史加载 | ≤ 400ms（P99，100条以内） |
| 字段编辑保存 | ≤ 200ms（P99） |

### 4.2 兼容性

- 浏览器：Chrome 100+、Edge 100+、Safari 15+
- 分辨率：1280×768 以上，支持 1920×1080 最优体验

### 4.3 数据完整性

- 任何通过接口触发的字段变更，必须在 `ticket_log` 中留有字段级记录；
- 变更记录写入失败时，整个操作事务回滚，不允许出现"操作成功但无历史记录"的情况。

## 五、UI 交互细节

### 5.1 内联编辑

右侧信息面板可编辑字段遵循以下交互：

```
[未编辑状态]  字段名  字段值            （悬停显示 ✎ 图标）
[编辑状态]   字段名  [输入框/选择器]  [✓确认] [✗取消]
[保存中]     字段名  字段值  Loading...
[保存成功]   字段名  新字段值          （短暂 green flash）
[保存失败]   字段名  字段值  [重试]     （提示错误信息）
```

### 5.2 变更历史条目展示

每个变更字段行展示格式：
- 字段名（加粗）+ 换行
- &emsp;旧值（灰色，删除线）→ 新值（正常色）
- 若旧值为空：直接展示新值（不显示箭头和旧值）
- 若新值为空（删除操作）：新值显示「（已删除）」（红色）

### 5.3 状态徽章

圆角矩形标签，颜色见 3.2.1 章节，字号 12px，padding: 2px 8px。

## 六、边界与约束

1. **变更历史只增不改**：ticket_log 的记录一旦写入，永不更新和删除；
2. **字段 label 一致性**：变更历史中展示的字段名（中文标签）需与右侧面板展示的字段名完全一致；
3. **旧值存储限制**：text/longtext 类型字段（如问题描述、复现步骤）的旧值，截断存储前 500 字符，避免 ticket_log 表膨胀；
4. **系统自动变更**：SLA 超期、定时任务触发的状态变更，变更类型记录为「系统自动」，变更人展示「系统」。
