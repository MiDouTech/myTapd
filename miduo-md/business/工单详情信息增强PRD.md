# 工单详情信息增强 — 产品需求文档（PRD）

> 版本：v1.0  
> 日期：2026-03-31  
> 所属系统：米多内部工单系统（Miduo Ticket Platform）  
> 范围：登录态工单详情页、公开工单详情页（免登录链接）、相关开放接口与入单渠道  
> 文档类型：产品需求（实施前需同步更新 `miduo-md/workflow/` 设计方案与接口对应关系）

---

## 一、文档概述

### 1.1 摘要

当前工单详情（尤其**公开详情页**）存在信息重复、处理轨迹不可见、接口类问题缺少结构化排障字段、证据链不完整等问题，导致提交方与协同方沟通成本高、关单后仍无法自解释。本需求在**不降低安全边界**的前提下，统一「动态/处理记录」数据源，补充**处理结论**、**证据展示**与**可选排障结构化字段**，并优化信息架构以减少冗余展示。

### 1.2 目标

| 目标 | 说明 |
|------|------|
| 信息完整 | 访客在公开链接上可看到：结论摘要、关键处理动态（脱敏）、可公开的附件/截图 |
| 少重复 | 标题、问题描述、工单描述在展示层按规则去重或折叠，避免三处同文 |
| 可排障 | 对「功能缺陷/接口异常」类工单，支持结构化记录 URL、状态码、TraceId 等（可选填） |
| 可对齐 | 登录态详情与公开态在「结论、动态、证据」语义上一致，仅字段可见范围不同 |

### 1.3 非目标（本期不做）

| 非目标 | 说明 |
|--------|------|
| 公开页直接流转工单 | 状态变更仍在登录态或系统内完成；公开页只读（除下文可选 P2） |
| 暴露全部内部日志 | 不返回研发内部备注、Git 分支、未脱敏客户隐私等 |
| 替代客服系统 | 不承载完整 IM 会话，仅工单域内信息 |
| 全类型工单强制结构化 | 排障扩展字段按**分类/模板**启用，避免所有工单堆砌表单项 |

## 二、背景与问题陈述

### 2.1 现状痛点

1. **重复信息**：标题、客服区「问题描述」、富文本「工单描述」内容高度重叠，阅读效率低。  
2. **处理记录空白**：公开页仅展示评论列表；若关单、流转仅记入内部日志或未同步为评论，则出现「已关闭但暂无处理记录」。  
3. **证据不足**：接口类问题缺少 URL、返回码、TraceId 等；截图与附件在公开侧可能未完整透出。  
4. **结论缺失**：关单后缺少对外的「处理结论/关闭说明」，不利于提交方确认与留档。

### 2.2 证据与依据

- 业务侧：公开链接常用于企微群外发，接收方多为未登录用户。  
- 产品侧：与《工单系统产品设计方案》中「流程透明、可追溯」一致，需补齐访客视角。

## 三、用户角色与场景

### 3.1 角色

| 角色 | 诉求 |
|------|------|
| 工单提交方（含商户/业务） | 免登录打开链接即可看清问题、证据、处理过程摘要与结论 |
| 客服 / 一线处理人 | 录入结构化排障信息，关单时填写结论，减少反复追问 |
| 研发 / 测试 | 在登录态继续使用完整面板；公开侧不暴露敏感实现细节 |
| 管理员 | 配置哪些分类启用排障字段、公开动态白名单策略 |

### 3.2 核心场景

**场景 A：外发公开链接**  
客服将 `WO-xxx` 公开链接发到群，商户打开后能看到：当前状态、问题与预期、截图/附件、关键处理节点、关闭原因（若已关单）。

**场景 B：接口报错类缺陷**  
创建或处理阶段补充：请求路径、HTTP 状态、错误码、TraceId、发生时间、客户端类型；研发在登录态可见全量，公开侧仅展示白名单字段。

**场景 C：关单验收**  
处理人关闭工单时填写「处理结论」（必填或按分类强校验），公开页顶部或独立卡片展示结论摘要。

## 四、方案概述

### 4.1 信息架构（页面区块顺序建议）

**公开详情页（免登录）建议顺序：**

1. 头部：状态、优先级、标题、工单编号  
2. **处理结论**（若状态为已解决/已关闭等且有内容）：摘要卡片，置顶  
3. 基本信息：分类、来源、创建人、处理人、时间等（与现网一致，可微调字段）  
4. 客户信息（缺陷模板）：商户编号、公司、账号、场景码、问题描述、预期结果  
5. **排障信息**（若该分类启用且存在数据）：键值对或折叠面板  
6. **证据**：问题截图 + **附件列表（只读）**（文件名、类型、预览；下载策略见安全章节）  
7. **处理动态**（新）：时间线，混合「系统事件（脱敏）+ 对外可见评论」  
8. **工单描述（原始录入）**：若与问题描述重复则折叠或显示「与问题描述一致」

**登录态详情页：**

- 在现有详情能力基础上：与公开侧**同源字段**保持文案与数据一致；排障扩展字段在对应缺陷分区或独立分组中编辑。  
- 处理结论在「关闭/解决」流程中录入，写入独立业务字段（或等价持久化方式），同时可镜像一条内部可见评论（可选，由研发评估避免重复）。

### 4.2 数据与接口原则

| 原则 | 说明 |
|------|------|
| 单事实来源 | 「处理结论」以独立字段为准，不在多处各自维护互不一致 |
| 动态统一 | 「处理动态」由后端组装：评论 + 允许公开的工单日志/流转事件子集 |
| 白名单 | 公开接口仅返回《§7.2 公开字段与脱敏规则》所列字段 |
| 分类驱动 | 排障结构化字段通过**工单分类/模板**配置启用，避免全量表单 |

## 五、功能需求详述

### 5.1 处理结论（关闭/解决说明）

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| F-01 | 在「关闭工单」或「流转至已解决/已关闭」等终态路径上，增加「处理结论」输入；支持多行文本，建议上限 2000 字（与研发对齐） | P0 |
| F-02 | 处理结论保存后，**公开详情接口**须返回该字段（或等价摘要字段）；登录态详情同步展示 | P0 |
| F-03 | 若历史工单无结论即已关单，展示为空或「暂无书面结论」，不阻塞页面 | P0 |
| F-04 | （可选）支持极简格式化（换行保留）；不支持任意 HTML 粘贴以降低 XSS 风险，或由研发统一富文本消毒策略 | P1 |

**验收要点：** 新建工单走关单流程后，公开链接在未登录条件下可见结论全文（脱敏后）。

---

### 5.2 处理动态（时间线）

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| F-10 | 公开详情页「处理记录」升级为「处理动态」，数据源 = **对外可见评论** + **系统事件（白名单类型）**，按时间升序 | P0 |
| F-11 | 系统事件至少包含（可分期）：工单创建、分派/转交、状态变更、关单（可带结论摘要或「已关闭」+ 操作人展示名） | P0 |
| F-12 | 每条动态展示：时间、操作者展示名（系统任务显示「系统」）、事件类型标签、摘要文案 | P0 |
| F-13 | 不包含：内部备注全文、IP、内部账号 ID、研发私有字段变更明细 | P0 |
| F-14 | 登录态详情可继续保留完整时间轴/变更历史；与公开动态在文案上可一致化「对外版摘要」 | P1 |

**验收要点：** 仅流转与关单、无人工评论的工单，公开页仍能看到至少一条关单/状态相关动态。

---

### 5.3 证据：附件公开只读列表

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| F-20 | 公开详情接口返回该工单**允许外发的附件**列表：文件名、文件类型、大小、预览 URL 或下载策略标识 | P1 |
| F-21 | 前端在「证据」区合并展示：`problemScreenshot` 多图 + 附件列表；图片可预览，非图可下载或提示登录下载（由安全策略定） | P1 |
| F-22 | 支持「附件不对公开链接暴露」标记（ per-file 或 per-ticket），默认策略由管理员配置 | P1 |

**验收要点：** 登录态上传的附件，在勾选/策略允许时，公开页可见且与截图区不重复混乱。

---

### 5.4 排障结构化字段（接口/功能缺陷类）

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| F-30 | 对指定分类（如：功能缺陷-接口异常）启用扩展字段组，字段建议包括：请求路径/URL、HTTP 状态码、业务错误码、TraceId/RequestId、发生时间、客户端类型（枚举：H5/小程序/App/PC/未知） | P1 |
| F-31 | 所有扩展字段均为选填，但可在分类配置上标记「推荐填写」并在创建/处理界面提示 | P1 |
| F-32 | 企微机器人/自动建单：在能力范围内解析消息中的 URL、码值写入对应字段；无法解析则留空 | P2 |
| F-33 | 公开接口返回扩展字段时遵循 §7.2；若含敏感 query（如 token）须由后端脱敏或禁止展示 | P0（安全） |

---

### 5.5 展示去重：标题 / 问题描述 / 工单描述

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| F-40 | 当 `description` 与 `problemDesc` 文本完全一致或一方为另一方子串时，公开页只展示**问题描述**区块，「工单描述」折叠为「与问题描述一致」或隐藏 | P0 |
| F-41 | 当二者均有独立有效内容时，两块均展示，并在 UI 上标注「工单描述（补充）」 | P1 |
| F-42 | 登录态可采用相同规则，避免运营困惑 | P1 |

---

### 5.6 空状态与文案

| 编号 | 需求描述 | 优先级 |
|------|----------|--------|
| F-50 | 「处理动态」为空且无任何系统事件时：展示「暂无处理动态」 | P0 |
| F-51 | 有结论无评论时：不得再显示笼统「暂无处理记录」掩盖结论存在 | P0 |

## 六、交互与体验（非像素级）

- 公开页窄屏友好：证据区横滑或网格；动态区卡片化。  
- 长内容：处理结论、问题描述默认展示 3～5 行，支持「展开全文」。  
- 可选：顶部锚点「结论 | 信息 | 证据 | 动态」，减少滚动。

## 七、安全、合规与权限

### 7.1 基本原则

- 公开链接仅靠 `ticketNo` 访问，**不得**因本需求扩大写权限（P2 单独评审）。  
- 所有新增公开字段必须经过白名单与脱敏评审。

### 7.2 公开字段与脱敏规则（初版）

**允许公开（示例，以终版安全评审为准）：**

- 工单基础：编号、标题、状态、优先级、分类名称、来源标签、创建/关闭时间、期望完成时间  
- 人员展示名：创建人、当前处理人（展示名，非登录账号）  
- 缺陷客服信息：商户编号、公司名、商户账号、场景码、问题描述、预期结果、问题截图 URL  
- 处理结论（终态）  
- 处理动态：事件类型、时间、展示名、摘要文案  
- 附件：文件名、类型、大小、经鉴权的临时访问链接（如有）

**禁止公开（示例）：**

- 内部操作日志全文、字段级 diff 中的敏感值  
- Git 分支、内部环境账号密码、完整请求体含 Cookie/Token  
- 未授权的客户隐私（如手机号全文，除非业务确认可外发）

**URL 脱敏：** 自动去除或遮蔽 `access_token`、`token`、`password` 等 query 参数。

## 八、成功指标与验收

### 8.1 指标（上线后 1～2 个迭代观察）

| 指标 | 说明 |
|------|------|
| 公开页二次追问率 | 同工单关联的重复咨询/催办次数下降（需基线） |
| 结论填写率 | 关单工单中「处理结论」非空占比 |
| 排障字段覆盖率 | 启用分类下新建工单的扩展字段至少一项有值占比 |
| 停留时长 | 公开页人均停留变化（辅助验证信息是否更可读） |

### 8.2 验收检查清单（UAT）

- [ ] 关单后公开链接可见处理结论  
- [ ] 无评论仅有流转时公开页仍有处理动态  
- [ ] 附件在策略允许时在公开页可见  
- [ ] 描述与问题描述重复时公开页无三块重复正文  
- [ ] 启用排障分类时创建/编辑可保存扩展字段且公开侧符合白名单  
- [ ] 安全抽检：公开响应中无内部敏感字段

## 九、依赖与风险

| 类型 | 内容 |
|------|------|
| 技术依赖 | 工单日志/评论/附件模型扩展；公开接口 DTO 变更；可能的 Flyway 迁移 |
| 产品依赖 | 分类与模板配置能力（若无则需简易后台开关） |
| 风险 | 附件直链泄露 → 用短期签名 URL + 可选登录下载  
| 风险 | 动态与评论重复展示 → 后端去重或统一事件模型  
| 风险 | 历史数据无结论 → 仅展示空状态，不强制补录 |

## 十、分期实施建议（与研发共创排期）

| 阶段 | 范围 |
|------|------|
| **P0** | 处理结论字段 + 公开接口/页展示；处理动态（白名单系统事件 + 评论）；描述去重规则；空状态文案 |
| **P1** | 公开附件只读列表 + 证据区合并；排障扩展字段（分类启用）+ 登录态编辑；URL 脱敏 |
| **P2** | 企微解析增强；访客 token 补传材料；关联工单展示（另立 PRD 评审） |

## 十一、开放问题

1. 处理结论是否**全部分类强制必填**，还是仅缺陷/客诉类强制？  
2. 附件公开默认策略：**默认公开** 还是 **默认不公开**？  
3. 动态中是否展示「处理人变更」的具体姓名（涉及隐私时是否只显示「已转交」）？  
4. 是否与现有「工单描述」富文本策略统一为 Markdown/纯文本以降低安全风险？
