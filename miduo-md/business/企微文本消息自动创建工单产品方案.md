# 企微文本消息自动创建工单 — 产品方案

> **版本**：v1.0  
> **日期**：2026-03-11  
> **需求来源**：通过企业微信发送文本消息，自动创建工单  
> **关联文档**：《工单系统产品设计方案 v1.3》第 4.6.4 节  
> **对应任务**：新增 Task023

---

## 一、背景与价值

### 1.1 现状分析

当前工单系统已支持企微群机器人通过 **@工单助手 + 模板格式** 创建工单（Task006），但该方式对发送人有以下限制：

| 痛点 | 说明 |
|------|------|
| **操作成本高** | 用户必须记忆并手动输入模板格式，容易出错 |
| **触发条件严格** | 必须 @机器人，且格式完全符合才能创建成功 |
| **灵活度差** | 不支持自由形式的描述文本直接转工单 |
| **场景局限** | 仅限群内交互，不支持企微私聊场景 |
| **非结构化文本被丢弃** | 大量有价值的业务消息因格式不符而未被采集 |

### 1.2 目标

实现"**零门槛建单**"：

> 用户在企微中**发送任意文本消息**（群内 @机器人，或私聊工单助手应用），系统自动提取关键信息，创建工单草稿，并引导用户确认。无需记忆任何格式。

### 1.3 业务价值

| 维度 | 价值 |
|------|------|
| 降低操作门槛 | 从"模板格式"降为"自然语言"，新用户 0 学习成本 |
| 提升建单完整性 | AI 辅助提取字段，减少关键信息遗漏 |
| 覆盖更多场景 | 支持私聊建单，不局限于已绑定的客服群 |
| 数据质量提升 | 结构化提取后自动补全，减少无效工单 |

## 二、用户角色与场景

### 2.1 用户角色

| 角色 | 建单入口 | 说明 |
|------|----------|------|
| 客服人员 | 企微群（客服问题群） | 将客户反馈的问题直接在群内转为工单 |
| 产品/研发 | 企微群（技术群/项目群） | 快速记录功能缺陷或需求 |
| 任意员工 | 企微私聊（工单助手应用） | 随时提交内部请求 |
| 群成员 | 企微群（已绑定工单分类的群） | 群内日常问题快速建单 |

### 3.1 整体流程

```
企微消息（群内@机器人 / 私聊）
        │
        ▼
  消息接收层（已有：WecomCallbackApplicationService）
        │
        ▼
  ┌─────────────────────────────────┐
  │     消息类型路由                 │
  │  ┌──────────────────────────┐  │
  │  │ 现有：格式化模板消息       │  │
  │  │  @工单助手 #分类 标题...  │  │
  │  └──────────────────────────┘  │
  │  ┌──────────────────────────┐  │
  │  │ 新增：自然语言文本消息    │  │
  │  │  任意自由文本             │  │
  │  └──────────────────────────┘  │
  └─────────────────────────────────┘
        │
        ▼ 自然语言消息
  ┌─────────────────────────────────┐
  │   NLU 文本解析器                │
  │   (WecomNaturalLangParser)      │
  │  1. 意图识别（建单/查询/其他）   │
  │  2. 关键字段提取                │
  │     - 工单类型/分类             │
  │     - 核心标题                  │
  │     - 优先级信号词              │
  │     - 关键实体（商户号等）      │
  │  3. 置信度评分                  │
  └─────────────────────────────────┘
        │
        ▼
  ┌─────────────────────────────────┐
  │   工单草稿构建                  │
  │  - 合并群默认分类               │
  │  - 智能推断优先级               │
  │  - 组装工单创建预览             │
  └─────────────────────────────────┘
        │
        ▼
  向用户推送「工单预览卡片」（待确认）
        │
        ├─ [确认创建] → 正式创建工单 → 回复成功卡片
        │
        ├─ [修改内容] → 进入交互式修改引导流程
        │
        └─ [取消] → 回复已取消消息，不建单
```

### 3.2 消息解析策略（NLU 规则引擎）

在不引入大模型 API 的前提下，采用**基于规则 + 关键字匹配**的 NLU 方案，保障系统的可控性、低延迟和零外部依赖。

#### 3.2.1 意图识别

| 意图 | 触发规则 | 示例 |
|------|----------|------|
| `CREATE_TICKET` | 包含问题/故障/申请/需求/缺陷等建单关键词，且无其他明确指令前缀 | "VPN连不上" / "申请权限" |
| `QUERY_TICKET` | 消息以"查询/查一下/工单状态"开头 | "查询WO-001状态" |
| `URGE_TICKET` | 消息以"催一下/催办/追一下"开头 | "催一下WO-001" |
| `HELP` | "帮助/怎么用/使用方法" | - |
| `UNKNOWN` | 无法识别意图 | 引导用户说明需求 |

#### 3.2.2 分类识别规则

系统通过关键词映射表自动推断工单分类：

| 关键词组 | 推断分类 | 置信度 |
|----------|----------|--------|
| 商户、客户、问题、空白、报错、无法、不能... | 缺陷修复 | 高 |
| 申请、需要、要一台、帮我... | IT支持 / 行政服务 | 中 |
| 功能、需求、新增、优化... | 研发需求 | 高 |
| 账号、密码、权限... | IT支持 > 软件问题 | 高 |
| 打印机、电脑、网络、VPN... | IT支持 > 硬件/网络 | 高 |
| 请假、打卡、工资、薪资... | 人事服务 | 高 |

**兜底策略**：置信度低于阈值时，不强行推断分类，改为在确认卡片中展示"分类待确认"，引导用户选择。

#### 3.2.3 优先级信号词识别

| 信号词 | 推断优先级 |
|--------|------------|
| 紧急、立刻、马上、急、P0、严重崩溃 | 紧急 |
| 影响全部、影响所有、高优、重要 | 高 |
| 尽快、较急 | 高 |
| （无特殊信号词，默认） | 中 |
| 不急、有空弄、优化建议 | 低 |

#### 3.2.4 实体提取

| 实体类型 | 提取规则 | 示例 |
|----------|----------|------|
| 商户编号 | 匹配 8-12 位纯数字（前置"商户"/"编号"关键词） | 10004506 |
| 手机号 | 1开头的11位手机号 | 13800138000 |
| 工单编号 | 匹配 WO-/BUG- 开头的编号格式 | WO-20260228-001 |
| 时间 | "明天/下周/YYYY-MM-DD"格式 | 下周一 → 映射为日期 |

### 3.3 交互式确认流程

#### 3.3.1 预览卡片（文本消息回复）

```
┌─────────────────────────────────────────────┐
│  📋 工单创建预览                              │
│  ─────────────────────────────────────────  │
│  标题：大转盘H5扫码预览空白                  │
│  分类：缺陷修复（研发需求）                  │
│  优先级：高                                  │
│  描述：商户10004506大转盘H5扫码预览空白...    │
│                                             │
│  回复 1 确认创建                             │
│  回复 2 修改分类                             │
│  回复 3 修改优先级                           │
│  回复 4 补充描述                             │
│  回复 0 取消                                 │
│                                             │
│  （60秒内未操作，自动取消）                  │
└─────────────────────────────────────────────┘
```

> **说明**：由于企微群机器人回调无法发送按钮交互卡片（需要企微特定API），确认流程采用**数字回复指令**的方式实现。

#### 3.3.2 会话状态管理

系统在 Redis 中维护**待确认工单草稿**的会话状态：

```
Key: wecom:draft:{chatId}:{userId}
Value: {
  "sessionId": "session_uuid",
  "draft": {
    "title": "...",
    "categoryId": null,
    "categoryPath": "缺陷修复",
    "priority": "high",
    "description": "...",
    "customFields": {...}
  },
  "step": "CONFIRM",   // CONFIRM / MODIFY_CATEGORY / MODIFY_PRIORITY / SUPPLEMENT_DESC
  "expireAt": 1234567890
}
TTL: 60秒
```

### 3.4 群聊 vs 私聊处理差异

| 维度 | 群聊 | 私聊（应用消息） |
|------|------|-----------------|
| 触发方式 | 必须 @工单助手 | 直接发送文本 |
| 默认分类 | 继承群绑定分类 | 无默认分类，需推断 |
| 回复方式 | 被动回复到群 | 应用消息推送到个人 |
| 确认超时 | 60 秒 | 5 分钟（私聊更宽松） |
| 身份校验 | 通过企微 UserId 匹配系统用户 | 同左 |

### 3.5 缺陷工单专属解析增强

当系统识别到**缺陷类**消息时，自动触发缺陷工单专属字段提取：

```
输入：@工单助手 商户10004506，北京新标酒业，账号13146232388
反馈大转盘H5第三步设置奖项时扫码预览空白，测试环境也能复现

自动提取：
  - 类型：#缺陷
  - 商户编号：10004506
  - 公司名称：北京新标酒业
  - 商户账号：13146232388  
  - 问题描述：大转盘H5第三步设置奖项时扫码预览空白
  - 场景：测试环境也能复现
```

## 四、技术设计

### 4.1 新增组件

| 组件 | 所在模块 | 说明 |
|------|----------|------|
| `WecomNaturalLangParser` | ticket-application/wecom | 自然语言消息解析器（NLU 规则引擎） |
| `WecomDraftSession` | ticket-application/wecom | 草稿会话 POJO |
| `WecomDraftSessionService` | ticket-application/wecom | 会话状态管理（基于 Redis） |
| `WecomInteractiveConfirmService` | ticket-application/wecom | 交互式确认流程处理器 |

### 4.2 改造组件

| 组件 | 改造点 |
|------|--------|
| `WecomBotMessageParser` | 新增消息类型路由：普通文本 → `NATURAL_LANGUAGE` 类型 |
| `WecomBotCommandService` | 新增 `NATURAL_LANGUAGE` 指令的处理分支 |
| `WecomMessageProcessor` | 新增会话状态判断：有待确认草稿时优先进入确认流程 |

### 4.3 关键词配置化

所有分类映射关键词、优先级信号词通过数据库/配置文件管理，支持运营人员动态修改，不需要重新部署：

```sql
-- 新增：自然语言解析关键词配置表
CREATE TABLE `wecom_nlp_keyword` (
  `id`           bigint NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `keyword`      varchar(50) NOT NULL COMMENT '关键词',
  `match_type`   tinyint NOT NULL COMMENT '匹配类型：1=分类 2=优先级 3=实体',
  `target_value` varchar(100) NOT NULL COMMENT '映射目标值（分类路径/优先级/实体类型）',
  `confidence`   tinyint NOT NULL DEFAULT 80 COMMENT '置信度(0-100)',
  `sort_order`   int NOT NULL DEFAULT 0 COMMENT '排序（高优先匹配）',
  `is_active`    tinyint NOT NULL DEFAULT 1 COMMENT '是否启用',
  `create_time`  datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `update_time`  datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `create_by`    varchar(50) NOT NULL DEFAULT '' COMMENT '创建人',
  `update_by`    varchar(50) NOT NULL DEFAULT '' COMMENT '更新人',
  `deleted`      tinyint NOT NULL DEFAULT 0 COMMENT '逻辑删除',
  PRIMARY KEY (`id`),
  KEY `idx_match_type` (`match_type`),
  KEY `idx_is_active` (`is_active`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='企微自然语言解析关键词配置表';
```

### 4.4 会话状态存储（Redis）

```
Key 格式：wecom:draft:{chatId_or_userId}:{wecomUserId}
类型：String（JSON序列化）
TTL：
  - 群聊：60 秒
  - 私聊：300 秒（5分钟）
```

### 4.5 数据库扩展

```sql
-- 在 wecom_bot_message_log 表增加字段，记录自然语言解析结果
ALTER TABLE `wecom_bot_message_log`
  ADD COLUMN `parse_type` varchar(20) DEFAULT NULL COMMENT '解析类型：template/natural_language',
  ADD COLUMN `nlp_confidence` tinyint DEFAULT NULL COMMENT 'NLU解析置信度(0-100)';
```

### 4.6 消息处理流程图（技术视角）

```
企微服务器回调
      │
      ▼
WecomCallbackController.receiveCallback()
      │（解密 + 去重已有）
      ▼
WecomMessageProcessor.process()
      │
      ├─ 检查 Redis 会话：有待确认草稿？
      │   ├─ YES → WecomInteractiveConfirmService.handleReply(数字指令)
      │   └─ NO  ↓
      │
      ├─ WecomBotMessageParser.parse() → 消息类型路由
      │   ├─ 格式化模板消息（#分类 标题）→ 现有 CREATE 流程
      │   ├─ 指令消息（帮助/查询/催办）→ 现有指令处理
      │   └─ 自然语言消息 → WecomNaturalLangParser.analyze()
      │                           │
      │                     返回 NLPAnalyzeResult
      │                     {intent, categoryPath, priority, 
      │                      title, description, customFields,
      │                      confidence}
      │                           │
      │                    confidence >= 阈值？
      │                     ├─ YES → 组装工单草稿 → 存 Redis
      │                     │        → 推送预览卡片给用户
      │                     └─ NO  → 推送"无法识别"+ 引导格式提示
      │
      ▼
  WecomBotCommandService.sendReply()（推送回复消息）
```

## 七、异常处理

| 异常场景 | 处理方式 | 机器人回复 |
|----------|----------|-----------|
| 解析置信度低，无法识别意图 | 不创建草稿 | "❓ 我没理解您的意思，请按以下格式：@工单助手 #分类 标题\n或发送"帮助"查看完整指南" |
| 用户未关联系统账号 | 拒绝建单 | "❌ 您尚未关联工单系统账号，请先访问系统完成企微授权登录" |
| 会话已超时（60秒无操作） | 清除草稿 | "⏰ 确认超时已取消，如需建单请重新发送" |
| 用户输入无效的确认指令 | 保持会话，提示 | "请回复数字 0-4 进行操作" |
| 分类无法推断且群无默认分类 | 在预览中标注"待确认" | 引导用户修改分类 |
| 工单创建失败（系统异常） | 记录日志，通知用户 | "❌ 工单创建失败，请稍后重试或到系统内手动创建" |

## 八、数据埋点与运营监控

### 8.1 关键指标

| 指标 | 计算方式 | 目标值 |
|------|----------|--------|
| 自然语言建单触达率 | 自然语言解析成功条数 / 总自然语言消息条数 | ≥ 70% |

| 分类识别准确率 | 用户未修改分类就确认 / 总预览次数 | ≥ 75% |
| 平均确认耗时 | 从发消息到确认的平均时间 | ≤ 20 秒 |
