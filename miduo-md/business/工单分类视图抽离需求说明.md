# 工单分类视图抽离需求说明

## 1. 背景

当前「所有工单」页面把普通工单、功能缺陷、监控告警放在同一个列表里展示。  
这就像医院把普通门诊、外科复查、急诊告警都排在同一个窗口，虽然都能办事，但值班人员需要先在一堆单子里分辨“哪个最急、哪个要走缺陷流程、哪个只是普通协助”，处理效率会下降。

从现有系统实现看：

1. 「所有工单」通过 `GET /api/ticket/page?view=all` 查询工单列表。
2. 工单分类来自数据库动态分类树 `ticket_category`，前端通过分类下拉筛选 `categoryId`。
3. 监控告警主要可通过 `source=alert`、告警状态 `alert_*`、告警分类识别。
4. 功能缺陷/技术缺陷主要可通过分类 ID、缺陷工作流状态、缺陷扩展信息识别。
5. 测试环境可能叫「功能缺陷」，生产环境可能叫「技术缺陷」，不能按名称写死判断。

## 2. 现状问题

| 问题 | 影响 |
|---|---|
| 告警和缺陷混在所有工单 | 普通处理人找单困难，列表噪音大 |
| 告警类工单没有独立急诊入口 | 值班人员难以快速聚焦高优先级告警 |
| 缺陷类工单没有独立缺陷工作台 | 测试/开发要在普通工单中反复筛选 |
| 分类依赖人工筛选 | 用户每次都要自己选择分类，操作成本高 |
| 分类名称跨环境不一致 | 如果按名称过滤，测试和生产容易口径错位 |

## 3. 目标

本次变更目标是把「所有工单」按分类管理中的一级分类动态拆成多个分类工单入口，让分类管理有多少个分类分组，系统就显示多少个分类工单入口：

```text
工单中心
├─ 我的工单
├─ 分类工单
│  ├─ 一级分类A
│  ├─ 一级分类B
│  └─ ...（来自分类管理）
├─ 全部工单（管理员）
└─ 工单看板
```

目标拆解：

1. **分类入口自动生成**：分类管理有多少个一级分类，左侧就展示多少个分类工单入口。
2. **分类查询包含子类**：点击一级分类后，展示该分类及所有子分类下的工单。
3. **缺陷/告警仍可识别**：分类绑定缺陷/告警工作流后，后端仍能识别它们的业务归属。
4. **管理员仍可查全量**：保留真正意义上的「全部工单」入口或高级筛选。
5. **跨环境稳定**：所有归属跟随分类管理配置，不依赖分类名称。

## 4. 非目标

本次不做以下内容：

1. 不重做工单详情页，缺陷、告警、通用工单仍复用现有详情能力。
2. 不拆分数据库主表，继续使用统一 `ticket` 表。
3. 不调整现有告警 Webhook 建单流程。
4. 不重构完整工作流引擎。
5. 不改变历史工单数据，只通过配置和查询口径改变展示入口。

## 5. 用户角色与使用场景

| 角色 | 主要入口 | 场景 |
|---|---|---|
| 普通处理人 | 分类工单 / 我的工单 | 按分类分组处理咨询、协助、配置等事项 |
| 测试人员 | 对应缺陷分类工单 | 查看待测试受理、测试复现、待回归缺陷 |
| 开发人员 | 对应缺陷分类工单 / 我的工单 | 查看待开发、开发中、临时解决等缺陷 |
| 运维/值班人员 | 对应告警分类工单 | 快速确认监控告警、处理恢复、跟踪 SLA |
| 管理员 | 全部工单 | 跨类型检索、统计核对、排查历史数据 |

## 6. 推荐产品方案

### 6.1 导航结构

推荐采用「动态分类工单入口」方案，而不是在代码里固定写几个入口。

原因：  
分类管理本身就是业务分组的来源。分类管理里有几个一级分类，系统就应该显示几个分类工单入口。这样分类一调整，菜单和查询口径自动跟着变。

```text
工单中心
├─ 我的工单
├─ 分类工单
│  ├─ 一级分类A
│  ├─ 一级分类B
│  └─ ...（来自分类管理）
└─ 工单看板

管理视角可额外显示：
└─ 全部工单
```

### 6.2 页面定义

| 页面 | 页面定位 | 默认数据范围 |
|---|---|---|
| 分类工单 | 按分类管理一级分类拆分的工作台 | 当前一级分类及所有子分类 |
| 全部工单 | 管理员全量检索 | 不默认排除任何类型 |

### 6.3 字段展示建议

#### 分类工单

| 字段 | 说明 |
|---|---|
| 工单编号 | 点击进入详情 |
| 标题 | 普通问题标题 |
| 公司名称 | 关联客户/公司 |
| 分类 | 普通分类 |
| 优先级 | 低/中/高/紧急 |
| 状态 | 当前流程状态 |
| SLA | 是否正常 |
| 创建人 | 提交人 |
| 处理人 | 当前负责人 |

#### 后续增强：缺陷类分类

当某个分类绑定「缺陷工单工作流」时，后续可在该分类工单页强化展示：

- 严重程度
- 测试受理人
- 开发负责人
- 是否有效反馈

#### 后续增强：告警类分类

当某个分类绑定「告警工单流转」时，后续可在该分类工单页强化展示：

- 告警等级
- 来源系统
- 最近触发时间
- 值班处理人

## 7. 配置口径

分类抽离必须跟随「分类管理」配置，不允许写死中文名称，也不要求运维额外维护一串分类 ID。

推荐规则：

| 分类管理配置 | 归属视图 | 说明 |
|---|---|---|
| 分类绑定「缺陷工单工作流」 | 缺陷工单 | 名称可叫功能缺陷、技术缺陷或其它业务名称 |
| 分类绑定「告警工单流转」 | 告警工单 | 告警也可通过 `source=alert` 兜底识别 |
| 分类绑定其它工作流 | 通用工单 | 默认进入普通协作列表 |

为什么这样做：

1. 测试环境叫「功能缺陷」，生产环境可能叫「技术缺陷」，但只要分类绑定的是缺陷工作流，系统就能自动识别。
2. 告警可能映射到不同子分类，只按名称容易漏单；绑定告警工作流或 `source=alert` 都能进入告警视图。
3. 分类归属由分类管理维护，运营调整分类绑定后，列表口径自动跟随变化。

## 8. 业务规则

### 8.1 分类工单规则

1. 左侧「分类工单」子菜单来自分类管理的一级分类。
2. 点击某个一级分类后，查询该分类及所有子分类下的工单。
3. 新增或停用一级分类后，刷新页面即可同步菜单。
4. 工单详情仍复用原详情页。

### 8.2 缺陷/告警归属规则

1. 分类绑定「缺陷工单工作流」时，系统可识别为缺陷类。
2. 分类绑定「告警工单流转」时，系统可识别为告警类。
3. 分类未单独绑定工作流时，继承最近父分类的工作流。
4. 告警工单额外通过 `source=alert` 兜底识别。

### 8.3 全部工单规则

1. 仅管理员或有权限用户可见。
2. 默认展示真正全量。

## 9. 验收标准

| 编号 | 验收点 | 预期 |
|---|---|---|
| AC01 | 分类管理存在 N 个一级分类 | 左侧「分类工单」下显示 N 个分类入口 |
| AC02 | 打开某个分类入口 | 只展示该分类及子分类下的工单 |
| AC03 | 该分类下有子分类工单 | 子分类工单也应出现在列表中 |
| AC04 | 管理员打开全部工单 | 可看到缺陷、告警、普通工单全量 |
| AC05 | 非管理员直接请求全部工单 | 后端拒绝 |
| AC06 | 测试/生产分类名不同 | 只要分类管理结构正确，入口和查询都正确 |
| AC07 | 顶部搜索 | 在当前分类分组内搜索；不在分类页时默认进入第一个分类分组 |
| AC08 | 工单详情 | 从任意分类列表进入详情均正常 |
| AC09 | 工单看板 | 若口径同步调整，看板与列表统计一致 |

## 10. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 一级分类过多 | 左侧菜单变长 | 后续可增加折叠、搜索或排序 |
| 分类路径历史数据缺失 | 子分类查询可能漏单 | 发布前抽样核对分类 path；必要时补齐分类 path |
| 顶部搜索目标不明确 | 用户搜索结果口径混乱 | 当前在分类页则搜当前分类；否则搜第一个分类分组 |
| 权限未控制全部工单 | 普通用户看到过多数据 | 全部工单仅管理员/授权角色可见 |
