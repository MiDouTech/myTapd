# 工单分类视图抽离需求说明

## 1. 背景

当前「所有工单」页面把普通工单、功能缺陷、监控告警放在同一个列表里展示。  
这就像医院把普通门诊、外科复查、急诊告警都排在同一个窗口，虽然都能办事，但值班人员需要先在一堆单子里分辨“哪个最急、哪个要走缺陷流程、哪个只是普通协助”，处理效率会下降。

从现有系统实现看：

1. 「所有工单」通过 `GET /api/ticket/page?view=all` 查询工单列表。
2. 工单分类来自数据库动态分类树 `ticket_category`，前端通过分类下拉筛选 `categoryId`。
3. 监控告警主要可通过 `source=alert`、告警状态 `alert_*`、告警分类识别。
4. 功能缺陷/技术缺陷主要可通过分类 ID、缺陷工作流状态、缺陷扩展信息识别。
5. 测试环境可能叫「功能缺陷」，生产环境可能叫「技术缺陷」，不能按名称写死判断。

## 2. 现状问题

| 问题 | 影响 |
|---|---|
| 告警和缺陷混在所有工单 | 普通处理人找单困难，列表噪音大 |
| 告警类工单没有独立急诊入口 | 值班人员难以快速聚焦高优先级告警 |
| 缺陷类工单没有独立缺陷工作台 | 测试/开发要在普通工单中反复筛选 |
| 分类依赖人工筛选 | 用户每次都要自己选择分类，操作成本高 |
| 分类名称跨环境不一致 | 如果按名称过滤，测试和生产容易口径错位 |

## 3. 目标

本次变更目标是把「功能缺陷」和「监控告警」从「所有工单」中抽离为独立工作入口，让不同角色像进入不同科室一样处理工单：

```text
工单中心
├─ 我的工单
├─ 通用工单
├─ 缺陷工单
├─ 告警工单
├─ 全部工单（管理员）
└─ 工单看板
```

目标拆解：

1. **通用工单更干净**：默认不展示缺陷和告警。
2. **缺陷工单更专业**：突出缺陷标题、严重程度、测试/开发负责人、回归状态。
3. **告警工单更像急诊室**：突出告警级别、触发时间、确认状态、SLA。
4. **管理员仍可查全量**：保留真正意义上的「全部工单」入口或高级筛选。
5. **跨环境稳定**：所有缺陷/告警归属通过分类 ID 和来源配置控制，不依赖分类名称。

## 4. 非目标

本次不做以下内容：

1. 不重做工单详情页，缺陷、告警、通用工单仍复用现有详情能力。
2. 不拆分数据库主表，继续使用统一 `ticket` 表。
3. 不调整现有告警 Webhook 建单流程。
4. 不重构完整工作流引擎。
5. 不改变历史工单数据，只通过配置和查询口径改变展示入口。

## 5. 用户角色与使用场景

| 角色 | 主要入口 | 场景 |
|---|---|---|
| 普通处理人 | 通用工单 / 我的工单 | 处理咨询、协助、配置等普通事项 |
| 测试人员 | 缺陷工单 | 查看待测试受理、测试复现、待回归缺陷 |
| 开发人员 | 缺陷工单 / 我的工单 | 查看待开发、开发中、临时解决等缺陷 |
| 运维/值班人员 | 告警工单 | 快速确认监控告警、处理恢复、跟踪 SLA |
| 管理员 | 全部工单 | 跨类型检索、统计核对、排查历史数据 |

## 6. 推荐产品方案

### 6.1 导航结构

推荐采用「独立菜单入口」方案，而不是只在所有工单中加筛选按钮。

原因：  
「功能缺陷」和「监控告警」不是普通筛选条件，它们有不同的处理节奏。缺陷像“维修工单”，告警像“急诊呼叫”，需要独立工作台。

```text
工单中心
├─ 我的工单
├─ 通用工单
├─ 缺陷工单
├─ 告警工单
└─ 工单看板

管理视角可额外显示：
└─ 全部工单
```

### 6.2 页面定义

| 页面 | 页面定位 | 默认数据范围 |
|---|---|---|
| 通用工单 | 普通协作工单列表 | 排除缺陷分类、告警分类、告警来源 |
| 缺陷工单 | 缺陷处理工作台 | 仅包含配置的缺陷分类 |
| 告警工单 | 告警处置工作台 | `source=alert` 或配置的告警分类 |
| 全部工单 | 管理员全量检索 | 不默认排除任何类型 |

### 6.3 字段展示建议

#### 通用工单

| 字段 | 说明 |
|---|---|
| 工单编号 | 点击进入详情 |
| 标题 | 普通问题标题 |
| 公司名称 | 关联客户/公司 |
| 分类 | 普通分类 |
| 优先级 | 低/中/高/紧急 |
| 状态 | 当前流程状态 |
| SLA | 是否正常 |
| 创建人 | 提交人 |
| 处理人 | 当前负责人 |

#### 缺陷工单

| 字段 | 说明 |
|---|---|
| 工单编号 | 点击进入详情 |
| 缺陷标题 | 缺陷主题 |
| 公司名称 | 反馈来源 |
| 缺陷分类 | 功能缺陷/技术缺陷及子类 |
| 严重程度 | P0/P1/P2 或优先级映射 |
| 当前状态 | 待测试、开发中、待回归等 |
| 测试受理人 | 最近测试受理人 |
| 开发负责人 | 当前开发处理人 |
| 是否有效反馈 | 关闭后复盘口径 |

#### 告警工单

| 字段 | 说明 |
|---|---|
| 工单编号 | 点击进入详情 |
| 告警标题 | 告警规则/对象 |
| 告警等级 | P1/P2/P3 或优先级 |
| 来源系统 | 夜莺/监控平台 |
| 首次触发时间 | 告警首次出现 |
| 最近触发时间 | 最近一次告警 |
| 当前状态 | 已触发、已确认、已恢复等 |
| SLA | 是否超时 |
| 值班处理人 | 当前告警负责人 |

## 7. 配置口径

分类抽离必须跟随「分类管理」配置，不允许写死中文名称，也不要求运维额外维护一串分类 ID。

推荐规则：

| 分类管理配置 | 归属视图 | 说明 |
|---|---|---|
| 分类绑定「缺陷工单工作流」 | 缺陷工单 | 名称可叫功能缺陷、技术缺陷或其它业务名称 |
| 分类绑定「告警工单流转」 | 告警工单 | 告警也可通过 `source=alert` 兜底识别 |
| 分类绑定其它工作流 | 通用工单 | 默认进入普通协作列表 |

为什么这样做：

1. 测试环境叫「功能缺陷」，生产环境可能叫「技术缺陷」，但只要分类绑定的是缺陷工作流，系统就能自动识别。
2. 告警可能映射到不同子分类，只按名称容易漏单；绑定告警工作流或 `source=alert` 都能进入告警视图。
3. 分类归属由分类管理维护，运营调整分类绑定后，列表口径自动跟随变化。

## 8. 业务规则

### 8.1 通用工单规则

1. 默认排除绑定「缺陷工单工作流」的分类。
2. 默认排除绑定「告警工单流转」的分类。
3. 默认排除 `source=alert` 的告警来源工单。
4. 用户手动进入「全部工单」时不应用上述排除。

### 8.2 缺陷工单规则

1. 默认只查分类管理中绑定「缺陷工单工作流」的分类。
2. 分类下拉只展示缺陷分类及其子分类。
3. 状态筛选优先展示缺陷相关状态。
4. 支持继续按编号、标题、公司、时间、优先级筛选。

### 8.3 告警工单规则

1. 默认查询条件为：
   - `source=alert`
   - 或分类管理中绑定「告警工单流转」的分类
2. 状态筛选优先展示告警状态：
   - 已触发
   - 待确认
   - 处理中
   - 稳定观察
   - 已恢复
   - 已关闭
3. 列表按优先级和最近触发时间突出急迫性。

### 8.4 全部工单规则

1. 仅管理员或有权限用户可见。
2. 默认展示真正全量。
3. 可提供开关：
   - 包含缺陷工单
   - 包含告警工单

## 9. 验收标准

| 编号 | 验收点 | 预期 |
|---|---|---|
| AC01 | 打开通用工单 | 不展示配置为缺陷/告警的工单 |
| AC02 | 打开缺陷工单 | 只展示缺陷分类内工单 |
| AC03 | 打开告警工单 | 展示 `source=alert` 或告警分类工单 |
| AC04 | 管理员打开全部工单 | 可看到缺陷、告警、普通工单全量 |
| AC05 | 分类配置为空 | 系统按兼容策略运行，不出现空页面误判 |
| AC06 | 测试/生产分类名不同 | 通过配置不同 ID 后结果正确 |
| AC07 | 顶部搜索 | 搜索时能按目标入口口径返回结果 |
| AC08 | 工单详情 | 从三个列表进入详情均正常 |
| AC09 | 工单看板 | 若口径同步调整，看板与列表统计一致 |

## 10. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 分类 ID 配错 | 列表漏单或多单 | 发布前通过分类树接口核对 ID |
| 告警只按分类过滤会漏掉 | 告警映射到非告警分类时不在告警列表 | 告警列表同时支持 `source=alert` |
| 通用工单突然少很多 | 用户误以为数据丢失 | 页面说明“已将缺陷/告警拆到独立入口” |
| 顶部搜索仍跳所有工单 | 用户搜索结果口径混乱 | 搜索入口增加目标视图或默认跳全部工单 |
| 权限未控制全部工单 | 普通用户看到过多数据 | 全部工单仅管理员/授权角色可见 |
