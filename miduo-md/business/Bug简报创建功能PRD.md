## 一、执行摘要

我们正在为米多内部工单系统的**测试/QA 角色和管理员**优化 Bug 简报创建流程，解决当前创建页面信息密度不足、字段引导不清晰、多工单关联信息无法快速整合的问题，最终目标是将**Bug 简报从创建到提交审核的平均耗时从约 20 分钟缩短至 10 分钟以内**，同时保证简报内容的规范性和可追溯性。

**核心改进方向**：
1. 增强工单关联时的**智能预填充**能力（问题描述、影响范围、解决方案自动聚合）
2. 补全**临时解决方案 + 临时解决时间**双轨制解决方案字段
3. 优化**缺陷归因**体系（二级联动级联选择器，覆盖配置错误/编码缺陷/设计缺陷等 7 个一级分类）
4. 完善**审核流程**可见性（状态日志时间轴、审核意见展示）

## 二、问题陈述

### 2.1 问题背景

米多内部工单系统已具备完整的 Bug 简报管理能力（`bug_report` 主表 + 责任人/工单/日志/附件从表），现有功能已上线并通过内部使用验证。但经过实际使用反馈，在**简报创建环节**存在以下核心痛点：

### 2.2 谁有这个问题？

- **主要用户**：测试/QA 人员（TESTER 角色），负责在缺陷修复后整理和提交 Bug 简报
- **次要用户**：管理员（ADMIN / TICKET_ADMIN / QA_MANAGER），负责审核和归档简报
- **观察者**：开发负责人（DEVELOPER），需要查看简报了解缺陷归因和解决方案

### 2.3 具体痛点

| 痛点 | 描述 | 影响程度 |
|------|------|---------|
| 手动整合多工单信息耗时 | 当一个简报关联多个工单时，需要手动从每个工单复制问题描述、影响范围等内容后拼接 | 高 |
| 解决方案字段单一 | 原有系统仅有"解决方案"一个字段，无法区分"临时规避方案"和"彻底修复方案"，造成内容混乱 | 中 |
| 归因选择缺乏引导 | 逻辑归因字段为自由文本输入，不同人员填写标准不一，统计分析价值低 | 中 |
| 严重级别定义不清 | P0-P4 各级别边界模糊，不同人员对同一问题可能给出不同级别 | 中 |
| 提交前无校验提示 | 提交审核前未明确提示必填项缺失，导致审核人反复驳回 | 低 |

## 四、战略背景

### 4.1 业务目标对齐

- **目标**：提升 Bug 管理规范化程度，支撑季度复盘和质量趋势分析
- **OKR 对应**：
  - KR1：Bug 简报提交率（已关闭缺陷中有对应简报的比例）从当前约 60% 提升至 85%
  - KR2：简报平均从创建到归档周期从 3 天缩短至 1.5 天
  - KR3：简报审核一次通过率从约 70% 提升至 88%

### 4.2 为什么现在做？

- 现有系统已经过 V6（初始化）和 V17（临时解决方案）两轮迭代，基础数据结构完备
- 近期 Bug 数量增加，QA 团队反馈简报创建体验急需优化
- 统计看板已上线，需要高质量的简报数据作为输入，当前数据质量参差不齐

### 4.3 竞品参考

| 工具 | 优势 | 与米多的差距 |
|------|------|------------|
| TAPD | 完整的缺陷生命周期 + 变更历史 + 字段引导 | 米多已对标实现变更历史；简报创建流程仍有差距 |

## 五、解决方案概述

### 5.1 功能定位

Bug 简报（Bug Report）是对一类相关缺陷的**汇总性分析文档**，不同于单条工单（Ticket）——工单记录的是单次缺陷事件，简报记录的是对这类缺陷的**系统性分析和处置结果**。

### 5.2 整体用户流程

```
创建简报
    │
    ├─ 1. 关联工单（必填，至少1个）
    │       └─ 自动预填：问题描述 / 影响范围 / 彻底解决方案 / 严重级别
    │
    ├─ 2. 填写/修正问题描述（必填）
    │
    ├─ 3. 填写逻辑归因（可选，级联二级选择）
    │       └─ 配置错误 / 编码缺陷 / 设计缺陷 / 第三方问题 / 数据问题 / 运维问题 / 测试遗漏
    │
    ├─ 4. 选择缺陷分类（必填）
    │       └─ 功能异常 / 交互异常 / 性能问题 / 数据异常 / 安全漏洞 / 兼容性问题 / 接口异常
    │
    ├─ 5. 填写问题周期（开始日期 → 解决日期）
    │
    ├─ 6. 选择严重级别（P0-P4，含说明文字）
    │
    ├─ 7. 填写临时解决方案 + 临时解决时间（可选）
    │
    ├─ 8. 填写彻底解决方案（可选，自动从工单预填）
    │
    ├─ 9. 选择责任人（可多选）
    │
    ├─ 10. 选择审核人（提交审核时必填）
    │
    └─ 11. 保存草稿 / 直接提交审核
                │
                ▼
         待审核 (PENDING_REVIEW)
                │
          ┌─────┴─────┐
          ▼           ▼
     审核通过       审核驳回
     (ARCHIVED)   (REJECTED)
                       │
                       └─ 编辑后重新提交
```

### 5.3 关键设计决策

**决策1：智能预填充保持可编辑**
- 工单关联后自动预填的字段（问题描述、影响范围、解决方案）均可手动覆盖修改
- 系统追踪"是否被用户手动修改"，二次修改工单关联时，仅更新未被手动修改过的字段

**决策2：双轨制解决方案**
- 「临时解决方案」= 权宜之计（如关闭某功能入口、发布紧急补丁）
- 「彻底解决方案」= 根本性修复（如重构相关模块、修复底层逻辑错误）
- 两者可以独立填写，也可以同时存在

**决策3：严重级别带说明文字**
- 每个 P 级别在选择器下方展示对应的定义文字，减少因理解偏差导致的错误评级

## 六、成功指标

### 6.1 主要指标

| 指标 | 当前值 | 目标值 | 衡量周期 |
|------|-------|-------|---------|
| 简报创建到提交耗时（P50） | ~20 分钟 | ≤ 10 分钟 | 上线后 30 天 |
| 简报审核一次通过率 | ~70% | ≥ 88% | 上线后 30 天 |
| 简报字段完整率（必填项全填率） | 未统计 | ≥ 95% | 上线后 30 天 |

### 6.2 辅助指标

- 智能预填充字段被手动修改的比例（反映预填准确性，目标 < 40%）
- 逻辑归因字段填写率（目标 > 80%，当前为可选字段）
- 审核驳回原因分布（用于后续优化简报模板）

### 6.3 护栏指标（不能变差的）

- 简报创建接口响应时间（P99 ≤ 300ms）
- 工单关联搜索响应时间（P99 ≤ 500ms）
- 前端页面首屏加载时间（P99 ≤ 1000ms）

#### F2.3 智能预填充规则

当用户关联工单后，系统自动从工单数据中提取信息预填对应字段：

| 目标字段 | 预填数据来源 | 预填逻辑 |
|---------|------------|---------|
| 问题描述 | `ticket.description` + `bugCustomerInfo.problemDesc` + `bugTestInfo.actualResult` + `bugTestInfo.reproduceSteps` | 将多工单信息整合为自然语言段落，前 1000 字 |
| 影响范围 | `bugTestInfo.impactScope` | 多工单用"；"连接，截取 500 字 |
| 彻底解决方案 | `bugDevInfo.fixSolution`（取第一个工单） | 最多 1000 字 |
| 严重级别 | `bugTestInfo.severityLevel` | 取所有关联工单中最高（最严重）级别 |

**预填覆盖规则**：
- 用户**新增**工单关联时，触发预填（仅更新"未被手动修改"的字段）
- 用户**删除**工单关联时，不触发预填（避免覆盖已手动修改的内容）
- 编辑已有简报加载时，不触发预填

#### F2.4 严重级别说明文字

在严重级别下拉框下方，以 5 行说明卡片形式展示各级定义，帮助用户准确评级：

| 级别 | 颜色 | 定义描述 |
|------|------|---------|
| P0（致命） | 红色 `#f56c6c` | 核心功能完全不能使用，影响大量商户，阻碍正常业务操作；财务等关键数据丢失无法追回；造成数据损坏泄露。 |
| P1（重大） | 橙色 `#e6a23c` | 核心业务部分不能使用，影响较多用户正常业务开展；数据异常但可追回；出现设计未预料的严重异常。 |
| P2（严重） | 蓝色 `#409eff` | 重要功能异常但有临时替代方案；中等范围用户受影响；有一定经济损失风险。 |
| P3（一般） | 绿色 `#67c23a` | 非核心业务部分不能使用；条件触发的偶发错误；无直接经济损失但影响用户体验。 |
| P4（轻微） | 灰色 `#909399` | 轻微影响非核心业务或功能；概率性影响用户体验，如文案排版、错别字等视觉问题。 |

### 7.5 功能模块 F4：Bug 简报状态流转

```
 创建
  │
  ▼
DRAFT（待填写）
  │
  ├─ 保存草稿（反复编辑）→ DRAFT
  │
  ├─ 提交审核 → PENDING_REVIEW（待审核）
  │                │
  │          ┌─────┴─────┐
  │          ▼           ▼
  │       ARCHIVED    REJECTED
  │       (已归档)    (已退回)
  │                     │
  │                     └─ 重新编辑 → DRAFT → 重新提交 → PENDING_REVIEW
  │
  └─ 作废 → VOIDED（已作废，终态）
```

**状态说明**：

| 状态 | 中文 | 描述 |
|------|------|------|
| DRAFT | 待填写 | 初始状态，创建后默认 |
| PENDING_REVIEW | 待审核 | 已提交，等待审核人处理 |
| REJECTED | 已退回 | 审核不通过，可重新编辑提交 |
| ARCHIVED | 已归档 | 审核通过，进入归档，只读 |
| VOIDED | 已作废 | 已作废，不可恢复 |

## 十一、数据模型参考

### 11.1 简报主表关键字段

```sql
CREATE TABLE `bug_report` (
    `id`                  bigint(20)   NOT NULL AUTO_INCREMENT,
    `report_no`           varchar(50)  NOT NULL,                 -- BR-YYYYMMDD-NNN
    `status`              varchar(20)  NOT NULL DEFAULT 'DRAFT', -- DRAFT/PENDING_REVIEW/REJECTED/ARCHIVED/VOIDED
    `problem_desc`        text,                                   -- 问题描述
    `logic_cause_level1`  varchar(50),                           -- 逻辑归因一级
    `logic_cause_level2`  varchar(50),                           -- 逻辑归因二级
    `logic_cause_detail`  text,                                   -- 归因明细
    `defect_category`     varchar(50),                           -- 缺陷分类
    `introduced_project`  varchar(200),                          -- 引入项目
    `start_date`          date,                                   -- 开始时间
    `resolve_date`        date,                                   -- 彻底解决时间
    `temp_resolve_date`   date,                                   -- 临时解决时间（V17新增）
    `solution`            text,                                   -- 彻底解决方案
    `temp_solution`       text,                                   -- 临时解决方案（V17新增）
    `impact_scope`        text,                                   -- 影响范围
    `severity_level`      varchar(10),                           -- P0/P1/P2/P3/P4
    `reporter_id`         bigint(20),                            -- 反馈人
    `reviewer_id`         bigint(20),                            -- 审核人
    `remark`              text,
    `submitted_at`        datetime,
    `reviewed_at`         datetime,
    `review_comment`      text,
    `created_by_user_id`  bigint(20),
    PRIMARY KEY (`id`)
);
```
