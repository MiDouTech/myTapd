# 企微图片消息工单关联 — 产品方案

> **版本**：v1.0  
> **日期**：2026-03-14  
> **需求来源**：用户通过企微群发送图片（截图）辅助建单，需支持图片自动关联到工单  
> **关联文档**：《企微文本消息自动创建工单产品方案》《工单系统产品设计方案》  
> **对应任务**：待拆解（建议 Task024）

---

## 一、背景与价值

### 1.1 现状分析

当前系统对企微回调消息存在以下限制：

| 痛点 | 说明 |
|------|------|
| **仅处理文本** | 企微回调只处理 text 类型，`msgType != "text"` 即丢弃 |
| **截图无法采集** | 实际反馈场景中，用户常附上截图描述问题，图片被直接忽略 |
| **信息不完整** | 仅文字描述难以复现问题，图片是缺陷工单的高频刚需 |

### 1.2 目标

> 支持**图片消息自动关联到工单**：用户在企微群发送「文字描述 + 截图」后，系统将图片持久化并关联到对应工单，实现完整建单体验。

### 1.3 业务价值

| 维度 | 价值 |
|------|------|
| 提升建单质量 | 缺陷工单可携带截图，便于研发复现问题 |
| 降低沟通成本 | 用户一次发送即可完成文字+图片提交，无需二次补充 |
| 覆盖真实场景 | 满足「先发图后描述」「先描述后发图」两种常见习惯 |

## 二、核心挑战分析

### 2.1 消息离散性

企微群消息是**逐条独立回调**，无天然关联：

```
用户操作：发文字 "#缺陷 登录页面崩溃" + 发截图1 + 发截图2
系统收到：3 个独立的 Webhook 事件（text、image、image）
```

需要**智能关联**：将同一用户、同一群、时间接近的文字与图片绑定到同一工单。

### 2.2 图片临时性

企微返回的图片资源有时效限制：

| 资源类型 | 说明 | 处理要求 |
|----------|------|----------|
| PicUrl | 企微临时外链预览 | 有时效，需及时转存 |
| MediaId | 需通过企微 API 下载 | 不及时调用 API 会失效 |

必须**及时下载 + 持久化存储**，否则图片会丢失。

### 2.3 语义关联性

用户行为顺序不确定：

| 场景 | 用户行为 | 系统处理逻辑 |
|------|----------|--------------|
| 先文后图 | 先发文字建单，再发截图 | 图片收到后，关联到最近 5 分钟内该用户创建的工单 |
| 先图后文 | 先发截图，再发文字建单 | 文字触发建单时，拉取该用户 5 分钟内暂存的图片并批量关联 |

需要**时间窗口 + 同用户同群**的关联策略。

## 三、用户角色与场景

### 3.1 用户角色

沿用现有企微工单体系角色：

| 角色 | 使用入口 | 图片需求 |
|------|----------|----------|
| 客服/产品 | 企微群 @工单助手 | 发送客户问题截图，辅助建单 |
| 研发/测试 | 企微群 @工单助手 | 发送 Bug 复现步骤截图、报错界面 |
| 任意员工 | 企微私聊工单助手 | 发送申请/问题相关图片 |

### 3.2 核心场景

#### 场景一：先描述后发图（推荐流程）

```
企微群对话：
  T+0s  用户：@工单助手 #缺陷 登录页面崩溃
  T+3s  用户：[发送截图1]
  T+5s  用户：[发送截图2]

工单助手回复（文字消息触发建单后）：
  ✅ 工单已创建：WO-20260313-001
  📋 标题：登录页面崩溃
  🖼️ 图片附件：2张（已上传）
  👉 处理人：张三 | 优先级：高
```

#### 场景二：先发图后描述（常见习惯）

```
企微群对话：
  T+0s  用户：[发送截图]
  T+10s 用户：@工单助手 #缺陷 上面是错误弹窗，商户10004506反馈的

工单助手回复：
  ✅ 工单已创建：WO-20260313-002
  📋 标题：错误弹窗（商户10004506反馈）
  🖼️ 图片附件：1张（已上传）
```

#### 场景三：仅发图片（等待关联）

```
企微群对话：
  T+0s  用户：[仅发送截图，未发文字]

可选回复（可配置为静默）：
  📷 图片已接收，请在5分钟内补充文字描述，图片将自动关联到新工单。
```

#### 场景四：图片超时未关联

5 分钟内未收到文字建单，超时策略（可配置）：
- **CREATE_TICKET**（默认）：静默创建「仅有图片」的工单，标题待补充
- **EXPIRE**：丢弃图片，标记为 EXPIRED
- **NOTIFY_USER**：回复用户「请补充文字描述」

## 四、功能设计

### 4.1 整体流程

```
企微回调 (POST /api/wecom/callback)
         │
         ├── MsgType = text
         │     └── [现有流程] 解析 → 创建工单 → 触发图片关联
         │
         └── MsgType = image  ← 新增处理分支
               │
               ├── Step1: 异步下载图片（企微 MediaId API / PicUrl 降级）
               ├── Step2: 上传到七牛云（持久化）
               ├── Step3: 暂存到 wecom_pending_image（status=PENDING）
               └── Step4: 尝试关联到近期工单（时间窗口）
                     ├── 找到 → 直接关联，status=LINKED
                     └── 未找到 → 等待后续文字消息触发关联
```

### 4.2 时间窗口关联规则

| 规则 | 触发条件 | 处理逻辑 |
|------|----------|----------|
| **规则A：先文后图** | 同一 chatId + 同一 fromUserId；图片发送时间在工单创建时间 [0, +5 分钟] 内 | 自动关联到该工单 |
| **规则B：先图后文** | 同一 chatId + 同一 fromUserId；文字消息触发工单创建时 | 查找该用户最近 5 分钟内 PENDING 的暂存图片，批量关联 |
| **规则C：超时未关联** | PENDING 超过 5 分钟 | 定时任务按策略处理：静默建单 / 丢弃 / 提醒用户 |

### 4.3 用户交互反馈

#### 4.3.1 文字 + 图片（图片追加到工单）

```
✅ 工单已创建：WO-20260313-001
📋 标题：登录页面崩溃
🖼️ 图片附件：2张（已上传）
👉 处理人：张三 | 优先级：高
```

#### 4.3.2 仅图片（等待关联，可选是否回复）

```
📷 图片已接收，请在5分钟内补充文字描述，图片将自动关联到新工单。
```

> 可通过配置 `notify-on-pending: false` 静默处理，避免打扰用户。

#### 4.3.3 图片超时未关联（默认策略：静默建单）

```
工单 WO-20260313-002 已创建
📷 包含1张图片，标题待补充（请@工单助手 更新工单描述）
```

### 4.4 配置项（Nacos）

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `wecom.image.enabled` | boolean | true | 是否开启图片消息处理 |
| `wecom.image.association-window-minutes` | int | 5 | 时间窗口（分钟） |
| `wecom.image.timeout-strategy` | enum | CREATE_TICKET | 超时策略：EXPIRE / CREATE_TICKET / NOTIFY_USER |
| `wecom.image.notify-on-pending` | boolean | false | 收到图片时是否立即回复用户 |
| `wecom.image.max-images-per-ticket` | int | 10 | 单工单最大图片数 |

## 五、数据设计

### 5.1 新增表：wecom_pending_image

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |

## 六、技术设计摘要

### 6.1 新增服务组件

| 组件 | 职责 |
|------|------|
| WecomImageHandlerService | 图片消息处理、暂存、关联逻辑 |
| WecomImageDownloadService | 企微图片下载（MediaId API / PicUrl 降级）+ 七牛上传 |

### 6.2 改造现有组件

| 组件 | 改造点 |
|------|--------|
| WecomCallbackApplicationService | 新增 image 消息类型处理分支，去除对非 text 的过滤丢弃 |

### 6.3 关键技术决策

| 问题 | 决策 | 理由 |
|------|------|------|
| 图片下载方式 | 优先 MediaId API，降级 PicUrl HTTP | MediaId 更稳定，PicUrl 为临时外链 |
| 关联策略 | 5 分钟时间窗口 + 同用户 + 同群 | 兼容先文后图 / 先图后文 |
| 超时策略 | 默认静默创建工单 | 比丢弃图片体验更好 |
| 存储 | 七牛云（复用现有能力） | 无需引入新依赖 |
| 并发 | 图片下载用 @Async 异步 | 不阻塞主流程，避免回调超时 |

## 七、接口设计

### 7.1 改造的现有接口

| 接口 | 改造内容 |
|------|----------|
| POST /api/wecom/callback | 内部逻辑扩展：新增 `MsgType=image` 处理分支，对外签名不变 |

### 7.2 内部接口（无对外 API）

图片处理为企微回调内部逻辑，无需新增对外 REST 接口。工单附件查询沿用现有 `/api/ticket/detail/{id}` 等接口，返回的附件列表中包含企微来源图片（`source=WECOM_BOT`）。

## 九、异常处理

| 异常场景 | 处理方式 | 用户感知 |
|----------|----------|----------|
| 企微 MediaId 下载失败 | 降级 PicUrl，若均失败则标记 FAILED | 工单创建成功但该图片未关联 |
| 七牛上传失败 | 标记 FAILED，记录日志 | 同上 |
| 时间窗口内无工单可关联 | 保持 PENDING，等待后续文字建单 | 收到「图片已接收」提示（如开启） |
| 单工单图片数超限 | 超出部分不关联，记录日志 | 仅前 N 张关联 |
| 企微回调超时 | 图片下载异步执行，主流程快速返回 | 无感知，图片可能稍晚关联 |
