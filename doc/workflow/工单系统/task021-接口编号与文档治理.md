# Task021：接口编号与文档治理

> **业务模块**：工单系统（治理专项）  
> **依赖**：Task001~Task020  
> **预估工时**：1~2天  
> **对应规范**：API 接口编号管理规范、项目文档管理规范

---

## 一、任务目标

解决当前“接口编号重复、文档状态冲突、多个来源不一致”的治理问题，建立单一可信文档源（SSOT）与自动校验机制，保证后续任务不再引入编号混乱。

## 二、现状问题（已识别）

1. **同一编号在不同模块重复使用**（如 API000001、API000020）  
2. **功能接口对应关系文档存在多版本冲突**  
   - `doc/workflow/工单系统/功能接口对应关系.md`
   - `doc/workflow/功能接口对应关系.md`
3. **部分接口状态与实际代码不一致**（文档标“待开发”，代码已实现）
4. **缺少自动检查，易在新增接口时再次污染**

## 三、治理范围

### 3.1 文档统一

- 指定单一主文档：`doc/workflow/工单系统/功能接口对应关系.md`
- 根目录同名文档改为：
  - 归档说明（指向主文档），或
  - 删除并在 README 中声明唯一入口

### 3.2 编号治理

- 按模块重划编号区间并保证全局唯一（建议）：
  - 工单核心：API000001~API000099
  - 工作流与分派：API000100~API000199
  - SLA与通知：API000200~API000299
  - Bug简报：API000300~API000399
  - 认证与用户：API000400~API000499
  - 开放能力与集成：API000500~API000599
- 输出“旧编号 -> 新编号”映射表，避免追溯困难

### 3.3 代码注释同步

- 后端：Controller 注释 + `@Operation` 描述统一
- 前端：`src/api/*.ts` 注释统一
- 文档：Task 文档与 API 设计文档同步更新

### 3.4 自动校验机制

- 增加编号校验脚本（建议）
  - 路径：`scripts/qa/check-api-numbering.mjs`
  - 校验项：
    - 编号重复检测
    - 缺失编号检测
    - 文档与代码编号一致性检测
- 将脚本纳入 CI 或发布前检查清单

## 四、详细实施步骤

### Step 1：盘点

- 提取后端 Controller、前端 API 注释、文档表格中的全部编号
- 形成《编号盘点清单（现状）》

### Step 2：规划

- 确认目标编号区间与映射方案
- 评审并冻结迁移规则（避免反复改号）

### Step 3：执行

- 批量更新代码注释与文档
- 确认主文档唯一性并处理重复文档

### Step 4：校验

- 运行校验脚本，确保无重复、无遗漏、无冲突
- 输出《治理结果报告》

## 六、验收标准

- [ ] 任一接口编号在全仓范围内全局唯一  
- [ ] 功能接口主文档唯一，其他位置仅保留索引或归档说明  
- [ ] 后端 Controller、前端 API 注释与主文档一致  
- [ ] 新增接口可通过脚本校验，防止重复引入  
- [ ] 治理过程有映射表，历史编号可追溯  

## 七、风险与应对

- **风险1：编号变更影响历史沟通与测试用例**  
  处理：保留映射表，在过渡期双写（旧编号注释 + 新编号注释）后再清理。

- **风险2：多文档维护人员并行修改导致回退**  
  处理：明确主文档 owner，其他文档只允许引用不允许重新维护编号表。

- **风险3：一次性全量修改引入遗漏**  
  处理：先小范围试跑（一个模块），校验通过后再全量执行。

## 八、与后续任务衔接

- Task022 将编号校验纳入发布基线与 CI 门禁  
- 后续新增 Task 必须先分配编号再开发，禁止“先写接口后补编号”
