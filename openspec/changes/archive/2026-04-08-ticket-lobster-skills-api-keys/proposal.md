# 变更提案：工单龙虾技能包分发与个人 API 密钥

## 背景

希望内部工单系统向「龙虾」等 AI 助手提供类似微盟 Admin Skills 的体验：用户可**下载技能包**（文档与工具约定），并通过**个人 API 密钥**安全调用工单接口完成查询、流转与报表类能力。此前已在架构层面明确：**技能包与敏感凭据必须分离**；本提案将需求收敛为可交付变更。

## 现状问题

1. 工单平台以 **JWT + 企微登录** 为主，缺少面向 **Agent/自动化客户端** 的长期凭据模型与自助发放入口。
2. 无官方 **技能包（manifest + 使用说明）** 的版本化分发路径，龙虾侧难以稳定对齐 API 与权限边界。
3. 若将密钥写入技能 zip 或仓库，会造成**不可吊销的泄露面**；缺少**按用户、按密钥实例**的管理与审计约定。

## 目标

- 登录用户可在工单 Web **自助创建 / 命名 / 禁用 / 删除** 个人 API 密钥；**完整 Secret 仅在创建时展示一次**。
- 提供**可公开或登录后下载**的技能包产物（无密钥）：含 README、`config.example`、工具/MCP 与 API 的对应说明、版本与兼容矩阵。
- Agent 请求使用 **API Key 鉴权**（或等效 Header），权限 **不高于** 该用户在系统中的角色；支持 **作用域（scope）** 与 **速率限制**（可先粗粒度）。
- 关键操作具备 **审计能力**（哪个 keyId、何时、调用了哪类接口），日志中**不落完整密钥**。

## 非目标

- 不规定龙虾具体运行时（Cursor / 企微机器人 / 自研网关），仅约定**客户端如何使用密钥与技能包**。
- 不在本变更中实现完整的 **OAuth Device Flow**（可作为后续增强）；首版以 **API Key** 为主。
- 不实现技能包的 **npm/cdn 自动发布流水线**（可先用仓库内 `artifacts/` 或静态资源路径，任务中可拆分）。
- 不替代现有 **服务账号/系统任务** 模型；个人密钥与服务账号分轨，后续可补充组织级密钥。

## 验收标准

1. 用户在「个人设置（或集成）」中创建密钥后，可用该密钥通过 HTTP 调用**已文档化的一组 Agent 专用接口**（或现有 REST 在 Key 鉴权下等价行为），且权限与浏览器登录用户一致。
2. 列表页仅显示密钥**脱敏前缀**、名称、创建时间、状态；支持禁用与删除；支持轮换（删除旧 + 建新）。
3. 发布的技能包内**不存在**任何真实密钥；`config.example` 仅含占位符与环境变量说明。
4. 安全评审项：HTTPS 传输、密钥存储哈希（不落库明文）、审计字段齐备；文档明确禁止将 Key 提交 Git。

## 影响范围

- **后端**：`ticket-platform` 鉴权模块、新增或扩展 Controller/Application、持久化（新表或扩展用户扩展表）、可选 `ticket-job`（与定时报表订阅若拆后续任务）。
- **前端**：`miduo-frontend` 个人中心 / 集成设置页（密钥 CRUD UI）。
- **文档**：`miduo-md/workflow/` 或 `ticket-platform` 下 Agent 集成说明；技能包内 README。
- **运维**：密钥长度、环境变量命名、日志脱敏规范。
