# 变更提案：工单详情评论 @ 提及与企微推送

## 背景

工单详情页的「评论与处理记录」已支持富文本评论（`POST /api/ticket/{id}/comment`），站内通知体系已具备 `NotificationType.COMMENT` 与用户通知偏好（含企微渠道开关）。产品期望交互对齐语雀等协作工具：在评论中可 **@ 指定同事**，被 @ 的人除站内感知外，还能收到 **企业微信应用消息**，避免漏看。

`miduo-md/workflow/工单系统/task006-企业微信深度集成.md` 已列出交付项「评论@提及」与企微应用消息推送，本变更将该项在工单 Web 详情场景落地。

## 现状问题

1. 评论仅提交 `content` 字符串，无结构化「被提及用户」列表，后端无法可靠识别要通知的人（纯文本 @ 昵称易重名、难解析）。
2. 发表评论后未对被 @ 用户触发企微 `message/send`（已有 `WecomClient.sendTextMessage` 等能力可复用）。
3. 前端评论编辑器无 @ 选人能力，用户只能手动打字，体验与语雀不一致。

## 目标

- 评论发表时支持 **显式传入被 @ 的用户 ID 列表**（与富文本展示解耦：正文中可同步高亮 @，以产品/UI 为准）。
- 对被 @ 用户（需为系统内有效用户）发送 **企微应用文本消息**，内容包含工单标识、评论摘要、跳转入口（与现有工单通知文案风格一致）。
- 与现有 **站内通知** 行为协调：可沿用或扩展 `COMMENT` 类型，确保通知中心可展示「谁在工单中 @ 了你」。
- **去重**：同一评论中同一用户仅通知一次；可选排除评论发布者本人。

## 非目标

- 不在本期实现企微群内 @、机器人侧解析评论。
- 不做全站「@ 我」独立聚合页（除非已有通知中心可满足，仅依赖通知列表即可）。
- 不强制要求评论 HTML 与 mentions 列表绝对同步校验（以服务端 `mentionedUserIds` 为准触发通知，正文由前端编辑器负责展示）。

## 验收标准

1. 工单详情评论框支持 @ 选择同事（键盘触发或按钮，行为接近语雀：选人后写入正文标记 + 提交列表）。
2. 提交评论后，被 @ 用户收到企微应用消息（在企微通知偏好开启且账号已绑定 `wecom_userid` 的前提下）。
3. 被 @ 用户收到站内通知（类型与文案可读，可点击进工单详情）。
4. `mentionedUserIds` 为空时行为与现网一致，无回归。
5. 非法用户 ID、重复 ID、未绑定企微的用户：跳过企微推送并记日志，不导致评论失败。

## 影响范围

- **后端**：`ticket-entity`（入参 DTO）、`ticket-application`（`TicketApplicationService.addComment` 及通知/企微发送）、可能复用 `WecomClient`、`NotificationApplicationService` 或既有工单通知门面。
- **前端**：`TicketDetailView.vue` 评论区、`ticket.ts` API、类型定义；列表/预览如需展示 @ 样式可后续迭代。
- **文档**：`miduo-md/workflow/功能接口对应关系.md` 等（扩展 API000508 请求体或新增接口编号，实施阶段按规范更新）。
