### Core Model（必须说清楚）

Miduo SSO uses **two tokens**:

- **exchange token**（交换 token / request field `token`）: short-lived, opaque, **one-time** token delivered to the browser via redirect (QUERY or FORM_POST).
- **sessionToken**: returned only after backend validation; must be stored and used **server-side only** (refresh/revoke).

Non-negotiables:

- `appSecret` is **backend-only**; never expose to browsers/mobile apps.
- Third party must not trust URL PII (`mobile`, `name`, etc.) as identity proof.
- Login bridge `redirectUri` must match **allowlist** (strict). **Production should require HTTPS**. (Current backend code accepts `http`/`https`, but HTTPS is strongly recommended.)
- `state` must be generated by the third party and verified on callback (CSRF/session binding).
- For `validate-login-token`, Miduo returns `wework_userid` when `valid=true`. This value is the authoritative Enterprise WeChat userid (企业微信用户ID).

- **第三方回调/落地（必须提供完整 URL）**  
  - `callbackUrl`（你方接收 exchange token 的入口页面/接口对应的页面）：`<https://third.example.com/sso/callback>`

- **登录桥回跳白名单（必须精确列出）**

    - `<https://third.example.com/sso/callback>`  
  - **确认口径**：是否要求精确匹配到 path（推荐）还是仅同 host（目前后端按 host 匹配；如需更严格必须提前约定）

- **state 策略（防 CSRF）**  
  - 第三方必须回答：`state` 如何生成、保存（cookie/session/db）、回调时如何校验与过期（建议 5–10 分钟）

- **第三方必须具备的后端接口/能力**  
  - `POST /auth/miduo/callback`（或等价）：前端把 `token/state` 交给后端  
  - 后端能调用：`validate-login-token` / `refresh-session-token` / `revoke-session-token`  
  - `sessionToken` **只存后端**（db/redis），不会写入浏览器存储

### P1 强烈建议确认（不上线会很痛）

- **时钟偏差窗口**：允许 `X-App-Timestamp` 偏差 ±`<300>` 秒  
- **nonce 去重窗口**：`<600>` 秒（第三方侧需要保证同 appCode 下 nonce 不复用）  
- **签名输出格式**：Base64（是否需要去掉换行、是否 URL-safe Base64）  
- **日志脱敏策略**：token/sessionToken/appSecret 全量禁止打印；最多输出 hash 或前 6 后 4

## Required Completeness Check（交付信息必须齐）

### A) Identity And Environment

The handoff should clearly contain:

- `baseUrl`
- `appCode`
- `appSecret`
- `shortcutId`
- `callbackUrl`（第三方落地页/回调入口）
- `redirectUri` allowlist（登录桥回跳白名单）
- token delivery mode: `QUERY` or `FORM_POST`
- exchange token TTL (seconds)
- sessionToken TTL (seconds)
- session max age if enabled (seconds)
- allowed claims (e.g. `userId`, `mobile`, ...)
  - Note: `allowedClaims` controls optional PII fields like `mobile`/`email`. When returned, these fields are **plain/original (not masked/encrypted)**, so treat them as sensitive.

Recommended (if missing, flag as P1):

- clock skew tolerance window (e.g. ±300s)
- nonce uniqueness window (e.g. 5–10 minutes)
- exact signature encoding (Base64 recommended; if hex allowed, clarify lowercase/uppercase)

### B) API Surface

The handoff should clearly contain:

- `POST /api/sso/redirect-url`（发放跳转 URL）
- `GET /api/gzt/workbench/shortcut/jump/{shortcutId}`（工作台快捷入口统一跳转）
- `POST /api/open/sso/validate-login-token`（校验 exchange token）
- `POST /api/open/sso/refresh-session-token`（续期 sessionToken）
- `POST /api/open/sso/revoke-session-token`（吊销 sessionToken）
- `GET /api/sso/bridge/redirect-url`（登录桥：失效后回米多再回第三方）

### C) Authentication And Signing（开放接口必签）

Open API headers (required):

```text
Content-Type: application/json
X-App-Code: <appCode>
X-App-Timestamp: <unix seconds>
X-App-Nonce: <random string>
X-App-Signature: <HMAC-SHA256 signature>
```

Canonical string:

```text
canonical = appCode + "\n" + timestamp + "\n" + nonce + "\n" + signedValue
signature = HMAC_SHA256(appSecret, canonical)  // Base64 output recommended
```

Reference implementations (copy-paste safe):

```javascript
// Node.js
import crypto from "crypto";

export function miduoSignatureBase64({ appCode, appSecret, timestamp, nonce, signedValue }) {
  const canonical = `${appCode}\n${timestamp}\n${nonce}\n${signedValue}`;
  return crypto.createHmac("sha256", appSecret).update(canonical, "utf8").digest("base64");
}
```

```java
// Java
import javax.crypto.Mac;
import javax.crypto.spec.SecretKeySpec;
import java.nio.charset.StandardCharsets;
import java.util.Base64;

public static String miduoSignatureBase64(String appCode, String appSecret, String timestamp, String nonce, String signedValue) throws Exception {
    String canonical = appCode + "\n" + timestamp + "\n" + nonce + "\n" + signedValue;
    Mac mac = Mac.getInstance("HmacSHA256");
    mac.init(new SecretKeySpec(appSecret.getBytes(StandardCharsets.UTF_8), "HmacSHA256"));
    byte[] raw = mac.doFinal(canonical.getBytes(StandardCharsets.UTF_8));
    return Base64.getEncoder().encodeToString(raw);
}
```

Signed value per endpoint:

- validate: `signedValue = token`
- refresh: `signedValue = sessionToken`
- revoke: `signedValue = sessionToken`

### D) Integration Behavior（落地逻辑必须正确）

The handoff should clearly explain:

- Browser only transports `exchange token`; browser must not hold `appSecret`
- Third-party backend must validate `exchange token`
- `exchange token` is one-time and should fail on second validate
- Third-party backend stores `sessionToken` server-side
- Third-party backend refreshes `sessionToken`
- Third-party logout revokes `sessionToken`
- When refresh fails (`success=false`), go through login bridge
- `redirectUri` must be HTTPS and allowlisted
- Token delivery mode:
  - `QUERY`: landing receives query params
  - `FORM_POST`: landing receives form body

### 5) 交付验收清单（第三方必须实现）

- callback 页面/接口能接收 `token` + `state`（QUERY 或 FORM_POST）
- 后端校验 `state`（必须），再调用 `validate-login-token`
- 校验成功后创建第三方本地会话，并将 `sessionToken` **仅存后端**
- 有 refresh 续期策略（定时或按活跃触发）
- logout 时 revoke `sessionToken`
- refresh 失败会走登录桥重认证
- 任何日志/监控里 **不打印 appSecret / token / sessionToken 全量**

### 1) Configuration（建议用环境变量/密钥管理）

- `MIDUO_SSO_BASE_URL`
- `MIDUO_SSO_APP_CODE`
- `MIDUO_SSO_APP_SECRET`
- `MIDUO_SSO_REDIRECT_URI`（你方回调地址，必须 https 且在 allowlist）
- `MIDUO_SSO_SHORTCUT_ID`（如需要从工作台快捷入口跳转）

### 2) Browser landing / callback（你方前端）

Responsibilities:

- Receive `token` (and optionally `app_code`, `state`, `source`) via QUERY or FORM_POST
- Immediately POST `token` + `state` to **your backend** over HTTPS
- Do not persist `token`/`sessionToken` in localStorage

### 3) Backend: validate exchange token → create local session

Minimum flow:

- Your backend endpoint (example): `POST /auth/miduo/callback` accepts `{ token, state }`
- Verify `state` matches what your backend issued/stored (CSRF protection)
- Call Miduo open API `POST {baseUrl}/api/open/sso/validate-login-token` with:
  - signed headers
  - JSON body: `{ "token": "<exchange token>" }`
- Only when business success is true, create your local session and store the Miduo `sessionToken` **server-side**

### 4) Refresh: keep Miduo session alive

- Call `POST {baseUrl}/api/open/sso/refresh-session-token` with body `{ "sessionToken": "<sessionToken>" }`
- If `success=false`, stop refreshing and redirect user to login bridge

### 5) Logout: revoke

- Call `POST {baseUrl}/api/open/sso/revoke-session-token` with body `{ "sessionToken": "<sessionToken>" }`
- Destroy local session and delete stored `sessionToken`

### 6) Login bridge: re-auth when refresh fails

- Browser navigates to:
  - `GET {baseUrl}/api/sso/bridge/redirect-url?appCode=<appCode>&redirectUri=<url-encoded https redirectUri>&state=<state>`
- Miduo returns an URL that redirects back to your `redirectUri` with a **new exchange token**
- Your callback handler repeats: validate → create session → store sessionToken

### 6.1) Login bridge encoding pitfalls（非常常见的“登录后不回跳”原因）

When the browser enters the bridge page through Miduo login like:

- `/login?redirect=<encoded internal path>`

the nested `redirect_uri` is often encoded inside another encoded string. If the login page redirects using SPA routing without careful decoding, the bridge page may receive:

- `redirect_uri = "http%3A%2F%2Fthird.example.com%2Fcallback"` (still encoded)

Symptoms:

- Bridge page shows “缺少必要参数/redirect_uri 不合法”
- Or frontend calls `/api/sso/bridge/redirect-url` with `redirectUri=http%3A%2F%2F...`
- Backend rejects because `redirectUri` cannot be parsed as a URI

Fix/mitigation guidance for third-party teams:

- Always treat `redirect_uri` / `redirectUri` as a URL value that may be percent-encoded once; **decode it once if it still looks encoded**.
- In logs/alerts, print only the **scheme + host** when diagnosing (never print full token/secret).
- Prefer generating the “login→bridge” URL by encoding only what is needed:
  - Encode the full `redirect` value once
  - Inside that, keep `redirect_uri` encoded at most once (avoid accidental double-encoding)

## Mandatory Risk Review（必须逐条检查）

Always inspect for these problems and call them out explicitly if missing or unclear:

- `appSecret` stored or used in frontend
- Third party trusting URL `mobile`, `name`, or other PII as login identity
- Missing backend callback endpoint to consume `token`
- No handling for second validate call returning invalid
- No refresh strategy, or refresh done only in browser
- No revoke strategy on logout
- Missing login bridge fallback when refresh fails
- `redirectUri` using `http://` or not allowlisted
- Lack of `state` generation + verification
- Missing clock skew handling for signed requests
- Reusing nonce (no anti-replay window)
- Logging full `token`, `sessionToken`, or `appSecret`
- Persisting `sessionToken` in browser storage
- Not masking secrets in docs/logs when audience is broad

## Required Third-Party Retrofit Checklist（实现产物清单）

Verify the third-party system has:

- Landing/callback handler that receives `token` (QUERY/FORM_POST)
- Backend endpoint that consumes `token` and validates via open API
- Local session creation after validate succeeds
- Server-side storage of `sessionToken`
- Refresh logic (timer or refresh-on-activity)
- Logout flow that revokes `sessionToken`
- Bridge re-auth flow when refresh fails
- Error handling for invalid/expired tokens, signature failures, bridge redirect rejection
- `state` strategy (issue + store + verify)
